## Current role of this folder in Mog

This folder is the **shell-level orchestration layer** that sits above the kernel and below React. It owns four concerns:

1. **Document set ownership (`DocumentManager`).** `create-document-manager.ts` is a singleton-per-shell factory holding open `DocumentHandle`s in `Map`s *outside* React's tree so documents survive component remounts. It is the only layer that knows the full set of open documents. It implements three near-identical open pipelines ŌĆö `loadDocument` (host-backed XLSX import / legacy CSV), `createDocument` (blank), `createCollaborationDocument` (room-backed) ŌĆö each with cache-hit dedup, in-flight promise dedup, a per-file operation chain (`enqueueFileOperation`), and a generation/abort protocol so a `dispose` arriving mid-open tears down resources before they become visible. It registers its `documents` map as a `lifecycle-state` active-docs provider so every open doc participates in the unload-flush and `__dt` persistence surfaces.

2. **Project / file management (`ProjectService`).** Business logic over a "smart service, dumb store" split: the service holds all orchestration (open/close project, open/save/rename/delete file, tab navigation, recent projects), the zustand store holds pure state. It calls the kernel only through `DocumentManager` and the platform only through an injected `ProjectIpc` (Tauri `invoke` / `secureInvoke`, with a `mock-ipc` for tests). `shell-service.ts` is a thin facade wiring `DocumentManager` + `ProjectService` + store together to eliminate `window.__SHELL__` reach-arounds in `apps/spreadsheet/` file/object handlers.

3. **Capability security (`ShellCapabilityRegistry`).** An in-memory implementation of the kernel `ICapabilityRegistry` contract: per-app capability grants with scope/expiry, expansion, event emission, and a paired in-memory audit log with retention/pruning and JSON/CSV export.

4. **Recovery and recency.** `TrapRecoveryCoordinator` isolates a wasm32-trapping document and replays its siblings onto a fresh WASM instance (one recovery per page lifecycle, fail-closed on a second trap). `lifecycle-state.ts` is a module-global gathering point for the `__dt.persistenceEnabled` getter (three independent signals). `recent-docs` mirrors the kernel Meta API into zustand; `active-document-recency` records active-tab transitions into Meta.

## Evidence (observed in the current tree)

- **`validatePathInProject` is a path-traversal check that does not prevent path traversal.** `project-service.ts:127-147` normalizes case + slashes then guards with `normalizedFile.startsWith(normalizedProject)`. Two defects: (a) **prefix-sibling escape** ŌĆö project `/home/u/proj` admits `/home/u/proj-secrets/x.xlsx` because `startsWith` has no path-boundary check (no required trailing separator / segment match); (b) **no `..` resolution** ŌĆö the "normalize" step only swaps `\`ŌåÆ`/` and lowercases; it never canonicalizes `..` segments, so a `newPath`/`savePath` carrying `../` is string-compared as-is. The function's own docstring claims it "Prevents path traversal attacks" (`:125-126`); it is the sole gate on `openFile`, `saveFile` (Save-As), `renameFile`, and `deleteFile` paths.

- **`saveMutex` serializes saves across *all* files, not per-file.** `project-service.ts:150-171` is a single hand-rolled boolean+queue mutex held for the whole `saveFile` body (`:524-566`). Saving file A blocks an unrelated save of file B end-to-end (including the `ipc.write_file` round-trip). The correct granularity is per-`fileId`; the contention is invisible until two tabs autosave concurrently.

- **`saveFile` ignores the `PlatformFileHandle` the shell facade stores.** `shell-service.ts:165-171` maintains a `fileId ŌåÆ PlatformFileHandle` map and its header (`:18-28`) states the handle is "the *live* object you can `.write()` through (FSA / Tauri / anchor-download)" so SAVE can write "without re-prompting." But `ProjectService.saveFile` (`:524-566`) only ever calls `ipc.write_file({ path })`. There is no code path that writes through the stored handle, so the facade's stated purpose (handle-based save on web FSA) is unwired ŌĆö two disconnected save mechanisms with no bridge.

- **Cache-hit short-circuits ignore the requested open options.** In `loadDocument` (`create-document-manager.ts:421-431`) and `createDocument` (`:528-538`), a cache hit returns the existing handle without checking that it was opened with compatible options. `createDocument`'s cache hit ignores `skipDefaultSheet`, `operation`, `documentId`, and `workbookLinkResolver`; `loadDocument`'s ignores `kind`/`csvOptions`. Only the *collaboration* path validates mode compatibility (`collaborationModeMatches`, `:241-250`). A second `createDocument(fileId,{skipDefaultSheet:true})` after a default `createDocument(fileId)` silently returns the wrong document shape.

- **`attachSidecar` is a mandatory interface method whose only implementation throws.** `document-manager.ts:121-124` declares `attachSidecar` on the public `DocumentManager` interface with a full docstring ("Attach a collab WS sidecarŌĆ” IdempotentŌĆ” After this call `getSidecar` returns the new sidecar"), but `create-document-manager.ts:826-833` always throws "attachSidecar is not a first-join API." It is a documented-but-dead contract that every consumer must know not to call ŌĆö a latent runtime trap.

- **Three open pipelines duplicate ~120 lines of cache/dedup/generation/finally machinery.** `loadDocument`, `createDocument`, and `createCollaborationDocument` each re-implement: the cache-hit-with-dispose-generation check, the in-flight-promise dedup with mode check, `allocateGeneration` + `setLoadingState('loading')` + `loadingModes.set`, the `enqueueFileOperation` body with its re-check after the queue, and the identical `finally` that deletes `loadingPromises`/`loadingGenerations`/`loadingModes` and calls `clearDisposeRequest`. The shapes differ only in the resource-acquisition middle and the mode payload. This is the highest-duplication hotspot in the folder and the most error-prone (the abort protocol must stay byte-identical across all three).

- **`disposeAll` partial-failure path clears all maps *and* rethrows.** `create-document-manager.ts:942-974`: on a non-collaboration failure it clears every map (`documents`, `hostAdapters`, `sidecars`, ŌĆ”) then throws an `AggregateError`. Clearing the `documents`/`hostAdapters` maps after a failed dispose drops the only references to handles/adapters whose `dispose()` rejected ŌĆö they can no longer be retried or cleaned up, so a failed teardown leaks the underlying WASM/host resources while reporting failure. The collaboration-failure branch (`:949-956`) instead resets `disposedAll=false` and throws *without* clearing ŌĆö two divergent partial-failure policies in one method.

- **`lifecycle-state.ts` is process-global mutable singleton state, and its teardown wipes co-tenants.** Module-level `_lifecycleHooksRegistered`, `_bootResolutionTerminal`, and `_activeDocsProviders` (`:41-105`) are realm-global. `setActiveDocsProvider(null)` (`:117-122`) **clears the entire provider Set** ŌĆö but the file's own rationale (`:90-104`) is that *multiple* subsystems register (`useAppDocument`'s cache *and* `DocumentManager`'s map). So an HMR teardown or test reset that passes `null` from one subsystem silently unregisters the other's provider too, defeating the documented multi-source design. The targeted unregister (returned for the non-null path) is correct; the `null` overload is a footgun.

- **`imported-pivot-metadata.ts` hand-rolls a full ZIP + OOXML reader (461 lines) in the shell layer, re-parsing bytes the import path already parsed.** `create-document-manager.ts:358` calls `extractImportedPivotMetadata(source.data)` on the same `Uint8Array` already handed to `importStandaloneBrowserHostBackedDocument`. The extractor re-implements EOCD scanning, central-directory walking, `deflate-raw` inflation, and `DOMParser` XML parsing (`imported-pivot-metadata.ts:96-199`). Robustness gaps: `readZipText`'s uncompressed-size mismatch branch returns the decoded bytes either way (`:158-161`, a dead guard); `parseXml` returns `null` when `DOMParser` is undefined and the caller treats that as "no pivots" silently; ZIP64 and data-descriptor (streaming) entries are unhandled. This is import-domain logic living in shell document services, duplicating capability the kernel/host import path already owns.

- **Test-only `mock-ipc` is exported from the production barrel.** `project/index.ts:44` re-exports `createMockFileSystem`, `createMockIpc`, `MockFileSystem` from `mock-ipc.ts` (435 lines). The only consumer is `project-service.test.ts:14`. Shipping a mock file system through the production `services/project` barrel pulls test scaffolding into the production module surface and bundle reachability.

- **`generateFileId` is duplicated verbatim across two services.** `shell-service.ts:72-81` and `project-service.ts:92-102` carry identical `crypto.randomUUID`-with-manual-`xxxx`-fallback implementations; the shell-service copy's comment (`:69-71`) explicitly notes it "Mirrors `project-service.ts:generateFileId()`" to keep IDs collision-free ŌĆö an acknowledged copy that should be one shared util.

- **`window.__SHELL__` migration is incomplete.** `shell-service.ts:4-6` states the facade exists "to eliminate the `window.__SHELL__` documentManager / projectService reach-arounds that file/object handlers in `apps/spreadsheet/` use today" ŌĆö present tense "today," i.e. the reach-arounds still exist. The facade is built but the migration that justifies it is unfinished.

- **Capability registry timestamp/eviction calls `Date.now()` per event with no injected clock.** `registry.ts` and `audit-log.ts` call `Date.now()` directly throughout (grant/revoke/expiry/audit timestamps, `:178/:334/:351`, audit `:67`). Expiry (`cleanupExpired`, `hasActiveGrant`) is enforced only lazily at access-time or via externally-driven `cleanupExpired()`/`pruneExpired()` ŌĆö there is no timer except the audit log's own prune interval (`audit-log.ts:56-59`). Grants therefore "linger" (still counted by `getGrants`) until something touches them.

- **Observability is unstructured `console.*` throughout.** `create-document-manager.ts` (load/create/dispose failures), `trap-recovery-coordinator.ts` (sibling-recovery failures), `project-service.ts:219/737` (auto-open / restore failures), `active-document-recency.ts:41`, `recent-docs-slice.ts:125/138`. No shared, tagged shell diagnostic channel, so silent-degradation paths (a swallowed restore failure, a dropped recency touch) are hard to attribute. Mirrors the kernel-plan finding (010).

- **Clean baselines worth preserving.** Zero `TODO`/`FIXME`/`HACK` markers in production code. The trap-recovery coordinator's re-entrancy race-guard (`trap-recovery-coordinator.ts:208-237`) is correct and well-documented (assigns `inFlight` synchronously before the async body runs, caught by an integration test). The collaboration room-URL canonicalization (`create-document-manager.ts:252-269`) validates protocol/segment-roundtrip carefully. `collab-room.ts` and `tree-utils.ts` are pure and clean. The dedup/generation/abort protocol, while duplicated, is correct. Improvements here are about **closing a security gap, fixing concurrency granularity, removing dead/duplicated surface, and relocating misplaced import logic** ŌĆö not behavior changes to the happy path.

## Improvement objectives

1. **Make `validatePathInProject` a real containment check.** Canonicalize (resolve `..`/`.`, collapse separators) and compare on path-segment boundaries, not raw `startsWith`. This is a security correctness fix on the only gate protecting file read/write/rename/delete.
2. **Fix save concurrency granularity and wire handle-based save.** Replace the global `saveMutex` with a per-`fileId` lock, and either bridge `saveFile` to the stored `PlatformFileHandle` (so the facade's FSA-write purpose works) or remove the unwired handle map ŌĆö no half-built mechanism.
3. **Validate open options on cache hits.** Make `loadDocument`/`createDocument` either return the cached handle only when the requested options are compatible, or reject with a clear `DocumentModeConflictError`-style error, matching the discipline the collaboration path already enforces.
4. **Remove the dead `attachSidecar` contract.** Drop it from the `DocumentManager` interface and the throwing implementation; collaboration documents are created via `createCollaborationDocument` and that is the only supported path.
5. **De-duplicate the three open pipelines** behind one parameterized `openPipeline` helper (resource-acquirer + mode-builder injected) so the cache/dedup/generation/abort/`finally` machinery exists once and is tested once.
6. **Make `disposeAll` partial-failure policy single and safe.** One policy: on any dispose failure, do not drop references to undisposed resources; surface the `AggregateError` while keeping retry-able state, and reset `disposedAll` consistently.
7. **Relocate imported-pivot extraction out of shell services** to the import/host layer that already parses the XLSX ZIP, eliminating the second parse and the hand-rolled ZIP reader; close its silent-failure and unsupported-entry gaps as part of the move.
8. **Tighten module boundaries:** move `mock-ipc` out of the production barrel (test-only export path), extract one shared `generateFileId`, and scope `lifecycle-state` teardown so unregistering one provider never wipes co-tenants.
9. **Standardize shell-service observability and the capability clock** by routing degradation through one tagged diagnostic channel and injecting a clock into the capability registry/audit log for deterministic expiry.

All are production-path: they close a security gap, fix concurrency and contract defects, remove dead/duplicated/misplaced surface, and strengthen invariants every consumer compiles against. None reduce scope or add shims.

## Production-path contracts and invariants to preserve or strengthen

- **DocumentManager identity invariant.** `handle.documentId === fileId` (verified at `create-document-manager.ts:332-336, 352-356, 732-736`) stays enforced on every open path; the open-pipeline extraction (objective 5) must keep these checks per-path.
- **Generation / abort protocol.** A `dispose` that arrives while an open is in flight must still abort publication and dispose the just-created resources (`requestDisposeGeneration`/`isDisposeRequested`/`clearDisposeRequest`, `:221-239`, `publishLoadedDocument` guard `:389-396`). The `openPipeline` extraction must reproduce this exactly; the dedup/abort behavior is the most safety-critical logic in the folder.
- **Per-file operation serialization.** `enqueueFileOperation` (`:271-285`) keeps create/load/dispose for one `fileId` strictly ordered; preserve FIFO and the self-deleting chain semantics.
- **Collaboration mode fences.** `createCollaborationDocument` mode-match on cache/dedup (`collaborationModeMatches`), the room-URL canonicalization and protocol/segment validation, and `detachSidecar`/`disposeDocument` refusing to detach room-backed docs (`:835-847, 884-892`) are load-bearing ŌĆö keep them. Objective 3's cache-option validation generalizes this discipline to the normal paths; it must not weaken the collab checks.
- **`lifecycle-state` "read live, never cache" contract.** Every read path (`readHasAnyAppendActive`, `iterateActiveDocsForFlush`, `readPersistenceStateSnapshots`, ŌĆ”) iterates the registry on each call so the `__dt` getters see live state. Objective 8's teardown scoping must not introduce caching; the dedupe-by-`documentId` semantics (`iterateActiveDocsForFlush:223-237`) stay.
- **Trap-recovery topology.** One recovery per page lifecycle, fail-closed on a second trap, originating doc stays in `error`, siblings recovered in parallel with per-sibling failure isolation, listeners re-attached on the swapped bridges (`trap-recovery-coordinator.ts` class doc + `_recoverImpl`). This is correct and must be preserved exactly; no change here beyond observability routing (objective 9).
- **Capability registry contract.** `ICapabilityRegistry` semantics ŌĆö capability expansion (`expandCapabilities`), scope matching, expiry-on-access, event emission ordering (`granted`/`revoked` to both per-app and global listeners) ŌĆö stay identical; objective 9 only injects the clock, it does not change when expiry is observed by callers.
- **"Smart service, dumb store" split** (`project-service.ts:6-10`). All business logic stays in the service; the store stays pure setters. The path-containment and save-lock fixes stay in the service.
- **`ProjectIpc` is the sole platform boundary.** `ProjectService` never imports Tauri/FSA directly; it goes through the injected `ipc`. Objective 2's handle-based save must respect this (handle write through the platform abstraction, not a direct FSA call in the service).
- **Recent-docs is observe-only on Meta.** The slice never calls `touchDoc` from boot (`recent-docs-slice.ts:11-15`); `active-document-recency` is the only shell path that touches on active-tab change, gated against collaboration and `skipLocalPersistence` docs (`active-document-recency.ts:30-32`). Preserve these gates.

## Concrete implementation plan

**Phase 1 ŌĆö Path containment (`project-service.ts`).** Replace `validatePathInProject` with a canonicalizing comparison: normalize separators, resolve `.`/`..` segments to a canonical absolute form, then assert the file path equals the project root or is a strict descendant on a segment boundary (project root + separator prefix, or segment-array prefix match). Apply it uniformly to `openFile`, `saveFile`-with-`savePath`, `renameFile`, `deleteFile`, and the new-file/folder creation paths. Add regression cases: prefix-sibling escape (`/p` vs `/p-evil`), `..` traversal, mixed-separator, and the legitimate single-file-mode no-project bypass (`:128-131`).

**Phase 2 ŌĆö Save concurrency + handle wiring (`project-service.ts`, `shell-service.ts`, `ipc-types.ts`).** (a) Replace the global `saveMutex` with a `Map<fileId, lock>` (or reuse a shared async-mutex util if one exists in `@mog/spreadsheet-utils`) so saves serialize per file. (b) Resolve the handle disconnect: extend `ProjectIpc`/the save path so that when a `PlatformFileHandle` is registered for a `fileId` (web FSA), `saveFile` writes through it, falling back to `ipc.write_file` by path for desktop; OR, if handle-based save is not yet a product requirement, delete `setDocumentHandle`/the `handles` map and the `getDocumentState().handle` surface from `shell-service.ts` so there is no dead mechanism. Decide via the blocked-evidence note below.

**Phase 3 ŌĆö Cache-hit option validation (`create-document-manager.ts`).** For `loadDocument`, compare the cached/in-flight `kind` (record it in `documentModes`/`loadingModes` like collab does) and reject incompatible re-opens; for `createDocument`, record and compare `skipDefaultSheet`/`operation`/`documentId`. Reuse the `DocumentModeConflictError` pattern. Add tests for "re-open same fileId with different kind/options."

**Phase 4 ŌĆö Remove dead `attachSidecar` (`document-manager.ts`, `create-document-manager.ts`).** Delete the method from the interface and the throwing implementation. Grep consumers (the facade and `apps/spreadsheet`) for any reference; none should exist since it always threw. Keep `detachSidecar`/`getSidecar` (those have real implementations).

**Phase 5 ŌĆö Extract `openPipeline` (`create-document-manager.ts`).** Introduce one internal helper taking `(fileId, modeBuilder, acquireResources, options)` that owns: cache-hit check (with option validation from Phase 3), in-flight dedup, generation allocation, `setLoadingState('loading')`, the `enqueueFileOperation` body with post-queue re-check, `publishLoadedDocument`, and the shared `finally`. `loadDocument`/`createDocument`/`createCollaborationDocument` become thin callers supplying their resource acquirer and mode payload. The abort/generation protocol lives in one place. Gate on the existing `create-document-manager.test.ts`.

**Phase 6 ŌĆö `disposeAll` single policy (`create-document-manager.ts`).** Make one partial-failure policy: collect failures, do **not** clear map entries for `fileId`s whose dispose rejected (so they remain retry-able and their resources tracked), clear only the successfully-disposed entries, reset `disposedAll` consistently with the chosen retry semantics, and throw the `AggregateError`. Add a test where one handle's `dispose()` rejects and assert the failed doc's entry survives while siblings are cleared.

**Phase 7 ŌĆö Relocate imported-pivot extraction (`imported-pivot-metadata.ts` ŌåÆ import/host layer).** Move ZIP/OOXML extraction to the layer that already opens the XLSX ZIP during `importStandaloneBrowserHostBackedDocument`, so the bytes are parsed once and the metadata is delivered alongside the imported handle. Keep the shell-side `attach/getImportedPivotMetadata` WeakMap accessor (it is a legitimate shell-side handle annotation) but feed it from the single import parse. As part of the move, replace the hand-rolled ZIP reader with the shared XLSX/ZIP utility the import path uses, and turn the silent `DOMParser`-undefined / `parsererror` fallthroughs into surfaced diagnostics on the `ImportedPivotMetadataSet.diagnostics` channel (which already exists, `:48-51`). If a clean import-layer seam does not exist yet, see the blocked-evidence note.

**Phase 8 ŌĆö Boundary tightening.** (a) Move `mock-ipc` exports out of `project/index.ts`; have `project-service.test.ts` import directly from `./mock-ipc` (or relocate the file under `project/__tests__/`). (b) Extract `generateFileId` into a single shared util (e.g. `shell/src/services/_shared/` or an existing utils module) consumed by both `shell-service.ts` and `project-service.ts`. (c) In `lifecycle-state.ts`, remove the `null`-clears-everything overload from `setActiveDocsProvider` (or split it into an explicit `__resetLifecycleStateForTests`-only clear, which already exists at `:163-167`) so production/HMR teardown only ever uses the targeted unregister returned at registration.

**Phase 9 ŌĆö Observability + capability clock.** Introduce a small shell diagnostic helper (subsystem-tag + level) and route the load/create/dispose, trap-recovery sibling-failure, project restore/auto-open, recency-touch, and recent-docs read failures through it. Inject a `now()`/clock into `createShellCapabilityRegistry` and `createShellCapabilityAuditLog` (default `Date.now`) so expiry/audit timestamps are testable and the registry can optionally drive a prune timer like the audit log already does.

## Risks, edge cases, and non-goals

- **Risk ŌĆö path-containment over-tightening (Phase 1).** Canonicalization must preserve the legitimate single-file-mode "no project open ŌåÆ allow" bypass and not reject valid Windows UNC / drive-letter paths. Mitigate with platform-specific path tests and by canonicalizing both sides identically.
- **Risk ŌĆö `openPipeline` extraction altering the abort protocol (Phase 5).** The generation/dispose-request interplay is subtle (the comment at `:447-449` and the synchronous re-entrancy concern mirror the trap-recovery race). Land the extraction behind the existing tests plus the new concurrent open+dispose test before relying on it; review that no `await` is introduced between generation allocation and `loadingPromises.set`.
- **Risk ŌĆö handle-based save scope (Phase 2).** Wiring FSA writes touches the platform boundary and product behavior (re-prompt vs silent write). If the product intent is unclear, the safe production-path interim is to remove the dead handle map rather than ship a half-wired save ŌĆö but that is a deliberate decision, not a default.
- **Edge case ŌĆö `imported-pivot` relocation and read-only pivots (Phase 7).** The `readOnly: true` metadata drives the field-panel "imported pivot" UI; the move must deliver identical metadata shape (`ImportedPivotTableMetadata`) so downstream rendering is unchanged. The current extractor's leniency (best-effort, diagnostics on failure) must be preserved so a malformed pivot definition degrades to "no pivots" with a diagnostic, never a hard import failure.
- **Edge case ŌĆö `lifecycle-state` global across realms (Phase 8).** Module-global state is shared per JS realm; the teardown fix must not change the "register once per page load, idempotent" semantics the `__dt` getters depend on.
