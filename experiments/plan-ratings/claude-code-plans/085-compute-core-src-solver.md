# 085 — Improve `mog/compute/core/src/solver` (solver bridge: goal seek, solver, what-if)

## Current role of this folder in Mog

This folder is the **spreadsheet ↔ numerical-engine bridge** for all optimization and root-finding features (Goal Seek, Solver, and the "what-if" family). It owns three concerns:

1. **The public solver contract** (`types.rs`). These are the cell-aware types every caller and the TS layer compile against. `Variable`/`Constraint` carry `CellId`; `SolverParams` names an `objective_cell`. This is the serialization boundary.
2. **Type translation** (`mod.rs`, `to_crate_*`/`from_crate_*`). It strips the spreadsheet identity (`CellId`, objective-cell) down to the crate's pure `FnMut(&[f64]) -> f64` + `SolverConfig` world, and lifts crate `SolverResult` back into the cell-aware `SolverResult`/`GoalSeekResult`.
3. **Method routing** (`solve()`). It decides what runs locally in Rust (root finding, NM, BFGS, L-BFGS-B, DE) versus what is declared out of scope for the local engine (`Simplex`, `MixedInteger`, and *any* constrained problem), which it returns as `TerminationReason::RequiresPython`.

`ComputeCore::solve`/`goal_seek` (scheduler) supply the evaluation closure by re-evaluating the objective cell's AST under temporary cell-value overrides (read-only; the `CellMirror` is never mutated). The Goal Seek path is implemented by calling the crate's `solve_root` **directly** in the scheduler and then converting via this folder's `from_crate_result_to_goal_seek`; it does **not** funnel through `solve()`.

Because this is a contract surface that crosses both the Rust↔Rust crate seam and the Rust↔TS seam, improvements here are about **routing correctness, contract truthfulness, and closing a dead-end terminal state** — not about reimplementing algorithms (those live in the crate).

## Evidence (observed in the current tree)

- **`RequiresPython` is a dead-end terminal state — no consumer dispatches to Python.** `types.rs:144-146` documents "The calling layer (Tauri/WASM) catches this and dispatches to Python sandbox," and `mod.rs:5,147-148` repeat it. A repo-wide search for `RequiresPython`/`requires_python`/`python.*sandbox`/`PythonSolver` finds **only** the producer in `solver/mod.rs` plus the doc comments — **zero handlers** in `mog/app`, `mog/infra`, or any TS. Git log on the folder shows a single "Initial public source import" commit. So every constrained problem, `Simplex`, and `MixedInteger` resolves to `converged: false`, `solution: vec![]`, and a terminal reason the product cannot act on. The feature is silently inert.
- **Bounds are silently dropped for explicit `NelderMead`/`BFGS`.** `to_crate_config` (`mod.rs:35-42`) always populates `bounds` from the variables. But the crate's dispatch comments (`compute-solver/src/dispatch.rs:13-15`) and method set establish that **only L-BFGS-B and Differential Evolution honor bounds**; plain Nelder–Mead and BFGS do not. So `solve()` with `method: BFGS` (or `NelderMead`) and a bounded variable can return an **out-of-bounds point reported as `converged: true`**. The bridge presents an answer that violates the user's stated bounds.
- **The documented `Integer`-constraint invariant is unenforced.** `types.rs:42-44`: "For `Integer` constraints, the `CellId` must match one of the `SolverParams::variables[i].cell_id` values. This is validated at runtime." `solve()` never validates this — it routes *any* non-empty `constraints` to `requires_python` (`mod.rs:163-166`). The invariant is documented but dead.
- **Public-contract default values disagree with the implementation.** `types.rs:96-99` document `precision` default `1e-6` and `max_iterations` default `1000`. `mod.rs:47-53` actually uses `max_evals = 10_000` (non-root) / `100` (root) and `ftol = 1e-8` (non-root) / `1e-6` (root). The numbers a caller reads in the contract do not match what runs.
- **Two divergent root-finding entry points for the same operation.** `types.rs:159-160` claims `GoalSeekParams` "Maps to `SolverParams` with `method: RootFinding`…", but the scheduler's `goal_seek` (`solver_methods.rs:139-156`) bypasses `solve()` and calls `compute_solver::solve_root` directly, setting `root_finding_step_limit = max_change` and `max_time_ms = 0`. The unified `solve()` RootFinding path (`mod.rs:183-185`) routes through `to_crate_config`, which leaves `root_finding_step_limit` at the crate default (`0.001`) and sets `max_time_ms = 30_000`. `SolverParams` has **no** step-limit / `max_change` field at all, so the two paths cannot be made to agree through configuration.
- **Unreachable / lossy `TerminationReason` variants.** `types.rs:138-141` declare `Unbounded` and `Infeasible`, but the crate never produces them and `from_crate_result` (`mod.rs:76-82`) never emits them — they are dead in every code path. Conversely the crate's `MaxEvaluations` is collapsed onto the bridge's `MaxIterations` (`mod.rs:78`), erasing the eval-vs-iteration distinction the result also reports separately in `evaluations`/`iterations`.
- **`dual_values` is always `None`.** `from_crate_result` hard-codes `dual_values: None` (`mod.rs:84`) and so does `requires_python` (`mod.rs:134`). There is no LP path, so shadow prices are never produced; the field is contract surface with no producer.
- **Error/`RequiresPython` paths return an empty `solution`.** `requires_python` (`mod.rs:124`) and the RootFinding-non-Target error (`mod.rs:174`) return `solution: vec![]`, but the success contract says `solution` is "parallel to `params.variables`" (`types.rs:108-109`). The scheduler returns this straight to callers; any consumer indexing `solution[0]` on a failed solve panics. The empty-vs-parallel contract is unstated.
- **Determinism is currently preserved — keep it that way.** `to_crate_config` never sets `config.seed`, but the crate's DE falls back to a fixed seed (`diff_evolution.rs:257-259`, `seed_from_u64(0)`), so `GlobalEvolution` from the bridge is deterministic across recalcs. This is load-bearing (spreadsheet recalc must be deterministic) and any seed plumbing added must default to a fixed seed.

## Improvement objectives

1. **Close the `RequiresPython` dead end on the production path.** Constrained problems and integer constraints are first-class Solver features; today they silently produce an inert result. The right production fix is to **solve them natively** in the local engine rather than route to a Python path that no consumer implements. Bound constraints are already supported by L-BFGS-B; general inequality/equality constraints and integer constraints need engine support (penalty / augmented-Lagrangian for continuous constraints; branch-and-bound or DE-with-rounding for integers). `RequiresPython` is retired in favor of native solving for the cases the engine can handle, and a *truthful* terminal reason (`Infeasible`/`Unbounded`/`Unsupported`) for the rest.
2. **Never report an out-of-bounds point as converged.** When bounds are present, route to a bound-aware method (L-BFGS-B) regardless of the requested unbounded method, or reject the request with a clear error. Honor the user's bounds as a hard invariant of the result.
3. **Enforce the documented `Integer`-constraint invariant** (and validate constraint `CellId`s against the variable set) at the bridge boundary, returning a deterministic, attributable error instead of silently dropping into the dead-end path.
4. **Make the contract defaults true.** Reconcile `precision`/`max_iterations`/`max_time_ms` doc-defaults in `types.rs` with the values actually applied in `to_crate_config`, in **one** documented place.
5. **Unify the two root-finding entry points.** Make `GoalSeekParams` genuinely map to `SolverParams`/`solve()` as documented (add the missing step-limit/`max_change` field to the unified contract), so Goal Seek and `solve(method: RootFinding)` are the same code path with identical termination semantics.
6. **Make `TerminationReason` and `solution`/`dual_values` truthful.** Either emit `Unbounded`/`Infeasible` when they actually occur (they become reachable once constraints are solved natively) or remove the dead variants; preserve the eval/iteration distinction; specify and uphold the `solution` length contract on every return path; and gate `dual_values` to the (future) LP path or drop it until then.

These are production-path improvements: they make a declared-but-inert feature actually work, remove a class of silently-wrong results, and align the contract with reality. None reduce scope, add shims, or are test-only.

## Production-path contracts and invariants to preserve or strengthen

- **Serialization stability.** `SolverParams`/`SolverResult`/`GoalSeekParams`/`GoalSeekResult` are emitted to TS via `bridge-ts`. Any field added (e.g. a root-finding step limit, an `Unsupported` termination reason) must be additive and the bridge-ts allow-list updated in lockstep; no field is renamed or removed without a coordinated `bridge-ts` change. (Editing the types means the generated TS bindings must be regenerated by the owning folder — flagged as a dependency, not done here.)
- **Read-only objective evaluation.** Solver/Goal-Seek must never mutate the `CellMirror`; the override-based evaluation closure (`solver_methods.rs`) is the contract. Any new constraint-evaluation closures (constraint cells) must use the same read-only override mechanism.
- **Recalc determinism.** Same inputs ⇒ same result, including the `GlobalEvolution`/DE path. Any seed plumbing defaults to a fixed seed; no wall-clock or RNG nondeterminism may leak into the result.
- **`solution` is parallel to `variables` on success.** Strengthen to a stated, total contract: on every non-success path, `solution` is either empty *by documented convention* or padded to `variables.len()` — pick one and document it so consumers cannot panic on `solution[0]`.
- **Objective-transform semantics.** `Maximize` = minimize `-f`; `Target(t)` = minimize `|f − t|` (crate `EvalHarness`). The bridge must not double-transform; `to_crate_objective` stays a pure 1:1 map.
- **NaN/non-numeric handling.** The evaluation closures coerce eval failures to `f64::NAN` and the crate harness maps NaN/Inf to a reject sentinel. Preserve this; constraint evaluation must follow the same rule.

## Concrete implementation plan

Ordered so each step is independently reviewable and the first steps are pure correctness fixes with no algorithmic dependency.

1. **Truthful defaults (contract fix, bridge-only).** In `types.rs`, correct the doc-comments for `precision`, `max_iterations`, `max_time_ms` to state the real applied defaults and that they differ for root finding vs. multi-variable. In `mod.rs`, lift the magic numbers in `to_crate_config` into named `const`s (e.g. `ROOT_MAX_EVALS`, `OPT_MAX_EVALS`, `ROOT_FTOL`, `OPT_FTOL`, `DEFAULT_MAX_TIME_MS`) referenced by the doc-comments, so the contract and the code share one source of truth.
2. **Bounds-honoring routing (bridge-only).** In `solve()`, when `params` has any finite bound and the requested method is bound-unaware (`NelderMead`/`BFGS`), redirect to `compute_solver::Method::LBFGSB` (which handles bounds natively) instead of passing bounds to a solver that ignores them. Document this redirect in the `solve()` doc-comment and assert (debug) that the returned solution satisfies the bounds before reporting `converged: true`; if a returned point violates bounds, downgrade to `converged: false` with `NumericalError` rather than emit a bound-violating "solution."
3. **Constraint validation at the boundary (bridge-only).** Add a `validate_params(params) -> Result<(), SolverResult>` step at the top of `solve()` that: (a) checks every `Constraint::Integer(cell_id)` and constraint `cell_id` resolves to a `variables[i].cell_id`; (b) checks variable bounds are well-ordered (`lower <= upper`); (c) checks `x0` finiteness. On failure return a deterministic error result with a precise message — never the dead-end path.
4. **Unify root finding (bridge + contract).** Add an optional `root_finding_step_limit: Option<f64>` (or `max_change`) to `SolverParams`, plumb it through `to_crate_config`, and reimplement the scheduler's `goal_seek` to construct a `SolverParams { method: RootFinding, objective: Target(target), variables: [single], … }` and call `ComputeCore::solve`, then convert via `from_crate_result_to_goal_seek`. This makes `types.rs:159-160`'s documented mapping literally true and removes the divergent `max_time_ms` (0 vs 30_000) behavior. (Touches `solver_methods.rs` — cross-folder; sequence after the bridge change lands.)
5. **Native constrained optimization (cross-folder: `compute-solver` crate).** Extend the crate with a constraint representation (continuous `g(x) ≤/≥/= v` and integer-valued variables) and a constrained solve path: penalty / augmented-Lagrangian wrapping the existing unconstrained solvers for continuous constraints, and branch-and-bound (or DE-with-integrality) for integer constraints. Then in the bridge, map `Constraint::{LessEqual,GreaterEqual,Equal,Integer}` into the new crate constraint type and route `Simplex`/`MixedInteger`/constrained `Auto` to it. The constraint values are themselves cell formulas, so the scheduler must build a constraint-evaluation closure parallel to the objective closure (read-only overrides). This is the headline objective and the largest lift; it can proceed in parallel with steps 1–4 because it does not alter the unconstrained contract.
6. **Reachable, truthful terminations.** Once step 5 lands, emit `Infeasible`/`Unbounded` from the crate where detectable and map them through `from_crate_result`; otherwise replace the two dead variants and `RequiresPython` with a single honest `Unsupported { reason }` for genuinely-unhandled problem classes. Preserve `MaxEvaluations` distinctly (add a bridge variant or map it without conflating with iteration-count exhaustion). Gate `dual_values` to the LP path or drop the field until an LP solver exists.
7. **`solution` length contract.** Decide and document: on non-success, pad `solution` to `variables.len()` with the last-evaluated point (preferred — consumers can always index) or keep it empty-by-contract. Apply uniformly across `requires_python`/error/`from_crate_result` and state it in `types.rs`.

## Tests and verification gates

(Per task constraints, this plan does not run any build/test commands; the gates below are what a follow-up implementation must satisfy.)

- **Routing tests (`tests.rs`).** Extend the existing routing suite: bounded `BFGS`/`NelderMead` request returns a solution that satisfies the bounds (or a documented `converged: false`), not an out-of-bounds converged point. Integer-constraint with a `cell_id` not in `variables` returns the new validation error, not `RequiresPython`. Constrained `Auto` returns a native result once step 5 lands.
- **Default-contract test.** A test asserts the applied `max_evals`/`ftol`/`max_time_ms` equal the documented constants for both root and non-root, guarding against future doc/code drift.
- **Goal-Seek unification test.** Assert `goal_seek(params)` and the equivalent `solve(SolverParams{method: RootFinding,…})` yield identical `solution_value`/termination on a shared fixture.
- **`solution`-length invariant test.** For every termination reason, assert `result.solution.len()` matches the documented contract.
- **Crate-level algorithm tests** for the new constrained/integer paths live in `compute-solver` (penalty convergence on a known constrained QP; branch-and-bound on a small MILP; bound-satisfaction property tests). Reuse the crate's existing proptest harness.
- **Determinism gate.** A test runs `GlobalEvolution` and the new constrained path twice and asserts byte-identical results.
- **bridge-ts regeneration.** Adding contract fields requires regenerating the TS bindings (owned by `mog/infra/rust-bridge`); the bridge-ts golden test must pass with the new fields present.

## Risks, edge cases, and non-goals

- **Risk — contract churn into TS.** Any `SolverParams`/`SolverResult` field change ripples to generated TS and the `bridge-ts` allow-list. Keep changes additive; sequence the bridge-ts regeneration as an explicit dependency.
- **Risk — constrained solver quality.** Penalty/augmented-Lagrangian and branch-and-bound are nontrivial; a poor implementation could report false convergence. Mitigate with bound/constraint-satisfaction assertions on the result and conservative termination (prefer `Infeasible`/non-converged over a constraint-violating "solution").
- **Edge cases:** equal upper/lower bound (fixed variable); empty `variables`; `Target` objective with a constant (non-reachable) function (already covered by the GoalSeek "not converged" test); NaN-producing objective/constraint cells; integer constraint on an unbounded variable (branch-and-bound needs finite bounds — validate or derive).
- **Performance.** Solver/Goal-Seek run synchronously on the recalc thread via `sync_block_on`; constrained solves multiply eval count. Keep the `max_evals`/`max_time_ms` budget enforcement (crate `EvalHarness`) as the backstop and surface budget exhaustion as a truthful termination.
- **Non-goals:** rewriting the numerical algorithms themselves (the crate's NM/BFGS/L-BFGS-B/DE are well-tested and out of scope); building an actual Python sandbox (explicitly rejected — the production direction is native solving, not a path no consumer implements); data-table (`data_table/`) feature work beyond the shared what-if framing; UI/Tauri wiring of new termination reasons.

## Parallelization notes and dependencies on other folders

- **Steps 1–3** (defaults, bounds routing, constraint validation) are **bridge-only** (`solver/mod.rs`, `types.rs`, `tests.rs`) and can be done immediately and in parallel; they have no crate dependency.
- **Step 4** (Goal-Seek unification) touches `mog/compute/core/src/scheduler/solver_methods.rs` and the `SolverParams` contract — sequence after step 1; coordinate the `bridge-ts` regeneration with `mog/infra/rust-bridge`.
- **Step 5** (native constrained optimization) is the major cross-folder dependency on `mog/compute/core/crates/compute-solver` and requires a new constraint-evaluation closure in `solver_methods.rs`. It can proceed in parallel with 1–4 since it does not change the unconstrained contract, but **step 6** (truthful terminations, `dual_values`) depends on it.
- **Coordination:** the `bridge-ts` golden bindings (owned elsewhere) gate any contract-field addition; the `compute-api` re-export module (`mog/compute/api/src/pure/solver.rs`) should be revisited once the contract changes, but only as a downstream consumer.
