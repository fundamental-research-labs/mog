## Current role of this folder in Mog

This crate is the **high-performance WASM-backed implementation of the chart data-transform pipeline**. The kernel chart compiler intercepts a `ChartSpec`'s `transform` array at compile time and, when the WASM exports are available, routes the inline data through `chart_apply_transforms(data, transforms)` (`chart-compiler.ts:390`) instead of the TypeScript grammar transforms. `data` is `Vec<DataRow>` where `DataRow = serde_json::Map<String, Value>` — a direct, zero-restructuring image of the JS `Record<string, unknown>` rows. The pipeline applies each `Transform` in order (filter → aggregate → bin → sort → calculate → fold → regression → density, in whatever sequence the spec lists) and returns transformed rows that replace `spec.data.values`.

Beyond the pipeline, the crate exposes the algebraic primitives the chart layer needs directly: `compute_stack` for bar/area/histogram stacking, `histogram`/`cumulative_histogram`/`normalized_histogram` for bin charts, `kernel_density_estimation`/`violin_shape` for density and violin marks, and `chart_compute_statistics` for box plots. `grouping.rs` provides first-seen-order group utilities.

Because the consumer silently **falls back to the TS grammar on any error or absence**, two correctness regimes must agree: divergence is invisible in tests that only exercise one path, but produces user-visible chart differences (different axis category order, different filtered row sets, different derived values) in production depending on WASM availability. **Behavioral parity with the TS grammar is therefore the dominant invariant of this crate, ahead of performance.**

## Evidence (observed in the current tree)

- **The filter expression comparison split is not string-literal-aware — predicates containing an operator inside a string silently mis-parse.** `evaluate_expression` (`filter.rs:146-155`) scans for comparison operators with `expr.find(op)` over `["==","!=",">=","<=",">","<"]` and splits at the **first raw byte match**, with no string/paren awareness (unlike `split_logical`, which is string-aware at `:164-210`). For `datum.tag != "a==b"`, `find("==")` locates the `==` *inside the string literal*, splits into `datum.tag != "a` / `b"`, both resolve to `Null` (bare-field lookups fail), and `Null == Null` returns `true` — the row passes unconditionally. Any string literal containing `==`, `>`, `<`, etc. corrupts the predicate.
- **The calculate operator split breaks scientific-notation and signed literals.** `split_binary_op` (`calculate.rs:109-154`) scans right-to-left for `+`/`-` (lowest precedence) at paren-depth 0 outside strings, requiring non-empty both sides. For `datum.x * 1e-5` it matches the `-` inside `1e-5`, splitting into `datum.x * 1e` / `5`; `1e` fails `parse::<f64>()` → the whole expression evaluates to `Null`. Likewise `datum.a + -3` and any unary minus (`-datum.x`) are unsupported/mis-split. Exponent and signed numeric literals are common in calculated fields.
- **Two divergent hand-rolled expression evaluators with overlapping-but-unequal grammars.** `filter.rs` supports `&&`/`||`/`!`/comparisons/truthy but no arithmetic; `calculate.rs` supports arithmetic/concat/parens but no comparisons/logical/negation. They duplicate `datum.`/`datum[...]`/literal resolution (`filter.rs:220-267` vs `calculate.rs:157-201`) with subtle differences (calculate strips wrapping parens and recurses; filter does not). Neither matches a single documented grammar, and the TS oracle is a Vega-expression subset — so each is a separate, untracked parity surface.
- **Aggregate output ordering is lexical-by-serialized-key, not first-seen — diverging from `grouping.rs` and from Vega.** `apply_single_aggregate` (`aggregate.rs:40-46`) groups into a `BTreeMap<String, Vec<&DataRow>>` keyed by `make_group_key` (`:74-84`, fields joined with `\x1f`), so result rows come out **sorted by the serialized key string**, not in first-seen order. `grouping.rs::group_by_fields` (`grouping.rs:38-61`) deliberately preserves first-seen order for the same operation. For a categorical bar chart this reorders the x-axis (e.g. `"East"` before `"West"` only by luck of lexical order; `"10" < "9"` lexically for numeric-looking categories). This is a determinism/parity defect, not a perf choice.
- **Group keys coerce numbers via raw JSON repr, so `1` and `1.0` (and `1` vs `"1"`) split into different groups inconsistently across modules.** `aggregate.rs:78-80` uses `v.to_string()`; `grouping.rs:46-50` uses `serde_json::to_string(&parts)`; `Distinct` (`aggregate.rs:233-244`) uses `v.to_string()`; the *filter* numeric path compares via `value_as_f64` (`filter.rs:57`). So filtering treats `1`/`1.0` as equal but grouping/distinct treat them as distinct, and the coercion rule differs from JS `String()` and from the TS grammar's keying. `value_to_string` (`utils.rs:6-14`) is JS-`String()`-flavored but is **not** used on the aggregate/distinct key path.
- **`ChartError` is dead code; the pipeline cannot surface diagnostics.** `error.rs` defines `TransformFailed`/`InvalidBin` but nothing constructs or returns them — `rg` finds zero non-definition uses. `apply_transforms` returns `Vec<DataRow>` unconditionally (`transforms/mod.rs:22-28`); bad specs degrade silently (missing filter field → all rows dropped, `filter.rs:37-40`; non-numeric calculate → `Null`; empty regression input → `[]`). The JS side has a `console.warn` fallback path (`chart-compiler.ts:374`) but the WASM path gives it nothing to warn *about* — failures look like legitimate empty/transformed output.
- **`apply_transforms` deep-clones the entire dataset at every pipeline step, then each transform clones again.** `apply_transforms` does `let mut result = data.to_vec()` then reassigns `result = apply_transform(&result, …)` per step (`transforms/mod.rs:22-27`); `apply_transform` borrows `&[DataRow]` and every transform allocates a fresh `Vec` of cloned `serde_json::Map`s (`filter.rs:31`, `calculate.rs:23-30`, `sort.rs:25`, `bin/rows.rs:40-64`, `fold.rs:35-57`). The bridge already hands ownership in (`bridge_pure.rs:568-572` takes `data: Vec<DataRow>` by value) only for it to be re-borrowed and re-cloned. For an N-step pipeline over M rows this is O(N·M) deep `Value`-map clones — the heaviest cost in the crate, all avoidable by threading ownership.
- **`ci0`/`ci1` use a fixed normal-approximation (z = 1.96), which is not Vega-Lite's bootstrap CI.** `aggregate.rs:203-231` computes `mean ± 1.96·(s/√n)`. Vega-Lite `ci0`/`ci1` are bootstrap confidence intervals (resampled), so for small/skewed groups the WASM bars differ from the TS oracle. Either the intended definition is normal-approx (then document it as a deliberate divergence and align the TS grammar) or it must become bootstrap to match — currently undocumented and unverified against the oracle.
- **Global aggregate over empty input drops the summary row Vega would emit.** `apply_single_aggregate` returns `Vec::new()` when `data.is_empty()` (`aggregate.rs:35-37`), so a no-`groupby` `count` over empty data yields `[]`, whereas Vega produces one row `{count: 0}`. Parity gap for empty-data charts.
- **`get_bin_boundaries` re-derives bins from only `[min, max]`, a footgun divergent from binning the real data.** `grid.rs:102-113` calls `calculate_bins(&[min, max], …)` — treating the two endpoints as the entire sample. With `nice` rounding the grid usually matches, but `maxbins`-driven step selection on a 2-element sample can differ from the step chosen over the full dataset, so boundaries returned here need not match the bins `histogram`/`apply_bin` actually use.
- **`value_to_string` can diverge from JS `String()` at the numeric extremes used as category keys.** `utils.rs:11` uses Rust `Number::to_string()`; JS renders large/small magnitudes in exponential form (`String(1e21) === "1e+21"`, Rust → `"1000000000000000000000"`). Rare, but these strings become category/group identities, so a mismatch shifts a series into the wrong category vs the TS path.
- **Mixed-type sort and stacking order are defined but ad-hoc vs JS semantics.** `compare_present_values` assigns a fixed type rank `Bool < Number < String < other` (`sort.rs:85-94`); JS/Vega comparison coerces differently. Nulls-last is correctly stable (Rust `sort_by` is stable, `sort.rs:26`), but cross-type ordering is an unverified parity assumption.
- **Otherwise clean and well-tested.** Zero `TODO`/`FIXME`/`HACK` markers in production code. `stacking.rs` is pure, sanitizes non-finite to 0, and avoids divide-by-zero (`:108`); `grouping.rs` is order-correct; bin grid math guards empty/degenerate ranges (`grid.rs:47-66`) and clamps `find_bin_index` (`:93-99`); every transform has focused unit tests. The improvements are about **parity correctness, surfacing diagnostics, and removing redundant clones** — not redesign.

## Improvement objectives

1. **Make the two expression evaluators correct and unified against one documented grammar.** Replace the byte-`find` comparison split and the `1e-5`-breaking arithmetic split with a single tokenizing/parsing expression module shared by filter and calculate, string/paren-literal-aware, supporting the operator set the TS Vega-expression subset exposes (comparison, logical, arithmetic, unary `!`/`-`, parens, `datum.*`/`datum[..]`, numeric incl. exponent, string, bool, null). This is the highest-value correctness fix.
2. **Restore first-seen group ordering and unify key coercion.** Make `aggregate` preserve first-seen order (reuse/extend `grouping.rs`) and route every group/distinct key through one canonical, JS-`String()`-aligned coercion so filter, group, distinct, and stacking key identically.
3. **Surface transform diagnostics instead of silent degradation.** Wire `ChartError` (or a non-fatal `diagnostics` channel) so unknown fields, unparseable expressions, and invalid bin configs are reported to the JS layer, while still returning best-effort rows. Decide and encode the contract: fail-soft with diagnostics (preferred — preserves the current never-throw behavior the JS fallback relies on) rather than hard error.
4. **Eliminate per-step dataset clones by threading ownership through the pipeline.** Convert `apply_transforms`/`apply_transform` to consume and return owned `Vec<DataRow>`, mutating in place where the transform is row-preserving (calculate, bin, sort, filter-retain).
5. **Close the statistical/empty-data parity gaps.** Align `ci0`/`ci1` with the TS oracle (bootstrap or documented normal-approx), emit the global-aggregate-over-empty summary row, and fix `get_bin_boundaries` to bin the real sample.
6. **Lock parity with a golden cross-check harness.** Add fixtures asserting WASM output equals TS-grammar output for a representative spec corpus, so future drift in either path is caught.

## Production-path contracts and invariants to preserve or strengthen

- **WASM ⇄ TS-grammar behavioral equivalence.** Same input rows + same `Transform[]` must yield identical output rows (values *and order*) on both paths. Strengthen via objectives 1, 2, 5, 6.
- **Never panic / never throw across the FFI boundary.** The JS consumer treats a throw as "fall back to TS" (`chart-compiler.ts:373-376`); panicking in WASM is worse (poisons the instance). All new parsing/eval code must be total (no `unwrap` on input-derived data, no index-out-of-range). Preserve.
- **Serde wire compatibility of `types.rs`.** `Transform` is `#[serde(tag="type", rename_all="lowercase")]`; field renames (`as`, `oneOf`), `camelCase` per-series configs, and tuple `as_fields` are the JS contract. Any new field must be `#[serde(default)]` and optional to stay backward/forward compatible. Preserve.
- **FFI signature stability in `bridge_pure.rs`.** Changing `apply_transforms`'s return type (objective 3) ripples to `chart_apply_transforms`; keep the externally-visible JS shape additive (e.g. return rows as today, expose diagnostics via a separate export or an out-param struct) rather than breaking the existing call.
- **Purity and finite-output discipline.** Transforms are pure functions of their inputs; non-finite values are sanitized (stacking) or nulled (calculate/bin). Strengthen the sanitization to be uniform across transforms.
- **Determinism.** Output must not depend on `HashMap` iteration order. `stacking.rs` and `aggregate.rs` accumulate in `HashMap`/`BTreeMap` but currently emit in input order (stacking) or sorted order (aggregate); after objective 2 both emit first-seen order deterministically.

## Concrete implementation plan

1. **Extract a shared expression module (`transforms/expr/`).**
   - Add `mod.rs` with a small lexer (tokens: `datum.field`, `datum[..]`, number incl. `[eE][+-]?digits`, string `'..'`/`".."`, bool, null, operators, parens) and a Pratt/precedence-climbing parser producing an `Expr` AST. Make lexing string/paren-aware so operators inside literals never split.
   - Evaluator returns `serde_json::Value` (calculate) and a truthiness view (filter). Implement JS-semantics comparison/arith/concat/division-by-zero-→null exactly once, matched to the TS grammar's documented operator behavior.
   - Re-point `filter.rs::apply_expression_filter` and `calculate.rs::apply_calculate` at the shared evaluator; delete the duplicated `split_logical`/`split_binary_op`/`resolve_value`/`resolve_token`. Keep the structured `FilterSpec` path unchanged.
   - Port the existing filter/calculate unit tests verbatim onto the new module, then add the failing cases from Evidence (`"a==b"` string literal, `1e-5`, `-datum.x`, `datum.a + -3`, nested parens, comparison chains the TS grammar allows).
2. **First-seen aggregate ordering + canonical keying (objective 2).**
   - Add a canonical `group_key(value) -> String` in `utils.rs` (JS-`String()`-aligned, with explicit numeric normalization so `1`/`1.0` agree, documented against the TS rule) and use it in `aggregate.rs`, `grouping.rs`, and `Distinct`.
   - Replace the `BTreeMap` in `apply_single_aggregate` with an insertion-ordered structure (reuse `grouping::group_by_fields`, or an index-map pattern as in `grouping.rs:38-61`). Keep aggregate-op computation unchanged.
   - Emit the global (`groupby == []`) summary row for empty input (objective 5).
3. **Diagnostics channel (objective 3).**
   - Introduce a `TransformOutcome { rows: Vec<DataRow>, diagnostics: Vec<ChartDiagnostic> }` (or thread a `&mut Vec<ChartDiagnostic>`), where `ChartDiagnostic` carries `{ transform, field, kind, message }`. Populate on: missing/typed-wrong fields, unparseable expressions (from the new expr module's parse errors), invalid bin config (use `ChartError::InvalidBin`).
   - Keep `apply_transforms` total and best-effort (never throw). Add a new `#[bridge::pure]` export (e.g. `chart_apply_transforms_diag`) that returns rows + diagnostics, leaving `chart_apply_transforms` intact for callers that don't need them; or have the existing export forward to the new one and discard diagnostics. Decide via the `bridge_pure.rs` owner.
4. **Ownership threading for performance (objective 4).**
   - Change `apply_transforms(data: Vec<DataRow>, …) -> Vec<DataRow>` and `apply_transform(data: Vec<DataRow>, …) -> Vec<DataRow>`. For row-preserving transforms, mutate in place: `calculate` inserts into each owned row, `bin` inserts `as`/`as_end`, `sort` sorts the owned `Vec`, `filter` uses `Vec::retain`. Row-multiplying/replacing transforms (`fold`, `aggregate`, `regression`, `density`) build a new `Vec` as today but consume the input.
   - Update `bridge_pure.rs::chart_apply_transforms` to pass its owned `data` straight through (drop the `&` borrow).
   - This is a semantics-preserving refactor; gate it behind the parity harness (objective 6).
5. **Statistical / bin parity (objective 5).**
   - Resolve `ci0`/`ci1`: confirm the TS oracle's definition; if bootstrap, implement a deterministic seeded bootstrap in `compute-stats` (cross-crate dependency — coordinate) and call it; if normal-approx is intended, document it in both crates and add a parity test pinning the constant.
   - Fix `get_bin_boundaries` to accept and bin the real value slice (or remove it if it has no production caller — verify with `rg` before deleting).
   - Audit `value_to_string` for JS-`String()` numeric parity at exponential magnitudes; align with the canonical `group_key` from step 2.
6. **Golden parity harness (objective 6).**
   - Add a fixture corpus of `(rows, transforms)` covering every `Transform` variant and the edge cases above, with expected outputs captured from the TS grammar. Assert equality (values + order). Place fixtures so both the Rust test and a TS-side test can consume them, making drift in either path fail loudly.

## Risks, edge cases, and non-goals

- **Parity risk cuts both ways.** Fixing the WASM path to be "more correct" can still *diverge* from a TS path that has the same bug. The golden harness must capture the **TS oracle's actual current behavior**; where WASM and TS both need fixing (e.g. the expression grammar, `ci0`/`ci1`), the change must land in lockstep across this crate and `mog/kernel/src/domain/charts`, or the fallback will reintroduce divergence. Treat the TS grammar as a co-change, not a fixed oracle, for those items.
- **Expression grammar scope creep.** The Vega expression language is large; the shared evaluator must implement exactly the subset the TS grammar supports — no more (unsupported syntax should produce a diagnostic + null, matching TS), no less. Pin the supported surface in a doc comment and tests.
- **FFI/serde compatibility.** Any new export or struct must not break existing `bridge-ts` codegen or the `compute-bridge.gen.ts` shape; additive only. `ChartSortOrder` is deliberately named to avoid a codegen alias collision (`types.rs:258-262`) — keep that discipline for new enums.
- **Float determinism.** Bootstrap CI (if adopted) must be deterministically seeded so WASM and any native run agree; never use unseeded RNG.
- **In-place mutation correctness.** Threading ownership must not change observed order or field-insertion order (serde `Map` preserves insertion order; calculate/bin appending columns must keep appending, not reorder).
- **Non-goals:** no new chart/transform *types* beyond the existing union; no redesign of `compute-stats`' core math (only `ci0`/`ci1` and quantile parity are touched, by coordination); no change to the stacking algebra (already correct); no removal of the TS fallback path; no test-only or shim fixes — every change is on the production transform path.
