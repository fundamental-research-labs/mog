## Current role of this folder in Mog

This folder is the **decomposed input-behavior layer** sitting between the orchestrator hook and the domain/coordinator. The orchestrator (`hooks/shared/use-grid-mouse.ts`) attaches native `pointerdown/move/up` listeners to the grid container (for `setPointerCapture`) plus React `onDoubleClick`/`onContextMenu`/`onMouseLeave`, performs hit-testing through the `SheetView` capabilities, and then **delegates the semantic decisions** to the hooks here:

1. **Cell sub-interactions** (`use-cell-interaction`) — given a hit cell and the click position within it, decide whether the click lands on a comment indicator, a checkbox, or a validation dropdown, and start/extend cell editing on double-click with the caret placed at the click point.
2. **Context menu** (`use-context-menu-handler`) — classify the right-click target, reconcile selection (replace selection when clicking outside it; preserve full-row/full-column selections), and hand the orchestrator a `ContextMenuOptions` payload, deferring to Radix for floating-object menus.
3. **Cursor feedback** (`use-cursor-manager`) — translate hit-test results into CSS cursors, writing them **directly to the DOM** on every mouse move to avoid React re-renders, with a `useMemo` fallback for the React `style` path.
4. **Drag micro-modes** (`use-formula-range-drag`, `use-warp-adjust-interaction`) — ref-based drag state machines that bypass React state for per-frame updates.
5. **Pure hit-test math** (`helpers/*`) — the geometry that maps a click to a UI sub-element or a caret index, kept side-effect-free for unit testing.

The folder's dominant non-functional concern, repeated in nearly every file's header, is **render isolation**: hooks read editor/selection state on-demand via `getSnapshot()`/getter functions instead of subscribing, and drag state lives in `useRef` so the per-frame path never triggers React renders. The benchmark suite (`__tests__/benchmarks.test.ts`) exists to defend this.

The folder is also explicitly **mid-refactor**: `index.ts` declares itself "the first step in refactoring the monolithic use-grid-mouse.ts hook," and the 2139-line orchestrator confirms the extraction is incomplete. This plan does **not** finish that extraction wholesale (that is a separate, larger effort owned by the orchestrator folder) — it hardens the pieces already extracted, removes dead/divergent code, and fixes correctness/perf defects that ship today.

### Key structural facts established by inspection (read-only)

- **Debug `console.log` in the production cursor path.** `use-cursor-manager.ts:271–281` logs `'[useCursorManager] cursor computation:'` and `'returning ink cursor:'` (with a per-call `inkCursor?.slice(0, 50)` string allocation) **inside the `cursor` `useMemo`**. This fires on every recompute of the cursor fallback — console spam in production and avoidable work. This is a concrete shipped defect.
- **Orphaned, divergent warp-adjust implementation.** `use-warp-adjust-interaction.ts` (375 lines) and its pure helpers `getWarpAdjustCursor`/`isWarpAdjustHandle`/`calculateWarpAdjustHandlePosition` have **zero consumers** outside the folder's own tests (`rg` across `apps/spreadsheet/src` excluding `grid-mouse/` finds none). The **live** warp-adjust feature is implemented as an XState flow in `systems/objects/machines/object-interaction-machine.ts` (`UPDATE_WARP_PREVIEW`, start/preview/commit/cancel events). So this file is a parallel, unwired duplicate of a feature that already ships through a different path — dead code that drifts from the real implementation.
- **Dead filter-button hit-test.** `isClickOnFilterButton` + the `FILTER_BUTTON` constants are exported but only re-exported by the `hooks/index.ts` barrel — no functional consumer. The file headers themselves state filter-button clicks "are now handled by DOM overlays (`FilterButtonOverlay`)." The hit-test and its constants are stranded after that migration.
- **Hand-synced renderer constants (drift risk).** `helpers/click-detection.ts` hard-codes `FILTER_BUTTON`, `VALIDATION_DROPDOWN`, and `COMMENT_INDICATOR` sizes with comments saying each "must match" `cells-layer.ts` / `validation-renderer.ts`. These are duplicated literals across the input layer and the render layer with only a comment enforcing agreement — silent mis-hit on any renderer change.
- **Duplicated word-boundary logic.** `use-cell-interaction.ts:542–575` defines `findPrevWordBoundary`/`findNextWordBoundary`, duplicating the canonical word-boundary functions in `systems/grid-editing/machines/editor/cursor-movement.ts`. Worse, the double-click word-select **re-implements selection as an O(n) loop of `editorActions.selectRight()` dispatches** (`use-cell-interaction.ts:453–455`) — one dispatch per character — rather than a single select-range command.
- **Per-call canvas allocation + duplicated measurer.** `helpers/cursor-position.ts` ships two near-identical functions: `calculateCursorPosition` (creates a **fresh `document.createElement('canvas')` on every call** — once per double-click) and `calculateCursorPositionWithMeasurer` (DI variant for tests). The production path uses the allocating variant; the testable variant is parallel code.
- **Async fire-and-forget in the click path.** `handleCellClick` (`use-cell-interaction.ts:387–391`) calls `void handleValidationDropdownClick(cell, clickPosition)` and immediately returns `false`, so the orchestrator proceeds with normal selection while the async dropdown open races behind it. Separately, `handleCellDoubleClick` and `handleValidationDropdownClick` are typed `boolean | Promise<boolean>`, an under-specified sync/async contract the orchestrator must remember to `await`.
- **Formula range drag is main-pane only.** `use-formula-range-drag.ts:254–261` synthesizes a hit-test region with `bounds {0,0,0,0}` and a comment admitting frozen-pane awareness "would require iterating over actual rendered regions — that is tracked separately." Dragging a formula range box whose cells live in a frozen pane misbehaves. A real product limitation, documented but unaddressed.
- **Stale/over-broad `useCallback` deps.** `handleCommentIndicatorClick` lists `activeSheetId` in its dep array but never reads it; `useContextMenuHandler`'s `handleContextMenu` lists `activeSheetId` (also unused in the body). These don't cause bugs but obscure the real dependency surface and defeat memoization reasoning.
- **Touch-tolerance logic lives in the orchestrator, not here.** `GridMouseEvent.pointerType` exists precisely to tighten hit tolerances for touch (per the `types.ts` comment), yet the `borderTolerance = e.pointerType === 'touch' ? 5 : 3` expression is duplicated **four times** in `hooks/shared/use-grid-mouse.ts` (lines ~954, ~1122, ~1454, ~1966). The folder owns the `GridMouseEvent` abstraction but not the tolerance helper that should accompany it.

## Improvement objectives

1. **O1 — Remove shipped debug logging from the cursor path.** Delete the two `console.log` calls in `use-cursor-manager.ts` so the `cursor` `useMemo` does only cursor computation. If ink-cursor diagnostics are genuinely wanted, route them through the app's existing debug-logger gate rather than raw `console.log`.
2. **O2 — Eliminate dead/divergent code.** Remove `use-warp-adjust-interaction.ts` and its exports (the feature ships via `object-interaction-machine.ts`), and remove the dead `isClickOnFilterButton`/`FILTER_BUTTON` hit-test (filter clicks ship via `FilterButtonOverlay`). Prune the corresponding barrel exports. This shrinks the module to what is actually wired and stops the duplicates from drifting.
3. **O3 — Make the cell-click contract synchronous and ordered.** Replace the `void handleValidationDropdownClick(...)` fire-and-forget and the `boolean | Promise<boolean>` unions with a single, awaited, deterministic decision so a dropdown-open click can no longer race normal selection. The orchestrator gets one `Promise<boolean> | boolean` it must await, or — preferred — a fully synchronous "handled?" decision with the async editor start kicked off only after the handled-verdict is returned.
4. **O4 — Unify caret-position measurement on a cached, injected measurer.** Collapse `calculateCursorPosition` and `calculateCursorPositionWithMeasurer` into one function backed by a **shared, reused** `CanvasRenderingContext2D` (module-scoped, lazily created once) so double-clicks stop allocating a canvas per call, and the production and test paths exercise the same code.
5. **O5 — Reuse the canonical word-boundary + selection commands.** Delete the local `findPrev/NextWordBoundary` copies in favor of the editor domain's canonical exports, and replace the O(n) `selectRight()` loop with a single select-range/select-word command (adding one if the selection/editor API lacks it — cross-folder dependency).
6. **O6 — Establish a single source of truth for render-aligned hit-test constants.** Move `FILTER_BUTTON`/`VALIDATION_DROPDOWN`/`COMMENT_INDICATOR` geometry to a shared constants module imported by both the renderer (`cells-layer.ts`, `validation-renderer.ts`) and `click-detection.ts`, so input hit-areas can never silently diverge from what is painted.
7. **O7 — Own touch tolerance in the abstraction that defines `pointerType`.** Add a pure `getBorderTolerance(e: GridMouseEvent)` helper (and any sibling tolerance helpers) to this folder and have the orchestrator call it, removing the four duplicated `=== 'touch' ? 5 : 3` expressions.
8. **O8 — Close the frozen-pane gap in formula range drag.** Replace the synthesized zero-bounds single region with iteration over the actual rendered regions (frozen + main panes) supplied by the geometry/viewport capability, so dragging a range box resolves the correct cell in any pane.
9. **O9 — Tighten memoization hygiene.** Drop unused `activeSheetId` (and any other unread) entries from `useCallback` dependency arrays so the dependency surface reflects reality.

Objectives are ordered by value/risk: O1–O2 are pure subtractions (highest value, lowest risk); O3–O5 fix correctness/perf; O6–O9 are structural hardening.

## Production-path contracts and invariants to preserve or strengthen

- **Render isolation (strengthen).** The per-frame path (mouse move, drag move, cursor write) must not trigger React re-renders. Drag state stays in `useRef`; cursor writes stay direct-DOM; on-demand state reads stay `getSnapshot()`/getter-based. O1 strictly reduces work on this path; O3 must not introduce a subscription. The `benchmarks.test.ts` thresholds are the guardrail.
- **`GridMouseEvent` duck-type (preserve).** Handlers must keep accepting both React `MouseEvent` and native `PointerEvent` via the minimal `GridMouseEvent` interface — no casts between the two. O7's tolerance helper must take `GridMouseEvent`, not a concrete event type.
- **Context-menu selection reconciliation (preserve exactly).** The subtle rules in `use-context-menu-handler.ts` must be kept verbatim: right-clicking outside the selection replaces it; right-clicking a column/row header only preserves the current selection when it is already a *full*-column/row selection containing that index (the `isFullColumn`/`isFullRow` guards), so `INSERT_ROW_ABOVE`/`INSERT_COLUMN` target the clicked index. Floating-object hits must **not** `preventDefault()` (Radix needs the native event); empty/frozen areas **must** `preventDefault()`. These are encoded in existing tests and must stay green.
- **Caret-placement fidelity (preserve).** Double-click-to-edit must continue to place the caret at the clicked character using the cell's actual canvas font (`getCellCanvasFont(format)`) and `DEFAULT_CELL_STYLE.padding`; O4 changes only *how* the measurement context is obtained, not the measured result. The midpoint-rounding behavior in `calculateCursorPosition` must be byte-for-byte preserved (unit tests pin it).
- **Formula-range edit semantics (preserve, then extend).** The active-sheet filtering (cross-sheet range boxes must not consume clicks meant to add a new reference) and the "only update if range actually changed" guard must remain. O8 extends pane coverage without altering which range-box a given pixel selects in the main pane.
- **Protected-cell handling (preserve).** Both edit-start paths surface `showProtectionAlert(reason)` when `startEditing` fails with a protected reason; refactors must keep this.
- **Async editor start ordering (strengthen).** After O3, the rule becomes explicit: selection/handled decisions are synchronous; only the editor *start* (which is legitimately async) is deferred, and it is never interleaved with a selection mutation for the same click.

## Risks, edge cases, and non-goals

- **Risk — removing warp-adjust hides a half-wired feature.** If product intends to switch warp-adjust to this hook, deletion loses the code. Mitigation: the live path is the machine; verify with `rg` (done — zero consumers) and capture the intent in the follow-up note. Re-derivation from the machine is cheap if ever needed.
- **Risk — shared constants module (O6) is cross-cutting.** Touching the renderer to import shared geometry can ripple into render snapshot tests. Mitigation: move only the numeric constants (no behavior), land behind renderer-owner review, and gate on the canvas render tests.
- **Risk — word-boundary semantics differ.** The canonical editor functions may treat punctuation/Unicode differently than the local `[\w]` regex. Mitigation: diff behavior on a fixture table; if they differ, adopt the canonical (keyboard-consistent) semantics deliberately and document the change rather than preserving the divergent copy.
- **Edge case — SSR/no-canvas (O4).** The cached 2D context can be `null` (no DOM / jsdom); the wrapper must fall back to `text.length` exactly as today.
- **Edge case — O3 microtask ordering.** Deferring the editor start must not let a subsequent pointerdown be processed against stale editor state; the handled-verdict must be synchronous so the orchestrator's `return` short-circuits before any competing handler runs.
- **Non-goals.** This plan does **not** finish migrating the 2139-line `hooks/shared/use-grid-mouse.ts` orchestrator into this folder (separate effort, owned by the shared-hooks/orchestrator review). It does not change selection/editor state-machine semantics, does not alter rendered output, does not touch the `FilterButtonOverlay` DOM-overlay approach, and introduces no compatibility shims or test-only fixes — every change is a production-path edit.
