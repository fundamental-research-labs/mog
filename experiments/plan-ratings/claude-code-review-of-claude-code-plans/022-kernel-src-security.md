Rating: 8/10

# Review — Plan 022: Kernel Security (Capability & Protection Enforcement)

## Summary judgment

This is a strong, unusually well-evidenced plan. Its central thesis — that `mog/kernel/src/security` ships an orphaned, contract-divergent, untrusted "v2" subject-based enforcement model alongside an older AppId-based system that is what `index.ts` actually re-exports and what production consumes — is **verified true in every checkable particular**. I independently confirmed:

- `index.ts` re-exports exclusively from `../services/capabilities/*`; none of the local `security/*.ts` service symbols (`CapabilityGrantService`, `CapabilityRegistryService`, `TrustPolicyService`, `LegacyGrantServiceAdapter`) are exported. The v2 files have no production importer.
- `grant-service.ts`: `findMatchingGrant` returns the first `Map`-iteration broader match (order-dependent); `revoke` deletes only the exact key; the injected `trustPolicy` and `registryService.scopeSchema` are unused on `grant`; implied caps are returned with no scope/expiry; `grantedAt: Date.now()` (number).
- `trust-policy.ts`: bundled/first-party returns an empty `autoGrantList` documented as "means all" and `canAutoGrant` returns `true` for *every* capability (including `mog:network:unrestricted`, `riskTier: 'critical'`) when `firstPartyAutoGrantCapabilities` is unset; `bundled` is a self-asserted flag.
- `capability-registry.ts`: `extractNamespace` returns `undefined` for <3-segment IDs (ownership bypass); namespace ownership is first-come; `getImplied` BFS de-dups but does not reject cycles or dangling `implies`.
- The `@mog-sdk/app-platform` contracts are real and the drift is exactly as described: branded `CapabilityId`/`PackageId`/`AppId`/`AppInstanceId`, `CapabilityGrant.timestamp: string` (ISO-8601), `TrustPolicyDecision.canAutoGrant: boolean` (single, with required `trustSource`), `evaluateTrust(packageId, source)`, `grant(...) => void`, `revoke(...) => void`. `scopeMatches` does exist in `services/capabilities/scope.ts`, and the four `__tests__` files exist.

Because the divergence sits on an authorization boundary, framing it as a correctness/security concern (not tidiness) is correct, and the prioritized soundness fixes — deny/most-specific precedence, revoke tombstones, implied-scope propagation, scope validation, no silent auto-grant of critical caps, namespace authority — are the right ones.

## Major strengths

- **Evidence discipline.** The plan reads as if written by someone who actually read every file; the appendix claims hold up to verification. This is rare and is the single biggest reason to trust the plan.
- **Genuine security findings, not cosmetics.** The unconditional first-party auto-grant of `critical`/unrestricted caps, the short-ID namespace-ownership bypass, the self-asserted `bundled` flag, and revocation defeatable by a surviving broader allow are all real, exploitable-class authorization defects.
- **Authorization semantics are specified precisely.** Deny-precedence, most-specific-wins, order-independence (no `Map`-iteration reliance), tombstones surviving broader allows, and implied-scope-as-upper-bound are stated as testable invariants rather than vibes.
- **Verification gates are above average.** The **type-level contract-conformance assertion test** (build fails if the kernel types drift from `@mog-sdk/app-platform` again) directly attacks the root cause — drift — rather than a symptom. The unit test matrix maps one-to-one onto the soundness fixes.
- **Honest risk section.** It calls out the persisted-grant data migration (timestamp type change), the silent match-set change when moving AppId→subject keying, and tombstone GC-on-regrant — all real and easy to miss.

## Major gaps or risks

- **Phase 6 is the crux and it is under-specified.** "Update `services/capabilities/*` so the AppId-keyed store becomes a thin internal detail behind the subject service (**or is replaced by it**)" hides the hardest decision in a parenthetical. The *live* system is materially richer than v2: it has `GrantChangeEvent`s, persisted stores (`stores/`), an audit logger, re-auth, a sensitive-capability handler, and a requester. The v2 `CapabilityGrantService` has **none** of these — no events, no persistence, no audit hook, no sensitive handling. The plan picks the *less capable* implementation as canonical without a mapping for the capabilities the canonical-today system has and v2 lacks. "Thin internal detail" badly understates this; the reconciliation, not the soundness fixes, is where the project will actually succeed or fail.
- **Scope is an epic dressed as a folder plan.** It mutates a cross-package contract (`app-platform`), the in-folder v2, the live `services/capabilities` system, ~8 shell consumers (`launch-app.ts`, `capability-context.tsx`, settings UI, `use-app-kernel.ts`), and a persisted-data migration. The plan acknowledges sequencing/parallelization, but there is no phase-to-PR decomposition or rollback story, and "best done by the same worker in lockstep" is a staffing wish, not a plan.
- **"Conform to the contract" vs "change the contract" is bidirectional and presented as more settled than it is.** The plan adopts the contract for timestamps/`grant` return, but proposes *evolving the canonical contract* to the impl's list-form `TrustPolicyDecision` (replacing `canAutoGrant`). That is defensible (the list form is more expressive), but it is the tail wagging the dog on the canonical type and a genuine design fork that the app-platform owner could reject — yet it is folded into Phase 0 as if it were a diff to resolve rather than a contended decision. The whole timeline gates on it.
- **Provenance for authenticating trust signals is asserted, not located.** Phase 4 rightly says `bundled`/`signature`/`enterprisePolicyId` must come from a verified record. But the canonical signature is `evaluateTrust(packageId, source: PackageSource)` — so that provenance must live inside `PackageSource`, and the plan never pins down what `PackageSource` actually carries or who establishes it. Without that, "authenticate the bundled signal" is a goal with no mechanism.
- **`expiresAt` is load-bearing but extra-contractual.** Several invariants (implied-grant expiry propagation, injected-clock expiry tests) depend on `expiresAt`, which the plan itself notes is *not* in the contract and must be proposed as an addition. If the contract owner declines, multiple Phase-2 behaviors lose their field.

## Contract and verification assessment

The contract analysis is the plan's best dimension: accurate field-level diffs, the right canonical target (branded types re-exported so kernel and contracts cannot re-drift), and a build-failing conformance gate to enforce it. Verification gates are concrete and tied to specific defects. Two gaps keep this from being airtight: (1) the persisted-store migration is named as a risk but has **no** corresponding test or migration step in the plan body (it should be a gate, not a footnote), and (2) the integration gate ("app-launch consent/grant/deny still flows after the adapter is removed") is correct but thin given that Phase 6 changes the keying model underneath every existing grant — one end-to-end scenario will not cover the silent-match-change risk the plan itself raises. Note also that the `legacy-adapter` "consumer" is documented in its own docstring but has **no actual importer** today (confirmed); the plan's careful wording ("currently single, documented call site") is honest, but Phase 5's migration work may be smaller (delete-only) than it reads.

## Concrete changes that would raise the rating

1. **Specify Phase 6 reconciliation explicitly (highest impact).** Decide and document whether the live system's stores/events/audit/re-auth/sensitive-handler are (a) re-pointed at subject-keyed storage behind the existing barrel, or (b) reimplemented on v2. Provide the mapping for each: where do `GrantChangeEvent`s, audit entries, and sensitive-handler hooks fire in the subject service today (they don't)? This is the difference between a plausible plan and a buildable one.
2. **Decompose into sequenced PRs with rollback.** e.g. PR1 contract additions (+`expiresAt` decision) → PR2 conform v2 impl + soundness fixes + conformance gate → PR3 reconcile live system behind frozen barrel → PR4 migrate consumers + delete adapter + data migration. State the green-to-green checkpoint for each.
3. **Resolve the `TrustPolicyDecision` shape as a decision, not a diff.** Get an explicit yes/no from the app-platform owner on list-form vs `canAutoGrant` before Phase 0 "completes," and record the fallback if they decline (e.g., keep `canAutoGrant` and express richer policy out-of-band).
4. **Pin down `PackageSource`.** Show what it carries and who populates `bundled`/`signature`/`enterprisePolicyId` such that they are verifiable, or add a phase to establish that provenance. Otherwise Phase 4's hardening has no foundation.
5. **Promote the persisted-grant migration to a real gate** with a migration step and a round-trip/read-old-write-new test, rather than a risk-section mention.
