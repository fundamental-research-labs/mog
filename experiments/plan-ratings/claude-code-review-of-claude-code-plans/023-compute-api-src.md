Rating: 8/10

# Review of Plan 023 — Harden and unify the `compute-api` boundary

## Summary judgment

This is a strong, unusually well-evidenced plan. Almost every concrete claim it makes about `mog/compute/api/src` is verifiable in the source as written, and the plan is honest about the limits of its own evidence (e.g. that the `Workbook`/`Sheet` facade has no production FFI consumer today). It correctly separates the one genuine owner-level decision (facade enforcement posture) from the mechanical hardening work, sequences the risky behavioral change last behind that decision, and ties each verification gate to a real test file or build target. The deductions are for a priority-framing tension around its top-ranked objective, an understated behavior-change risk in the write-path change, and a couple of imprecise framings in the address/range objective — sharpenings, not structural flaws.

I verified the plan against the code rather than taking its word: `set_cell` does take `impl Into<String>` and dispatches `set_cell_value_parsed(&str)` (sheet.rs:82-96); `set_range` blindly materializes `start_row + i`, `start_col + j` from a possibly-jagged `&[Vec<String>]` and discards `_end_row`/`_end_col` (sheet.rs:99-118); `CellRange::resolve`/`parse_a1_range` neither normalize inverted corners nor bound-check the `Bounds` variant (address.rs:96-198); `sheet_names` does 1+N `query_engine` round-trips (workbook.rs:121-137); public signatures leak `compute_core::storage::engine::search::WorkbookTable`, `engine_types::DataBounds`, `ZOrderEntry`, and `BridgeSortOptions` (workbook.rs:149-170, sheet.rs:249-266, 434); the five named stub modules exist and advertise no capability; `error.rs` has two independent `SecurityDenied` reconstruction paths (`From<SecurityError>` at 193-224 vs `rehydrate_security_denied` at 236-263) that do not share a constructor; and `to_bridge_value`'s tests cover only 4 of 9 variants. I also confirmed the consumer claims: all four binding crates import `ComputeService`, no non-test code outside the crate uses the facade, `gated = true` is set in `bridge_service.rs`, and `formula-eval` uses only `compute_api::pure::pivot_convert`.

## Major strengths

- **Evidence grounding.** Specific, falsifiable claims throughout, and they hold up. This is the difference between a plan that read the code and one that pattern-matched a template.
- **Honest scope and consumer analysis.** The in-scope/out-of-scope list is precise, and the plan explicitly states the facade has no production FFI consumer — the single most important fact for judging the severity of Step 1.
- **Correct identification of the real fork.** Step 1 is framed as "investigate-then-commit" with an explicit owner decision (committed Rust SDK → enforce; internal-only → feature-gate). The plan does not pretend to know the answer, which is the right call.
- **Invariants section.** The preserve/strengthen split is excellent — WASM memory discipline (`default-features = false`), bridge auto-generation as a consumed contract for four crates, the `SdkValue` semantic table, and `Principal` non-serializability are all load-bearing and correctly flagged.
- **Verification gates are concrete.** Golden snapshot over *all* variants (closing the 4/9 gap), a counting `Dispatch` wrapper to lock the O(1)-hop claim, enforcement-parity through `security_e2e`, and a cross-target cargo-tree check that `native` doesn't leak into WASM.
- **Sequencing is reasoned, not arbitrary.** Land isolated low-risk steps (3/5/6) first, then write-intent (2), then lock the contract (4), then the behavioral change (1) behind its decision, robustness (7) last.

## Major gaps or risks

1. **Priority/severity tension on Step 1.** The plan ranks "close the security-bypass divergence" as objective #1 and "highest priority," yet its own evidence shows the divergent surface has *no production consumer* — the only in-process callers are `formula-eval` (internal tooling) and the crate's own tests. A bypass reachable only by trusted in-process Rust is materially lower production-path severity than "highest priority" implies. The plan partly resolves this by offering path (b) feature-gating (likely the correct answer given no consumer) and by sequencing the work last, but the framing should be reconciled: rank by production-path exposure, and state plainly that path (b) is the default unless an owner commits the facade as a public SDK.

2. **Step 2 understates a behavior change for existing string callers.** Switching `set_cell` from `impl Into<String>` to `impl Into<SdkValue>` is described as source-compatible and intent-preserving, but it silently *changes runtime behavior* for callers passing an empty `String`: today `""` flows through `set_cell_value_parsed("")` (Parse), whereas the new path maps `Text("")` → `Literal("")` per the `SdkValue` table. That is precisely the intended fix, but it is a semantic change, not just a type change, and the plan's risk note only mentions the type-inference angle (`String`-typed variables). It should call out the empty-string semantics flip explicitly and add a regression assertion for it.

3. **"Byte-identical wire shapes" between the two SecurityDenied paths may be unachievable at the edges.** The two paths consume different inputs: `From<SecurityError>` derives tags from `principal.tags()` (structured) and `operation` from `operation.to_string()`, while `rehydrate_security_denied` splits a flat `principal_tags` string on `,` and takes `operation` as a raw `&str`. Sharing one *constructor* (the plan's proposal) converges the tail, but a tag containing a comma, or any lossy string encoding upstream, can still diverge at the *input-parsing* step. The gate should test convergence at the construction boundary and acknowledge that input-parsing fidelity is a separate, upstream concern.

4. **Minor imprecision in the address/range objective.** Objective 3 says `parse_a1_range` accepts "unbounded ranges," but `parse_a1_range` actually requires both endpoints to be full A1 cells (`A:A` fails at `parse_a1("A")` with "no row number"). The unboundedness is real only for the numeric `Bounds(u32,…)` variant (which accepts arbitrary/`u32::MAX` corners). Separately, adding `MAX_ROWS`/`MAX_COLS` validation to the `Bounds`/`Position` pass-through is itself a behavior change that could reject deliberately-large whole-column-style bounds some caller relies on; the plan should state how whole-row/whole-column intent is represented post-change, since `parse_a1_range` rejects `A:A`/`1:1` today.

5. **Step 7 is appropriately hedged but thin.** The unbounded-channel backpressure and `Drop`-without-join concerns are real, but "evaluate a bounded channel" and "make Drop join or detach with rationale" are decisions deferred without a recommended default. Given it's ranked last and gated behind benchmarking, this is acceptable, but a stated leaning (e.g. "document the unbounded contract; do not change semantics without a load benchmark") would tighten it.

## Contract and verification assessment

The contract analysis is the plan's strongest dimension. It correctly identifies the four consumed contracts that must not regress (the descriptor-group list and `__bridge_descriptor_ComputeService_*` macros, the error envelope's PascalCase-`kind`/camelCase-field shape with `Compute(inner)` pass-through, the `SdkValue` semantic table, and `Principal` non-serializability) and these match the code. The error-envelope contract is the highest-leverage one because it is mirrored by hand in `bridge-error.ts` and grepped by app-eval scenarios for human substrings ("part of an array formula") — the plan's golden-snapshot-plus-cross-language-fixture gate directly defends it, and the existing test at error.rs:444 confirms the substring-survival requirement is real.

Verification gates are concrete and mostly sufficient: address/range unit cases (inverted/out-of-bounds/jagged), write-intent regression extended to the facade path, all-variant golden snapshot, enforcement parity via `security_e2e`, a round-trip-counting `Dispatch` wrapper, and the WASM-feature-leak cargo-tree check. The plan correctly defers actually *running* cargo/clippy/test to the implementation worker. Two gates need sharpening: the "byte-identical" SecurityDenied assertion (see gap 3), and the enforcement-parity gate should specify what the *expected* posture is per chosen path (a vs b), since the test that proves "denied through the facade is rejected" only makes sense under path (a) — under path (b) the parity test becomes "the unenforced facade is not reachable in a default build," a different assertion the plan doesn't spell out.

## Concrete changes that would raise the rating

1. **Reconcile Step 1's priority with its exposure.** Re-rank by production-path reachability, name path (b) as the default given the absence of any facade FFI consumer, and make the owner decision a yes/no gate ("does any shipping product call `Workbook`/`Sheet` directly?") rather than an open architectural question.
2. **Add an explicit empty-string-semantics regression to Step 2** and revise the risk note to state that existing `""` string callers change from Parse to Literal behavior — verified across the workspace with `cargo check` *and* a behavioral test, not just compilation.
3. **Re-scope the SecurityDenied convergence gate** to assert equivalence at the shared constructor and separately document the input-parsing fidelity limit (comma-in-tag, lossy encodings) as out of scope or as a distinct upstream fix.
4. **Fix the address/range framing**: attribute unboundedness to the `Bounds`/`Position` numeric variant (not `parse_a1_range`), and specify the post-change representation/handling of whole-row/whole-column ranges so the new bounds check doesn't silently reject legitimate large ranges.
5. **Make the enforcement-parity test conditional on the chosen path** (a: facade-denied surfaces typed `SecurityDenied`; b: facade absent from default build), so the regression guard is well-defined regardless of the owner decision.
6. **Give Step 7 a stated default** (document the unbounded contract; require a load benchmark before any channel-semantics change) so it is actionable as written rather than purely investigative.
