Rating: 9/10

# Review of Plan 025 — Harden the recalc scheduler (`mog/compute/core/src/scheduler`)

## Summary judgment

This is an excellent, evidence-grounded plan that I would hand to an implementer with high confidence. Every concrete claim I spot-checked against the source is accurate, down to line numbers and constant values. It correctly identifies that this is a mature, well-commented subsystem whose real risks are concentrated in *silent degradation paths, a documented cross-sheet correctness gap, structural duplication, and panic-on-invariant points* — not gross algorithmic defects — and it scopes the work accordingly. The plan is genuinely production-path: it strengthens correctness, observability, and maintainability of the live recalc engine rather than building a shim or test harness. It explicitly enumerates the invariants that must survive, sequences the work to avoid same-file churn, and bakes in investigation gates that can *descope* an objective if the evidence doesn't support it. The only reasons it isn't a 10 are that a couple of objectives rest on Phase-0 findings that could shrink them, the verification gates are necessarily unexecutable under task constraints, and one objective (5) slightly understates what already exists in the code.

## Major strengths

- **Verifiable specificity.** I confirmed the cited facts: `spill.rs` `MAX_DEPTH = 5` with the silent `Ok((Vec::new(), …))` returns on both depth-overflow (`spill.rs:197`) and `past_deadline` (`spill.rs:207`); the `TODO(P1)` cross-sheet schema gap in `schema_validation.rs:107-118` (the comment even pre-sketches the exact fix the plan adopts); the two `.expect("SUMIFS cache epoch must be initialized…")` panics at `passes.rs:35` and `:712`; the "any changes to the per-level body must be mirrored" warning in the `passes.rs:11-19` header; the duplicated `topo_evaluate_pass` (`:20`) / `topo_evaluate_pass_with_levels` (`:461`) functions; `metrics.timed_out` and the `#CALC!` marking at `passes.rs:283-307`; and that `RecalcMetrics` already carries `timed_out`, `agg_prepass_groups/cells`, and `projection_conflicts`. A plan whose factual substrate holds up this completely is rare.
- **Correct severity triage.** It explicitly separates the *correctness* risk (projection-stabilization aborts leaving stale spilled values) from the *performance* risk (agg-prepass bail-out), and treats them differently in Phase 1. This is the right instinct and avoids over-engineering the perf path.
- **Invariant catalogue is the best part.** The "contracts and invariants to preserve or strengthen" section is a precise, sourced list (single evaluator core, result completeness/ordering, four-strategy boundary, `WriteTrust` region atomicity, cycle seeding semantics, thread-local cache hygiene, WASM/native result parity). This converts "don't break things" into checkable assertions and directly anchors the regression gates.
- **Honest, self-limiting refactor.** Objective 3 (dedup `passes.rs`) is the highest-risk item, and the plan says so, mandates a line-by-line Phase-0 diff, and pre-commits to *descoping to documentation* if the bodies turn out not to be truly identical. That is exactly how a risky refactor of mature code should be framed.
- **Sequencing and parallelization are real.** Landing Objective 3 before Objective 5's edits to the same file, sequencing Objective 4's epoch change after Phase 3, and grouping independent file-disjoint objectives for concurrency all reflect actual file-conflict reasoning, not boilerplate.
- **Wire-contract awareness.** It flags that new `RecalcMetrics` fields cross the `snapshot-types` boundary into the kernel/TS layer, must be default-valued/backward-compatible, and may require a `@mog-sdk/contracts` declaration rollup — and correctly refuses to edit those out-of-scope files.

## Major gaps or risks

- **Objective 5 under-credits existing behavior.** The plan frames timeout cells as "indistinguishable downstream from genuinely-errored cells." In fact the timeout path already pushes a `CellErrorInfo { error: "Recalculation timeout exceeded" }` (`passes.rs:303-306`), distinct from a genuine `#CALC!` error info. What is *not* distinguishable is the in-cell `CellValue::Error(CellError::Calc, None)` written into the mirror. The plan's Phase-0 framing ("confirm whether the engine/UI surface `timed_out` distinctly") would surface this, so it is not fatal — but the objective should state the gap more precisely (in-cell value vs. side-channel error info) rather than implying no signal exists.
- **Two objectives are contingent on Phase 0 and could collapse.** Objective 1's `MAX_DEPTH` fix branches between "report truncation" and "iterate to fixpoint," and the plan admits it can't choose until it knows whether real workbooks exceed depth 5. Objective 3 may reduce to a doc change. This is responsible, but it means the *delivered* scope is less certain than the headline objective count suggests. The plan would be stronger if it stated a fallback estimate of value if both contingencies resolve to the smaller option.
- **No measured baseline for Objective 7.** The plan correctly gates allocation tuning on profiling, but offers no current numbers or a concrete benchmark command/fixture beyond "263K / stress-many-sheets style." Without a named benchmark target the gate is hard to enforce; this objective is appropriately lowest-priority but the weakest-specified.
- **Verification gates are unexecutable here and partly aspirational.** Per task constraints the plan does not run tests, so claims like "must fail on `main` and pass after the fix" (Objective 2) and "byte-identical computed values" (Phase 3) are assertions the implementer must validate, not facts established. The plan is transparent about this, but the WASM-vs-native parity gate in particular is operationally heavy and under-detailed (no command for the WASM single-threaded config).

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. The preserve/strengthen invariants are individually sourced and actionable, and the non-goals fence is crisp (no eval-semantics changes, no `DependencyGraph` algorithm changes, no new public `ComputeCore` API, no four-strategy rework, no compatibility shims beyond default-valued field additions). The result-completeness-and-ordering contract (`entrypoints.rs:99-110`, seed-before-dependents, dedup) is explicitly tied to viewport patching and re-asserted as a Phase-3/4 gate, which is the correct linkage.

Verification gates are well-targeted per objective: each objective has a matching test obligation (truncation flag flips on a synthetic >5-deep cascade; cross-sheet constraint revalidation; non-panicking epoch path + deferred-graph debug-assert firing; timeout-tagged cells distinct from `#CALC!`). The "results identical across the named `scheduler_tests` suites" gate is the right backstop for the refactor. The main weaknesses are operational, not conceptual: the WASM/native parity run lacks a concrete invocation, and the performance gate lacks a named baseline. The static gates (clippy/fmt) are correctly delegated to the implementer.

## Concrete changes that would raise the rating

1. **Restate Objective 5 precisely:** acknowledge the existing distinct `CellErrorInfo.error` string and scope the fix to the *in-cell value* legibility (e.g., an `extra_flags` bit or a distinct error reason on the `CellValue`), so the objective targets the genuine gap rather than implying none exists.
2. **Give Objective 1 a decision rule, not just an investigation:** state the concrete observable Phase 0 will check (e.g., "do any existing spill/projection tests or corpus fixtures drive `projection_stabilize` past depth 3/5?") and the threshold that selects fixpoint-iteration over truncation-reporting, so the branch isn't left fully open.
3. **Name the performance baseline for Objective 7 / Phase 3:** cite the specific benchmark binary/fixture and a current timing (or state that capturing it is the first Phase-4 step), so the "no measurable slowdown" gate is enforceable.
4. **Specify the WASM parity command:** add the exact feature flags / target used for the single-threaded WASM test run so the parity gate is reproducible rather than aspirational.
5. **Add a rollback/landing note for the `RecalcMetrics` wire change:** since the new fields ripple to kernel/TS consumers, note whether the Rust-side additions can land independently (default-valued) ahead of the contracts rollup, to de-risk the cross-repo sequencing the plan already flags.

These are refinements, not corrections — the plan's diagnosis, scoping, and contract discipline are already strong enough to execute against.
