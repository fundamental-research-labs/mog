# Review — Plan 027: Harden the `ParseOutput → WorkbookSnapshot` import lowering pipeline

## Summary judgment

This is an unusually well-grounded plan. Every factual claim I spot-checked against `mog/compute/core/src/import` holds: the `MixedCbor` encoder pushing `0x00` for `Array`/`Control`/`Image` (`classifier.rs:538`), the two hand-rolled tag/discriminant tables that must agree byte-for-byte (encoder `classifier.rs:548-562` vs decoder `mirror/range_view.rs:651-699`, discriminants `0..=10` matching exactly), the duplicated ID-minting (`u128_to_hex32` vs `format!("{:032x}")` in `sheet_lowering.rs:77` / `:109`), the caller-side re-derivation diff (`construction/xlsx.rs` "Pass 3: Collect ranged positions per sheet"), `data_table_lowering.rs:25-26` `unwrap_or_default()`, `table_lowering.rs:40` `SheetId::from_raw(0)`, the pivot re-scan `output.sheets.iter().find(|s| s.name == ...)` (`pivot_lowering.rs:46`), and the `row_ids.iter().position(...)` inner-loop scans plus the "check all sheets" simplification comment in `name_lowering.rs:134,181-182`.

The plan correctly identifies the highest-value problem on this path — a silent, total-corruption risk in a hand-synchronized wire codec, plus actual silent data loss for image/control/array cells — and proposes the right structural fix (single shared codec) with a property-test gate as the headline verification. It is honest about scope boundaries, cross-crate dependencies, and what it is deferring. The objectives are correctness/robustness on the production ingest path, not cosmetic.

It loses a point only because it is a large, multi-crate program of work nominally scoped to one folder, and two of its objectives (the data-loss remediation mechanism and the codec's home) are left as "investigate and decide" rather than resolved — appropriate for a plan, but they are the load-bearing decisions and remain open.

## Major strengths

- **Evidence-first, accurate.** The folder inventory, the pipeline diagram, and the specific line-level claims are all verifiable and correct. This is not a hallucinated plan; it was written against the real code.
- **Correct prioritization.** It leads with the codec drift / silent data-loss pair (objectives 1, 2, 8), which is the genuine production hazard: a one-byte divergence between `classifier.rs` and `range_view.rs` silently corrupts every imported mixed column, and the `Array|Control|Image => push(0x00)` arm silently drops values today. That is the right thing to fix first.
- **Contracts are explicit and testable.** C1–C10 promote currently-implicit invariants (identity-space unity, anchored-never-ranged, lossless promotion, axis bounds, encode/decode tag agreement, caller-diff equivalence) into named, assertable properties. C8 and C6 in particular convert "two computations that happen to agree" into "one source of truth," which is the correct framing.
- **Sequencing and parallelization are real, not decorative.** Step 1 is correctly called out as critical path (and the only step that must edit production code outside the folder). The migration bridge — keep the snapshot-vs-parse-output diff as a `debug_assert_eq!` until the caller change lands — is a genuinely good "no flag day" mechanism. The per-file independence claims (Step 3→classifier, 4→pivot, 5→data_table/table, 6→name) match the actual file boundaries.
- **Honest non-goals and risk register.** Row/2D promotion, `TableSpec.range_ref` typing, enum-variant rename, and upstream sparkline string typing are all explicitly deferred with reasons. The wire-format-stability risk (are these payloads persisted to Yrs verbatim?) is exactly the right question to ask before changing tags, and the `i64::MAX as f64` / `2^53` boundary callout for the round-trip test is precise.

## Major gaps or risks

- **Scope vs. folder boundary.** Despite the "stays inside the public folder" framing, the plan's flagship step necessarily edits `mirror/range_view.rs` and possibly `cell-types`, and Step 2 requires a follow-up in `storage/engine/construction/xlsx.rs`. This is disclosed, but it means item 027 cannot land independently of the `mirror` / `cell-types` / `construction` queue items. The plan should state more firmly that Step 1 is *blocked on* coordinated review by those owners, not merely "should be reviewed by" them.
- **The data-loss fix is under-specified at the decision point.** Objective 2 / Step 1 offers (a) a new faithful tag vs (b) prove-unreachable + `debug_assert!` + per-cell fallback, and leans toward (b) for `Image`. But "keep the cell per-cell instead of dropping its value" inside `flush_run` is a non-trivial control-flow change (the run is already being collapsed), and the plan does not specify how a single value is extracted back out of a promoted run mid-flush. If (b) is chosen, the mechanism needs a concrete sketch; otherwise the implementer may discover the fallback is awkward exactly where the plan assumed it was cheap.
- **Codec home is an unresolved architectural fork.** "`cell-types` vs a dedicated `compute-core` module" is left to be resolved "before Step 1." Since everything downstream depends on it, an unresolved dependency-edge question at the head of the critical path is the plan's single biggest schedule risk. A pre-step to confirm the `cell-types` dependency graph (does it already see `value-types`/`CellValue`?) would de-risk this.
- **Gate 2 ("fails to compile if tags diverge") is aspirational as written.** A shared exhaustive `match CellError` gives compile-time safety for the *error discriminants*, but the value *tag* bytes (`0x00..0x04`) are integer literals — a shared codec makes drift impossible by construction, but no compile error enforces "tag N means variant M." The plan should reframe gate 2 as "single source of truth removes the divergence class" rather than implying the compiler will catch a tag-number swap.
- **Determinism contract (C5) leans on allocator seed.** The `id_map.is_none()` path mints IDs via `STORAGE_ID_ALLOC.next_u128()` (`sheet_lowering.rs:78,109`), which is process-global and not seeded by `ParseOutput`. C5's "identical allocator seed ⇒ byte-identical" is fine for the `id_map.is_some()` production path but the plan should note the fallback path is not byte-deterministic across processes, so the determinism test must pin the `id_map.is_some()` case.

## Contract and verification assessment

The contract set is the strongest part of the plan and is well matched to the verification gates: gate 1 (round-trip property test) backs C3/C8, gate 4 backs C6, gate 5 backs C7, gate 6 backs objective 5, gate 8 backs objective 7's parity requirement, and gate 9 extends `deterministic_output` (which already exists) for C5. The insistence that the optimized name-linkage produce *identical* `linked_range_id` assignments as the current implementation (gate 8) is the correct way to make a performance refactor safe.

Two weaknesses: (1) the gates list a `xlsx-roundtrip` corpus report and the `test_deferred_xlsx_import::range_streaming` integration suite as the end-to-end fidelity check, which is right, but the plan doesn't state a pass criterion (zero-regression vs. a tolerance) — "no regression in imported value fidelity" needs a baseline. (2) The no-silent-loss gate (gate 3) is binary-correct in intent but, as noted, depends on a fix mechanism that isn't fully specified, so the test may be easy to write and hard to make pass.

The migration `debug_assert_eq!` bridge (Step 2 / gate 4) deserves explicit credit: it makes C6 self-enforcing during the transition window rather than relying on the implementer to remember to re-check.

## Concrete changes that would raise the rating

1. **Resolve the codec-home fork before publishing the plan as final**, or add an explicit Step 0 that inspects the `cell-types` dependency graph and records the decision, so Step 1 starts unblocked.
2. **Sketch the (b)-path data-loss fallback concretely** — show how `flush_run` extracts a single `Image`/`Control` value back to a per-cell entry mid-promotion, or commit to path (a) (a faithful tag) and specify the new tag's wire layout. This is the load-bearing correctness change and should not be left as "investigate."
3. **Pin pass criteria for the integration/corpus gates** (e.g. "byte-identical `RangeData` on the `id_map.is_some()` path; zero net fidelity regression on the corpus baseline X").
4. **Reframe gate 2** to claim "single source of truth eliminates the drift class" rather than compile-time enforcement of tag numbers, and add an explicit exhaustive-`CellError`-match assertion as the part that *is* compile-checked.
5. **State the cross-crate landing order as a hard dependency**, not a courtesy review: Step 1 cannot merge without coordinated `mirror`/`cell-types` changes, and Step 2's caller edit must either ride with it or sit behind the `debug_assert_eq!` bridge until the construction-folder item lands.
6. **Note C5's process-global allocator caveat** so the determinism test scopes itself to the production (`id_map.is_some()`) path.
