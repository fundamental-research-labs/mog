# Review of 028 — Consolidate and harden the compute-core identity facade

## Summary judgment

This is a strong, evidence-grounded plan. Every concrete claim it makes about the current state is verifiable in the source: the folder really does contain only `mod.rs`, `grid_index.rs` (a one-line `pub use compute_document::identity::*;` glob), and `tests.rs`; `mod.rs:8` still carries the stale "Rust port of the TypeScript `GridIndex`" framing; the dual-import split is real (33 files reach `crate::identity::GridIndex`, 27 reach `compute_document::identity` directly inside `compute/core/src`); `dimensions/rows.rs:43` does map a `None` row identity to `ComputeError::SheetNotFound`; and `AxisIdentityStore` genuinely splits `Explicit(Vec<Id>)` / `Runs(CompactAxisIdentityStore)` with an `identity_at` that returns `Option`. The plan correctly diagnoses a real, known bug class (resize/format failing with a misleading "sheet not found" on data-dependent compact axes) and proposes a coherent architecture: collapse the facade, unify the import path, introduce a typed error, and push the root-cause invariant fix down to the layer that owns it. The honesty about scope, alternatives, and the central open risk is above average.

It falls short of a 9–10 mainly because the substantive correctness fix is deferred to dependency folders and rests on an invariant the plan itself admits may be false, and because the headline regression test is underspecified at exactly the point that matters (how to obtain a compact-`Runs` fixture).

## Major strengths

- **Accurate, falsifiable problem statement.** The plan does not hand-wave; it names files, counts, and the exact `ok_or_else(SheetNotFound)` site. I confirmed each. This is the difference between a plan grounded in the tree and one written from memory.
- **Correct architectural framing.** It recognizes that the folder is a *facade* over an extracted implementation, and that the bug's home is the boundary type the dimensions code consumes. Choosing to centralize the `Option -> error` decision (`require_row_id`) rather than leave it scattered in `rows.rs`/`cols.rs` is the right instinct.
- **Two-tier error model.** Separating "make in-bounds⇒identity hold" (root cause) from "if it genuinely can't, surface `MissingAxisIdentity` not `SheetNotFound`" (diagnosability) is exactly the right decomposition for a defect whose reproducibility is data-dependent.
- **Test de-duplication is principled.** It correctly identifies that `tests.rs` only ever exercises the `Explicit`/`new` path and duplicates a subset of `compute-document/src/identity/tests/*`, and insists coverage strictly increase before deletion. That guards against the common "delete tests, lose coverage" failure.
- **Sequencing and dependencies are explicit and correct.** Step 1 before Step 2; axis-layer root cause landing with the facade helper so the regression is green; hard dependencies on `cell-types` and `compute-document` named with the actual files. Parallelization notes are realistic (per-sub-area import migration is independent).
- **Disciplined non-goals.** No data-model change, no compat shim, no parallel TS model. This keeps a potentially sprawling refactor bounded.

## Major gaps or risks

- **The central correctness fix is not in this folder.** Step 3's root cause ("hydration of a `Runs` axis yields full coverage for `0..len()`") lives in `cell-types`/`compute-document`. So the *production* value of this plan is gated on sibling plans whose existence and correctness this plan asserts but cannot guarantee. The in-folder deliverable reduces to: facade collapse, consumer import migration, a new error variant, and a regression test. That is real work, but the plan's most important promise (fixing the user-visible bug) is delegated.
- **The load-bearing invariant may be false — and the plan knows it.** Risk bullet 5 concedes that compact persisted axes might be "legitimately sparse," in which case in-bounds⇒identity cannot hold and the fix flips to on-demand materialization. This is the crux of the whole plan, yet it is left unresolved. A plan this thorough should have spent one investigation step deciding whether a `Runs`/compact axis can legitimately have an in-bounds index without an identity (inspect `CompactAxisIdentityStore` segment semantics and the hydration path) before committing to "the invariant holds." As written, the primary fix and the fallback fix are both live, which means the design is not actually settled.
- **Regression-test fixture is underspecified.** Step 5 and the integration gate hinge on "a workbook whose persisted axes hydrate as compact `Runs`," but the plan never says how to construct or obtain one. `GridIndex::new` always yields `Explicit`, so the test author needs a `from_axis_stores`/`from_runs` constructor path or a real imported fixture that triggers compaction. Without naming that mechanism, the regression test — the single most valuable artifact here — risks silently testing the `Explicit` path again and passing vacuously.
- **`MissingAxisIdentity` blast radius is acknowledged but thin.** Adding a `ComputeError` variant touches the WASM/serde boundary and any caller matching `SheetNotFound`. The plan flags this but does not enumerate the serde derivation site or whether the TS error mapping needs a new case. For a variant that crosses the wasm boundary, "ensure it serializes" is a hand-wave where a file pointer is needed.
- **Minor signature inaccuracy.** The plan writes `identity_at(i)`, but the real signature is `identity_at(&self, sheet_id: SheetId, position: u32)`. Harmless conceptually, but the proposed `require_row_id` helper and the invariant assertion `(0..len()).all(|i| identity_at(i).is_some())` will need the sheet id threaded through — worth getting right since it affects the helper's call site.

## Contract and verification assessment

The "preserve" list is genuinely useful: lazy CellIds, dense axis identities, remove-all-then-insert-all shift correctness, the `sort_rows` (fixed RowIds) vs `reorder_row_ids` (permuted) split, and `Arc<IdAllocator>` monotonicity with the virtual-CellId `ensure_past` skip are all real, load-bearing contracts and are stated precisely enough to test against. The "strengthen" list is the right target.

Verification gates are mostly concrete: unit additions placed in the *implementation's* test dir (correct home), a boundary suite for path resolution, an integration regression, and a grep gate asserting zero `compute_document::identity` imports remain under `compute/core/src` outside the facade. The grep gate is the strongest single check — it makes "one canonical path" mechanically enforceable. The weakness, again, is that the most important gate (the compact-axis regression) depends on a fixture the plan doesn't pin down, and the invariant assertion in `invariants.rs` can only be written once the sparse-vs-dense question is answered.

## Concrete changes that would raise the rating

1. **Resolve the invariant question first.** Add an explicit investigation step: inspect `CompactAxisIdentityStore` and the `from_yrs_arrays`/`from_axis_stores` hydration path and state, as a finding, whether an in-bounds index can legitimately lack an identity. Commit to either "invariant holds, assert it" or "sparsity is legal, materialize on demand" — not both. This is the single change that would most increase confidence.
2. **Name the fixture mechanism.** Specify exactly how the regression test forces a `Runs` axis (which constructor, or which imported workbook triggers compaction), and add an assertion that the store under test is actually `Runs` (mirroring the existing `let AxisIdentityStore::Runs(compact) = store else { ... }` pattern already in `axis_store.rs` tests) so the test can't pass vacuously on the `Explicit` path.
3. **Pin the error-variant edit sites.** List the `ComputeError` definition file, its serde derivation, and the TS/wasm error-mapping location, plus the search for existing `SheetNotFound` matchers that must learn about the new variant.
4. **Correct the `identity_at` signature** throughout (thread `SheetId`), so `require_row_id` and the invariant assertion are implementable as written.
5. **Clarify the in-folder vs cross-folder split of value.** State plainly that, absent the `cell-types`/`compute-document` root-cause fix, this plan delivers a boundary cleanup + diagnosable error but not the bug fix — so reviewers can sequence the sibling plans accordingly rather than expecting this one to close the defect alone.
