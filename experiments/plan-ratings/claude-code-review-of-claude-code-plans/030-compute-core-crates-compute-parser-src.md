# Review of Plan 030 — `mog/compute/core/crates/compute-parser/src`

## Summary judgment

This is a strong, production-path plan grounded in genuine source inspection rather than speculation. Almost every claim it makes is verifiable at the cited `file:line`, the objectives target real latent defects (not cosmetic churn), the sequencing is deliberate and risk-aware, and the contract/invariant section is unusually disciplined. It reads like it was written by someone who actually traced the winnow grammar, the `last_error_kind` channel, and the `identity_transform` sentinel.

The two things that hold it back from a 9: (1) the headline evidence for Objective 1 — "zero producers" for a list of variants — is factually wrong for three of the nine variants, and that inaccuracy directly mis-scopes which variants are safe to delete; and (2) the consumer-contract risk framing rests on a premise (`compute-cf`/`compute-table` branch on `ParseErrorKind`) that I could not substantiate. Neither is fatal, but the first is the kind of error that, taken at face value, leads an implementer into a correctness trap the plan itself warns about elsewhere.

## Major strengths

- **Accurate, citable diagnosis.** I verified the load-bearing claims directly:
  - `parser.rs` error paths return `Span::new(pos, input.len())` with `pos = input.len() - remaining.len()` (input-relative), while the success path uses `state.offset(remaining)` (body-relative, origin 0). The mixed-coordinate / hardcoded-end defect in Objective 2 is real (parser.rs:263–294).
  - The `last_error_kind` channel is exactly as described: `pratt::stash_if_empty` is first-write-wins (pratt.rs:19–24), while `alpha/args/atom/arrays/postfix/range_ops` call `.last_error_kind.set(Some(..))` directly (last-write-wins). Objective 3's "two competing disciplines" framing is correct.
  - `references.rs` rejects out-of-bounds col/row with bare `backtrack()` and carries the literal comments the plan quotes ("`InvalidColumnNumber` exists but `backtrack()` is correct here", refs at lines 64, 80, 136, 154). Objective 3's "cut only after the parse has committed" is the right nuance, and the plan even captures the `try_parse_row_range` post-`:` commit subtlety.
  - `try_parse_structured_ref` discards the inner parser's `MalformedStructuredRef` via `let Ok(..) = parse_structured_ref(..) else { return Err(backtrack()) }` (references.rs:561), and correctly distinguishes the `find_outer_matching_bracket` → `None` "not a structured ref" case (references.rs:550). Objective 4 is precise.
  - The `CURRENT_SHEET = SheetId::from_raw(0)` sentinel and its standing TODO exist (identity_transform/refs.rs:25), and `CellRef::Positional { sheet: SheetId, .. }` in `formula-types/src/refs.rs:25` confirms Objective 6's cross-crate blast radius is real.
  - `reference_tokens.rs` `.expect(...)` panics at lines 116, 119, 133 feeding editor highlighting from arbitrary user text — Objective 7 is justified.
- **Excellent sequencing and risk isolation.** The one high-blast-radius, cross-crate change (Objective 6, `Option<SheetId>`) is explicitly deferred to last, framed as a standalone PR, and given a concrete fallback (debug-assert tripwire + newtype) if the ripple proves too large. Steps 1–5/7/8 are correctly identified as confined to the crate and parallelizable.
- **Disciplined contract section.** It distinguishes "preserve" from "strengthen" per invariant, explicitly states the accepted language must not change (diagnostics-only), and ties each objective back to a named contract (`parse_formula` totality, `ParsedExpr::classify` totality, depth bound, lexer fast paths).
- **Honest about the constraint.** It states up front that it cannot run cargo/test and therefore specifies gates rather than executing them, including a benchmark before/after on the named hot-path cases.

## Major gaps or risks

- **Objective 1's central evidence claim is overstated and partly wrong.** The plan asserts "a grep across non-test, non-`parser.rs` source shows zero producers for `InvalidRowNumber`, `InvalidColumnNumber`, `InvalidCellReference`, `UnknownSheetName`, `MalformedNumber`, `MalformedArrayLiteral`, `InvalidReference`, `ExpectedClosingBracket`, `ExpectedClosingParen`." In fact `identity_transform/refs.rs` produces `InvalidRowNumber` (lines 140, 146), `InvalidColumnNumber` (lines 171, 177), and `InvalidReference` (line 65), and `identity_transform/external.rs:29` produces `InvalidReference`. These live on the A1→`IdentityFormula` conversion path, which is in-scope, non-test production code. The genuinely-dead-everywhere set is narrower: `InvalidCellReference`, `UnknownSheetName`, `MalformedNumber`, `MalformedArrayLiteral`, `ExpectedClosingBracket`, `ExpectedClosingParen`. The underlying observation (the *winnow classification path* in `references.rs` backtracks instead of producing precise kinds) is correct — but the plan conflates "not produced by the main parse path" with "zero producers anywhere," and that conflation matters: an implementer who deletes `InvalidRowNumber`/`InvalidColumnNumber`/`InvalidReference` as "dead variants" (Objectives 1/8) would break the identity-transform path. The plan needs to reconcile these two producers explicitly.
- **The consumer-contract premise looks unsubstantiated.** The plan repeatedly leans on `compute-cf`/`compute-table`/`compute/core` matching on `ParseErrorKind` (Risk 1, the "consumer compile gate," and the `non_exhaustive` audit). A workspace-wide search for `ParseErrorKind` outside `compute-parser` returned no matches. If no external code branches on the kind, then Risk 1 ("some consumer may branch on the generic `UnexpectedToken`") is largely hypothetical and the variant-removal audit is cheap rather than load-bearing. This is over-caution (safe) but it weakens the plan's credibility on exactly the contract it foregrounds; it should either cite the actual matcher sites or downgrade the risk.
- **`Empty` and `TrailingInput` span origins are left unaddressed.** Objective 2 fixes the main `Err(_e)` arm, but the early `Empty` returns use `Span::new(0, ..)` / `Span::new(1, ..)` and `TrailingInput` uses the same input-relative `pos`. If the goal is one coherent coordinate system, these paths should be named so they aren't left inconsistent after the fix.
- **Unmeasurable verification.** By constraint the gates (clippy, criterion, consumer compile) are advisory. The "hot path must not regress" assertion is reasonable (changes touch failure paths) but unproven; the plan correctly flags this, so it's a noted limitation rather than a defect.

## Contract and verification assessment

The contract surface is the best part of the plan. It correctly identifies the two totality guarantees that the error-precision work must not break (`parse_formula` never panics; `ParsedExpr::classify` never escapes to `Unparseable`), and it ties Objective 7 directly to the first. The span coordinate contract is stated precisely and matches the code's documented "byte offsets relative to the formula body after `=`."

Verification gates are comprehensive and well-matched to the objectives: per-objective regression tests with concrete inputs (`=A1048577` → `InvalidRowNumber`, `=XFE1` → `InvalidColumnNumber`, unterminated `=Table1[[#Data]` → `MalformedStructuredRef`), determinism test for the channel change, proptest for structured-ref agreement and UTF-16 totality, round-trip preservation for Objective 6, and a benchmark gate. The gaps: the consumer compile gate rests on the unverified matcher premise above, and the test for "a real `SheetId(0)` is treated as cross-sheet" (Step 6) is exactly the right test but presumes the fixture can even construct `SheetId::from_raw(0)` — worth confirming that constructor is reachable in test code.

## Concrete changes that would raise the rating

1. **Correct the producer audit.** Replace the blanket "zero producers" sentence with the verified split: variants live only in the `identity_transform` path (`InvalidRowNumber`, `InvalidColumnNumber`, `InvalidReference`) vs. variants dead everywhere (`InvalidCellReference`, `UnknownSheetName`, `MalformedNumber`, `MalformedArrayLiteral`, `ExpectedClosingBracket`, `ExpectedClosingParen`). Explicitly exclude the identity-transform-live variants from the Objective 8 deletion candidates, or state that Objective 3 should make `references.rs` produce the *same* kinds `identity_transform` already emits (which would actually unify the taxonomy rather than just trimming it — a stronger outcome).
2. **Substantiate or soften the consumer-contract claim.** Either cite the specific `ParseErrorKind`/`parse_structured_ref` match sites in `compute-cf`/`compute-table`/`compute/core`, or note that no external matchers were found and downgrade Risk 1 and the consumer compile gate accordingly. (`parse_structured_ref` *is* used by `compute-table` tests and `bridge_pure`; `ParseErrorKind` appears unused externally — say so.)
3. **Name the `Empty`/`TrailingInput` paths in Objective 2** so the coordinate-system fix is total across all error exits, not just the main `Err` arm.
4. **Add a one-line note that the Step 6 fixture can construct `SheetId::from_raw(0)`** (or whichever constructor) so the sentinel-regression test is known-feasible.
5. **State the intended end-state for the identity-transform error kinds** under the new `Option<SheetId>` representation, since Objective 6 touches `identity_transform/refs.rs` — the same file that produces the bounds kinds — and the two objectives overlap there.
