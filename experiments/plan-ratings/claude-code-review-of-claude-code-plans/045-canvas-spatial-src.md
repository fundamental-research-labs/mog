# Review of plan 045 — `mog/canvas/spatial/src` (grid spatial index + hit-test pipeline)

## Summary judgment

This is a strong, evidence-grounded plan that I was able to verify almost line-for-line against the live source. Every concrete claim it makes about the current code checks out: `packCellKey` is exactly `((cellX & 0xffff) << 16) | (cellY & 0xffff)` at `grid-index.ts:58-60` with the `[-32768, 32767]` doc claim at `:56`; `MAX_CELLS = 1000` (`:33`) and `MAX_QUERY_CELLS = 10_000` (`:39`); the dead-threshold bug is real (a 1001–10000-cell query passes `!isQueryOversized` at `:255` but `getCellsForBounds` returns `null` at `:131` for >1000 cells, dropping into the full-scan branch at `:266-273` with the misleading "Item bounds itself is oversized" comment at `:267`); the nondeterministic tie-break is real (`pipeline.ts:20` `getZIndex(b) - getZIndex(a)` over `Set`-ordered candidates from `queryPoint` at `grid-index.ts:309`); the NaN-ghost path, the per-query `number[]`/`Set` allocations (`:135`, `:253`, `:261-263`), and the unconditional `oversized` scans (`:292-296`, `:318-322`) are all accurately characterized. The architecture-doc layer reference (Layer 2, `mog/docs/architecture/README.md:347`/`:353`) and the `import-boundaries.cjs` `hardware`-layer pin for `@mog/spatial` (line 101) are both correct, as is the `{@mog-sdk/contracts}`-only dependency set in `package.json`. Test files live in `__tests__` (three files) as stated, and the `jest` + `check-types` (`tsc --noEmit`) gates exist as scripts.

The plan diagnoses genuine, non-obvious weaknesses (the silent broad-phase *selectivity collapse* framing is the standout — it correctly distinguishes "wrong output" from "degenerates to O(n)"), proposes a clean fix per weakness, preserves the stable `SpatialIndex<T>` contract, and sequences the work into landable PRs with explicit parity-test safety nets. It is a near-ideal harden-without-rewrite plan.

## Major strengths

- **Verifiable, precise evidence.** Line citations are accurate and the causal chains (e.g. NaN → `cellCount = NaN` → `isOversized = false` → empty loops → ghost entry) are correct. This is the difference between a plan that sounds plausible and one a reviewer can trust.
- **Correct root-cause separation.** It treats Step 1 (key collision) as a *performance/selectivity* fix, not a correctness fix, because the final `boxesOverlap`/`boundsContainPoint` filter guarantees output correctness regardless of key scheme. That nuance drives the right verification strategy (parity, not output-change).
- **Strong contract discipline.** No `SpatialIndex<T>` signature changes; new capabilities (`stats()`, `forEachCell`) kept off the interface and on the concrete class; purity and the `hardware`-layer / single-dependency invariants explicitly preserved. The `seq`-on-`SpatialEntry` risk is flagged as a *public type change* with a named fallback (keep `seq` internal, pass index into `hitTestPipeline`).
- **Sound key-scheme math.** `S = 2**26`, bias `B = S/2`, key `= (cellX+B)*S + (cellY+B)` keeps the product under `2**52`, is a true bijection in range, and stays an integer `Map` key (no string allocation). Alternatives (nested maps, string keys) are considered and rejected with hot-path reasoning.
- **Verification gates are concrete and consumer-aware.** It names the exact consumer test suites and typechecks (drawing-canvas hit-map, ink spatial-index + diagnostics, charts hit-tester, kernel) and ties them to the `seq` risk. Sequencing (core PR → pipeline determinism PR → consumer gate → docs follow-up) is explicit and correct.

## Major gaps or risks

- **Input-validation policy is the weakest specification (Step 4).** "Reject non-finite bounds (throw in dev, and in production route to `oversized`)" leaves the *mechanism* for distinguishing dev from production unspecified in a package that is deliberately pure with no env access, no logger, and no build-time `NODE_ENV` convention established in the plan. A spatial index cannot tell "dev" from "prod" without importing something — which collides with the stated purity/zero-dependency invariant. The plan should commit to a single deterministic policy (most defensibly: normalize/clamp-or-route-to-`oversized` always, so behavior is identical everywhere and no env branch is needed), or specify exactly how the dev throw is gated. As written, an implementer has to invent this.
- **Step 1's out-of-range guard interacts subtly with the count-based oversized path, and the plan under-specifies it.** Today `isOversized` is purely cell-*count* based. A tiny box at coordinate ~2e9 spans one cell but is out of the new `[-B, B)` range; the plan's "route to `oversized`" guard must therefore live inside the cell enumerator/`forEachCell`, not in `isOversized`, and `getCellsForBounds`/`forEachCell` must return the "oversized/null" signal for *out-of-range* as well as *too-many-cells*. The plan gestures at this ("route to oversized rather than aliasing") but does not state that `remove`/`updateBounds` must consistently treat such items as oversized on the way out (they happen to, via `oversized.has(id)` checks at `:189`/`:214`, but the plan doesn't note this is what keeps removal correct).
- **`stats().lastQueryFullScan` adds query-time mutable state to an otherwise stateless query.** It is a minor wrinkle, but the plan presents `query` as pure-output; recording a per-call flag is a side effect (fine for a diagnostic, but worth a sentence on thread/worker reuse semantics, since the package is explicitly designed to run in workers).
- **`seq` overflow / monotonic counter lifetime is unaddressed.** A `number` counter incremented per insert is safe for any realistic session, but the plan adds a "fresh `seq` on re-insert" rule without noting the counter never resets across `clear()` (or whether it should). Trivial, but it's the kind of thing the determinism tests should pin.
- **No explicit micro-benchmark or before/after perf assertion.** The plan's headline justifications (allocation reduction, mid-size query speedup) are perf claims, yet the gates are correctness/typecheck only. The plan even says the agent won't run benchmarks — reasonable, but it leaves the perf objectives (Objectives 2 and 5) without any falsifiable gate beyond "parity tests stay green," which proves correctness, not speed.

## Contract and verification assessment

Contract clarity is excellent. The `SpatialIndex<T>` stability invariant, the exact result-set semantics (`query` = inclusive-edge intersection matching `boxesOverlap`; `queryPoint` = inclusive containment), idempotent re-index, and purity/layer/dependency invariants are all stated precisely and match the code. The two *new* invariants — total hit order (z desc, then `seq` desc = "drawn last wins") and the documented finite coordinate domain — are well-formed and testable. Candidate-order-is-unspecified-except-`findNearby` is a correct and honest statement of current behavior.

Verification gates are above average for this experiment: real commands (`pnpm --filter @mog/spatial test`, `check-types`, `tsc -b` for the `composite`/`declaration` build), named regression tests mapped one-to-one to each fix (collision regression, enumeration-vs-full-scan with `stats().lastQueryFullScan`, deterministic tie-break with an unrelated-insert reshuffle, invalid-input policy, parity), and a consumer typecheck/test matrix. The gaps are: (1) the input-policy test can't be written until the dev/prod ambiguity is resolved; (2) no perf gate for the perf-motivated steps; (3) the app-eval integration check is correctly scoped as optional/owner-run, which is the right call but means end-to-end determinism is not gated here.

## Concrete changes that would raise the rating

1. **Resolve the dev/prod input-validation ambiguity (Step 4).** Pick one deterministic, env-free policy for non-finite bounds (recommended: always route to `oversized` so the item is discoverable on every path, eliminating the ghost; optionally `console.warn`-free) and one for negative width/height (normalize by swapping, or reject in the factory's documented domain). State it as a single rule so the Step-4 test is writable. This is the only change that meaningfully blocks an implementer.
2. **Make the out-of-range routing explicit in the enumerator contract.** Specify that `forEachCell`/`getCellsForBounds` returns the oversized signal for *either* >`maxCells` *or* any cell coordinate outside `[-B, B)`, and add one sentence noting `remove`/`updateBounds` stay correct because they branch on `oversized.has(id)` first.
3. **Add a falsifiable perf check for Objectives 2 and 5**, even a coarse one: e.g. a test asserting a 3600-cell query touches only the enumerated cells (`stats().lastQueryFullScan === false`) — already present — plus an allocation-sensitive assertion or a documented micro-benchmark the owner runs, so the perf claims aren't gated solely by correctness parity.
4. **Pin `seq` semantics across `clear()`** in the plan text and in the tie-break test (does the counter reset on `clear()`? state it), and note the `number`-counter safe-integer ceiling as a non-issue explicitly.
5. **One sentence on `stats()` statefulness** in worker/reuse contexts, since the package advertises worker-safety and `lastQueryFullScan` is per-call mutable state.

None of these are structural; the plan is fundamentally correct and well-sequenced. Resolving item 1 (and to a lesser extent 2) is what separates this from a 10 — as written, the input-validation step hands the implementer an unresolved design decision inside an otherwise fully-specified plan.
