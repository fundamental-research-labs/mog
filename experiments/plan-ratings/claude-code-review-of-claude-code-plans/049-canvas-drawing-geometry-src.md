# Review of Plan 049 — Harden `mog/canvas/drawing/geometry/src`

## Summary judgment

This is an unusually strong hardening plan. It is built on concrete, verifiable evidence rather than speculation: every file:line citation I spot-checked against the live source is accurate. The plan correctly identifies that this package is the dependency-free math kernel beneath on-screen hit-testing, kernel computation, and PDF export, and it reasons correctly that the high blast radius (a defect causes *cross-surface divergence* between screen and export) is the justification for the work. The objectives are ordered de-risking-first, each step is independently shippable and test-gated, and the plan is scrupulous about preserving observable contracts (export surface, matrix/angle conventions, diagnostics codes) while strengthening internal invariants. The few weaknesses are at the margins — the perf/allocation "contract" is the least concrete part, and a couple of objectives defer real decisions rather than make them — but these are honestly flagged, not hidden.

I verified the central claims:
- `path.ts:408–445` — the `A`/`a` arc branches consume 7 params and emit `segments.push({ type: 'L', x, y })` with the exact "not yet implemented; emit a lineTo" comment. **Confirmed.**
- `path.ts:87` — tokenizer regex `([MLCQZHVSTAmlcqzhvsta])|(-?\d*\.?\d+(?:[eE][+-]?\d+)?)`, which cannot split packed flag fields. **Confirmed.**
- `path.ts:458–475` — `pathToSvgString` switch has cases `M/L/C/Q/Z` and **no `default`/`never` guard**; ignores `closed`/`subPaths`. **Confirmed.**
- `path.ts:665–682` — `pointAtLength` runs a 30-iteration binary search, each iteration re-sampling the cubic from t=0 in 20 sub-steps, after a separate `cubicLength` call. **Confirmed.**
- `bounded-cache.ts:18–26` — `get` bumps recency only when `value !== undefined`, conflating absent vs present-undefined. **Confirmed.**
- `matrix.ts:55` uses `1e-12` for singular; `diagnostics/validators.ts:150` independently uses `1e-12` (and `1e6`/`1e8` extremes). Two unlinked literals. **Confirmed.**
- `point.ts:151–162` — `reflect` documents the unit-normal precondition only in a comment, no enforcement. **Confirmed.**
- `connector-routing.ts:211–315` — `routeBend` returns 3 or 4 points, `routeCurve`/`routeLongCurve` return 4 Bézier control points, and `routeConnector` flattens all into a bare `Point2D[]`. **Confirmed.**
- `custom-geometry.ts:35,264,411` — the `arcTo`→bezier conversion does exist in the shapes package, so the plan's "one implementation, not two" framing is real. **Confirmed.**
- Contracts shim (`contracts/src/viewport/geometry.ts`) re-exports from `@mog/types-viewport/geometry`; `connector-rerouting.ts` exists; 73 files import `@mog/geometry`. **Confirmed.**

The plan's evidentiary discipline is the main reason for the high score.

## Major strengths

1. **Evidence quality.** Problems are stated with file:line anchors that hold up under inspection, and several are linked to the consumers that make them dangerous (text-effects calling `pointAtLength` per glyph; PDF export reaching persisted output). This converts abstract "tech debt" into ranked, justified risk.
2. **Correct blast-radius reasoning.** Identifying that arc stubbing silently corrupts not just rendering but `pathLength`/`pathBoundingBox`/hit-testing/PDF export — with no error signal — is the highest-value finding, and the plan rightly makes it the largest objective.
3. **Contract-preservation discipline.** The "Preserve vs Strengthen" split is excellent: export names/signatures, matrix layout, the `pointInArc` angle convention (0 at 12 o'clock, clockwise — chart picking depends on it), and diagnostics codes are all explicitly fenced as must-not-regress.
4. **Sequencing.** Steps 1–3 (epsilon, BoundedCache, exhaustiveness) are genuinely independent and low-risk; 5 correctly builds on the flattening introduced in 4; 6/7 are flagged as coordinated cross-folder edits. The dependency graph reasoning ("package sits at the bottom of the drawing graph, re-test consumers regardless") is sound.
5. **It resists silent behavior change.** The boundary-semantics objective (7) is deliberately scoped to *document and dedup*, not change picking results, with an explicit call-out that any actual convention flip is a separately-reviewed product decision with charts/engine sign-off. That restraint is the right call.
6. **Data-model decision is surfaced, not buried.** Step 4 explicitly weighs "emit cubic `C` segments (no contract change)" vs "add a first-class arc `PathSegment`" and recommends the minimal-blast-radius option while naming the cost of the alternative (every `switch (seg.type)` across the monorepo). That is exactly the right framing.

## Major gaps or risks

1. **The perf/allocation "contract" is the softest objective.** Objective 9 and step 9 ask for a "hot functions / allocation budget" section and a CI-gated micro-benchmark, but the plan never proposes concrete numbers (target ns/op, allocation counts, regression threshold percentage). "Add a micro-benchmark as a non-blocking informational gate first, then a regression threshold" is honest but defers the hard part. Without a chosen threshold this risks becoming a perpetually-informational gate that catches nothing. The plan would be stronger if it at least specified *how* the threshold is derived (e.g. baseline ± X% on the named functions).
2. **`pointAtLength` accuracy bound is asserted but not quantified.** Objective 3 and the verification table promise an "error bound vs sample count," but no target tolerance is named, and the LUT-vs-old-loop comparison strategy on non-arc paths ("equivalence with pre-change values") is in tension with the acknowledged numerical drift risk. The plan needs to say whether non-arc paths must be bit-stable or within ε — it currently implies both.
3. **Cross-folder coordination for arc-sharing is under-specified as a deliverable.** Step 4 says to "factor it so it can be shared with / replace the duplicate in `custom-geometry.ts`," but `custom-geometry.ts`'s arc is an OOXML *visual-angle* `arcTo` (rx/ry/startAngle/sweepAngle), whereas SVG path arcs use *endpoint parameterization* (rx, ry, x-rotation, large-arc, sweep, endpoint). These are different input parameterizations of the same underlying ellipse-to-bezier math. The plan treats "share the implementation" as straightforward, but the shared core is the *center-parameterization → ≤90° cubic* step, not the public signature. The plan should name that the unifiable kernel is the center-form sweep, with two thin adapters, otherwise "do not silently leave two copies" may be impractical and the step stalls.
4. **No rollout/flagging story for the arc behavior change.** The plan correctly flags that golden-image/PDF snapshots will diff, but stops at "review every snapshot diff." For a change that alters *persisted/exported* geometry, it would be safer to consider whether the cubic-emission can be validated against an independent oracle (analytic ellipse sampling) beyond just re-baselining snapshots — the verification mentions "bounding box vs analytic ellipse," which is good, but tangent/endpoint continuity against the spec's reference would harden it further.
5. **Tokenizer fix is described but not designed.** Problem 2 is correctly diagnosed, but step 4's "parse arc parameters field-by-field with single-digit flag handling" is the only design detail. Given that this is the subtlest correctness trap (packed flags like `0150`), a concrete sketch of the arc-parameter scanner (consume rx, ry, rotation as numbers; then exactly one char each for the two flags; then x, y) would reduce implementation ambiguity.

## Contract and verification assessment

**Contract clarity: strong.** The discriminated-result type for `routeConnector` (`{ kind: 'polyline'; points } | { kind: 'cubic'; start; cp1; cp2; end }`) is the single biggest API-clarity win and is precisely specified, with a sensible backward-compat plan (keep lower-level helpers as raw-point or adapter forms, migrate the one consumer in a paired edit). The shared `epsilon.ts` with named constants, with `matrix.invert` and `validateTransform` importing the *same* `SINGULAR_DET_EPS`, closes a real latent drift bug and is well-scoped. The `never`-exhaustiveness guard on `pathToSvgString` is correct and future-proofs the serializer against any new `PathSegment`.

**Verification gates: good, with the noted softness on perf.** The plan reuses the existing per-file Jest suites (which I confirmed exist for all 11 source files plus `__snapshots__`) and extends rather than replaces them. The proposed invariant tests are the right ones: `compose(decompose(m)) ≈ m` across reflection/composite matrices, `parse∘serialize` stability, arc endpoint/tangent continuity, `invert`/`validateTransform` agreement at the shared threshold, and no-input-mutation determinism checks. The cross-consumer regression list (charts pick, engine spatial-query, text-effects, shapes presets, file-io PDF) is accurate and matches the real importer set. The gate commands are correct for the package layout, and the plan correctly notes the contracts declaration-rollup build dependency *only if* the arc-segment alternative is taken. The one weakness is that "snapshot diffs are the evidence the fix works" is a qualitative gate; pairing it with an analytic oracle would make the arc verification objective rather than reviewer-judgment-based.

## Concrete changes that would raise the rating

1. **Quantify the perf gate.** Name the hot-function baseline methodology and a concrete regression threshold (e.g. "fail CI if any benched function regresses >15% vs the committed baseline"), and state whether the bench runs in the existing Jest harness or a separate runner. This is the single change that would most move the plan toward 10.
2. **Pin the `pointAtLength` numerical policy.** State explicitly: non-arc paths must match old output within ε = <value> (or be bit-identical), and define the LUT accuracy bound as a function of sample count, so reviewers have a pass/fail criterion rather than "looks close."
3. **Redesign step 4's sharing as "unify the center-parameterization kernel."** Clarify that the shared code is the ellipse-center-form → ≤90° cubic split, with two adapters (SVG endpoint-form and OOXML visual-angle-form), rather than implying a single public function serves both call sites.
4. **Sketch the arc tokenizer scanner.** Add the field-by-field consumption order (3 numbers, 2 single-char flags, 2 numbers, repeat) so the subtle packed-flag fix is unambiguous.
5. **Add an independent arc oracle to verification.** Beyond snapshot re-baselining, assert sampled points of the emitted cubics lie within ε of analytic ellipse points and that endpoint tangents match the spec, since this change reaches exported/persisted output.
6. **Name an owner/sign-off step for the export-affecting change.** The plan mentions "confirm with charts/shapes/export owners" — make that a gating checklist item, not prose, given the persisted-output blast radius.
