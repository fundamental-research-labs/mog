# Review of 059 — Improve `mog/runtime/sdk/src`

## Summary judgment

This is a strong, production-grounded plan. Its central virtue is that nearly every claim is verifiable in the tree, and I confirmed the load-bearing ones directly: the fire-and-forget `wb.dispose = () => { ...; void readyHandle.dispose().catch(...) }` against the awaited `close`/`[Symbol.asyncDispose]` (`boot.ts:334–354`); the two `as any` casts at `boot.ts:536` and `boot.ts:557` on the `createHeadlessDocument`/`importHeadlessDocumentFromXlsx` calls; `CollaborativeEngine` building on the deprecated `createHeadlessEngine` path (`collaborative-engine.ts:17,281,289,354`) including the load-bearing `from_yrs_state` "same Yrs items" rationale (`:272–274`); the single `registerExternalWorkbookSession` call site inside `HeadlessEngine.initWorkbook` (`boot.ts:691`) and never from `createWorkbook`; the eager `import rawApiSpec from './generated/api-spec.json'` plus `apiSpec ... as unknown as ApiSpec` (`api-describe.ts:1,40`) re-exported from the barrel (`index.ts:174`); the 4.95 MB git-tracked spec; the duplicated `interface MogSdkLogger` (`boot.ts:197`, `node-headless-host.ts:120`); the `Array.from(bytes)` fingerprint round-trip (`node-headless-host.ts:227–231`); the `JSON.parse`-or-object probe and `catch (e: any)` in the coordinator wrapper; and the absence of any schema/validate path in `api-describe.ts`. The five named `__tests__` files all exist. The plan reads like it was written from the code, not from a description of it.

The work is correctly framed as contract/safety/cost hardening rather than feature work, sequenced low-risk-first with the high-ripple path consolidation last, and it respects the run's editing constraints (src-only, no config/doc/generated edits, no build commands). The weaknesses are concentrated in the two most ambitious steps (4 and 6), which defer or under-specify their hardest technical decision.

## Major strengths

- **Evidence is real and precisely located.** Line references are accurate to within an off-by-one, and the most consequential claims (dispose asymmetry, the `as any` seam, the deprecated-path-as-collaboration-substrate inversion, the eager 5 MB import) are all confirmed. This is the difference between a plan that hardens the actual contract and one that hardens an imagined one.
- **The invariant section is the best part.** "Production-path contracts and invariants to preserve or strengthen" pins the frozen `createWorkbook` overload set, fail-closed creation, import durability, trust-profile constants, single-site brand cast, introspection output stability, and TZ defaulting. This turns otherwise-risky refactors into bounded ones by naming what must *not* change.
- **Sequencing and parallelization are explicit and correct.** Step 1 before Step 6 (consolidated factory inherits a correct dispose), Step 5 before Step 6 (typed coordinator before rerouting collaboration), and the disjoint-file parallel set (2/3/4/7) are all defensible and matched to the actual file boundaries.
- **Cross-folder coordination is named, not hand-waved.** The contracts `Workbook.dispose` signature (Step 1), the native NAPI coordinator binding as the authoritative source for `CoordinatorAddon` (Step 5), and the `@mog/kernel-host-internal` option shapes (Step 6) are each flagged as owned elsewhere — the plan refuses to "fix" them with a cast.
- **Security-adjacent change handled with care.** Step 3 correctly insists the fingerprint issuer and resolver change atomically, notes the failure is closed (import rejected) not open, and pairs it with a round-trip + tamper-rejection test.

## Major gaps or risks

- **Step 6 defers its core decision.** The consolidation offers "(a) rebuild through `createNodeHeadlessHost` + typed factories" or "(b) retain as a thin typed wrapper," but never commits or gives criteria for choosing. This is the highest-ripple, highest-value change and the plan leaves its architecture as a fork. A plan can legitimately defer this pending the kernel-host signature review, but it should at least state the decision rule (e.g. "prefer (a) unless CRDT/Yrs-state sharing semantics can't be preserved through the host boundary, in which case (b)"). As written, the reader cannot tell what "done" looks like for the most important step.
- **Step 4's lazy-loading mechanism is under-specified against module-format reality.** `import rawApiSpec from './generated/api-spec.json'` is a *static* binding; making it lazy means a dynamic `import()`/`require()` whose ergonomics differ across the package's ESM and CJS outputs (tsup dual build). The plan asserts tree-shaking will improve and that returned shapes stay identical, but it does not address that a static JSON import generally cannot be made conditionally lazy without switching to dynamic import, nor whether `apiSpec` (a current *eager const* export) can remain a plain export once the source is lazy. The verification ("tree-shaking confirmed") is deferred to implementation, so the plan never demonstrates the win is achievable as described — only that it's desirable.
- **Step 4's schema validation has an unweighed cost tension.** Validating the 5 MB JSON against `api-spec.schema.json` at the load seam pulls a validator (e.g. ajv) and the schema into the runtime graph — partly working against the bundle-cost objective the same step pursues. The "or a generated runtime type guard" alternative is the better answer but is offered as an aside; the plan should pick, and should clarify whether validation runs at publish/CI time (cheap, drift caught before ship) vs every consumer load (recurring cost).
- **The "zero `as any` in `boot.ts`" grep gate is blunt.** It's a fine smoke check, but it can be satisfied by relocating the cast rather than eliminating the unsafety. The real gate is that the kernel-host option object *type-checks* against the imported signature — the plan says this elsewhere but encodes the weaker grep as the verification.
- **Evidence #11 (README/llms.txt drift) is acknowledged but creates a slightly awkward scope seam.** The plan correctly excludes those files from the src-only edit set, but #11 is the one finding with no actionable step in this plan at all — it's logged for a different owner. That's honest, but it means a reader should not expect the doc drift to be fixed by executing this plan.

## Contract and verification assessment

Contract clarity is high. The plan distinguishes stable / experimental / deprecated surface explicitly (Step 7), names the brand-cast confinement and fail-closed guarantees as hard invariants, and treats the `MogDisposable`/`MogAsyncDisposable` contract as the standard `dispose()` must meet. The Step 1 honesty about a possible `Workbook.dispose(): void → Promise<void>` contracts change — flagged as cross-folder and possibly breaking, with an explicit "do not work around it with a cast" — is exactly the right posture.

Verification gates are concrete and mostly well-matched: the named existing tests map to the steps they guard (`sdk-logging` → Step 2, chart-export tests → Step 3 neighborhood, `kernel-boundary` → trust seam for Steps 3/6), and the new tests target the actual risk for each step (dispose-awaits-teardown spy, fingerprint round-trip + tamper rejection, byte-identical introspection output + lazy-load probe + corrupted-spec fixture, coordinator round-trip + missing-method typed failure, colab-eval for Step 6). The static gates (typecheck, api-report review, verify-build/smoke/publish) are appropriate for an external package surface. The main verification weakness is that the two claims most central to the plan's payoff — Step 4's tree-shaking win and Step 6's CRDT-semantics preservation — are both deferred to "the implementing change" with the plan providing the test *intent* but not evidence the goal is reachable as specified.

## Concrete changes that would raise the rating

1. **Commit Step 6 to a decision rule.** State when (a) full host-backed rebuild applies vs (b) thin typed wrapper, keyed on whether Yrs-state/SheetId/history sharing survives the host boundary. Define the done-state for the deprecated path (retired vs documented substrate) as an output of that rule, not an open question.
2. **Resolve Step 4's lazy mechanism against the dual build.** Specify dynamic `import()` vs `require()` per output format, confirm `apiSpec` can stay an export under lazy sourcing (or change it to a getter), and state the fallback if the static JSON cannot be made lazy without a dynamic import boundary.
3. **Pick the spec-validation strategy and its timing.** Choose generated type guard vs schema-validator, and make validation a publish/CI-time gate (or a dev-only assertion) so it doesn't reintroduce the per-consumer cost Step 4 is removing.
4. **Strengthen the Step 6 verification beyond the grep.** Replace/augment "zero `as any` in boot.ts" with an explicit assignability assertion that the option object `satisfies` the imported `@mog/kernel-host-internal` parameter type.
5. **Quantify the cost claim, even roughly.** A measured "import of `@mog-sdk/node` parses ~5 MB / adds N ms today" baseline (read-only, no build needed to estimate) would let the plan show, not just assert, the startup win — and would calibrate how much validator cost is acceptable in Step 4.
6. **Note the Step 3 fingerprint must also fold `sizeBytes` and reject length-extension ambiguity.** The plan keeps `sizeBytes` in the identity but should state the digest binds it (e.g. `.update(bytes).update(sizeBytes)`) so two different buffers can't collide on the cheaper path.
