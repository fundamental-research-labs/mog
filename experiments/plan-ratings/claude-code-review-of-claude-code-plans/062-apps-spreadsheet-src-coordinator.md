Rating: 8/10

# Review of plan 062 — `mog/apps/spreadsheet/src/coordinator`

## Summary judgment

This is a strong, evidence-dense refactoring plan for the spreadsheet composition root. Its central claims are not hand-waved — I verified the most load-bearing ones directly against the tree and they hold up: the 1,173-line `sheet-coordinator.ts` whose own header still claims "~250-line pure composition root" (`sheet-coordinator.ts:3-4`), the genuinely-orphaned `mutations/equation.ts`/`mutations/diagram.ts` (zero importers outside their colocated tests; the live equation/diagram write paths resolve to `actions/handlers/{equation,diagram}` and `selectors/diagram`), the duplicated floating-object projection (`receipt-processing.ts` re-implements the same classify → `applyBatch` → `applyPatches` sequence as the push path, minus the bounds-backfill and generation guards), and the unsound `actor-access` coordinator overload typed as `Record<string, unknown>` with last-wins spread merges while the typed `*FromBundle` path is the one marked `@deprecated`. The plan is well-sequenced (delete-first, then projection extraction, then wiring extraction), preserves a careful list of real invariants, and correctly distinguishes edit targets from cross-folder ripple. It loses points mainly on the under-specified design of its highest-blast-radius step (actor-access re-typing) and on a few objectives that are framed as decisions rather than resolved.

## Major strengths

- **Evidence is real and checkable.** Line counts match exactly (sheet-coordinator 1,173; diagram 583; equation 292; their tests 1,431 + 750 + a 62-line text-effects boundary test). The "false header" claim, the orphan claim, the projection-duplication claim, and the `Record<string, unknown>` actor-access claim all reproduce on inspection. This is a plan grounded in the current tree, not in a remembered version of it.
- **Correct, well-scoped sequencing.** Step 1 (delete orphans) is genuinely the cheapest/highest-signal change and shrinks the surface every later step touches. The instruction to do step 2 (projection) before step 3 (wiring) to minimize churn on the same file is right, and the "pure move first → add tests → unify in a separate commit" mitigation for the riskiest step is exactly the discipline that prevents the ghost-object regression it warns about.
- **Invariant preservation is specific, not generic.** The plan names the subtle invariants that a careless refactor would break: subscribe-before-populate coalescing, per-flush generation guards, `created`-vs-`updated` classification keyed on cache membership at apply time, bounds-backfill for object types Rust omits, structural sharing, the `formulaEditing`-only return-to-origin semantics, the pending-format reapply guard, dual TS-native/OOXML connection shapes, and the deliberately-swallowed `refreshActiveCell` race during teardown ("do not 'fix' the empty catch into a throw"). This is the difference between a plan that has read the code and one that has skimmed it.
- **Honest about what it can't decide.** It flags the `ShellCoordinator` UIStore wiring as a product decision needing owner sign-off, the `actor-access` key-collision as a possible latent bug in a *system* folder (out of scope, "flag it, don't paper over it"), and the text-effects boundary test as something to verify before deleting. I confirmed that last caution is warranted: `text-effects-kernel-boundary.test.ts` is a live boundary guard that scans `src/coordinator/mutations` for kernel-internal imports and asserts no duplicate `text-effects.ts` exists — it must NOT be deleted, and the plan was right to gate its removal on verification rather than listing it unconditionally.

## Major gaps or risks

- **Step 5 (re-type `actor-access`) is the widest blast radius and the least-designed.** The plan says "a generic typed merge over the systems' `access` bundles" but doesn't sketch the actual type. A spread-merge of five systems' `ActorAccessors`/`ActorCommands` does not naturally produce a sound named type, and surfacing key collisions *at compile time* (versus a dev-time runtime assertion) is non-trivial — TypeScript intersection/spread of record types silently last-wins just like the runtime does. Given this step "should be reviewed against the full importer list" (the plan's own words), it deserved a concrete type proposal, not a one-line gesture. This is the single largest correctness risk in the plan and the thinnest part of it.
- **The post-refactor line budget is aspirational and unquantified.** Objective 2 says the file should "approach its claimed role" and "update the file header to describe the real responsibilities and line budget," but doesn't commit to a number. The existing "~250-line" claim is itself unrealistic given construction + ordered `start()` + dispose; replacing one false number with an unstated one is a weak success criterion. A target ("≤400 lines, composition + delegation only") would make the objective verifiable.
- **Verification depends entirely on downstream CI by constraint.** The plan correctly notes it cannot run build/test/typecheck, so all its gates are "what a reviewer/CI must execute." That is honest, but it means the plan ships zero executed evidence that the refactor is green — and several steps (projection unification, actor-access retyping) are exactly the kind that typecheck-break in non-obvious ways. The app-eval regression list is good but the plan leans on "use existing app-eval scenarios" without confirming those scenarios actually exist for each named behavior (e.g. merge-anchor snap, connector re-route on shape move).
- **Step 4 diagnostics could change behavior subtly.** Emitting a warning per absent dependency is described as "no feature enabled/disabled differently — only made observable," but the choice between `onMetric` and `console.warn` and the de-dup of repeated warnings (these wiring paths can run per sheet-switch) is unspecified. A naive implementation could spam logs on every legitimately-optional config.

## Contract and verification assessment

The contract section is the plan's strongest dimension. The public `SheetCoordinator` surface it pledges to keep stable (`grid/renderer/objects/input/ink`, `floatingObjectCache`, `workbook`, `uiStore`, `setRendererDependencies`, `handlePointerUp/Cancel`, `processReceipts`, `deleteSheet`, `dispose`, `isActive`) matches the methods present in the file, and the construction/dispose ordering invariant is real. The DAG constraint (no `coordinator/ → views/`, hence `view-clipboard-data.ts` + `CoordinatorViewAdapter`) and the `@internal` `SheetView.gridRenderer` boundary are correctly identified as things to preserve.

Verification gates are appropriate in *kind* (typecheck, keep existing receipt/input coordinator tests, add push-path + projection unit tests, dead-code grep gate, app-eval sweep, no new `any`) but soft in *enforcement*: "should target the unified projection helper," "add an equivalent push-path test," "prove zero importers before removing." These are the right gates; what's missing is a crisp pass/fail bar (e.g. "the new `FloatingObjectProjection` test must assert all five enumerated invariants" is stated, but the wiring-module tests are left as "one per `wireX`" without the specific assertion each must make). The dead-code deletion gate is the most concrete and is correctly scoped.

## Concrete changes that would raise the rating

1. **Specify the `actor-access` merge type.** Provide the actual generic signature (or a `satisfies`-based collision check) and state explicitly whether collision detection is compile-time or a dev assertion. Name the canonical function and confirm via importer search whether `createActorAccessLayerFromBundle` has any remaining callers before promising to retire it. This is the change that most reduces the plan's real risk.
2. **Commit to a numeric line/responsibility budget** for post-refactor `sheet-coordinator.ts`, so "shrink to a composition root" is verifiable rather than rhetorical.
3. **Confirm the named app-eval scenarios exist** (or mark which must be authored) for each at-risk behavior — merge-anchor snap, connector re-route, floating-object insert/move/delete repaint, sheet-switch restore — rather than assuming coverage.
4. **Pin down the wiring-diagnostic mechanics:** `onMetric` vs `console.warn`, stable prefix, and de-duplication so per-sheet-switch wiring doesn't emit repeated warnings.
5. **Make the per-`wireX` test assertions concrete:** for each extracted wiring module state the one invariant its test must assert (e.g. "wireFindReplaceHighlight: assert highlight invalidation fires on workbook find-replace event and cleanup unsubscribes"), mirroring the precision already shown for the `FloatingObjectProjection` test.
6. **Resolve, don't defer, objective 6's default** more firmly: the plan already recommends (b) delete-and-document; stating the exact comment and the tracking-issue target for the `FOCUS_GRID→FOCUS_BASE_VIEW` rename would close it cleanly without owner round-trips for the no-op-deletion portion.

Overall: a high-quality, production-path plan that I would approve to start (beginning with the low-risk step 1) while requiring the actor-access type design and a concrete line budget before steps 2 and 5 land.
