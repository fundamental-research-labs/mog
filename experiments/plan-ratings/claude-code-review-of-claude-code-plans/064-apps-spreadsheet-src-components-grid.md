# Review of Plan 064 — `mog/apps/spreadsheet/src/components/grid`


## Summary judgment

This is a strong, evidence-grounded plan for refactoring a mature, performance-sensitive React⇄canvas bridge. I independently verified its central factual claims against the source, and they are accurate to the line: `SpreadsheetGrid.tsx` is 1059 lines; `useRenderContextConfig.ts` is 481; `InlineCellEditor.tsx` is 777. The module-level globals `_lastKnownCellRect`/`_measurementCanvas`/`_measurementContext` exist at lines 69–81/284–292; the two `exhaustive-deps` disables are at 230/386; the `as unknown as SplitViewportConfig` cast is at 430 and the inline-`import(...)` floating-object cast at 531; the `paperSizeMap` is exactly `{1,5,9,8}` with `?? 'letter'` fallback at line 690–691; the two "Split view not yet implemented" `console.log`s are at `ScrollContainer.tsx:215,251`; the `T | Promise<T>` getter unions are present in the options interface; the "migration pending" out-of-band `updateContext` calls are at lines 267/396/417/435/446/476; `getAutoPageBreaks` is genuinely `async` (line 674); the rich-text editor's "TODO: Store segments directly" / "Track character selection" comments exist verbatim; and the dangling `@see` doc targets (`ARCHITECTURE-CHECKLIST.md`, `09-SPREADSHEET-GRID-DECOMPOSITION.md`, `RENDERER-INSTANCE-OWNERSHIP.md`, etc.) are indeed absent from `mog/docs/internals/spreadsheet/`, where only `ARCHITECTURE.md` and the `renderer/` subfolder survive. This is among the most faithful plan-to-code mappings I would expect to see.

The plan correctly diagnoses the folder's defining constraint (render isolation / the "842 re-renders/sec" history), respects it as a hard invariant, and sequences the work so behavior-locking precedes refactoring and the one behavior-changing item is isolated last. The objectives are well-motivated refactors, not a rewrite.

## Major strengths

- **Verifiable, specific evidence.** Nearly every assertion carries a file+line anchor that checks out. This makes the plan trustworthy and the scope auditable.
- **Contract-first framing.** C1–C9 articulate the invariants that matter (render isolation, synchronous paint-path getters, `data-testid` eval contract, renderer lifecycle ordering, WYSIWYG single-sourcing). The `data-testid` enumeration (C7) is exactly the kind of de-facto contract that silently breaks eval harnesses, and naming it is good practice.
- **Sequencing discipline.** Phase 0 characterization tests as the C1/C5 oracle before any extraction; pure extractions (O1) before wiring consolidation (O2); the synchronous-getter narrowing (O3) explicitly gated as a prerequisite for O2; the behavior-changing rich-text work isolated last.
- **Honest dependency accounting.** O2 depends on an out-of-folder `setContextConfig` capability the plan cannot guarantee, and R3 specifies a concrete degradation (partial consolidation, update the comment to a tracked dependency, no shim). This is the right way to handle a cross-folder blocker.
- **Type-level enforcement of a runtime invariant.** O3.3 narrowing the getter unions from `T | Promise<T>` to `T` turns C3 from convention into a compile error — a genuinely good move.

## Major gaps or risks

- **C1's verification has no teeth.** The plan promises canvas output is "byte-for-byte the same" for outline levels, page breaks, filter icons, floating-object previews, etc., but the only gates are (a) characterization unit tests over the *extracted pure functions* and (b) app-eval, which drives DOM/`data-testid`s through the input path — neither proves the rendered canvas pixels are unchanged. There is no screenshot/pixel-diff gate. The pure-function snapshots lock the *inputs* to the renderer, which is a reasonable proxy, but the plan should say so explicitly rather than claim byte-for-byte parity it does not gate. This is the single biggest gap between what is promised and what is checked.
- **O7 (rich-text) is over-scoped and under-specified relative to the rest.** It is the only data-correctness change, it depends on a rich-text segment model/contract the plan admits it has not confirmed ("confirm the API surface before implementing"), and it gets far less concrete detail than the refactor objectives. Bundling a user-visible feature with a "no behavior change" refactor plan dilutes the safety story. This belongs in its own plan, or at minimum behind an explicit feature flag with the contract confirmed first.
- **O4's mechanism is hand-wavy.** "Per-instance state via a small grid-scoped context (or hoisted into a hook)" is offered as an alternative without a concrete design, yet this is the most fragile file in the folder (the memory record notes caret-occlusion and click-commits-formula bugs). The unmount-survival requirement (C4/R5) is precisely what makes "just move it off the module" non-trivial — the rect must outlive the editor's mount but not the grid's. The plan flags the risk but does not commit to a mechanism, leaving the hardest part to implementation-time discovery.
- **The "two concurrent grids" motivation for O4 may be hypothetical.** The plan justifies de-globalizing on split-panes / multi-workbook, but split view is itself documented as half-landed (O5.1). If no two grids ever mount concurrently today, O4 is risk taken for a future that O5 hasn't decided to ship. The plan should reconcile these two objectives — the split-view shipping decision (O5.1) gates whether O4's cross-contamination scenario is real.
- **Grab-bag breadth.** Eight objectives spanning pure refactor, type hygiene, doc-pointer cleanup, dead-code removal, a cross-folder API request, and a net-new feature. O5.3 (doc hygiene) is near-zero-risk busywork sitting beside high-risk editor surgery. This is acceptable but the plan would be cleaner split into a "safe refactor" plan and a "rich-text feature" plan.

## Contract and verification assessment

The contract set is unusually well-developed and mostly enforceable. C2 (render isolation) is backed by a concrete regression-test proposal and the `[coordinator]`-dep-array discipline. C3 (synchronous getters) is enforced at the type level. C7 (testid/ARIA) is enumerated. C8 (paper-size preservation) is precise. R2 correctly anticipates the first-frame "not yet resolved" case and prescribes the existing placeholder-then-invalidate pattern.

The weakness is the oracle for the headline invariant. C1 visual parity is the thing most likely to silently regress under O1–O3, and it is the thing least operationalized: characterization tests cover the pure math, and app-eval covers interactive DOM, but nothing asserts the canvas frame is identical. For a folder whose entire reason to exist is feeding the canvas, the verification story should either add a canvas/visual-parity check or explicitly downgrade C1's language to "renderer inputs are unchanged, app-eval covers interactive parity." Similarly, C9 (lossless rich-text round-trip) is asserted as the O7 acceptance bar but the segment contract it depends on is unconfirmed, so the gate cannot be specified yet.

## Concrete changes that would raise the rating

1. **Resolve the C1 verification gap.** Either (a) add a canvas snapshot / pixel-diff gate for the affected render paths, or (b) restate C1 honestly as "renderer *inputs* are unchanged (locked by Phase 0 snapshots); interactive parity by app-eval" and drop the "byte-for-byte canvas" claim. Name which app-eval scenarios actually exercise outline rendering, page-break preview, and floating-object preview, so coverage is auditable rather than assumed.
2. **Split O7 out** into its own plan (or hard-gate it behind a flag) and make confirming the rich-text segment contract a Phase-6 entry/exit gate, not an implementation-time discovery. Define the C9 round-trip oracle in terms of that contract.
3. **Commit to an O4 mechanism.** Pick context vs. hoisted-ref and sketch where the fallback rect lives so it survives editor unmount but not grid unmount, with a named test for the `animateScrollTo` window. Tie this to the O5.1 split-view decision: if split view is gated off, state whether O4 is still in scope or deferred until concurrent grids are real.
4. **Add an explicit ordering gate between O5.1 (split-view decision) and O4**, since the multi-grid cross-contamination risk that motivates O4 depends on whether multiple grids can ever co-mount.
5. **For O3, replace "most already do" with a per-getter table** (getter → backing cache hook → does it already expose a synchronous resolved read + `onCacheUpdate`? → new work needed). Today only `hasTableColumnFilter`'s new cache hook is called out concretely; the others are asserted.
6. **Specify the app-eval baseline command** (or reference the harness scenario list) the reviewer runs for C7, so "must pass unchanged" is a concrete checklist rather than a hope.
