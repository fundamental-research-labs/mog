# Review of Plan 073 — Collapse the duplicated renderer contract surface in `mog/contracts/src/rendering`

## Summary judgment

This is a strong, evidence-driven consolidation plan. It correctly diagnoses a real, shipped defect: `mog/contracts/src/rendering` is a half-finished migration in which most files are already one-line re-export shims onto leaf packages (`@mog/types-rendering`, `@mog/types-viewport`), but four heavyweight files remain full local copies. The barrel pulls *types* from the leaf packages (`export type * from '@mog/types-rendering'`) while pulling *runtime values* from the stale local copies — a genuine split-brain. The plan's stated goal (one canonical owner per contract, drift fixes folded in, gates taught to prevent recurrence) is the right one, the direction of consolidation matches the established pattern, and the phasing isolates the zero-risk de-dup from the behavior-changing convergence.

I independently verified the plan's central factual claims and they hold. The plan is accurate where accuracy matters most, which is the main reason for the high score. Points are withheld for a non-exhaustive drift enumeration, a soft Phase 3 mechanism, and a visual sign-off that leans on an unconfirmed assumption.

## Verification performed (read-only)

- **Identical-file claims confirmed.** `diff -q` shows `constants.ts`, `data-source-types.ts`, `grid-renderer-primitives.ts`, and `visual-feedback.ts` are byte-identical to their leaf twins (`@mog/types-viewport/rendering/*` and `@mog/types-rendering/visual-feedback`). Phase 1 is therefore genuinely a pure de-dup.
- **`getCellRenderedSize` drift confirmed.** Present in `mog/types/rendering/src/grid-renderer.ts:315`, absent from the contracts copy.
- **Dead-but-misleading claim confirmed.** `index.ts:1` is `export type * from '@mog/types-rendering'`; there is no `GridRenderer` value/type export sourced from `./grid-renderer`, so the local interface copy is dead. A repo-wide grep finds no deep importer of `@mog-sdk/contracts/rendering/grid-renderer` (only the module-banner string matches), corroborating the plan's grep claim.
- **Skin value drift confirmed** for every pair the plan lists (formula-ref fill `0.12/0.16`→`0.08`; `commentIndicator #d93025`→`#fbbc04`; `validationDropdown`/`filterIcon`→`#5f6368`; `checkboxCheck`→`#ffffff`; `checkboxBackground`→`#1a73e8`; `hiddenIndicator #999999`→`#9aa0a6`).
- **`data-sources.ts` provenance divergence confirmed** (`../document/search` vs `@mog-sdk/types-document/document/search`; `./data-source-types` vs `@mog/types-viewport/rendering/data-source-types`); interface shapes match.
- **Subpath + test claims confirmed.** `package.json#exports` exposes `./rendering`, `./rendering/sheet-view-skin`, `./rendering/coordinates`, `./rendering/constants`, each with a `development` condition resolving to source `.ts`; `__tests__/constants.test.ts` asserts `DEFAULT_ROW_HEIGHT === 20` via `../constants`. A real consumer of `DEFAULT_RESOLVED_SHEET_VIEW_SKIN` exists at `mog/views/sheet-view/src/capabilities/skin.ts`, so the visual-regression concern is not hypothetical.

## Major strengths

- **Defect is real and precisely characterized.** Every concrete divergence in the plan checks out against source. The split-brain framing (types from leaf, values from local) is exactly correct and is the key insight.
- **Right consolidation direction, low-risk pattern.** Re-exporting onto leaf packages matches the already-shipped shims (`coordinates.ts`, `render-context.ts`, etc.), so the plan extends an established convention rather than inventing one. No new build machinery.
- **Phasing is sound and de-risked.** Phase 1 (identical files → shims, byte-identical output) is correctly separated from Phase 2 (drifted files, observable value change), and Phase 3 (gates) is correctly sequenced after the owner set is stable. This ordering lets the safe work land independently.
- **Behavior change is surfaced, not buried.** The plan explicitly flags that Phase 2's skin convergence changes emitted runtime colors and is the one place observable output changes — and even names the inverse-fix branch if product intent turns out to favor the contracts values.
- **Verification ties to the right gates.** It anchors on the actual build pipeline (`check-contracts-declaration-identity`, `verify-runtime-exports`, `check-contracts-runtime-inventory`, `check-contract-runtime-imports`), a before/after `.d.ts` diff, the existing unit test, and downstream typechecks — and respects the recorded build-order gotcha (build contracts before consumers typecheck).
- **Good edge-case coverage.** `SearchHighlight` duplicate-identity risk, subpath-must-still-emit-dist risk, and the `getObjectBoundsSync`→`boundsReader` non-goal are all called out.

## Major gaps or risks

- **Drift enumeration in Objective 2 is presented as exhaustive but is not.** The canonical `ResolvedSheetViewOverlaySkin` defaults also diverge: `pastePreviewFill` (`rgba(33,115,70,0.10)`→`rgba(26,115,232,0.08)`), `pastePreviewBorder` (`selectionBorder`→`#1a73e8`), and `dragGhostBorder` (`dragTargetColor`→`#1a73e8`) all change under convergence — these are not in the plan's bullet list. The plan is partly protected by the Phase 0 "re-diff to capture every divergence" instruction and the verification rule that "any other delta is a defect," so this won't silently slip. But the verification gate says the *only* intended `DEFAULT_RESOLVED_SHEET_VIEW_SKIN` deltas are the enumerated ones; an implementer following the headline list literally would flag the paste/drag-ghost changes as defects and stall. The enumeration should either be completed or explicitly labeled non-exhaustive.
- **Phase 3 is the weakest, least-committed part.** "Extend / configure the gates… or note the smallest extension needed" defers the actual mechanism to Phase 0 discovery. Given that the plan's own framing is "the gates already exist yet failed to catch this," the most valuable deliverable — a concrete enforcement that fails on re-divergence — is left as a contingent TODO. The plan does not say what single-ownership enforcement would even look like for these gates (allow-list entry? brand-owner registration? identity assertion against the leaf `.d.ts`?), so the reader cannot judge whether Phase 3 is a one-line config or a script change.
- **Visual sign-off rests on an unverified assumption.** "The canonical values are newer and assumed correct" is the load-bearing premise for shipping a user-visible color change, but the plan provides no evidence the canonical skin is the *intended* product state (e.g., no git-blame/date check on the two tables, no product reference). It flags this as a risk and names the inverse fix, which is the right hedge — but the determination of which owner is correct is exactly the decision that should be resolved (or at least scoped to a named approver) before this is implementable, not deferred to "run app-eval and eyeball it."
- **`development` export condition under-discussed.** The `./rendering/*` subpaths resolve to source `.ts` in dev mode, so converting `sheet-view-skin.ts` to a shim changes what dev-mode consumers see *immediately* (before any build), not just the rolled-up `.d.ts`. The plan reasons mostly about the built dist artifacts; it should confirm the dev-condition path also forwards values correctly so dev and built builds stay consistent.

## Contract and verification assessment

The contract reasoning is the plan's best quality. It enumerates the must-preserve public surface by name (interfaces, the `RenderPriority` value enum, constant values, skin DTOs, default objects), pins runtime invariants (`RenderPriority` numeric values, `DEFAULT_ROW_HEIGHT === 20` tied to the Rust `points_to_pixels(15.0)`, the platform `DEFAULT_COL_WIDTH` branch), and correctly insists that subpath shims must still emit dist artifacts so `package.json#exports` don't 404. The distinction between "preserve exactly" (Phase 1) and "intentionally converge" (Phase 2) is crisp and matches what the build's declaration-identity diff can actually police.

Verification gates are well-chosen and largely sufficient: full build pipeline as the regression guard, structural `.d.ts` before/after, the existing constants test, downstream typecheck across kernel/shell/canvas-engine, app-eval chrome scenarios for the skin change, and a grep gate for stale deep imports. The one verification weakness mirrors the gap above: the "only intended deltas" assertion is keyed to an incomplete enumeration, so the gate as written would mis-classify the legitimate overlay-color convergence.

## Concrete changes that would raise the rating

1. **Complete (or explicitly mark non-exhaustive) the skin-drift list** to include `pastePreviewFill`, `pastePreviewBorder`, and `dragGhostBorder`, and reconcile the verification rule "the only intended `DEFAULT_RESOLVED_SHEET_VIEW_SKIN` deltas are X" with the full set, so the `.d.ts`/value diff gate doesn't false-positive on legitimate convergence. (→ 9)
2. **Commit Phase 3 to a concrete mechanism.** State the expected enforcement shape for `check-contracts-declaration-identity` / `check-contracts-runtime-inventory` (owner-list entry vs. script change), with a fallback if the gate has no owner-list concept. Even a provisional mechanism beats "note the smallest extension needed."
3. **Resolve the canonical-vs-contracts ownership question before Phase 2, or assign it.** Add the cheap evidence the plan already implies (which table is newer / who owns the product decision) so the direction of the color convergence is decided, not assumed, and name who signs off on the visual change.
4. **Address the `development` export condition explicitly** for `sheet-view-skin.ts` and `constants.ts`: confirm the dev-mode source-resolved subpath forwards both types and values post-shim, so dev and built consumers see identical output.
5. **Pin the leaf-package rebase dependency more concretely.** The parallelization note flags that a sibling worker restructuring `@mog/types-rendering`/`@mog/types-viewport` forces a rebase; naming the exact import paths this plan depends on (so a path change is a mechanical find/replace, not a re-plan) would harden the cross-folder coordination.
