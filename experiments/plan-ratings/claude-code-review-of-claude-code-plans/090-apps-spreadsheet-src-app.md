Rating: 8/10

# Review of Plan 090 — `mog/apps/spreadsheet/src/app`


## Summary judgment

This is a strong, evidence-driven plan for the spreadsheet app's composition root. I verified its central factual claims against the live tree and they hold up with unusual precision: the folder is 7 files (`CoordinatorProvider.tsx` is 834 lines, not the stated 833 — a one-line slip), the keydown routing tree lives where claimed (`:292-450`), the IME guard is genuinely first (`:294`), the Escape-with-open-dialog carve-out exists (`:336-338`), the `RangeSelectionCoordinatorSetup` no-op TODO callbacks are real (`:527-535`), `SpreadsheetIndicators` is fed `exportNotification={null}` at `index.tsx:817` and short-circuits to an empty fragment, `SpreadsheetCoordinatorProvider` is rendered at `index.tsx:758` with no `onUIAction`, the two type-escape casts exist verbatim (`:259-263`), `coordinator-keydown-capture.ts` does not yet exist, and `Shell`/`ViewSwitcher`/`RecordDetailSidebar` have no component callers (only the `ui-store/slices/navigation` actions reference the *concept*) while a live `views/` registry family (`registry.ts`, `HybridViewContainer.tsx`, `container.tsx`) does exist. The diagnosis is accurate, the contracts-to-preserve section is the strongest part of the document, and the phasing is sensibly ordered (lowest-risk, self-contained extractions first).

The plan loses points for treating two of its six phases (4 and 5) as fork-in-the-road decisions ("implement OR delete", "converge OR fold in") that defer the actual product call rather than resolving it, and for verification gates that are described well but explicitly handed off to an integrator the plan cannot bind.

## Major strengths

- **Exceptional grounding.** Nearly every assertion carries a line citation and the citations are correct. This is a plan written from the code, not from a summary of it. The "Enter moves down 2 cells" regression, the Windows Alt-menu focus-theft (`:464-475`), and the `dismissCommentPopover` Escape-bubble interaction are named as concrete blast-radius hazards, not abstractions.
- **The contracts/invariants section is production-grade.** It enumerates the exact semantics a refactor must not break: capture-phase single-source-of-truth for nav keys, IME-guard-first ordering, `async`/`await` rejection preservation (no regressing to `.then(() => {})`), the `errorStyle → enforcement` mapping, the circular-reference `setIterativeCalculation(true)`-before-`onEnableIterative` ordering, and focus-layer routing. These are the things a careless extraction silently destroys, and the plan front-loads them as acceptance criteria.
- **Right sequencing and parallelism.** Phase 1 (keydown extraction) and Phase 2 (editor-deps extraction) are correctly identified as file-disjoint and concurrently runnable; cross-folder and product-gated work (Phases 4–5) is pushed late with explicit dependency callouts.
- **Phase 1's `decideKeydownRouting` design is the high-value core** and it is specified concretely: a pure function over a plain `ctx` snapshot, returning a route decision, with a truth-table test enumerating each branch. This mirrors the existing, already-tested `coordinator-keyup-capture.ts` precedent — a credible, low-risk pattern the codebase already endorses.

## Major gaps or risks

- **Phases 4 and 5 specify a decision procedure but not an outcome.** "Thread real state OR delete the dead prop" and "converge OR fold into `index.tsx` and remove copies" are both reasonable, but the plan leaves the choice to product intent it never resolves. For Phase 4's `onUIAction` this is partially defensible (it is genuinely coupled to the in-flight `charts.ts:271` migration), but a plan that ends a phase at "decide X" is incomplete as an executable spec — an implementer can't start Phase 4/5 without first running an out-of-band decision the plan doesn't own. These should either be split into a decision-stub plus a follow-on, or the plan should state a default and let the decision override it.
- **Phase 1's purity claim needs one more guardrail.** `decideKeydownRouting` is to be "free of DOM side effects and of `keyboardCoordinator`," but several branch conditions read `document.activeElement?.tagName` and `document.querySelector('[role="dialog"]')` (`:326`, `:336`). The plan correctly proposes passing `activeTag` and `hasOpenDialog` into `ctx`, but does not explicitly say the DOM *reads* must move to the call site (only that side effects must not). A truth-table test cannot exercise `document.querySelector` cleanly; the plan should state that all DOM interrogation is hoisted into `ctx` construction, leaving the pure function operating purely on the snapshot. As written this is implied but not nailed down, and it's exactly where a "pure" extraction quietly leaks impurity.
- **Editor-deps "move verbatim" understates the testability win it claims.** Phase 2 says move `:613-782` verbatim while also promising unit tests for `errorStyle→enforcement`, datetime-fraction, and auto-fill gating. But those behaviors are tangled inside closures over `workbook` and the dialog actions; "verbatim" extraction with injected deps is plausible, yet the plan doesn't acknowledge that making the *transforms* unit-testable may require pulling the pure mapping logic (e.g. `errorStyle→enforcement`) out as standalone helpers separate from the dependency-bearing commit functions. The test list implies that decomposition without committing to it.
- **Verification gates are unenforceable from within the plan.** The plan defers typecheck, Jest, and app-eval to "the integrator, not in this planning task." That is honest, but it means the plan ships no self-contained definition of done — there is no stated pass/fail bar the implementing agent runs before declaring a phase complete. For a refactor whose entire risk thesis is "don't silently regress keyboard semantics," the absence of an in-loop gate is the biggest verification weakness.
- **Phase 3 again offers two paths** (store-driven flag vs. delete-and-document). Better than the current TODO no-op, but the plan should pick the store-source-of-truth option (which its own evidence favors, since dialogs already read `rangeSelectionMode`) rather than re-presenting the fork.

## Contract and verification assessment

The contract clarity is the plan's best dimension: the "preserve or strengthen" list is specific, cites lines, and names the failure mode for each invariant. The DAG-boundary discussion correctly identifies the two casts as moving errors from compile to runtime and proposes typed adapters rather than surface-widening. The `EditorDependencies` shape is explicitly held fixed (only construction moves), which protects the editor-machine contract.

Verification *design* is good in intent — new Jest truth-table tests, the existing keyup regression kept green, and a targeted app-eval list covering the genuinely integration-critical keyboard/focus behaviors (BODY-focus shortcuts, Ctrl+PageUp/Down from chrome, CommentPopover Escape, Alt KeyTip promotion). The weakness is purely that execution and acceptance thresholds are outsourced; the plan names *what* to test but binds *no one* to a green bar. The contracts-rollup caveat (`pnpm --filter @mog-sdk/contracts build` before downstream typecheck) is correctly flagged for Phase 6.

## Concrete changes that would raise the rating

1. **Resolve, don't defer, the Phase 4/5 forks.** State a default outcome (e.g. "delete the dead `SpreadsheetIndicators` render and props unless export-notification state lands this cycle"; "fold the multi-view capabilities into `index.tsx` and delete `Shell`/`ViewSwitcher`/`RecordDetailSidebar` unless a multi-view product decision lands"), and mark the override as a gating question rather than leaving both branches live. For `onUIAction`, name the concrete check that decides it (is the `charts.ts` JSON-string protocol fully retired across all listed handlers? — enumerate them) so the decision is mechanical, not editorial.
2. **Make Phase 1 purity explicit:** state that all DOM reads (`activeElement.tagName`, `querySelector('[role="dialog"]')`) are hoisted into `ctx` at the call site, and that the pure function and its tests touch no `document`.
3. **Add an in-loop definition of done per phase** — even if the integrator runs the full suite, state the minimal command set the implementing agent must pass before marking a phase complete (the targeted Jest path + the named app-eval scenarios), with an explicit "no regression in `coordinator-keyup-capture.test.ts`" gate.
4. **Acknowledge the decomposition Phase 2 actually requires:** call out that unit-testing the transforms means extracting the pure mapping helpers (enforcement mapping, datetime fraction, auto-fill predicate) as standalone exports, not just relocating the closure — otherwise "verbatim move" and "unit test the transforms" are in tension.
5. **Pick the store-as-source-of-truth path for Phase 3** outright and specify the `rangeSelectionMode.active`-derived collapsed flag and which dialog components subscribe, rather than restating the implement-or-delete choice.
6. Minor: correct the `CoordinatorProvider.tsx` line count (834, not 833) and re-verify the handful of `:###` citations that are ±1 off after the count drift, so downstream agents anchoring on line numbers don't miss.
