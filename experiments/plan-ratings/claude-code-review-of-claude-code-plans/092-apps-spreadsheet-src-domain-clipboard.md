Rating: 8/10

# Review of Plan 092 — `mog/apps/spreadsheet/src/domain/clipboard`


## Summary judgment

This is a strong, unusually well-evidenced refactor plan. It demonstrates real reading of the folder rather than pattern-matching: the two-model split (`ClipboardPayload` dense cross-view pipeline vs. `ClipboardData` sparse in-document engine pipeline) is described accurately, and nearly every concrete claim I spot-checked against source is correct — the `getClipboardDimensions`/`getPayloadDimensions` barrel collision (`index.ts`), the unescaped `font-family: "${format.fontFamily}"` interpolation (`serializers.ts:372`) and raw color emit (`:378`/`:381`), the `legacyCopyToClipboard` + `document.execCommand('copy')` path (`clipboard-service.ts:371`), the `console.warn`/`catch {}` swallowing at the system-clipboard boundary (`:108`, `:301`), and the load-bearing `useCoreCopyRange` gating with its format-only-entries and cross-sheet reasoning (`paste-executor.ts:865`–`893`). The plan correctly identifies what must NOT change and encodes the executor's hard-won invariants as preservation contracts rather than candidates for "simplification." Risk sequencing (land low-risk objectives 1/4/5/6/7 first, defer the inference change and the executor decomposition behind their own tests) is exactly right, and the non-goals section wisely refuses the tempting-but-wrong "merge the two models" mega-refactor.

What keeps it from a 9–10 is a soft spot at the center of its headline objective: the comma-fragmentation justification for unifying the tokenizer doesn't clearly apply to the parser it names, and the cross-path inference-parity work — arguably the highest correctness risk in the whole plan — is deferred to a cross-folder "reconcile with the machine" step that is described more by intention than by mechanism.

## Major strengths

- **Evidence-grounded.** Claims are specific and verifiable, down to function names and behaviors. The "must preserve" list reads like it was distilled from the executor's own comments, and it is.
- **Correct risk model.** It names `executePaste` decomposition as the highest-risk change, gates it behind a per-phase decision-table test suite, and insists the orchestrator call order stay byte-for-byte equivalent. The cross-sheet blank-clear and format-only-entry guards are explicitly protected from naive extraction.
- **Honest scoping.** Phase 0 is read-only provenance tracing; non-goals exclude model unification, infra relocation, and any Rust/contract change; the contracts-rollup gotcha is correctly noted as *not* triggered.
- **Genuine defect surface, not cosmetics.** The HTML style-attribute injection (objective 6) and the silent-failure error handling (objective 5) are real, and the identifier-preservation issue in `inferValue` is a real data-corruption class (zip codes, account numbers).
- **Verification gates are concrete and objective-aligned** — each new test maps to a specific objective, and the app-eval/api-eval regression list covers the right round-trips (paste-special, cut-move, filtered rows, cross-sheet rebasing, external TSV/CSV/HTML, image).

## Major gaps or risks

- **The tokenizer-unification rationale targets the wrong divergence.** Objective 1 asserts that "any `ClipboardPayload` ingest path that runs CSV through `tsvToCells` ... will mis-split unquoted formulas/text containing commas." But `tsvToCells` (`serializers.ts:48`) splits only on `\t` — it never tokenizes on commas, so a `=SUM(1,2)` cell arriving as a single TSV field is *not* fragmented by it. The comma-fragmentation risk lives in the comma-delimited path (`parseCSV` in `clipboard-parser.ts`), not in `tsvToCells`. Phase 0 honestly flags this as an open question ("determine whether `tsvToCells` is reachable with comma-delimited data"), which is good — but objective 1 states the conclusion with far more confidence than Phase 0's own investigation warrants. The plan should resolve this tension before committing: the *actual* defensible win from routing `tsvToCells` through the shared tokenizer is quote/escape-handling consistency and lossy `inferValue` dedup, not formula-comma protection. As written, the marquee objective is partly justified by a mechanism that may not exist on the path it names.
- **Inference parity is the real correctness hazard and is under-specified.** Objective 2 + Phase 2 propose changing what external paste produces, then "reconcile with how `clipboard-machine` types parsed `string[][]`." That reconciliation — making both ingest routes emit the same `CellValue` — is the single most behavior-visible and most easily-divergent piece of the plan, and it is cross-folder (the machine is explicitly out of scope/edit). The plan hand-waves it ("if the machine has its own coercion, route it through the shared contract") without confirming whether the machine *does* coerce, where, or whether routing it through this folder creates a new dependency edge the machine's owners will accept. This deserves its own Phase-0 finding, not a conditional aside.
- **A guard the plan proposes to add already partially exists.** The standard-number branch already guards round-trip safety: `!isNaN(num) && isFinite(num) && trimmed === String(num)` (`serializers.ts:309`), so plain `"001234"` is *already* preserved as text. The leakage is specifically the comma-thousands branch (`:313`–`:320`), which lacks that guard (`"001,234"` → `1234`). The plan's objective-2 framing ("strips ... coerces ... silently mangles") overstates the current breakage; the precise fix is narrower (extend the existing guard to the comma/currency branches). Stating it precisely would lower implementation risk and avoid re-deriving a guard that's already there.
- **Objective 6's threat model is asserted, not established.** The plan treats unescaped style emit as an injection vector, but `cellsToHTML` output is written to the OS clipboard for cross-app paste — the plan does not show where, if anywhere, this HTML is re-parsed into *our* trusted DOM such that attribute breakout is exploitable rather than merely malformed. Sanitization is still worth doing (defense-in-depth, round-trip stability), but Phase 0 should confirm the consumer so the change is scoped as "robust serialization" vs. "security fix" correctly, and so the test ("must not break out of the `style` attribute on re-serialize") asserts against a real consumer.

## Contract and verification assessment

The contract section is the plan's best part. It enumerates the behavioral invariants that actually matter — `ClipboardPayload` shape stability, the full `useCoreCopyRange` condition set (verified against `paste-executor.ts:885`–`893`), phase ordering (unmerge-before-write, values-before-formats, comments-after-values), cut single-use lifecycle, signature-based own-clipboard detection, image-paste priority, and the `writeToSystemClipboard` user-activation/no-swallow contract — and it correctly insists `ClipboardService.createPayload` produce identical `text`/`html` after the serialization refactor. That last point is the right regression anchor and should arguably be promoted to an explicit golden-output test, which the plan only implies.

Verification gates are concrete and well-mapped: existing tests named individually, new tests tied to each objective, and a realistic app-eval/api-eval regression list. Two gaps: (1) there is no characterization/golden test proposed to *capture current `inferValue` output before changing it*, which is the safest way to make an intentional behavior change auditable; (2) the per-phase executor tests are described as asserting a "decision table" but the plan doesn't commit to extracting the `useCoreCopyRange` decision into a pure, separately-callable predicate — without that extraction, the decision table remains as hard to test as it is today. The plan says `decideCorePath(...)` will exist, which is the right instinct; it should state that this predicate is pure and side-effect-free as an explicit deliverable.

## Concrete changes that would raise the rating

1. **Resolve the objective-1 rationale before Phase 1.** Either (a) confirm via Phase 0 that a comma-delimited payload actually reaches `tsvToCells`/`parseExternalClipboard` in production and restate the formula-fragmentation claim accordingly, or (b) re-found objective 1 on its defensible benefits (single quote/escape implementation + single inference step) and drop the comma-fragmentation framing for `tsvToCells` specifically. Make the marquee objective's justification match the code path it edits.
2. **Promote inference-parity to a Phase 0 finding with a concrete mechanism.** Document where (file:line) `clipboard-machine` types parsed strings today, whether it coerces at all, and the exact shared-contract entry point both paths will call — plus a parity test that feeds the same `string[][]` through both routes and asserts identical `CellValue`. Get sign-off from the machine's owners noted as a precondition, since the machine is out-of-edit-scope.
3. **Narrow objective 2's fix to the unguarded branches.** Acknowledge the existing `trimmed === String(num)` guard (`serializers.ts:309`) and scope the change to extending it to the comma-thousands and currency branches, with a captured golden test of current `inferValue` output first.
4. **Confirm the HTML consumer for objective 6.** Add a Phase 0 step locating where `cellsToHTML` output is re-parsed, and label the change as security vs. robustness based on the answer.
5. **Commit to a pure `decideCorePath` predicate** as an explicit Phase 6 deliverable so the `useCoreCopyRange` decision table is unit-testable in isolation, and add the `createPayload` byte-identical `text`/`html` golden test as a named regression anchor.
6. **Add a rough size/effort signal per phase.** `paste-executor.ts` is 1,330 lines and `executePaste` ~500; an estimate of which extractions are mechanical vs. judgment-heavy would help reviewers gate the riskiest phase.
