# Review of Plan 093 — Cell & Formula Editor Domain (`apps/spreadsheet/src/domain/editor`)

## Summary judgment
This is a strong, evidence-grounded plan. Nearly every factual claim it makes is independently verifiable in the source, and the improvement objectives are real defects or genuine architectural hazards rather than speculative refactors. The sequencing is correct (low-risk consolidation first, then correctness, then independent parallel tracks), the contracts-to-preserve section is unusually thorough, and the verification gates map cleanly onto the changes. The single weak spot is Phase 2 (a new shared formula tokenizer), which is by far the largest and riskiest piece and is under-specified relative to the precision the rest of the plan shows.

I verified the plan's central claims directly:
- **Exact-duplicate files confirmed.** `diff systems/shared/utils/formula-context.ts domain/editor/formula-context.ts` and the parser equivalent are both byte-for-byte IDENTICAL. `rg` finds exactly two physical definitions each of `analyzeFormulaContext` and `extractFormulaRanges` (the domain copy + the shared/utils copy).
- **Split import paths confirmed.** `systems/grid-editing/machines/editor/autocomplete.ts:15` imports `analyzeFormulaContext` from `../../../shared/utils`, while UI/renderer consumers import the same functions from `domain/editor`. The divergence hazard is exactly as described.
- **`updateFormulaReference`/`buildReferenceText` bug confirmed.** `endColAbsolute` is computed at line 286 and then unconditionally overwritten at line 289; the per-axis `$` detection is ad-hoc regex (`startRowAbsolute = startRef.match(/\$\d/)` etc.) that will not round-trip mixed references.
- **Metadata N+1 confirmed.** `loadFormulaMetadata` fetches `ws = await wb.getSheet(name)` to read the id, then in a second `Promise.all` does `wb.getSheetById(id)` to re-fetch the same worksheet for `tables.list()` — a redundant double lookup, plus a per-sheet `tables.list()`.
- **DOM coupling confirmed.** Both `cursor-position.ts` and `selection-manager.ts` reach `window`/`document`/`getSelection` directly; `getTextNodesInOrder` is invoked at lines 166, 245, and 320 (plus internally), supporting the O(n²) claim.
- **Allowlist coupling confirmed.** `tools/platform-dependency-allowlist.jsonc` has four entries pointing at `apps/spreadsheet/src/domain/editor/cursor-position.ts`, so the "keep signatures additive" constraint is real.
- **`sortSuggestions` mutation confirmed.** It calls `suggestions.sort(...)` in place (line 196).

The plan's status note ("Evidence gathered by reading all seven source files…") is accurate, not boilerplate.

## Major strengths
- **Correctly prioritized.** Objective 1 (collapse the duplicate source) is genuinely the highest-leverage and lowest-risk change, and the plan front-loads it with a concrete bridge strategy (re-export shim → migrate machine imports → delete physical copies → `rg` acceptance check). The "land Phase 1 promptly to avoid silent divergence with in-flight fixes" risk is the right framing.
- **Defects are real and specific, with line numbers.** The `endColAbsolute` double-assignment, the string-literal blindness in `extractFormulaRanges` (regex runs over the whole formula with no string tracking — verifiable: `CELL_REFERENCE_PATTERN` is applied directly in the `exec` loop), and the metadata double-lookup are all confirmable bugs, not style preferences.
- **Contract preservation is exhaustive and testable.** The plan pins the `extractFormulaRanges` output shape (`{range,color,startPos,endPos,text,index}`, `endPos` exclusive, cycled colors), the cache snapshot semantics (shared in-flight promise, monotonic `version`, idle/null on invalidate, sheet-id normalization), disposal idempotency, and caret-clamp safety. These match the actual code and the test specs that exist (`__tests__/{formula-context,formula-range-parser,formula-metadata-cache}.test.ts`).
- **Behavioral exposure is named.** It explicitly states the only user-visible app-eval changes should be fixed string-literal highlighting and corrected absolute markers — a clear, falsifiable expectation that reviewers can check.
- **Parallelization/dependency notes are accurate.** Phases 4/5/6 touch disjoint files and can run concurrently after Phase 1; the contracts-build dependency for the shared A1 util (objective 5) is correctly flagged as a cross-folder sequencing constraint.

## Major gaps or risks
- **Phase 2 (shared tokenizer) is under-specified relative to its risk.** This is the largest change and the one most likely to regress test-pinned autocomplete heuristics, yet the plan describes it only at the level of a token-type list. It does not pin the tokenizer's *interface* as a deliverable (the token enum, the exact start/end/field shapes, how sheet-prefix + `!` is tokenized, how operators that legitimately start a function context are classified, how the existing `LOG10`/`CEILING.MATH`/`A1` disambiguation rules become tokenizer classification rules). Compare this to the precision of the contracts section — Phase 2 deserves the same. The mitigation ("port heuristics verbatim, gate on the full suite") is sound but informal; there is no feature-flag/rollback path if the tokenizer subtly diverges in production beyond what unit tests cover.
- **Phase 4 changes observable snapshot semantics, and the plan hedges.** Coalesced/debounced invalidation means the current per-event `version` bump + `idle`/`null` reset stops firing once per event. The plan says "preserve … per event … or document the new batched semantics explicitly" — that "or" leaves a real contract decision unmade. Some subscriber may depend on seeing an `idle` transition per structural event; the plan should commit to one behavior and pin it with a test, not defer.
- **Existing golden/snapshot tests that may pin current buggy behavior are not surveyed.** The string-literal and absolute-marker fixes change output. The plan asserts no other app-eval change but doesn't confirm whether any existing unit/app-eval assertion currently encodes the *buggy* highlighting (e.g. a test expecting a range inside `="A1"`). If one exists, it must be updated, and the plan doesn't budget for finding it.
- **Objective 5 scope-creep risk.** "Route ~20 files through one shared A1 utility / bounds constant" is named as motivation but the plan scopes the actual edit to this folder ("adopt it here first"). Good — but the framing could be read as a 20-file sweep. Worth an explicit statement that the other ~20 callers are out of scope for this item.
- **Cursor-position semantics across length-changing edits.** `updateFormulaReference` returns `newCursorPosition` for the edited reference, but replacing a reference shifts the positions of later references in a multi-ref formula. The round-trip tests cover the edited ref; the plan doesn't note whether downstream highlight recomputation handles the shift (it likely re-runs `extractFormulaRanges`, but this is unstated).

## Contract and verification assessment
The contract section is the plan's best feature and is accurate against the code and tests. Verification gates are well-matched: a consolidation guard (single-definition assertion), string-literal cases, tokenizer parity against the existing suites, absolute round-trip cases including mixed (`A$1`, `$A1`, `A$1:$B2`), cache debounce + error-retry tests, and selection-manager round-trip/injected-root tests. The note that typecheck/Jest/app-eval are run by the reviewer (not this planning task) is appropriate given the no-build constraint.

Two gaps: (1) the tokenizer "parity" gate is the load-bearing safety net for the riskiest phase, but parity is asserted at the level of "existing cases pass" without requiring a characterization-test pass *before* the rewrite to lock in current behavior on inputs the existing suite may not cover (R1C1-adjacent, full-column `A:A`, 3-D refs, quoted-sheet-with-apostrophe — several of which the plan itself lists as edge cases). (2) The Phase 4 semantics decision (above) has no committed test because the behavior isn't committed.

## Concrete changes that would raise the rating
1. **Specify the tokenizer contract as a deliverable.** Add the token-type enum, the exact emitted shape per token (type, start, end, text, classification), and the rules for sheet-prefix, operator-as-context, and the `LOG10`/`A1`/`CEILING.MATH` disambiguation — at the same fidelity as the "contracts to preserve" section. Add a pre-rewrite characterization-test step that snapshots current `analyzeFormulaContext`/`extractFormulaRanges` output across the listed edge cases, so the tokenizer is gated against a frozen baseline rather than only the pre-existing suite.
2. **Commit Phase 4's snapshot semantics.** Decide whether coalesced invalidation emits one `idle` transition per burst or preserves per-event transitions, state it, and pin it with the debounce test. Audit the cache's subscribers (renderer/store) to confirm none depend on per-event `idle`.
3. **Survey for golden tests encoding the current bugs.** Add a step to `rg` existing unit/app-eval assertions for string-literal highlighting and absolute-marker output, and list any that must change — converting "no other behavior change" from an assertion into a checked fact.
4. **Explicitly bound objective 5.** State that the ~20 external callers are out of scope and only `domain/editor` adopts the shared util now (with a follow-up note), so the change set stays small.
5. **Address multi-ref cursor/position shift.** One sentence confirming that highlight positions for trailing references are recomputed after a length-changing `updateFormulaReference`, with a multi-ref edit test.

Net: a careful, accurate, well-sequenced plan whose only material weakness is that its single biggest phase is specified more loosely than the rest. Tightening Phase 2's contract and committing Phase 4's semantics would make it a 10.
