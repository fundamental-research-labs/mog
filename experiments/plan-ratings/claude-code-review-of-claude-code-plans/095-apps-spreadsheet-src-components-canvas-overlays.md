# Review — 095 Canvas Overlay Layer

## Summary judgment

This is a strong, evidence-grounded plan. Every defect it names was verifiable against the live source with the cited line numbers essentially intact: the `ValidationDropdownPlaceholder` `console.log` / `// TODO: Wire this to actual cell mutation` (D1, `ValidationDropdownOverlay.tsx:101-110`); `aria-hidden="true"` on the `CanvasInteractiveOverlay` container at `:92` and the `OutlineToggleOverlay` container at `:219` wrapping focusable, labeled `<button>`s (D2); the `scrollTick` re-render pattern in both `OutlineToggleOverlay.tsx:137-143` and `HarnessOverlayMirrors.tsx:43-53` versus the imperative `translate3d` model in `FormControlLayerContainer.tsx:119-135` (D3); `computeOutlineRects` re-deriving outline-renderer math plus the in-comment admission that double-dispatch is "a future cleanup" (D4); the per-control sequential `await ws._internal.getCellPosition` chain and nested per-cell `getCell` loops with a `cancelled`-boolean-only guard (D5, `:204-302`); the hardcoded `#217346 // Excel green`, `Calibri`, `#ababab` and "approximate Excel appearance" comments (D7); `setTimeout(() => setIsOpen(false), 150)` in ComboBox (D8); and the `sparkline-edit`/`hyperlink` union members at `public-types.ts:949-950` that only `console.warn` under `isDev()` (D9). The grid mount points (`SpreadsheetGrid.tsx:981`/`:991`) and the existence of a real `components/grid/ValidationDropdown.tsx` picker are both real. The diagnosis is not hand-waved — it is the highest-credibility part of the plan.

The plan also gets the architecture right: it correctly frames pixel-alignment and faithful interaction routing as the two load-bearing properties, enumerates the four genuinely-distinct coordinate spaces, and treats `data-testid`s and the harness-mirror shape as external contracts. It earns its rating on accuracy of diagnosis and quality of the invariants section. It loses points on a few under-specified contracts in the fix steps and one phase whose cost/risk is larger than its framing implies.

## Major strengths

- **Defect inventory is real and precisely located.** This is not a generic "improve the code" plan; D1–D10 each point at a concrete shipped behavior with a line reference, and the most important one (D1) is a genuine user-facing broken path, not a flagged stub. Prioritizing D1 as Phase 1 is the correct call.
- **Invariants section is the best part.** "Alignment is the contract," the four coordinate-space conventions, `pointer-events` discipline, `data-no-grid-pointer` (correctly tied to the formula-edit commit hazard), linked-cell-as-source-of-truth, and idempotent action routing are exactly the properties a refactor of this folder must not break. Naming them up front is what makes the later phases safe to parallelize.
- **Honest dependency mapping.** Phase 4 (capability-contract extension across `views/sheet-view` + `canvas/grid-renderer`) and Phase 6 (worksheet internal API batch read in `types/api`) are correctly flagged as cross-package and sequenced last, with an interim de-risking step for Phase 4 (import renderer helpers rather than re-implement). The recommended order (2→3→1→7→5→8, then 6, then 4) is sensible.
- **Non-goals are explicit and disciplined.** Refusing the "leave the placeholder but hide it" and "compatibility wrapper for both checkbox names" easy-outs shows the plan is fixing defects at the source rather than papering over them.
- **Verification gates map to defects.** Most phases have a corresponding gate (single-dispatch assertion for D4, alignment-rect-within-tolerance for geometry drift, axe/role check for D2, render-count probe for D3/D5, rapid-refresh race gate for D6→D5).

## Major gaps or risks

- **Phase 1.2 leaves the mutation contract implicit.** It says "reuse the existing setter the in-grid `ValidationDropdown` already calls; do not invent a new mutation entry point" — good guardrail — but never names that setter. The folder's own `FormControlLayerContainer` writes through `ws.setCell(row, col, value)` after resolving a `cellId`; `ValidationDropdownOverlay` has `metadata.row`/`metadata.col` directly. The plan should state the expected commit path concretely (and confirm whether the in-grid picker actually shares it) so an implementer doesn't reverse-engineer it. As written this is the single highest-value change and its contract is the least pinned down.
- **Phase 4.1 is heavier than the plan's tone implies.** Extending `ISheetViewInteractiveElements` to emit `outline-level-button`/`outline-toggle` variants is the right end state, but it requires the renderer to emit group metadata (groupId, collapsed, level, summary-index) that today is only computed app-side. The plan asserts this de-risks alignment but doesn't acknowledge that the renderer would then need to own summary-index resolution (`getSummaryIndex`, `summaryRowsBelow`/`summaryColumnsRight`) — i.e. it moves real logic across the package boundary, not just bounds. The interim step (4.2) mitigates, but 4.3 (decommissioning the canvas hit-test) is the genuinely risky deletion and its coverage gate ("touch, collapsed groups, hidden headers") is named but not turned into enumerated scenarios.
- **Phase 6.1 hedges on a contract it should verify.** "add/most-likely-already-exists a bulk `getCellPositions`" — the plan didn't confirm whether the bulk API exists. `getCellPosition` is per-call async; if no batch API exists this is a new `types/api` + kernel surface, which is materially more work than "batch the lookups." A read-only check of `worksheet/internal.ts` would have resolved this and should be a precondition, not an aside.
- **D3/Phase 5 scroll-invariance claim needs a caveat.** The plan says outline level buttons are scroll-invariant (corner) and can skip the transform, while per-group toggles anchor to scrolling cells. That's correct, but the toggle's *cross-axis* coordinate (gutter X for row toggles, gutter Y for col toggles) is scroll-invariant while the along-axis coordinate scrolls — a pure container `translate3d` translates both axes. The plan's "translate only per-group anchored elements" is right in spirit but glosses that outline toggles need single-axis translation, unlike form controls which translate on both. Worth one sentence so the implementer doesn't apply a 2D transform to a 1D-scrolling element.
- **HarnessOverlayMirrors scope is slightly fuzzy in Phase 5.** The plan folds `ValidationCirclesMirror` into the scroll-unify work and correctly insists the `data-testid`/`data-row`/`data-col` contract and `getValidationCircles`/`getFlashFillPreviewCells` observers stay stable. Good. But converting a zero-size aria-hidden mirror from re-render to imperative transform is a different shape than a visible control layer, and the flash-fill ghost mirror isn't separately analyzed — it's lumped under "validation circles." Minor, but the mirror is a harness contract and deserves its own line.

## Contract and verification assessment

The contract section is the plan's strongest dimension: it correctly elevates `data-testid`s, the harness-mirror shape, the four coordinate spaces, and `data-no-grid-pointer` to API status and forbids silent renames. The idempotent-routing invariant (overlays dispatch the same coordinator/worksheet actions, never duplicate business logic) is exactly right and is the thing Phase 4.3 must not violate.

Verification is good but uneven. The mechanical alignment gate (DOM rect vs painted-button center within tolerance under scroll+resize+grouping) is the single best gate in the plan and directly defends the load-bearing property. The single-dispatch assertion for D4 is concrete and testable. Weaker spots: the new validation-dropdown scenario is described behaviorally but not named/located, and the Phase 4.3 coverage gate ("every group/level/summary-position case, touch, collapsed, hidden headers") is asserted as a precondition for a deletion without being decomposed into the specific scenarios that would prove exhaustiveness — which is exactly where a coverage gap would hide. The a11y gate correctly notes axe is necessary-but-insufficient and asks for a real AT pass; the "flood of buttons to screen readers" risk is well anticipated.

## Concrete changes that would raise the rating

1. **Pin the validation commit path (Phase 1.2):** name the exact setter/coordinator call the in-grid `ValidationDropdown` uses, confirm by inspection that it's reachable from `metadata.row/col`, and state it as the contract — this is the most valuable fix and currently the vaguest.
2. **Verify the batch API precondition (Phase 6.1):** check `types/api/.../worksheet/internal.ts` for an existing bulk position/range read and state definitively whether Phase 6 is "use existing API" or "add new API + kernel impl," since that changes the phase's cost and owner.
3. **Decompose the Phase 4.3 deletion gate** into the enumerated scenarios (row/col level buttons, per-group toggle, collapsed group, summary-above vs summary-below, hidden row/col headers, touch target) that must be green before the canvas-side hit-test is removed — make exhaustiveness checkable, not asserted.
4. **Acknowledge that Phase 4.1 moves logic, not just bounds,** across the package boundary (summary-index/group metadata) and say who owns it, so the "single geometry source" win is costed honestly.
5. **Add the single-axis-translation note for outline toggles** in Phase 5 so the scroll-sync refactor doesn't translate the scroll-invariant gutter axis.
6. **Give the flash-fill ghost mirror its own line** alongside validation circles in Phase 5, restating that both harness contracts (testids, row/col attrs, observers) must remain shape-stable post-refactor.
