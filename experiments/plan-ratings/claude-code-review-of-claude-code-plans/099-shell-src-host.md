# Review of 099-shell-src-host.md — Hardening the Shell App-Host

## Summary judgment

This is an unusually strong, evidence-grounded plan. I verified every concrete code claim it makes against the live source in `mog/shell/src/host`, and they are all accurate — not approximate, but line-for-line correct. The plan reads like it was written by someone who actually traced the launch state machine, the document cache, and the setup hook rather than summarizing from filenames. It identifies real, high-blast-radius defects (un-guarded async `setState`, silent ungated-kernel fallback, swallowed setup errors, orphaned `DocumentHandle`s), states them as named invariants (I1–I9), and sequences fixes into phases with explicit dependency and parallelization notes. The verification section is honest about not running anything and instead specifies the gates a follow-up PR must add. It is the kind of plan an implementer could pick up and execute with low ambiguity.

It is not a 10 only because of a few gaps detailed below — chiefly a naming collision it doesn't disambiguate, under-specified embedder coordination for the behavior-breaking Phase 3, and the fact that it is a large 7-phase / 19-step program whose "land in parallel" framing slightly understates the coordination cost.

## Major strengths

- **Invariant-first framing (I1–I9).** Separating "preserve" from "strengthen" turns vague hardening into checkable contracts, and the test section maps tests back to specific invariants (I2/I3/I4/I5/I6/I7). I7 in particular is correctly marked *preserve exactly* — the plan resists the temptation to refactor the already-correct §6.1 unload-flush path.
- **The generation-token pattern is the right fix** for I2 and is applied consistently (launch body, the `tables.list` effect, and the `useAppDocument` in-flight resolve). It also flags the deadlock failure mode (stale-only suppression) and assigns it a test.
- **Fail-closed capability gating is correctly identified as a security posture, not a cleanup.** Making `allowUngatedFallback` an explicit, auditable prop (default `false`) is the correct design, and the plan honestly labels it behavior-breaking and routes the mitigation (set the flag at first-party sites in the same PR, release notes).
- **Setup atomicity (Phase 4) attacks a genuine data-integrity bug**: the empty-bindings-to-`ready` path is real and the throw-and-route-to-error fix is correct.
- **Sequencing and risk sections are concrete**, not boilerplate — e.g. Phase 2 step 6 depends on Phase 1 step 4, Phase 6 step 18 deferred last pending importer search, and the document-eviction destructiveness explicitly flagged for product sign-off.

## Major gaps or risks

- **`ensureAppTables` naming collision is not disambiguated.** A monorepo search shows `ensureAppTables` also appears in `mog/types/api/src/apps/api.ts` as an *interface method* `ensureAppTables(): Map<string, AppTableId>` (line 678) plus a doc comment (651) — a distinct API from the host's free function. The plan says "search the monorepo for importers before removal" (good), but an implementer scanning hits will trip over this homonym. The plan should name the exact symbol to delete (`host/app-setup.ts#ensureAppTables` + its `index.ts` re-export at line 29 + the shell barrel re-export at `mog/shell/src/index.ts:48`) and explicitly exclude the `types/api` interface method from scope. As written, Phase 6 step 18 risks either over-reaching into a contracts type or stalling on a false-positive caller.
- **Phase 3 embedder coordination is under-specified.** The plan says set `allowUngatedFallback` "at the known first-party embedding sites (e.g. dev-app / first-party shell)" but never enumerates them or confirms they exist. Since this is the one behavior-breaking change, the plan should pin down the actual list of `AppSlot`/`ShellHost` mount sites (out of scope for editing, but in scope to *name*), or the "same PR" mitigation is a promise without a checklist.
- **Date-based document IDs interact with the eviction design.** IDs are `app-{appId}-{Date.now()}`; the proposed "dispose+delete the previous fresh document for the same appId" relies on tracking the prior ID, which the plan covers via the ref, but it doesn't address documents orphaned in IndexedDB from *prior sessions* (the cache is memory-only and per-session). The unbounded-leak claim is therefore only partially closed by the in-session policy; cross-session accretion needs either a persisted index or a startup sweep, which the plan doesn't mention.
- **Scope/size.** Seven phases, 19 steps, touching nearly every file in the folder plus cross-folder contracts (step 11 row-hint) and embedder sites (Phase 3). The "Phases 1, 2, 6 land in parallel" claim is reasonable for the files, but several phases share `AppSlot.tsx` and the host-hook prop surface, so the parallelism is more constrained than stated. This is a planning risk, not a correctness one.

## Contract and verification assessment

- **Contract clarity: high.** The host-hook contract (`onAppLaunch`/`onAppReady`/`onPermissionDenied`/`onAppError`/`onAppCrash`) is precisely specified with firing conditions and an explicit "optional, side-effect-only, no behavior change for ignorers" guarantee. The `ErrorBoundaryProps` extension (`onError`, `errorInfo`-threaded `fallback` signature) is concrete. The `allowUngatedFallback` prop default and JSDoc requirement are stated.
- **Verification gates: well-chosen but unrunnable here (by constraint), and appropriately framed as "what a follow-up PR must add."** The unit/component/contract/registry/static/app-eval split maps to the invariants. Strong specifics: the race test (stale resolution must not call `setLaunchState`), the once-then-succeeds vs deterministic-thrower retry distinction (correctly tests that remount-on-retry doesn't infinite-loop), and the grep gate for residual `console.log`. The plan correctly references the contracts declaration rollup and app-eval harness gotchas from memory.
- **Gaps in verification:** no gate asserts the *rollback* path of Phase 4 step 10 (partial-table failure tears down the sheet) — the test list covers throw-to-error but not the cleanup-actually-happened assertion. No gate covers Phase 3's embedder flag being set at the real sites (only the AppSlot-level behavior). No test named for step 11's schema-derived placement (the `SheetNotFound`/overlap risk it itself raises). These are the highest-value missing tests given the data-integrity stakes.

## Concrete changes that would raise the rating

1. **Disambiguate `ensureAppTables`**: name the exact symbol and re-export sites to remove (`host/app-setup.ts`, `host/index.ts:29`, `mog/shell/src/index.ts:48`) and explicitly exclude the `types/api/src/apps/api.ts` interface method (678) from scope.
2. **Enumerate the Phase 3 embedder sites** that must set `allowUngatedFallback` (read-only discovery, but list them) so the "same PR" mitigation is actionable, and add a verification gate that those sites compile with the flag set.
3. **Close the cross-session document leak**, not just the in-session one: specify either a persisted document index for app-scoped fresh docs or a startup/`enabled`-transition sweep, since the memory-only cache can't evict prior-session orphans.
4. **Add the two missing high-value tests**: (a) Phase 4 rollback actually deletes the partially-created sheet/tables on mid-loop failure; (b) Phase 4 step 11 schema-derived placement does not overlap and does not regress into the `SheetNotFound` class of issues.
5. **Right-size the parallelism claim**: note that Phases 1/2/3/6 all mutate `AppSlot.tsx` and the props surface, so they are better landed as an ordered set within one or two PRs than truly in parallel.
