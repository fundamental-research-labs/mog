# Review of Plan 001: Improve `mog/contracts/src/api`

## Summary judgment

This is a strong, evidence-grounded plan. Its central diagnosis — that `contracts/src/api` is a thin public facade over `@mog/types-api`, that `contracts/src/api/types.ts` has drifted from the canonical `types/api/src/api/types.ts`, that the package export map exposes only a subset of physical shims, and that public JSDoc contains mangled generated wording — all check out against the actual source. The plan is specification-rich, production-path-aware, and proposes durable governance (inventory, drift checker, conformance check) rather than one-off edits. It loses points for unresolved scope: a key public-policy decision is deferred to an "Integrator," the weak-type hardening work bleeds well beyond the nominal folder into sibling packages, and one documentation recommendation conflicts with a project naming convention.

## Verification performed (read-only)

- **Drift claim confirmed precisely.** `diff` of `contracts/src/api/types.ts` against `types/api/src/api/types.ts` shows the canonical file adds exactly what the plan names: the `WorkbookId`/`WorkbookSessionId`/`DocumentId`/`LinkId`/`ActorId` stable-ID block, chart snapshot exports (`ChartExportOptionsSnapshot`, `ResolvedChartSpecSnapshot`), `WorkbookSettingsPatch`, and the `WorkbookPolicyPreservedEvent` (the `'cells:policy-preserved'` workbook event). The contracts copy is 2616 lines vs 2629 canonical and is the stale side. The plan's risk list is accurate, not speculative.
- **Facade structure confirmed.** `workbook.ts`/`worksheet.ts` are 4-line `export type *` shims; `index.ts` re-exports root types, `export type * from '@mog/types-api/api'`, four local enums from `./types`, three receipt guards, and six format helpers — exactly as described.
- **Export-map subset confirmed.** `contracts/package.json` exposes `./api`, `./api/mutation-receipt`, `./api/worksheet/handles`, and `./api/worksheet/handles/index`, but none of the `workbook/**` or `worksheet/**` deep shims. The "mixed public/development-only subpath policy not enforced by a gate" claim is real.
- **Mangled comments confirmed.** `worksheet/format-mappings.ts` literally contains `spreadsheet special-cell typeAPI` and `spreadsheet special-cell typeuses` — botched automated text replacement, as the plan asserts.
- **README target confirmed.** `types/api/src/api/README.md` exists, so step 9's "update existing README" instruction is grounded.

This level of corroboration is the plan's biggest credibility asset.

## Major strengths

- **Accurate, falsifiable diagnosis.** Nearly every concrete claim is verifiable in the tree and verified true here. Plans that name specific missing symbols rather than gesturing at "drift" are far easier to execute and audit.
- **Production-path framing.** It enumerates real consumers (`kernel/src/api`, `apps/spreadsheet`, `views/sheet-view`, `runtime/sdk`, fixtures) and insists fixes land in implementation code, not declaration-only narrowing or `as any`. The conformance check (step 6) tying public interfaces to kernel implementation classes via ts-morph/compiler API is the right durable guard against future drift.
- **Strong, ordered verification gate list.** 22 gates including typecheck, build, runtime-inventory, declaration-identity, runtime-import, app/kernel tests, API-spec generation, snapshots, and external fixtures. It also honestly states the planning worker did not run them.
- **Distinguishes legitimate from accidental weak types.** Step 4 classifies `any`/`unknown` by intent (extension data, JSON payload, bridge escape hatch) and explicitly refuses to "pretend every payload is statically knowable" — a mature stance that avoids over-tightening schemaless boundaries.
- **Single-source-of-truth invariants** (branded identity owners, no private runtime imports leaking into dist) are precisely the failure modes declaration bundling produces, and they are called out as mandatory gates.

## Major gaps or risks

- **A central decision is left unmade.** Step 3 offers "Option A (root-only)" vs "Option B (broad subpaths)" and defers the choice to the Integrator. This is the single highest-leverage architectural decision in the plan (it changes the public surface area, the fixtures, and the gates), and leaving it open means eight parallel agents cannot safely converge — Agent A's policy checker and Agent H's fixtures depend on an answer that does not yet exist. A reviewer-grade plan should at minimum recommend a default with rationale.
- **Scope bleeds beyond the nominal folder.** The queue scope is `contracts/src/api`, but steps 4, 6, 7, and 10 substantially rewrite `types/api/src/api/**`, kernel implementation classes, `runtime/sdk` generators, and `apps/spreadsheet`. The weak-type audit alone ("slicer position, table style metadata, filter unique values, conditional-format clone rules, pivot readback state, floating-object update payloads…") is a multi-week cross-package effort. This is arguably correct (the folder is just a facade, so the real types live upstream), but the plan does not flag that the bulk of the work is outside the assigned folder, nor scope it down to what is achievable in one slice.
- **Documentation recommendation conflicts with a project convention.** Step 9 proposes replacing mangled phrases with "Office.js", **"Excel-compatible"**, "A1 notation", etc. The repo convention is to avoid referencing "Excel" in source comments (the mangled `spreadsheet special-cell type` text appears to be exactly a botched find-replace away from such a reference). Recommending "Excel-compatible" risks re-introducing the banned term. The plan should either pick a sanctioned phrasing or note the constraint.
- **No definition-of-done or sequencing between parallel agents and gates.** The 8-agent split and the 22-gate sequence are listed independently. Several gates depend on decisions other agents own (e.g. gate 3 "subpath policy checker" depends on the unresolved Option A/B). There is no dependency ordering among agents (B depends on the inventory from A; G depends on B/C/D's final shapes), so the parallelization section reads more like a wish than an executable schedule.
- **Risk of churn in source-scanning tooling is acknowledged but not bounded.** Step 2/risks note that removing copied declarations from `types.ts` "may affect tools that scan physical source files." Which tools, and whether `generate-api-spec.ts` / `generate-api-reference.ts` currently read the stale copy, is asserted but not pinned to a file/line — the one place the plan substitutes hypothesis for the evidence it otherwise supplies.

## Contract and verification assessment

The contract model is sound: one declaration owner per type, runtime values owned locally with no private runtime imports into dist, and a facade that projects rather than copies. The drift checker (step 2) and the implementation conformance check (step 6) together close the two loops that matter — facade-vs-canonical and contract-vs-implementation. The format-helper hardening (step 5) correctly targets currently-undefined behavior: the existing `clampIndent` rounds-then-clamps but says nothing about `NaN`; `officeJsAngleToOoxmlRotation` silently clamps and has no defined response to fractional/negative-OOXML input. Specifying these and testing them through the built package (not source paths) is the right call and matches how the runtime helpers are actually exported from `index.ts`.

Verification coverage is comprehensive and ordered from cheap (typecheck) to expensive (external fixtures, repo-wide typecheck). The insistence on testing through the built public package and on kernel/app tests covering the production path (not just generated specs/mocks) is the correct guard against green-but-wrong outcomes. The main weakness is that three of the listed gates are themselves new tooling to be built (facade inventory checker, subpath policy checker, weak-contract inventory checker, metadata agreement checker) — so the verification plan partly depends on deliverables of the same implementation pass, with no fallback if those checkers slip.

## Concrete changes that would raise the rating

1. **Resolve the subpath policy.** Recommend Option A or B explicitly with rationale (current evidence — only 4 subpaths exported, deep shims unreferenced in the export map — points toward root-only + an import-boundary gate). Make every dependent gate and fixture reference that decision.
2. **Scope the slice honestly.** Either narrow this plan to what is achievable for the `contracts/src/api` facade (inventory, drift removal, helper semantics, JSDoc, export policy) and split the cross-package weak-type/kernel-conformance work into clearly-labeled downstream plans, or add an explicit "this plan owns N packages, here is the staging order" statement.
3. **Add an agent dependency graph.** State which agents block which (A → B; B/C/D → G; all → Integrator) and tie each of the 22 gates to the agent/phase that unblocks it, so the parallelization is executable rather than aspirational.
4. **Fix the documentation recommendation.** Replace "Excel-compatible" with a sanctioned phrase (e.g. "spreadsheet-compatible" / "A1 notation" / "OOXML") and add a note that the mangled comments must not be repaired by re-introducing a banned product name.
5. **Pin the tooling-impact claim to evidence.** Name the exact generators/lines that read the stale `types.ts` copy (e.g. confirm whether `generate-api-spec.ts` resolves through the shim or the physical file today), matching the evidentiary standard the rest of the plan sets.
6. **Add explicit acceptance criteria / definition of done** (e.g. "zero accidental `any` remaining in the public root graph per the new checker; drift checker passes; conformance check green; runtime dist has no `@mog/types-*` import") so completion is objectively testable rather than judgment-based.
