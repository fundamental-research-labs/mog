# Review: 004-contracts-runtime-services-src


## Summary judgment

This is a strong, unusually well-grounded plan. Its diagnosis of `@mog-sdk/runtime-service-contracts` is accurate in every checkable particular: the package really is a private, type-only, six-file surface with no implementation and only a `typecheck` script; `contracts/src/runtime/*` really does carry a byte-for-byte duplicate of the error envelope and audit event; and the production `types/host/src/kernel.ts` handoffs really are materially stronger than the service-contract restatements. The plan correctly identifies the core defect — a weaker, parallel contract restated in plain strings and ISO timestamps that could metastasize if a future service adopts it — and prescribes alignment with the production host/kernel authorization model before that adoption happens. The security posture (fail-closed raw bytes, time-unit consistency, principal-drift caution, redaction invariants) is thoughtful and matches real concerns in the host gate.

The main thing holding it back from a 9–10 is a scope-to-value mismatch: this is a 10-step, 6-agent program of work, including consumer migration and declaration-snapshot tooling, aimed at a package that currently has **zero importers** anywhere in the tree. The plan's own framing ("specification staging area with high leverage") is defensible, but it then overreaches into work whose payoff is deferred and partly speculative, and it defers the single most important architectural decision rather than making it.

## Major strengths

- **Factually accurate against the codebase.** I verified the key claims: the duplicate error envelope/audit event in `contracts/src/runtime/`, the stronger production handoff (`operation`, `decisionId`, `correlationId`, `sessionId`, `nonce`, numeric `expiresAt`, `VerifiedPrincipal`, `resourceContext`, `sourceHostId`, `rawBytesPolicy`, `exportMaterialization`), the ISO-string vs epoch-ms expiry drift, and the existence of every named adjacent file (`identity.ts`, `capabilities.ts`, `host-operation-gate.ts`, the host adapters, `package-inventory.jsonc`, both docs). A plan that survives this level of spot-checking is rare.
- **Security-first contract reasoning.** The raw-byte discriminated-union requirement, the explicit rejection of "empty allowlist means all," the demand that denied decisions cannot carry grant fields, and the insistence that authorization depend on verified provenance rather than display labels are all correct and security-relevant.
- **Concrete, runnable verification gates.** The exact `pnpm --filter` invocations, the runtime-inventory/runtime-imports checks, and `check:publish-readiness:fast` are real and appropriate. The "type-only vs value-bearing" tripwire (add protocol helpers → must update inventory/docs/import-checks) is a genuine, easy-to-miss hazard the plan catches.
- **Clear sequencing and dependency map.** "Canonical source decision lands before broad import rewrites" and "docs updated in the same change as any disposition/value-export change" are the right ordering constraints.
- **Disciplined non-goals.** Explicitly refusing to ship a server, routes, OpenAPI, or to weaken the host gate to fit lighter contracts keeps the reserved/not-shipped boundary intact.

## Major gaps or risks

- **The package has no importers, which undercuts part of the plan.** `rg` finds zero references to `@mog-sdk/runtime-service-contracts` outside its own `dist`. Step E ("consumer migration in shell, runtime SDK, runtime test-host, and kernel tests") therefore migrates consumers that do not currently consume anything — it is net-new adoption disguised as migration. The plan should say plainly that no consumer uses these types today and treat adoption as an explicit, separately-justified decision, not a cleanup.
- **Several "real production path" verification gates don't actually exercise these contracts.** Authorizing an export through `HostDocumentOperationGate`, consuming the nonce, and proving replay rejection validates the *existing host gate* — which already works — not the runtime-service contract. These gates only become meaningful after consumers adopt the new types, which the plan defers. As written, the strongest-sounding behavior gates would pass without proving anything about the package under change.
- **The central ownership decision is deferred, not made.** Step 1 lists options (split package / neutral shard / `contracts/src/runtime`) but does not pick one, even though every later step and the parallelization plan depend on it. For a single-developer codebase the plan could and should commit to a recommendation (with a fallback), rather than leaving the keystone decision open.
- **Scope vs. leverage.** Six parallel agents, declaration-snapshot tooling, negative `@ts-expect-error` fixtures, consumer compile fixtures, and docs across three files is a large investment in a ~11 KB, six-file reserved package nobody imports. The risk of over-engineering a not-yet-used contract is real and is only partially mitigated by the "high leverage" framing.
- **Duplicate-resolution ambiguity.** The plan correctly finds the `contracts/src/runtime/*` duplicate but doesn't establish whether that copy is itself reachable/exported today (it could already be dead or already a shim). Without that, "convert to re-export shim or remove if dead" can't be actioned without re-investigation.

## Contract and verification assessment

Contract clarity is high. The plan names specific weak fields (`ServicePrincipal` as `{ principalId, principalType, tenantId, displayName? }`, `TenantScope` as three optional strings, `expiresAt` as ISO string, `outputFormat`-only export) and maps each to the stronger production counterpart, which I confirmed exists. The proposed `RuntimeServiceHandoffBase` + specialization is sound and mirrors the discriminated host handoff union. The error/audit/diagnostics strengthening (required `correlationId`, decision-vs-operation event phase, structured dependency states, redacted config fingerprints) are all reasonable and testable.

Verification is the plan's strongest dimension on paper but has the gap noted above: typecheck/inventory/import gates are solid and will catch shape/leak regressions, but the behavior gates conflate "host gate works" with "service contract is correct and adopted." The plan should add a gate that actually compiles a consumer against the new types (the step-9 "consumer compile fixtures" hint at this but are gated behind adoption that may never happen in this slice).

## Concrete changes that would raise the rating

1. **State the zero-importer fact up front** and reframe step E from "consumer migration" to "optional pilot adoption," with an explicit decision on whether any consumer should adopt now or whether this slice is alignment-only. This single change resolves most of the value tension.
2. **Make the canonical-ownership decision in the plan**, not at implementation time: recommend one home (e.g. keep runtime-services as canonical and convert `contracts/src/runtime/*` duplicates to re-export shims), with a stated fallback. Everything downstream depends on it.
3. **Split the verification gates** into "shape/leak gates" (valid now: typecheck, inventory, import-checks, `@ts-expect-error` fixtures) and "adoption-dependent behavior gates" (only meaningful after a consumer compiles against the types), so the plan doesn't claim production validation it can't deliver in this slice.
4. **Add a cheap consumer-compile fixture** that imports only the public entrypoint and `satisfies`-checks a handoff built from a real host gate result — this is the smallest gate that actually proves the contract matches production output without a full migration.
5. **Right-size the agent split**: collapse the six agents to ~three (ownership/dedup, host-alignment + fixtures, docs/inventory) given the package's size, and call out explicitly which work is alignment-only vs adoption-gated to avoid building tooling around an unused surface.
6. **Confirm the `contracts/src/runtime/*` duplicate's reachability** as the first investigative step so the "shim or delete" branch is decidable without re-discovery.
