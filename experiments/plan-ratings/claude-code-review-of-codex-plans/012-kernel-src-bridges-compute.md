Rating: 8/10

# Review of Plan 012: Kernel Compute Bridge Boundary Improvements


## Summary judgment

This is a strong, unusually well-grounded plan. Its description of `kernel/src/bridges/compute` matches the actual folder: every file it enumerates exists, every test it cites exists, and its specific factual claims check out against the source. I verified that `compute-bridge.ts` is ~63KB (far past the README's claimed "~200 lines" composition root — README.md:46 and README.md:96 both say `~200 lines`), that `_forceRecomputeRefErrorCells()` and the direct `compute_full_recalc` bypass exist (compute-bridge.ts:1032, :1114, :1132), that 34 direct `transport.call` sites live in the hand-written bridge, that session/principal methods are hand-written with `{ tags }` unwrapping (compute-bridge.ts:1627, :1683), that `manifest.gen.ts` currently only encodes `lifecycle`/`read`/`write` kinds (no `session`/`structural`), and that `sync-mutation-result.test.ts` documents the `rebuild_from_yrs_after_sync` → `MutationResult::empty()` gap (test header lines 7–13). The plan's "improve generation/descriptors, don't add shims" thesis is the architecturally correct one, and the invariant list is the best part of the document — it reads like it was extracted from the code, not invented.

The plan's central weakness is scope and acceptance-criteria precision, not correctness. As written it is a multi-quarter epic spanning Rust compute-core, bridge-ts generation, transport, kernel, SDK, and docs — packaged as one plan. Several objectives are directionally right but lack falsifiable "done" definitions, and the riskiest items (decompose `ComputeCore` with "no observable behavior change," remove TS workarounds only after Rust emits complete results) depend on characterization tests the plan asks for but does not require *first*.

## Major strengths

- **Accurate, evidence-based scoping.** The folder inventory, adjacent-path map, and "current role" section are correct and specific. This is not a generic refactor template; it is clearly written against the real code.
- **Excellent invariant catalogue.** The "Production-path contracts and invariants to preserve or strengthen" section (mutation pipeline ordering, the two-independent-data-planes rule, inclusive→exclusive viewport bound conversion, trap-once semantics, idempotent destroy with docId generation safety, flat-tag principal wire shape) is the kind of contract specification that makes a refactor safe. These are the hardest things to recover after the fact, and the plan captures them up front.
- **Correct root-cause framing.** It repeatedly insists the fix belongs in Rust descriptors / bridge-ts generation / transport metadata rather than in more hand-written TS — and the non-goals reinforce this ("do not hand-edit generated files as the lasting solution," "do not add compatibility wrappers that preserve wrong return shapes"). This is the right architectural instinct for a generated boundary.
- **The contract-inventory-first sequencing is right.** Step 1 (build the method disposition matrix as *test metadata*, fail when a new command has no disposition) is the correct foundation: it turns the boundary from tribal knowledge into an enforced gate and unblocks the parallel agents.
- **Verification gates are concrete and real.** The named focused tests exist; the cargo `bridge-ts` and `compute-api`/`compute-core` gates are plausible; integration/WASM/NAPI/UI tiers are enumerated. The plan honestly states no commands were run (planning-only), which is appropriate.
- **Sync gap is correctly elevated to a production contract.** Treating `sync-mutation-result.test.ts` as a real contract rather than a permanent expected-failure suite, and recognizing the fix likely lives in Rust `rebuild_from_yrs_after_sync`, is exactly right and matches the test's own header comment.

## Major gaps or risks

- **Scope is a single plan in name only.** Thirteen workstreams, seven parallel agents, spanning TS + Rust + codegen + transport + docs. There is no phasing into shippable milestones and no MVP cut. A reviewer cannot tell what "v1 of this plan" delivers versus the full vision. This is the primary reason it is not a 9–10.
- **Acceptance criteria are often qualitative.** "Turn the bridge into an auditable boundary contract," "make generated metadata the source of truth," "promote normalization to enumerated contracts" — these need measurable exit conditions (e.g., "zero un-dispositioned `transport.call` sites outside the matrix," "manifest emits `session`/`structural` kinds and N methods reclassified"). Some steps do imply tests, but the objective-level success bar is fuzzy.
- **"No observable behavior change" decomposition is asserted, not safeguarded.** Step 4 splits `ComputeCore` into 5–6 services. The plan rightly notes (Risks) that "moving logic out of `ComputeCore` can break ordering even if types still compile" — but it does not make a behavior-characterization/golden test suite a *precondition* of the split. As ordered, the decomposition could land before the ordering assertions exist.
- **Rust-before-TS dependency ordering is under-specified.** Removing `_forceRecomputeRefErrorCells` and the sheet-copy `compute_full_recalc` bypass requires the Rust side to first emit complete mutation results + viewport patches. The plan acknowledges this risk but the implementation steps and the Agent A–G split don't encode the hard dependency edge (Agent D's Rust fixes must land and be verified before Agent C/others delete the TS compensations). Without that gate, a regression in UI freshness / undo grouping / provider persistence is likely.
- **Cross-repo ownership boundary.** Much of the highest-value work (sync rebuild, sheet-delete invalidation, sheet-copy recalc) is in `compute/core` (Rust), outside this folder. The plan flags this, but a plan titled for `kernel/src/bridges/compute` that depends on substantial Rust core changes risks ownership/coordination stalls that aren't addressed beyond a dependency list.
- **`ComputeCore` is still exported for trap-recovery tests** (index.ts:54). Step 13 says "keep `ComputeCore` export only for the trap-recovery integration tests... if production needs it, define a narrow interface" — but doesn't resolve whether the decomposition (Step 4) breaks that shell-side test contract. The interaction between Steps 4 and 13 needs to be made explicit.
- **Effort/risk weighting is flat.** Floating-object/chart contract enumeration (Step 10) and the `ComputeCore` decomposition (Step 4) are treated as peers, but they have very different blast radii. No prioritization signal tells an implementer where to spend first.

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. The invariants are concrete enough to write tests against, the return-shape taxonomy (`bytes+mutation`, `id+mutation`, `id+config+mutation`, `multi-value+mutation`, `void`, `bytes`, `subscription`) is well-chosen and matches the manual-override categories actually present in the code, and the principal `{ tags }` envelope contract is correctly captured (verified at compute-bridge.ts:1627–1643). The proposed contract-inventory test, direct-transport-call audit, and `types.ts` leaf import-boundary test are the right enforcement primitives and would prevent the exact silent-regression class the plan describes ("a new method can silently choose the wrong path").

Verification gates are credible: the listed focused tests all exist on disk, and the bridge-ts/cargo/integration tiering is appropriate for a boundary that fans out to WASM/NAPI/Tauri. The main verification gap is ordering: the plan lists *which* tests to run but not *which must be green before which destructive step*. For a "preserve observable behavior" refactor, the gating order (characterization green → decompose → re-green; Rust result complete → remove TS workaround → regression test) is as important as the test list itself, and it is largely left implicit.

## Concrete changes that would raise the rating

1. **Phase the plan into shippable milestones with an explicit MVP.** E.g. M1 = contract inventory + direct-call audit + leaf-import test (pure observability, no behavior change); M2 = generation of non-standard return shapes + session kind; M3 = Rust result/patch completeness + workaround removal; M4 = `ComputeCore` decomposition; M5 = floating-object/chart + docs. State what each milestone independently delivers.
2. **Make a behavior-characterization (golden ordering) suite a hard precondition for Step 4.** Require tests that assert the full mutation-pipeline ordering (patch → mirror/events → validation → derived refresh → provider drain) to be green *before* any `ComputeCore` extraction, and re-green after.
3. **Encode the Rust-before-TS dependency as a gate, not a note.** Tie each TS workaround removal (`_forceRecomputeRefErrorCells`, sheet-copy bypass) to a named, verified Rust mutation-result/viewport-patch contract that must land and pass its regression test first.
4. **Add falsifiable exit criteria per objective.** Quantify: number of `transport.call` sites remaining (and each justified in the matrix), manifest kinds emitted, deep `.gen.ts` import count reduced to N, README line-count/section parity with reality.
5. **Resolve the Step 4 ↔ Step 13 interaction for the `ComputeCore` export.** State whether the shell trap-recovery tests migrate to a narrow `TrapState` interface, and whether that interface ships in M1/M2 so the decomposition doesn't break the shell contract.
6. **Add a rollback/feature-flag story** for the riskiest behavior-affecting changes (viewport refresh policy table, workaround removal), since the plan itself warns "a stale viewport is worse than an extra fetch."
7. **Prioritize the parallel agents.** Mark Agent A (inventory) and Agent D (Rust completeness) as critical-path, and gate C/E/F/G on their outputs explicitly rather than declaring the whole thing "naturally parallelizable after the inventory."
