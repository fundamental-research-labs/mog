Rating: 8/10

# Review of Plan 023 — Compute API Source Boundary Improvements

Source folder: `mog/compute/api/src`
Plan reviewed: `mog-internal/plans/active/experiments/plan-ratings/codex-plans/023-compute-api-src.md`

## Summary judgment

This is a strong, unusually well-grounded plan. Every structural claim it makes about the folder was verifiable against the actual source: the placeholder modules (`workbook/protection.rs`, `workbook/styles.rs`, `sheet/hyperlinks.rs`, `sheet/pivots.rs`, `pure/solver.rs` are all `// TODO` stubs holding only a `Dispatch`), the `bridge_delegate::delegate!` facade in `bridge_service.rs`, the session-local security principal state (`active_principal: ArcSwap<Option<Principal>>`, `principal_pool`, `security_active`), the unchecked `.expect("dispatch type mismatch")` downcast in `dispatch.rs:83`, the repeated `.map(|(_vp, m)| m).map_err(ComputeApiError::from)` mutation-extraction boilerplate (e.g. `sheet/structure.rs:36-49`), the loose `&str`/`serde_json::Value` entrypoints (`set_calculation_mode(mode: &str)`, `can_do_structure_op(operation: &str)`, `outline get_groups(axis: &str)`, `find_cell_value -> Option<serde_json::Value>`), and the native-actor/WASM-`Rc<RefCell>` dispatch split. It also correctly identifies that an audit-style regression harness already exists (`tests/coverage_audit.rs`) and that PyO3 already ships a generated `api_surface.json` with explicit dispositions (statusCounts include omitted/unsupported/python_only). The plan reads like it was written from the code, not from a summary.

Its principal weakness is scope discipline, not correctness. It is closer to a multi-quarter program than a single executable plan, it nominates a "first implementation slice" but never actually carves one, and its objectives over-weight the ergonomic `Workbook`/`Sheet` facade — which the plan itself honestly notes is exercised mainly by `compute/api` tests, not by runtime production callers — relative to the bridge/transport path that production actually flows through.

## Major strengths

- **Evidence-based specification.** Claims are concrete and verifiable: named modules, named descriptor groups (`core_cells`, `core_sync`, `core_theme`, `objects_floating/groups/hyperlinks/z_order`, `screenshot`), named generated artifacts (`compute-bridge.gen.ts`, `manifest.gen.ts`, `command-metadata.gen.ts`), and named TS consumers (`infra/transport/src/factory.ts`, `napi-transport.ts`, `kernel/src/bridges/compute/compute-bridge.ts`). This is what makes the plan auditable rather than aspirational.
- **Excellent invariants/contracts section.** "Production-path contracts and invariants to preserve or strengthen" is the best part of the plan: single-FFI-surface rule, cheap-clone handles sharing one engine, native-serialize/WASM-synchronous split, `compute-core` default features off except via `native`, `ComputeApiError::Compute` flattening the inner wire shape, principal never serialized, no in-band sentinels for clear/null. These are real, testable invariants and they match the code.
- **Builds on existing patterns rather than inventing.** Objective 1 (contract matrix) is a natural generalization of the existing `coverage_audit.rs` regression anchor; objective 2's PyO3 disposition contract dovetails with the already-generated `api_surface.json` dispositions. The plan respects the macro/delegate architecture instead of proposing to hand-write parallel surfaces (explicitly a non-goal).
- **Honest about reality.** It states out loud that the ergonomic facade is test-covered rather than production-covered, that placeholder removal can break downstream imports, and that tightening typed inputs will surface callers needing migration in the same change set. The risks/non-goals section is mature.
- **Realistic parallelization.** Agents A–F map cleanly onto the inventory-first-then-fan-out shape, and the cross-folder dependency list (compute-core, rust-bridge, wasm/napi/pyo3, kernel, types/api) is accurate.

## Major gaps or risks

- **No carved first slice.** The plan says workbook-rewrites and shim-adding are "out of scope for the first implementation slice," yet the 13-step implementation section is the whole program with no MVP boundary, no "land this PR first," and no per-step definition-of-done. As written, an implementer cannot tell where slice 1 ends. This is the single biggest thing holding the rating below 9.
- **Production-path weighting.** Objectives 2 and 3 (complete the ergonomic facade, typed inputs into `Workbook`/`Sheet`) consume most of the implementation budget, but the plan itself concedes that facade is not on the runtime hot path. The highest-leverage, genuinely production-facing items — descriptor parity (step 2), generated-artifact freshness gates (step 12), and consistent security-error promotion across bindings — are buried mid-list rather than prioritized first. A reader optimizing for production impact would reorder.
- **Unbounded "completeness" objective.** "Implement every engine-supported workbook/sheet operation" depends on the compute-core surface, which the plan does not enumerate and which is out of scope to change. The placeholders it wants to close (protection/styles/hyperlinks/pivots) are documented in-code as *engine-not-yet-supported*; closing them is therefore gated on compute-core work the plan explicitly excludes. The plan half-reconciles this ("remove or replace placeholders whose engine support already exists") but leaves the actual gating ambiguous — risk of either scope creep or no-op steps.
- **Deferred behavior decisions.** Several steps say "decide X" rather than deciding: reversed-range normalize-vs-reject (step 7), whether dates/times cross this layer (step 9). A plan of this quality should pre-commit to the call or state the decision criterion, since these are observable contract changes.
- **Placeholder behavior-change risk under-quantified.** Numeric out-of-bounds rejection (step 7 / risks) is called out as a behavior change, but there is no enumeration of which current callers rely on passthrough, so the "migrate callers in the same change set" promise has no work-list behind it.

## Contract and verification assessment

Contract clarity is the plan's strongest dimension: the invariants section gives crisp, testable preconditions, and the proposed contract tests (matrix coverage, facade coverage, descriptor parity, artifact freshness, error-wire shape, address/range and SDK-value property tests) map directly onto stated invariants. The error-wire test in particular is well-targeted given `ComputeApiError::Compute` flattening and the camelCase bridge fields in `error.rs`.

Verification gates are comprehensive and correctly tiered (Rust → bridge/binding → TS → contract). The two weaknesses: (1) several commands are placeholders rather than exact invocations — "the repo's configured `target-wasm` path," "the repo's bridge generation command" — which is precisely the gap the freshness gate (step 12) needs nailed down to function; the plan should name the actual generation command. (2) The full gate matrix (cargo + clippy + no-default-features + compute-core + wasm + napi + pyo3 + kernel + pnpm typecheck) is not runnable per-slice; without a slice boundary, "which gates gate which PR" is undefined. The contract tests are also themselves to-be-written, so "done = tests pass" is partly circular until the matrix exists.

## Concrete changes that would raise the rating

1. **Carve and freeze slice 1.** Pick the smallest production-valuable cut — recommended: (a) the contract matrix + audit test from step 1 (extending `coverage_audit.rs`), (b) descriptor parity disposition for PyO3 (step 2) since the disposition machinery already exists, and (c) the dispatch downcast hardening (step 8). Define done-criteria and the exact gate set for that PR. Defer facade completion and typed-input migration to explicitly numbered later slices.
2. **Reorder objectives by production leverage.** Promote descriptor parity, freshness gates, and security-error-promotion consistency above ergonomic-facade completion, and state that the facade work is SDK-ergonomics/test value, not runtime-path value (the plan already knows this — make the prioritization match).
3. **Name the generation command and gate commands exactly.** Replace "the repo's bridge generation command" / "target-wasm path" with literal invocations, so step 12's freshness gate is implementable as specified.
4. **Resolve the deferred decisions.** Pre-commit to reversed-range behavior (recommend reject-to-match-A1 with a normalize helper) and to whether dates/times cross this layer, with the rationale.
5. **Bound the "completeness" objective against a real engine-method list.** Reference or attach the compute-core bridge-descriptor enumeration the facade is being measured against, and split "engine supports it today" from "needs compute-core work (out of scope)" so placeholder steps don't silently expand into engine work.
6. **Attach a caller work-list for behavior changes.** Enumerate the current callers of out-of-bounds numeric coordinates and loose string/JSON entrypoints that the typed-input and bounds-validation steps will break, so the "migrate in the same change set" commitment has concrete targets.
