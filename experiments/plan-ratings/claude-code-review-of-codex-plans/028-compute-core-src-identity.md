Rating: 8/10

# Review of Plan 028 — compute-core `src/identity`

## Summary judgment

This is one of the stronger plans in the set. It correctly recognizes that the requested folder (`mog/compute/core/src/identity`) is a near-empty public facade — `mod.rs` exports `GridIndex`, `grid_index.rs` re-exports `compute_document::identity::*` — and that the real production work lives in `compute/core/crates/compute-document/src/identity/` plus the compute-core storage/mirror/sync/structural call sites. It then writes the plan against the production path rather than padding the facade. Every concrete code claim I spot-checked is true of the current tree, which gives the plan unusual credibility: the dense-materialization problem in `axis_insert_explicit`, the debug-only permutation bijection checks in `sort_rows`/`reorder_row_ids`, the `AxisIdentityStore::{Explicit, Runs}` split, lazy `ensure_cell_id`, virtual `CellId` derivation with `is_virtual()`/`ensure_past` guarding, and the `posToId`/`idToPos` codecs scattered across storage are all real. The proposed contract types (`GridIndexError`, `GridIndexValidationReport`, `validate()`) genuinely do not exist yet, so this is net-new hardening, not a re-description of existing code.

The plan loses points because its two highest-risk items — the new error/validation contract surface and the "mixed compact/explicit" axis representation — are named but not specified to the level where an implementer could build them without re-deriving the design, and because the verification gates degrade to generic `cargo test`/`clippy` plus deferred "once the touched call sites are known" integration gates.

## Major strengths

- **Accurate folder/scope diagnosis.** Correctly identifies the facade-vs-implementation split and redirects effort to `compute-document` + integration paths. Verified: `src/identity/mod.rs` and `grid_index.rs` are exactly the thin re-export described.
- **Invariants are precise and correct.** The "contracts to preserve" list maps onto real engine behavior: `CellId` is identity not position; empty cells need no `CellId` (matches lazy `ensure_cell_id`); `cell_at_pos`/`cell_to_pos` must be exact inverses; `posToId` is the CRDT winner and `idToPos` a derived mirror; virtual IDs must stay disjoint from allocator output and not advance the high-water mark (matches the `is_virtual()` guard in `register_cell`).
- **Correct identification of the worst current wart.** `axis_insert_explicit` (in `axis_mutations.rs`) does exactly what the plan says — on a `Runs` store it materializes the entire axis into an `Explicit` `Vec` on a single insert. Targeting compact-preserving structural ops is the right high-value objective.
- **Correctly flags release-vs-debug validation gap.** `sort_rows`/`reorder_row_ids` only check bijection under `#[cfg(debug_assertions)]`; the proposed validated `RowPermutation` value that proves bounds in release and leaves state unchanged on failure is a real improvement, and the plan correctly keeps the two sort semantics distinct (per-cell remap vs `rowOrder`-driven identity reorder).
- **Production-relevant verification breadth.** Test matrix names the right hydration adversarial inputs (stale `idToPos`, duplicate `posToId`, invalid hex, compact + legacy + mixed axes) and the right integration scenarios (writes beyond bounds, undo/redo, remote sync replay, range sort, relocate/cut-paste, XLSX roundtrip).
- **Honest non-goals and risk framing.** Explicitly refuses to change public A1/numeric APIs, rewrite the TS facade, or add test-only identity paths, and flags that fail-fast may break documents relying on silent repair.

## Major gaps or risks

- **Contract types are under-specified.** `GridIndexError`, `GridIndexInvariant`, and `GridIndexValidationReport` are named with a checklist of checks, but the actual error payload shape is not defined: which variants, what fields they carry (sheet/key/position/reason), and crucially *how* they map onto the existing `ComputeError` (the plan says "map to `ComputeError`" without locating that type or its variant conventions). For a contract-hardening plan, the contract itself is the deliverable and should be sketched as a type signature.
- **The "mixed compact/explicit" axis representation is the riskiest item and the least specified.** A `CompactAxisIdentityStore` with run segments already exists. The plan proposes a third representation that holds compact runs *plus* explicit inserted segments, with new persistence under `gridIndex/rowAxis`/`colAxis`, but gives no serialization schema, no statement of whether this is a new enum variant or an extension of `CompactAxisIdentityStore`'s segment model, and no migration story for already-persisted compact documents. This is where the project will actually get hard, and it is hand-waved.
- **Internal tension on the production assertion point.** Item 5 adds "a single production assertion point after structural rebuilds" that cross-checks `GridIndex`, `CellMirror`, layout index, and formula references, while the risk section says validation must not live in hot paths. On a multi-million-row sheet a full cross-structure check after every structural commit could be expensive; the plan does not bound its cost or say what runs in release vs debug.
- **Scope is very broad and not staged for incremental delivery.** The work spans `cell-types`, `compute-document`, storage, mirror, scheduler, parser, and graph. The parallelization section splits by *agent role* but not into independently-mergeable milestones, so there is no clear "ship error contract first, axes later" sequence — raising the chance of a long-lived mega-branch.
- **Verification gates are weak relative to the plan's ambition.** `cargo test -p compute-document` / `cargo clippy` are fine as a floor, but the integration gates are deferred to "once the touched call sites are known," and no concrete test names, assertion locations, or pass/fail criteria are given. Property-test invariants are described in prose but not as concrete properties.

## Contract and verification assessment

The invariant catalogue is the plan's best asset and is materially correct — it reads like it was written from the code, not from a template. The verification *intent* is good (adversarial hydration inputs, post-rebuild consistency assertion, property tests over operation sequences). The weakness is the gap between intent and executable specification: the new error/report types lack signatures, the new axis representation lacks a schema, and the gates are generic. An implementer could start confidently on items 1, 2, 4, 6, and 7, but would have to design item 3 (mixed axes + persistence) largely from scratch. Because item 3 is also the migration-and-serialization risk, that under-specification is the main thing holding the rating below 9.

## Concrete changes that would raise the rating

1. **Sketch the contract types.** Give a Rust signature for `GridIndexError` (its variants and the sheet/key/position/reason payloads), `GridIndexValidationReport`, and the `validate()` return type. State the exact `ComputeError` variant the error maps to and where that mapping lives.
2. **Specify the mixed-axis representation.** Say whether it is a new `AxisIdentityStore` variant or an extension of `CompactAxisIdentityStore` segments, give the `gridIndex/rowAxis`/`colAxis` serialization layout, and add an explicit forward/backward compatibility statement for already-persisted compact and legacy-dense documents (with the migration test it implies).
3. **Resolve the assertion-cost tension.** State which validation runs in release vs debug, bound the post-rebuild assertion to affected sheets/ranges, and quantify or cap its cost so it cannot regress structural-op latency on large sheets.
4. **Stage delivery into mergeable milestones.** E.g. M1: error contract + `validate()` + map to `ComputeError` (no behavior change); M2: validated `RowPermutation`; M3: centralized `GridPositionKey` codec + hydration repair/report; M4: compact-preserving axis ops + persistence. Each shippable on its own.
5. **Make the gates concrete.** Name the specific integration tests (sync replay, range structural, relocate viewport patches, XLSX roundtrip), state the property-test invariants as explicit properties (map inversion holds, no cell outside current axes, allocator high-water never advanced by virtual IDs), and define the post-rebuild consistency assertion as a named gate rather than a future TODO.
6. **Locate the call-site blast radius now.** The plan defers enumerating fallible-signature call sites; a concrete inventory (the ~40 files touching `posToId`/`idToPos` are already discoverable by grep) would de-risk the "mutator signatures return errors" change and tighten the estimate.
