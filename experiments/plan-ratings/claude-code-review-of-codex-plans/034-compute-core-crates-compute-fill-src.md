Rating: 8/10

# Review of 034 — Compute Fill Source Improvement Plan

Reviewed against the live source at `mog/compute/core/crates/compute-fill/src` and adjacent production paths (`mutation_handlers/fill.rs`, `engine_types/fill.rs`, `contracts/src/fill`, and the spreadsheet app flash-fill code). Multiple non-trivial claims in the plan were checked directly against code.

## Summary judgment

This is a strong, unusually well-grounded plan. Its "current role" and "observations from source inspection" sections are not hand-waving — almost every specific architectural claim I spot-checked is correct, including the subtle ones. It correctly preserves the crate's purity boundary, proposes a sensible work breakdown, and pairs the work with concrete, named verification gates that run through the real production mutation path rather than crate-local mocks. The two things holding it back from a 9 are (a) scope that is realistically a multi-quarter program presented as one plan, and (b) a recurring "decide whether X or Y" pattern that defers many of the actual contract decisions instead of specifying them — which slightly undercuts its claim to be a contract specification.

## Major strengths

- **Source accuracy is excellent and verifiable.** I confirmed:
  - `SourceCell` carries `format: Option<CellFormat>` and `ref_positions`, yet `detect_fill_pattern(values: &[CellValue], ...)` (`patterns/mod.rs`) receives values only — exactly the "value-only inference" gap the plan calls out (objective 2, step 5).
  - `FillResult.detected_pattern` is a single `FillPattern`; there is no per-lane exposure (`types.rs:153`), matching the "first-lane summary" observation (step 9).
  - The `RectRange` finding is real and well-spotted: `IdentityFormulaRef` *does* have a `RectRange` variant, storage's `resolve_identity_ref_to_fill_position` maps it to `RefPosition::Range`, but `shapes::adjust_single_ref` has **no** `(RectRange, Range)` arm and falls through to `mismatch_adjusted_ref(...) { out_of_bounds: true }`. So rect-range fills are silently flagged out-of-bounds today (step 10).
  - Storage placeholder positions are real: `FullRow => RefPosition::FullRow { row: 0 }` and `FullCol => RefPosition::FullCol { col: 0 }` in `fill.rs` (step 10, and the "combined contract" risk).
  - Flash fill is genuinely duplicated and drifting: `computeBridge.flashFill` exists in the generated bridge but the app uses a local TS `detectFlashFillPattern()` path and never calls the Rust bridge (step 14).
  - The lane-order concern is accurate: `group_value_cells_by_lane` sorts lane *keys* via `BTreeMap` but pushes values within a lane in `source_cells` iteration order, so within-lane output is sensitive to caller vector order (step 3).
- **Purity discipline is correct and explicit.** The invariants ("must not read storage, allocate cell IDs, mutate mirrors, or depend on mog-internal"; "result must not depend on source_cells order") match the crate's actual design intent and are the right things to lock down.
- **Verification gates are specific and production-aimed.** Naming `cargo test -p compute-core auto_fill`, `auto_fill_integration`, `auto_fill_formula_regression`, bridge freshness gates for `compute-bridge.gen.ts`/`compute-types.gen.ts`, and insisting flash-fill be exercised through real keyboard/mouse/clipboard E2E paths shows the author understands where regressions actually surface.
- **Edge-case catalogue is thorough and concrete** (month-end seeds Jan31/Feb29, 1900 leap-year convention, midnight rollover, overlapping source/target, asymmetric cross-axis tiling, non-finite results → `CellValue::Error`).
- **Non-goals are disciplined** — explicitly forbids parallel app/kernel autofill engines, compatibility shims for known-wrong behavior, and `mog`→`mog-internal` dependencies.

## Major gaps or risks

- **Scope is far beyond a single executable plan.** Seventeen implementation steps spanning the pure crate, storage reference resolution, bridge/contract types, kernel request shaping, and a TS/Rust flash-fill unification (plus a from-scratch ranked-program synthesis engine) is a program of work, not a plan. The 6-agent parallel split helps, but there is no phasing/milestone ordering — e.g. which subset constitutes a safe, shippable first increment. The "contract matrix first" gate is the only sequencing anchor.
- **Many contract decisions are deferred, not made.** The plan repeatedly says "decide whether" rather than specifying the answer: `filled_cell_count` meaning (step 13, invariant), whether `Values`/`WithoutFormats` include formulas, month-end anchoring semantics (step 7), `Series` with `step_value = 0` behavior, `LinearTrend`/`GrowthTrend` on non-numeric sources. For a document whose stated goal is "a complete, testable contract," leaving the headline semantic choices open means the spec is really a high-quality investigation + decision agenda. Strong, but a true contract spec would commit to the spreadsheet-correct answer for each.
- **Step 15 (ranked flash-fill program search) is under-specified relative to its difficulty.** Multi-token extraction, case/substring transforms, delimiter insertion, date/phone reformatting, multi-column combination, Unicode tokenization, ranked candidates with confidence and tie-breaking is a research-grade subsystem compressed into one bullet. It needs its own plan; folding it in inflates risk and obscures the cost.
- **No per-step acceptance criteria / definition-of-done.** Gates are listed crate-wide at the end rather than mapped to each step, so it's hard to tell when any individual step is complete.
- **Existing-test churn is acknowledged but not enumerated.** The plan flags that directional-anchoring and date-detection fixes "can reveal existing tests that assert wrong behavior," but does not inventory which current tests (`engine_tests.rs`, `series/tests.rs`, `patterns/tests.rs`, `formula_adjust/tests.rs`) would need to change, leaving a real "how much will break" unknown.
- **The external-ref contract is vague.** `IdentityFormulaRef` also has `ExternalCell`/`ExternalRange`/`ExternalName` variants; the plan says external refs should be "preserved/fail-closed by contract" but doesn't define what `AdjustedRef` should carry for them (today they hit the silent `out_of_bounds: true` mismatch arm). This deserves an explicit variant or warning kind, not a phrase.

## Contract and verification assessment

- **Contract clarity (invariants):** Very good *as invariants*, weaker *as decisions*. The "contracts and invariants to preserve or strengthen" section is the best part of the plan — coordinate-driven semantics, merge/hidden-cell skipping, cross-axis tiling asymmetry, finite-value enforcement, and no-`#REF!` registration are all the right contracts. But several invariants are conditional ("either count X or count Y; the selected meaning must be reflected…"), so they constrain rather than define.
- **Type/data-contract realism:** The plan correctly identifies that exposing per-lane patterns and richer diagnostics requires coordinated changes across `FillResult`/`FillResultSummary` (`types.rs`), `contracts/src/fill`, the generated bridge, and kernel callers, and warns against private-only summary shapes TS can't consume. That cross-layer awareness is accurate and rare.
- **Verification gates:** Strong and concrete (named cargo test filters, clippy on both crates, bridge freshness, typecheck, real E2E for fill handle/flash fill, bounded large-fill perf tests, conformance fixtures shared between TS preview and Rust commit). Main weakness is the lack of step-level mapping and no stated baseline (current pass counts / which gates exist today vs. must be created).
- **Risk handling:** Genuinely good. It calls out behavior-change risk from format-aware date detection, the pure-crate↔storage coupling for rect/full-row-col refs, and flash-fill preview/commit drift — all real.

## Concrete changes that would raise the rating

1. **Commit to the deferred decisions.** Replace each "decide whether" with the chosen spreadsheet-correct contract (e.g. state the `filled_cell_count` definition, whether `Values` includes formulas, the month-end anchoring rule, `step_value = 0` behavior) and reference the conformance behavior. This converts the document from a decision agenda into the contract it claims to be.
2. **Phase the work into shippable increments.** Add an explicit ordering with milestones, e.g. M1 = contract matrix + coordinate-stable lanes + RectRange/full-row-col fix (highest-value, lowest-risk correctness wins) before M2 = cell-aware detection (behavior-changing) and M3 = flash-fill unification/rewrite. State the safe first PR.
3. **Split out flash fill (steps 14–15) into its own plan.** The ranked-program synthesis engine is a separate effort; keep step 14 (unify ownership + shared conformance fixtures) here and reference a dedicated plan for the synthesis upgrade.
4. **Add per-step done-criteria and map gates to steps.** Each step should name the specific new/changed tests that prove it, so completion is checkable.
5. **Inventory the existing tests that will change.** List the current assertions in `engine_tests.rs`, `series/tests.rs`, `patterns/tests.rs`, and `formula_adjust/tests.rs` expected to flip under directional-anchoring and date-detection fixes, to bound the churn.
6. **Specify the external-ref adjustment contract.** Define exactly what `AdjustedRef`/warning is emitted for `ExternalCell`/`ExternalRange`/`ExternalName` and for the `(RectRange, Range)` arm, rather than relying on the silent `out_of_bounds: true` fallback.
