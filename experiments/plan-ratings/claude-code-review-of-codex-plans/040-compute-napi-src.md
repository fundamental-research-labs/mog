Rating: 8/10

# Review of Plan 040: Compute N-API Source Improvements

Source folder reviewed: `mog/compute/napi/src` (`lib.rs`, `coordinator.rs`, `chart_render.rs`) plus the adjacent transport, SDK, codegen, and packaging surfaces the plan names.

## Summary judgment

This is a strong, evidence-grounded plan. I independently verified its most load-bearing claims against the live source and the surrounding infrastructure, and they hold up — including the parts that are easy to get wrong. The plan correctly identifies real defects (not speculative ones), proposes contract-first fixes rather than shims, names the actual production consumers, and supplies a concrete, layered verification gate set with a sensible integration order. It is materially better than a generic "improve this folder" plan because it reasons from the bridge/descriptor architecture that actually generates these bindings.

The main reasons it is not a 9–10: the scope is very large for a single "folder" plan (six parallel agents touching compute-api, compute-core clock plumbing, transport, SDK, codegen, and the publish workflow), several objectives are substantial cross-cutting refactors that are under-de-risked, and there are a couple of small naming inaccuracies that a reader could trip on.

## Major strengths

- **The lifecycle accessor bug is real and precisely diagnosed.** I confirmed the split the plan calls out: constructor-created engines are read via `engine.compute_take_init_result()` (smoke-test.mjs lines 126/131/195/217 and napi-transport.ts lines 352/413), while `initFromYrsState`-created engines are read via `takeLifecycleResult()` (napi-transport.ts line 433, with an in-code comment acknowledging the divergence). Objective 4 / step 5 target a genuine, shipping inconsistency, and the plan is right that mocks can pass while the real `.node` fails — hence its insistence on snapshotting against the built binary.
- **The serde-metadata claim is accurate.** `infra/transport/src/napi-transport.ts` does merge a generated map (`NAPI_SERDE_PARAM_INDICES` from `command-metadata.gen.ts`) with a hand-maintained override map (`getNapiSerdeIndices`, lines 163–166), and the manual entries match the plan's examples (security policy IDs `compute_wb_security_*`, `text_to_columns`, etc., with comments explaining the UUID-as-exponent hazard). Objective 3 / step 4 fixes a concrete maintenance liability, and the plan correctly notes this likely requires bridge-IR changes, not just a TS edit.
- **The prelude recommendation aligns with documented intent.** `compute/core/src/bridge_types.rs` already exists and its own header says downstream consumers ("bridge-delegate, bridge-wasm, bridge-napi") should import `compute_core::bridge_types::*` as the single source of truth — yet `napi/src/lib.rs` instead hand-maintains a dozen `use compute_*::...` lines (lines 22–54). Objective 2 / step 3 is not invented; it is finishing a migration the codebase already started.
- **Coordinator risk is correctly characterized.** `coordinator.rs` does use a `LazyLock<Mutex<Vec<Option<Mutex<SyncCoordinator>>>>>` with `u32` indices and slot reuse (`insert_coordinator` reuses `None` slots), so a stale handle can address a freshly created room after dispose. The fail-closed/ownership redesign in objective 6 / step 8 addresses a real correctness hazard for long-lived Node processes.
- **The hand-written wire contracts are accurately described.** `coordinator_join`/`push`/`active_locks` assemble JSON strings with `.unwrap()` (lines 142, 175, 184, 335), and `chart_render.rs` takes a `request_json: String`. Objectives 7/9/11 (typed `#[napi(object)]` returns, versioned chart request) target what the code actually does.
- **Verification gates are layered and target the real failure surface.** Rust/codegen, native-addon/Node behavior, TS transport/SDK, and packaging gates are separated, and the "behavior gates that should be added" list (ABI freshness, descriptor parity, serde completeness, lifecycle parity, clock isolation, stale-handle safety, chart goldens) maps one-to-one to the objectives. The smoke test today has zero coverage of coordinator/chart/awareness/locks, so objective 9 fills a verified gap.
- **Sequencing is coherent:** land the failing ABI/metadata contract first so downstream agents inherit a shared gate, stabilize lifecycle/serde before coordinator/chart SDK changes, then land typed APIs behind the snapshot. The parallelization split assigns clean, mostly non-overlapping ownership.

## Major gaps or risks

- **Scope is large enough to threaten shippability as one unit.** Objective 5 (move clock injection onto the compute actor thread) and objective 6 (coordinator class redesign) are each plausibly their own plan. The plan acknowledges the clock change "may require changes outside `compute/napi/src`," but does not de-risk it: there is no investigation of whether `ComputeService` dispatch currently exposes a pre-recalc hook, nor a fallback if session-scoped clock injection on the actor thread is not cleanly threadable. A reader cannot tell how big step 7 actually is.
- **Minor naming inaccuracy.** The plan repeatedly calls the manual override "the `DEFAULT_NAPI_SERDE_PARAMS` override list," but the live symbol is an exported map `n: NapiSerdeParamMap` consumed via `getNapiSerdeIndices`. The substance is right; the name will not grep. Likewise "lifecycle-result accessor" is described as `take_lifecycle_result` at the Rust level, but the constructor path is exported as `compute_take_init_result` — the plan treats the rename as a clean normalization without flagging that it is an observable ABI break requiring a coordinated SDK/transport/API-snapshot bump (it does say this generally, but not tied to the specific current name).
- **ABI snapshot brittleness is unaddressed.** Snapshotting `Object.getOwnPropertyNames(prototype)` against the built `.node` is the right idea, but napi-rs version bumps and added bridge methods will churn the snapshot constantly. The plan does not say how the snapshot is regenerated, who owns updates, or how to distinguish an intended ABI change from an accidental one beyond "it fails."
- **Step 6 (retire `yrs_state_to_snapshot_json`) is presented as near-trivial but is a behavior change.** The helper builds a snapshot that "shares the same CellIds and Yrs document origin." Switching collaboration boot to `initFromYrsState` must preserve that origin-sharing guarantee; the plan asserts the lifecycle path "preserves the lifecycle contract directly" without evidence that CellId/origin parity is maintained. That is the kind of subtle regression that passes smoke tests and breaks collaboration convergence.
- **"Strengthen native binary integrity" (objective 8) leans on tools the plan does not inspect.** It references `check:binary-wrapper-surfaces`, `build-public-artifacts`, and `check:private-leaks` as integration points but does not state what those scripts currently assert, so it is unclear how much new work objective 8 actually is versus wiring into existing checks.

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. It enumerates the invariants that must hold at the binding boundary — single `ComputeService` engine surface, one one-shot `RecalcResult` accessor shared by both boot paths, the `[serde]`/`[str]`/`[parse]`/`[prim]`/`[bytes]` parameter tag contract, the packed byte-tuple framing (4-byte LE length + bytes + UTF-8 JSON), per-instance principal state, fail-closed coordinator ownership, and versioned chart-mark validation. These are specific and testable, not aspirational, and each ties to a named gate.

The verification gates are credible: real cargo crates (`bridge-napi`, `bridge-ts`'s `verify_up_to_date`, `compute-core-napi`, `compute-coordinator`, `compute-chart-render`, `compute-api`), real Node gates run against the built addon (not mocks), and packaging gates plus a publish-matrix dry run for the seven triples I confirmed exist under `compute/napi/npm/`. The explicit "must run through the built `.node` and the SDK loader path, not the smoke/mock path" guidance is exactly the lesson the lifecycle bug teaches.

Two gate-level gaps: (1) there is no gate proving CellId/Yrs-origin parity for the `initFromYrsState` migration (step 6), which is the riskiest silent regression; and (2) the ABI-snapshot gate has no stated maintenance/regeneration contract, so it risks becoming a rubber-stamp `--update` habit that defeats its purpose.

## Concrete changes that would raise the rating

1. **Split or stage the two heavy refactors.** Promote clock-on-actor-thread (step 7) and coordinator class redesign (step 8) to clearly optional later slices, or carve them into their own plans, so the core contract work (ABI snapshot, prelude, serde metadata, lifecycle unification) can land and ship independently. State which subset is the minimum shippable increment.
2. **De-risk step 7 with a concrete dispatch investigation.** Before committing, confirm whether `compute-api` dispatch exposes (or can cheaply expose) a pre-recalc hook on the actor thread, and specify the fallback if session-scoped injection is not cleanly threadable. Cite `compute/api/src/dispatch.rs` findings rather than asserting feasibility.
3. **Fix the symbol names.** Replace `DEFAULT_NAPI_SERDE_PARAMS` with the actual `n` / `NapiSerdeParamMap` + `getNapiSerdeIndices` merge in napi-transport.ts, and state explicitly that renaming `compute_take_init_result` → `takeLifecycleResult` is an ABI break requiring synchronized updates to transport, SDK mocks, smoke-test, README, and `check:api-snapshots`.
4. **Add a CellId/origin-parity gate for step 6.** Require a Node test proving that an engine booted via `initFromYrsState` shares CellIds/Yrs origin with the source engine identically to the old `yrs_state_to_snapshot_json` → `new ComputeEngine(json)` path, before removing the helper.
5. **Specify ABI-snapshot maintenance.** Document the regeneration command, who reviews snapshot diffs, and how an intended export change is distinguished from an accidental one (e.g., snapshot diff must accompany a descriptor or hand-export change in the same commit).
6. **Quote the existing packaging checks.** Summarize what `check:binary-wrapper-surfaces`, `check:private-leaks`, and `build-public-artifacts` already assert so objective 8's incremental work is scoped, rather than implied.
