# Review of 041 — Canvas Grid Renderer Source Improvement Plan

## Summary judgment

This is a strong, unusually well-grounded plan. I verified its central claims against the live source and almost all of them hold: the hardcoded header dimensions in the grid hit-test registration, the divergent dual hit-test path, the sticky-`undefined` update semantics, the per-render (not per-frame) frame-state clears, the "MVP/Future" scrollbar approximations, and the eslint coordinate-boundary invariant are all real and correctly located. The plan correctly identifies the highest-value defects — geometry/hit-test truthfulness and contract duplication — and separates them from cleanup. It is held back from a 9–10 by (a) one materially stale premise about the cells-layer decomposition, (b) scope that is really a multi-quarter program packaged as a single plan, and (c) verification gates that are thorough on correctness but soft on quantified performance acceptance criteria.

## Major strengths

- **Evidence-accurate problem statements.** Confirmed in source:
  - `rowHeaderWidth: 50` / `colHeaderHeight: 24` hardcoded at `grid-canvas/src/renderer/grid-renderer.ts:1078-1079`, exactly as objective 2 claims.
  - `hit-test/grid-hit-test.ts` subtracts fixed `rowHeaderWidth`/`colHeaderHeight` and reads a dimension provider (`dim.getColLeft`, etc.) — the linear-scan / fixed-offset concern is real (lines 112–462).
  - Two coordinate stacks coexist: `CoordinateSystemImpl.classifyPoint` (`coordinates/coordinate-system.ts:788`) plus `documentToViewport`/`documentToLayerViewport`/`rangeToViewport`, with a documented historical "double offset" bug, while `GridRendererImpl.hitTest()` (`grid-canvas/.../grid-renderer.ts:2516`) tries `engine.hitTest` first and then falls back to `coords.classifyPoint` at line 2561 — precisely the divergent fallback objective 3 targets.
  - `CellsLayer.render()` calls `clippedCells.clear()` / `interactiveElements?.clear()` (`layers/cells.ts:342,346`) inside the per-region render entry point, and config updates use `if (config.interactiveElements !== undefined)` (line 270) — confirming both the per-region-clear bug (objective 3) and the sticky-`undefined` bug (objective 4).
  - `viewports/viewport.ts` carries the "For MVP, assume all rows use default height / Future: ..." comments (lines 183–192) and `calculateViewport`/`pixelToCell` exports (objective 9).
  - The eslint coordinate invariant exists (`grid-renderer/.eslintrc.cjs`) and forbids direct `scrollOffset.x|y` / `viewportOrigin.x|y` reads — objective 10's enforcement claim is real.
- **Strong contract framing.** The "contracts and invariants to preserve or strengthen" section is the best part: dependency direction, region-local vs canvas-absolute coordinates, binary buffer as single source of truth, doc-space-vs-canvas-space dirty rects, and shared handle geometry/tolerance constants are all stated as testable invariants, not vibes.
- **Recommendations build on existing primitives.** `hitTestLayout` and `canvasToCell` over `ViewportLayout` already exist in `viewports/hit-testing.ts` (lines 115, 250), so objective 2's "route through layout-aware hit testing" is a consolidation onto real helpers, not greenfield invention. This makes the plan credible.
- **Good sequencing and parallelization.** The dependency notes (geometry contract first; engine frame-lifecycle hook before `CellsLayer` aggregation; optional-reader clearing landing early to remove ambiguity; painter decomposition last) are coherent and match the actual coupling.
- **Explicit non-goals** (no engine replacement, no dimension-provider fallback reintroduction, no flag-gated wrong geometry, no `mog → mog-internal` deps) sharply bound scope creep where it matters.

## Major gaps or risks

- **Stale premise on cells-layer decomposition (most important).** Objectives 5–6 are framed as "split `CellsLayer` into focused painters," but the per-feature painters already exist: `cells/` contains `fills.ts`, `borders.ts`, `text.ts`, `rich-text.ts`, `rotated-text.ts`, `shrink-to-fit.ts`, `text-wrap.ts`, `data-bars.ts`, `icon-sets.ts`, `indicators.ts`, `interactive-elements.ts`, `sparklines.ts`, `merged-cells.ts`, `center-across.ts`, etc., and `layers/cells.ts` already imports and orchestrates them. The genuinely-remaining work is narrower: the typed `CellFrameRenderPlan`, the recyclable `CellVisualRecord` replacing `CellRenderInfoExtended`, and numeric-key hot-path maps. The plan should re-baseline objectives 5–6 to "introduce a frame render plan and recyclable record on top of the existing painters," not "split the file."
- **Inaccurate inventory figure.** The plan asserts "98 TypeScript source files and 31 test files." Actual: ~67 non-test source files and 31 test files. The test count is right; the source count is off by ~30, which is the same staleness signature as the decomposition framing — suggesting the plan was written against an earlier tree or an over-counted glob. Not fatal, but it undercuts the "already substantial tests, problem is duplication" argument's precision.
- **Scope is a program, not a plan.** Ten objectives, eight parallel agent tracks (A–H), engine contract changes, a contracts/viewport-payload migration spanning `views/sheet-view` and Rust producers — this is multiple milestones. There is no phased "land this first, ship, measure, then continue" cut line, and no explicit MVP/critical-path subset. The correctness fixes (hardcoded header dims, hit-test path unification, sticky readers, per-region clears) are independently shippable and far higher value than the allocation/metadata work; the plan would be stronger if it declared those as Phase 1 with the rest deferred.
- **Performance gates lack acceptance thresholds.** Objective 8 and the verification section ask to "record frame time, visible-cell count, allocations, dirty rect count … before and after," but define no pass/fail target. A refactor that preserves correctness but regresses frame time by 15% would pass every listed gate. There is no rollback trigger tied to a number.
- **Engine frame-lifecycle hook is cross-package and under-specified.** `beginFrame/endFrame` do not exist in `engine/src` today (confirmed). Adding a layer-lifecycle hook to `canvas-engine` is a shared-contract change affecting every layer; the plan acknowledges this as a risk but does not specify the hook signature, who calls it, or how existing layers opt out, leaving the riskiest cross-package change as the least-detailed.
- **Metadata migration depends on out-of-scope producers.** Objectives 6–7's payload migration explicitly needs Rust viewport-buffer and contracts changes. That work cannot be scheduled from this folder's plan alone, so it should be marked "blocked / coordinate separately" rather than listed as an in-plan objective.

## Contract and verification assessment

- **Contract clarity: high.** Invariants are concrete and mostly testable, with named transforms (`docToRegionXY`, `cellRectInRegion`, `rangeRectInRegion`, `snapDocXToPixelGrid`) and a clear source-of-truth statement for the binary buffer. The `SetOrClear<T>` / `null`-to-clear vs `undefined`-to-leave patch contract (objective 4) is well specified and directly fixes a verified bug.
- **Verification gates: good on correctness, soft on perf.** The enumerated `pnpm test` / `pnpm typecheck` commands across both packages plus the cross-package `mog` typecheck are appropriate. The contract-test list is genuinely strong — especially the `grid-canvas` composition test proving `engine` grid provider and `GridRendererImpl.hitTest()` agree for the same point (this is the precise regression risk implied by the line-2561 fallback), and the multi-region frame-accumulation tests. Gaps: no quantified perf threshold, no explicit gate that the eslint coordinate invariant still passes after new geometry files are added (only mentioned as an allowlist edit), and browser verification is described qualitatively with no scripted scenarios or success criteria.
- One internal tension: objective 4 says optional-reader clearing "can make transient missing-reader frames visible … correct if the buffer is absent," and the invariants say missing viewport IDs "may skip cells while loading." The plan asks tests to "distinguish real loading gaps from stale-reader regressions" but does not define the observable signal that separates the two — this needs a concrete contract (e.g., a metric or a debug assertion), not just a test aspiration.

## Concrete changes that would raise the rating

1. **Re-baseline objectives 5–6** against the existing `cells/` painter modules: drop "split the file," keep only the frame-render-plan, recyclable `CellVisualRecord`, and numeric-key changes, and reference the current painter imports in `layers/cells.ts` so implementers don't re-do completed work. Fix the "98 source files" figure.
2. **Declare a Phase 1 critical-path subset**: header-dim removal + layout-aware `grid-hit-test` + unify `GridRendererImpl.hitTest()` onto one path + sticky-reader `SetOrClear` + per-frame (not per-region) clears. These are independently shippable, evidence-backed correctness fixes; everything else becomes Phase 2+.
3. **Add quantified perf acceptance criteria and a rollback trigger**: e.g., "frame time at p50/p95 must not regress >X% on the production `grid-canvas` path; allocations per frame must not increase," with the measurement harness named, before merging the cells refactor.
4. **Specify the `canvas-engine` frame-lifecycle hook**: proposed signature, caller, default no-op behavior for existing layers, and a test that a non-participating layer is unaffected. This is the highest-risk cross-package change and should be the most precise, not the least.
5. **Mark the metadata/payload migration (objectives 6–7) as cross-team / blocked** on Rust viewport-buffer + contracts changes, with the coordination owners named, so it isn't treated as schedulable within this folder.
6. **Define the loading-gap-vs-stale-reader observable signal** concretely (a debug metric or assertion contract), so objective 4's "distinguish" test has a real target.
7. **Add an explicit gate that the eslint coordinate invariant passes** after introducing the `RendererGeometry`/`GridGeometryContext` type and any new geometry files, including the allowlist diff, rather than leaving it as an aside.
