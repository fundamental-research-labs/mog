# Review of Plan 042: Grid Canvas Orchestration and Renderer Integration

## Summary judgment

This is a strong, evidence-grounded refactoring plan for `mog/canvas/grid-canvas/src`. I verified its factual claims directly against the source and they hold up to an unusual degree:

- `renderer/grid-renderer.ts` is exactly 2,844 lines, as stated.
- All nine named data-source adapter classes exist verbatim (`CellDataAdapter`, `SelectionDataAdapter`, `SheetDataAdapter`, `CollaborationDataAdapter`, `TraceDataAdapter`, `FloatingObjectDataAdapter`, `GroupingDataAdapter`, `PageBreakDataAdapter`, `OverlayDataAdapter`).
- The "untyped dispatch table" critique is real: `private readonly fieldHandlers: Record<string, (value: any) => void>` (line 969) built by `buildFieldHandlers()` (line 1096), driven by `updateContext()` (line 2392).
- The `invalidateCells()` full-layer concern (step 9) is real: it calls `this.markAllDirty()` at line 2415 rather than delegating to the scheduler's precise rects.
- The `resume()` / `isRunning` invariant (preserve-list) is real and explicitly commented in the code (lines 2272–2282): no `isRunning` gate because sheet-view owns engine start policy.
- Scene-graph methods (`applySceneGraphPatches`, `syncSceneGraph`, `buildSceneObject`, `getAllObjectBounds`), the `FloatingObjectPatch.bounds` production path, and the missing-bounds skip-with-warning behavior all exist as described.
- Tests reach into internals via `GridRendererImpl.prototype as any` (step 8 motivation), confirmed across four test files.
- `RenderContextConfig` and the data-source interfaces are owned in `@mog-sdk/contracts/rendering` (re-exported through `mog/types/rendering`), matching the dependency notes.

The plan correctly frames the package as a facade and scopes the work to making the implicit orchestration matrix explicit, typed, and independently testable while preserving the public export surface. Scope, out-of-scope, invariants, sequencing, and verification are all present and coherent. The main limitations are soft verification gates on the one genuinely risky axis (hot-path allocation/perf), an unresolved contract decision parked into implementation (step 9), and parallelization that understates merge contention on a single monolith.

## Major strengths

1. **Accuracy and specificity.** The plan reads like it was written against the actual file, not a guess. Class names, method names, line-level behaviors, and the contract ownership graph are correct. This dramatically lowers the risk of an implementer discovering the plan is built on false premises.
2. **Correct dependency-first sequencing.** Step 1 (the `RenderContextConfig` handler/disposition inventory) is genuinely the root dependency — every adapter, router, and parallel agent keys off it. The plan recognizes this and gates the parallel fan-out on the inventory landing.
3. **Invariant preservation is explicit and load-bearing.** The "production-path contracts and invariants to preserve or strengthen" section is the best part: O(fields) `updateContext`, callback-identity-does-not-dirty, write=invalidate, scroll-invalidation avoiding static overlay repaints, `computeViewportLayout()` purity, stale-sheet hit suppression, `resume()` delegation. These are the exact behaviors the existing tests lock, so the plan is refactoring against a real safety net rather than inventing one.
4. **Anti-shim discipline.** Repeatedly insists on updating the public contract and callers directly rather than adding compatibility layers or duplicate APIs (out-of-scope item, step 15, non-goals). For an internal monorepo with no external consumers this is the right call and avoids the classic refactor outcome of "two parallel paths forever."
5. **Test-quality upgrade is concrete.** Moving away from `prototype as any` toward testing extracted pure helpers (router handlers, invalidation helpers, scene-graph sync, composition factory) plus a thin public-facade integration test is the correct shape and is well justified by the current test debt.
6. **Edge-case and risk enumeration is thorough and domain-aware** — merged-cell boundary dirty extents, freeze+split+zoom combinations, late async sync after sheet switch, CJK/quoted/intrinsic-weight fonts, JSDOM-vs-browser CSS variable listener divergence.

## Major gaps or risks

1. **The highest-risk axis has the softest gate.** The single most dangerous outcome of this refactor is regressing the `updateContext()` hot path (50–200+ calls/sec, must stay O(fields) with no per-call allocation/clone). The plan flags this as a risk but the verification is only "benchmark production update sequences before and after *if performance-sensitive changes land*." There is no baseline capture step, no allocation assertion, no threshold, and no committed micro-benchmark harness. A typed router built with closures/object spreads can silently add per-call allocation that no functional test catches. This should be a hard gate with a captured baseline, not a conditional suggestion.
2. **Step 9 defers a contract decision instead of making it.** "Decide whether public `invalidateCells()` should delegate to the scheduler or remain a full-layer API" is exactly the kind of contract question a plan should resolve. It states a preference (delegate when a position index exists, documented full-dirty fallback otherwise) but leaves the decision open. Better to commit to the contract and list the one caller behavior that constrains it.
3. **Parallelization understates merge contention.** Agents B (adapters), C (invalidation), and D (scene-graph sync) all extract code out of the *same* 2,844-line `grid-renderer.ts` concurrently. "After the contract inventory lands" gates the *start* but not the *collision*: three agents moving large blocks out of one file will conflict heavily. The plan needs either an explicit extraction order (e.g., adapters first to a new dir, then router, then invalidation, then scene-graph) or a statement that B/C/D are sequential edits to the monolith even if their test work is parallel.
4. **Exhaustiveness mechanism is named but not committed.** Step 1 wants a test that "fails when `RenderContextConfig` gains a field that has no explicit disposition." A runtime fixture test cannot enforce compile-time exhaustiveness; a `keyof RenderContextConfig` mapped type / `satisfies` with a `never` fallback can. The plan mentions both flavors ("executable test fixture or generated metadata", "Unknown keys should be impossible at compile time") without committing to the compile-time mechanism, which is the only one that actually prevents silent drift.
5. **No golden/snapshot safety net for behavior preservation.** For a refactor whose explicit contract is "do not change observable dirty output," the strongest possible gate is a golden capture of emitted dirty rects across a representative input sequence (selection moves, scroll, geometry change, scene patch, theme change) asserted identical pre/post. The plan relies on the existing partial tests plus new per-helper tests; it never proposes a structural before/after equivalence check on the facade's observable output.
6. **Thin articulation of user-facing payoff vs. cost.** This is overwhelmingly a structure/testability refactor: almost all listed value is "drift-resistance," "explicit dispositions," "independently testable." For a 2,844-line teardown spread over seven agents, the plan would be stronger if it named the concrete defect classes this prevents (e.g., "new render-context field silently mapped to wrong dirty layer ships a stale-paint bug") and any latent bug it expects to surface and fix, rather than leaving the justification largely architectural.

## Contract and verification assessment

**Contract clarity: strong.** The plan correctly locates the authoritative contracts in `@mog-sdk/contracts/rendering` / `mog/types/rendering`, distinguishes facade-level view extensions (`getEngine()`, `getGridLayers()`, `getCellExpander()`) from the contracts-level `GridRenderer` interface, and is precise about which package owns what (grid-renderer = painting + null defaults, drawing-canvas = scene graph mutation, engine = dirty-rect accumulation). The write=invalidate contract and the scene-graph bounds source-of-truth (`getAllObjectBounds()` for batch, `FloatingObjectPatch.bounds` for incremental, legacy async callback demoted to overlay/selection fallback) are stated with enough precision to implement against. The `setViewportLayout(..., { invalidation: 'scroll' })` distinction between metadata update and structural full-dirty is correctly captured.

**Verification gates: good breadth, uneven depth.** Per-package `pnpm test` / `pnpm typecheck` commands are enumerated with sensible conditionality (only run drawing-canvas tests if scene-graph contracts move, etc.), plus a root `mog` typecheck for cross-package changes and a real-UI E2E requirement that bans seeding state via private method calls. This is the right matrix. The depth gaps: (a) no perf/allocation gate as noted above; (b) no named command or CI hook for the new contract-inventory exhaustiveness test; (c) no coverage threshold or golden-output equivalence check to prove the refactor is behavior-preserving rather than merely "tests still pass." The E2E "real keyboard/mouse/clipboard/pointer" mandate is good and aligns with how app-eval scenarios should drive UI.

## Concrete changes that would raise the rating

1. **Make the hot path a hard gate.** Add a step that captures a baseline for a fixed `updateContext()` sequence (e.g., allocation count and time over N iterations of a scroll+selection+geometry batch) and asserts the post-refactor router stays within a stated bound. Commit to "no object/array allocation per field beyond the value passthrough" as a testable invariant.
2. **Resolve step 9 in the plan.** Commit to: `invalidateCells()` delegates to `markCellsDirty()` precise rects when a position index is present, with a documented full-dirty fallback otherwise — and name the caller(s) (`InlineCellEditor`, etc.) whose expectations constrain the choice.
3. **Specify the extraction order to remove B/C/D merge contention.** State that block extractions from `grid-renderer.ts` are serialized (adapters → router → invalidation → scene-graph), even though each agent's *test* authoring runs in parallel.
4. **Commit to compile-time exhaustiveness.** Specify a `keyof RenderContextConfig` mapped-type / `satisfies` construct with a `never`-typed fallback so a new field is a typecheck failure, and keep the runtime fixture only as documentation of dispositions.
5. **Add a behavior-preservation golden test.** Propose a before/after snapshot of emitted dirty rects (and request-frame calls) over a canonical input sequence, asserted identical, as the primary safety net for the "no observable change" contract.
6. **Strengthen the why.** Add one short subsection naming the concrete defect class(es) this refactor prevents and any latent bug expected to surface, so the seven-agent cost is justified by more than architecture cleanliness.

Addressing items 1–3 would move this to a 9; all six would make it a 10.
