# Review of 044 вЂ” Canvas Overlay Source Improvement Plan

## Summary judgment

This is a strong, evidence-grounded plan. I cross-checked its central observations against the actual source in `mog/canvas/overlay/src` and the production adapter in `mog/canvas/grid-canvas/src/renderer/grid-renderer.ts`, and the diagnosis is accurate in nearly every specific it makes. The plan correctly identifies the package's defining structural weakness вЂ” render and hit-test paths independently re-query the data source and rebuild geometry, so they can (and demonstrably do) drift вЂ” and proposes a coherent fix (a single typed frame snapshot consumed by both paths, plus shared geometry descriptors). The contract/invariant section is unusually precise, the verification gates are concrete and runnable, and the sequencing is sensible. It loses points for breadth/ambition relative to a single workstream, a couple of genuinely open product decisions surfaced but not resolved, and speculative engine-dependent dirty-rect work whose payoff is uncertain.

## Major strengths

- **The observations are real, not hand-waved.** I verified the load-bearing claims:
  - `_lastCtx` coupling: `OverlayLayer` stashes the render context (`overlay-layer.ts:97,137`) and `hitTest()` early-returns null without it (`:248вЂ“249`), then mutates the transform on it for hit testing. The plan's call to make hit tests pure-geometry and drop `_lastCtx` is well-founded.
  - **Custom-handle objectId/handleId conflation is a true bug.** `hitTestHandles` returns `objectId: handle.id` for a custom handle (`handle-hit-testing.ts:70`) вЂ” i.e. the handle instance ID is reported as the object ID. The plan flags exactly this.
  - **Render/hit lock mismatch for group handles is real.** Render computes `allLocked` and passes it to `getHandleVisibility` (`overlay-layer.ts:172вЂ“173`), but the hit path hard-codes `getHandleVisibility(groupBounds, false, config)` (`handle-hit-testing.ts:77`). So an all-locked group renders no handles but can still be hit вЂ” precisely the "hidden handles must not be hittable" invariant the plan elevates.
  - **Custom-handle rotation asymmetry is real.** `renderCustomHandles` applies `applyRotation(ctx, bounds)` (`custom-handles.ts:104`), but `buildCustomHandlePath(handle, expansion)` in the hit path takes no parent bounds/rotation. Render and hit geometry diverge for rotated objects.
  - **Adapter is genuinely incomplete.** `getGuides`/`getRubberBand`/`getDragPreview`/`getInkPreview`/`getConnectionPointIndicators` all return `[]`/`null` in `OverlayDataAdapter` (`grid-renderer.ts:882вЂ“900`), and chrome is suppressed via `interactionState === 'operating'` (`:857`). The plan's framing вЂ” "don't silently return null; either feed real data or explicitly mark unsupported" вЂ” is the right discipline.
  - **Static custom handles** are indeed passed as layer config (`overlay-layer.ts:60,94`) and only rendered for single-selection, matching the "promote to a dynamic provider" objective.
- **Invariant discipline.** The "contracts and invariants" section reads like a spec: balanced `save`/`restore`, identical render/hit visibility for locked/tiny objects, synchronous-only render, explicit group-rotation semantics, no `mog-internal` leakage into public packages. These are the right things to pin, and several map directly to verifiable code conditions.
- **Verification is production-path-oriented.** It explicitly rejects mock-call-count assertions, demands browser-backed canvas tests (real `CanvasRenderingContext2D`/`DOMMatrix`/DPR), adapter projection tests in `grid-canvas`, and E2E via real input вЂ” not direct state mutation. The performance note ("target the path registered through `GridRendererImpl`, not mock renderers") is a meaningful guardrail.
- **Edge-case catalogue is thorough and concrete** (rotation angles incl. negative, threshold-boundary dimensions, fractional scroll, mixed/all-locked groups, frozen panes/sheet-switch ordering, lasso < 2 points, eraser radius).

## Major gaps or risks

- **Scope is large for one plan.** Five agents (AвЂ“E), six dependency stages, and changes spanning `overlay`, `contracts`, `types`, `grid-canvas`, and `apps/spreadsheet`. The plan acknowledges this and gates parallelization on agreed snapshot/descriptor contracts, which mitigates it вЂ” but the cross-package public-type changes (HandleRegion в†” ObjectHitRegion, OverlayHitResult в†’ floating-object hit target) are the riskiest part and the plan rightly notes downstream cursor/dispatch must migrate together. This is a coordination hazard more than a design flaw.
- **Open product decision left open.** Group-rotation semantics ("explicit product decision") is correctly surfaced as a risk, but the plan can't proceed on items 2/6/contracts cleanly until it's resolved; it would be stronger to state the default it will implement if no decision arrives (e.g. omit group rotation from both render and hit until product confirms).
- **Dirty-rect work is speculative and conditional on engine support.** Item 7 hinges on `canvas-engine` gaining a screen-space/coordinate-tagged dirty hint. The plan handles this responsibly (keep full-dirty until the engine contract exists) but the payoff is uncertain and it touches a folder outside scope. It reads as a stretch goal that could be deferred entirely without weakening the core fix.
- **No mention of the existing `__tests__` baseline.** The plan names new test files but doesn't enumerate what's currently in `canvas/overlay/src/__tests__` or commit to not regressing existing coverage; a brief "current tests and what changes" inventory would de-risk the test refactor (it proposes rewriting assertions away from call counts).
- **"Where those interactions exist" hedging.** For guides/rubber-band/connection points, the plan defers to "when the object system has those states." It does not confirm whether the spreadsheet object system actually produces these today вЂ” so an implementer could spend effort wiring adapter methods with no upstream producer. A short discovery step ("confirm which interaction states the object machine emits") should be a stage-0 task.

## Contract and verification assessment

Contract clarity is the plan's best dimension. It defines the snapshot as the single source of truth, specifies `HandleDescriptor` fields (owner ID, group flag, optional handle ID, `ObjectHitRegion`, visual + expanded hit geometry, priority, visibility reason, dirty bounds), and proposes a hit-target shape (`{ type: 'floatingObjectHandle', objectId, region, isGroup, handleId? }`) that resolves the objectId/handleId conflation I confirmed in code. Aligning the local `HandleRegion` union with the public `ObjectHitRegion` is the correct anti-divergence move. The one underspecified contract is the group target shape (`objectId: null` vs an explicit group target) вЂ” the plan appropriately conditions it on "downstream can consume consistently" but doesn't pick one; that choice should be nailed down before Agent A finalizes types.

Verification gates are appropriate and runnable: per-package `pnpm test`/`typecheck` for `overlay`, `grid-canvas`, `apps/spreadsheet`, root typecheck after public-type changes, plus browser/E2E and a production-path performance target. The insistence on real `Path2D`/`DOMMatrix`/DPR coverage directly addresses the current test gap the plan calls out (Node-mock-only). The fixture-first guidance for the Path2Dв†’pure-geometry migration ("define intended CSS-pixel hit envelope before migrating") is exactly right and turns a boundary-shift risk into a regression-testable change.

## Concrete changes that would raise the rating

1. **Add a stage-0 discovery task** confirming which interaction states the spreadsheet object machine actually emits today (guides, rubber band, connection points, ink), so adapter work targets live producers and "unsupported" is a deliberate, documented choice rather than a guess.
2. **State the default for group rotation** the plan will implement absent a product decision (recommend: omit from both render and hit), so implementation is unblocked.
3. **Pick the group hit-target shape now** (`objectId: null` + `isGroup: true`, or a distinct group target) and note the downstream consumers that must migrate, rather than leaving it conditional.
4. **Inventory the existing `__tests__`** and add an explicit "no coverage regression" gate alongside the new test files, since the plan rewrites assertion style.
5. **Demote the engine dirty-rect work to an explicit optional follow-up** (or split it into a separate plan) so the core render/hit-unification doesn't appear to depend on out-of-scope `canvas-engine` changes.
6. **Cite file:line evidence inline** for the four concrete bugs (lock mismatch `handle-hit-testing.ts:77`, objectId/handleId `:70`, custom-handle rotation asymmetry `custom-handles.ts:104` vs hit path, adapter nulls `grid-renderer.ts:882вЂ“900`) and make "fix these specific defects" a tracked acceptance criterion вЂ” the plan describes them but doesn't pin them as discrete, checkable deliverables.
