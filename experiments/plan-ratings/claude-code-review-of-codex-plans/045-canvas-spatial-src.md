Rating: 8/10

# Review: 045 — canvas/spatial/src Improvement Plan


## Summary judgment

This is a strong, evidence-grounded plan. Every substantive factual claim it makes about the source was verifiable against the actual code, including the central correctness bug. It correctly distinguishes a genuine production defect (cell-key truncation) from a list of contract-hardening improvements, respects package-layering boundaries instead of importing canvas code into kernel by convenience, and pairs each change with concrete, package-scoped verification gates plus oracle/differential tests. It is held back from a 9–10 by a few unresolved design decisions (the invalid-bounds failure mode), a missing performance analysis of its proposed key replacement, and breadth that bundles one critical fix with much optional API redesign without crisp prioritization.

## Major strengths

- **The headline bug is real and correctly diagnosed.** `packCellKey` (grid-index.ts:58–60) does `((cellX & 0xffff) << 16) | (cellY & 0xffff)`, truncating both coordinates to 16 bits. Two distinct cells whose coordinates differ by 65536 collide, exactly as the plan states. For document/world coords this is reachable (cell = coord/50, so collision at ~3.2M coordinate separation). The plan correctly frames this as a *correctness* issue, not a perf nicety, and refuses a compatibility shim that preserves the limitation.
- **The threshold-confusion claim is accurate and subtle.** `query()` (grid-index.ts:252–281) guards with `isQueryOversized` (`MAX_QUERY_CELLS = 10_000`), but then calls `getCellsForBounds()`, which internally returns `null` once `isOversized` (`MAX_CELLS = 1000`) trips. So a query spanning between 1000 and 10000 cells silently full-scans. The plan calls this out precisely (objective/step 4, line 86) and proposes a clean separation via `maxCellsPerEntry` / `maxCellsPerQuery`.
- **Consumer inventory is correct.** Verified: charts `hit-tester.ts` (`findNearby`), overlay `handle-hit-testing.ts` (`testPointInPath`), drawing-canvas `hit-map.ts`, ink re-export, and the separate kernel `GridSpatialIndex` with `Map<string, Set<StrokeId>>`. The non-atomic update path the plan targets is confirmed at hit-map.ts:185–187 (remove + insert, with a comment admitting `updateBounds` is bounds-only).
- **Canvas helper gap is real.** `testPointInPath`/`testPointInStroke` (canvas-hit-test.ts) call `save()`/`resetTransform()`/.../`restore()` with no `try/finally`; a throwing `isPointInPath` leaks transform state. The plan's fix is correct and minimal.
- **Architecture discipline.** It refuses to pull a canvas-layer package into kernel without ownership sign-off, offers the lower-level-extraction alternative, and explicitly forbids `mog -> mog-internal`. The `@mog/geometry` dependency is proposed conditionally on dependency-direction approval rather than assumed.
- **Verification gates are concrete and per-package** (spatial, ink, drawing-canvas, overlay, charts, plus root typecheck), and tests are sequenced before consumer changes. The randomized differential-vs-linear-oracle test is the right tool for this kind of index.

## Major gaps or risks

- **Invalid-bounds failure mode is unspecified.** The plan repeatedly asserts negative dimensions "should fail fast before reaching cell enumeration" but never says *how* — throw, dev-only assert, or drop. This matters: `insert` is on a UI hot path, and throwing in production could crash hit-testing on a malformed drag rectangle. The plan acknowledges the risk (line 161) but leaves the mechanism undecided, which is exactly the kind of contract a plan should pin down. Recommend specifying a dev-only invariant assert + production no-op/skip, or an explicit thrown error documented as a caller contract.
- **No perf analysis of the proposed key replacement.** The source comment justifies the packed integer key as "better Map performance." The plan prefers nested `Map<number, Map<number, Set>>` "for easier diagnostics," adding a second map lookup per cell on every query iteration, but never weighs that hot-path cost. A non-truncating single-number key is feasible without nesting — JS safe integers hold 53 bits, so a wider pairing (or `cellX * K + cellY` with a guarded range) fixes the collision while keeping one lookup. The plan lists string keys as an alternative but doesn't analyze the tradeoff. This is the one place where the chosen solution may be worse than a simpler fix to the same bug.
- **Breadth dilutes the critical fix.** Steps 1–8 mix one must-fix correctness bug with a large amount of optional surface (upsert vs update API redesign, `has`/`get`, comparator-based pipeline options, diagnostics snapshots, stroke-option plumbing). Many are correctly hedged with "only if a real caller needs it," which is honest, but a plan ideally resolves those conditionals rather than deferring them to implementation. A clearer phase-0 (collision + threshold + try/finally — the three confirmed defects) ahead of the discretionary hardening would improve executability.
- **Determinism claims are conditional.** "Preserve deterministic query order ... if callers rely on stable ordering" — the plan doesn't establish whether any consumer actually does. Without that, the insertion-sequence tracking is speculative work. A quick consumer audit would settle it.
- **`hitTestPipeline` tie-break.** The current pipeline sorts by z-index only (pipeline.ts:20); `Array.prototype.sort` stability then makes ties depend on `queryPoint` emission order, which itself depends on `Set` iteration. The plan correctly wants deterministic ties, but the proposed remedy (insertion sequence) requires the index to expose it — a cross-file contract the plan should state explicitly as a dependency between Agent A and Agent B.

## Contract and verification assessment

Contracts are unusually well-specified for an index: inclusive point containment, inclusive edge-touching `query` intersection (correctly noted as *different* from `@mog/geometry` `Rect.overlaps()`, and backed by the real assertion at grid-index.test.ts edge-boundary test), `contains`-mode full containment, `findNearby` edge-distance semantics with distance 0 for containing boxes, and oversized-entry-always-queryable. These match the code and existing tests. The main contract hole is the invalid-bounds behavior noted above.

Verification is the plan's strongest dimension: specific commands per affected package, tests-before-consumers ordering, a differential oracle for the mutation/query surface, targeted regression tests for the collision (65536 separation) and the threshold path (`maxCellsPerQuery` vs `maxCellsPerEntry`), and a `finally`-on-throw test for the canvas helpers. It honestly records that gates were not run because the task disallowed it. The E2E note (drive real pointer input, don't mutate state directly) is appropriate.

## Concrete changes that would raise the rating

1. **Pin the invalid-bounds contract**: state the exact failure mode (recommend dev-only invariant assert + production skip, or a documented thrown error), and list which known callers may emit negative-dimension drag rects so they can be normalized at the boundary first.
2. **Add a key-design tradeoff paragraph**: compare nested numeric maps vs a non-truncating single-number pairing (53-bit safe-integer pack) vs string key, on hot-path lookup cost, and justify the choice rather than defaulting to nesting for diagnostics.
3. **Introduce an explicit phase 0** = {collision fix, threshold separation, try/finally} as the confirmed-defect core, with the API/diagnostics work as clearly-optional later phases gated on a consumer-needs audit.
4. **Resolve the determinism conditionals**: a one-pass consumer audit to confirm whether any caller relies on stable `all()`/`query()` order and on pipeline tie ordering; drop the insertion-sequence work if none do.
5. **State the cross-agent contract** that Agent A must expose insertion sequence (or render-order metadata hook) for Agent B's deterministic tie-break, so the parallelization doesn't deadlock on an undefined interface.
