# Review of 048 — Canvas Drawing Shapes Source Improvement Plan

## Summary judgment

This is a strong, evidence-grounded plan. I spot-checked its central factual claims against the live source and every one held up:

- The `upArrow` extraction gap is real. `__tests__/presets/ooxml-coverage.test.ts` declares `UNSUPPORTED_PRESETS = new Set(['upArrow'])` with the exact comment "missing from preset-shape-data.json (extraction gap — only 186 of 187 shapes extracted)", asserts the canonical list length is 187, and skips `upArrow` from the per-shape `isValidShapeType` checks. The plan's framing of this as the keystone defect is accurate.
- The duplicated OOXML guide evaluator is real. `canvas/drawing/geometry/src/connection-points.ts` independently implements built-in variable creation (`l/r/t/b/hc/vc/...`), `resolveGuideRef`, single-formula evaluation, and full guide evaluation — parallel to shape-engine's `evaluateGuides()`. The "unify the math" objective targets a genuine drift surface.
- The downstream fallback duplication is real. `canvas/drawing-canvas/src/shape-rendering-info.ts` imports `getRegisteredShapeTypes` from `@mog/shape-engine` and unions it with `HARDCODED_SHAPE_TYPES`, with an inline comment saying the hardcoded set can be removed "once shape-engine has full coverage." The plan quotes this intent precisely.
- The bridge `ShapePreset` inventory is real and matches the plan: `infra/rust-bridge/bridge-ts/generated/ooxml-types.ts` lists 187 OOXML tokens (including `upArrow`) and terminates with `textBox` as the Mog extension — exactly the "187 OOXML + textBox extension" contract the plan asks to formalize.
- The stale comments exist: `spec-presets.ts` header says "all 186 OOXML preset shape definitions" and `index.ts` says "all 188 OOXML shape preset values" — the two-number inconsistency the plan flags as evidence the catalog count was never pinned.

Because the diagnosis is correct and the remediation is sequenced around the real root cause (regenerate the data so the full unique token set is proven, rather than hand-adding one shape), this plan would actually move the folder forward rather than paper over the symptom. The deductions are for an unresolved critical-path blocker and for scope that strains the per-folder framing.

## Major strengths

- **Root-cause discipline.** The plan explicitly forbids the tempting one-line fix ("Do not patch only `upArrow` by hand while leaving the extraction and coverage contract broken") and instead requires the generator to prove the full 187-token set. It also anticipates that `upArrow`'s absence may be a duplicate/parsing edge case in the source XML, not a simple omission — a sophisticated read.
- **Production-path framing.** Section "Production-path contracts and invariants to preserve or strengthen" is the best part. It pins the right invariants: pure-computation package boundary, dependency direction, single `generateShapePath()` path for tests and renderers, explicit-error (not silent-rectangle) on unknown shapes, and "tests and diagnostics call the same generator used by renderers." The closing instruction — "Do not count a regenerated snapshot suite alone as verification" — directly attacks the weak-test failure mode.
- **Catalog contract first.** Step 1 demands the canonical `ooxmlPreset` / `mogCustomShape` / `nonShapeRendererPrimitive` classification be written before generation code changes, and forces a decision on the genuinely ambiguous entries (`textBox`, `connector`). This is the correct ordering and matches the real source-of-truth fragmentation across Rust, bridge, contracts, and the canvas fallback list.
- **Honest about its own blocker.** The plan states up front that the source spec XML's documented paths were absent in this checkout and makes "locate or restore the authoritative spec XML" the first risk and an explicit precondition. I confirmed `scripts/` contains only `extract-preset-shapes.py` and `extract-connection-points.ts` and no spec XML — so this caveat is accurate, not boilerplate.
- **Concrete, real verification gates.** The `pnpm --filter` targets and the named kernel test (`shape-computation-e2e.test.ts`) are plausible and specific, and the parallelization section lists genuine inter-agent dependencies (classification before generator; generator before tightening exact-count tests, "otherwise tests will encode the current 186-shape gap").

## Major gaps or risks

- **The whole plan is gated on an artifact that may not exist.** Objective 2 and Step 2 hinge on regenerating `preset-shape-data.json` from `presetShapeDefinitions.xml`, which the plan itself says is missing from the checkout. The plan says "accept a checked-in spec XML path or explicit environment override and fail with a precise missing-source message" — but that is the *generator's* error handling, not a recovery path for the team. If the canonical XML genuinely cannot be located/restored, the entire "single deterministic generator" objective stalls and the plan offers no fallback (e.g., reconstruct from the existing JSON + the Rust inventory + targeted re-extraction of just the missing `upArrow`). This is the single biggest execution risk and deserves a concrete contingency, not just a fail-fast message.
- **Scope strains the "one src folder" frame.** This is nominally a plan for `mog/canvas/drawing/shapes/src`, but it edits or adds tests across `@mog/geometry` (moving guide math), `drawing-engine`, `drawing-canvas` (fallback removal), `kernel/src/domain/shapes`, `diagram`, and the Rust `ooxml-types` contract. That is a multi-package refactor. The six-agent split is reasonable, but the plan should acknowledge that the geometry-math move and the Rust-contract touch are the highest-blast-radius items and could be carved into a separate, independently-shippable change so the shapes-folder improvements aren't blocked behind a cross-package refactor.
- **No characterization step before the guide-math move.** Step 4 moves formula evaluation into `@mog/geometry` and rewires both shape-engine and connection-points to it. The plan asks for "tolerance-based tests" but does not mandate capturing the *current* output of both evaluators as a golden baseline *before* the move. Without a pre-move snapshot, "tolerance tests pin the intended semantics" risks pinning whatever the post-move code happens to produce. Recommend an explicit "characterize current behavior of both evaluators on a shared formula corpus, then assert the moved evaluator reproduces it within tolerance."
- **Generated-data determinism vs. timestamp tension.** Step 2 wants a manifest including a "generation timestamp," but Objective 2 demands "deterministic sorted JSON" reviewable as data diffs. A timestamp in a checked-in artifact defeats reproducible diffs and will churn on every regen. The hedge ("if the repo's generated-file conventions allow it") is too soft; the plan should default to excluding the timestamp from the committed file (or isolating it to a non-committed run log).
- **`cargo test -p ooxml-types shape_preset_spec_xml_roundtrip` is asserted but unverified.** I did not confirm that specific test name exists. If it doesn't, the verification gate is illusory. Low cost to verify; worth pinning.
- **UI verification is vague.** "Run the app or drawing lab and exercise shape insertion" names no command, skill, or harness. Given the repo has app-eval scenario infrastructure, the plan would be stronger naming a concrete verification surface (e.g., an app-eval scenario inserting `upArrow` and a representative arrow/flowchart/star/callout family) rather than manual exploration.

## Contract and verification assessment

The contract section is the plan's strongest dimension. Invariants are explicit, falsifiable, and aimed at the real seams: package purity, dependency direction, single generation path, command-order/coordinate-space preservation, arc-conversion semantics, deterministic non-negative text insets, and — critically — "connection point resolution must use the same guide values as the shape path for the same dimensions and adjustments," which is exactly the drift the duplicated evaluator can introduce.

Verification is mostly well-specified: focused per-package filters, a clear separation of focused vs. final gates, and a strong qualitative acceptance signal ("complete catalog coverage plus production renderer, hit-test, text, and connection metadata behavior"). The exact-count tightening is correctly sequenced behind the regeneration so tests don't ossify the 186-shape gap. Weaknesses: the unverified Rust test name, the timestamp/determinism conflict undermining the "review as data diffs" gate, the absence of a pre-move characterization gate for the guide-math relocation, and the hand-wave on UI verification. None of these are structural; they are precision gaps in an otherwise rigorous gate set.

## Concrete changes that would raise the rating

1. **Add a contingency for a missing source XML.** Specify what happens if `presetShapeDefinitions.xml` cannot be located: e.g., reconstruct the authoritative set from the existing JSON plus the Rust/bridge inventory, targeted-extract only the missing token(s), and vendor a checked-in spec file under `scripts/` so future regen is hermetic. Make "spec XML located/restored and committed" an explicit Step 0 deliverable, not just a risk bullet.
2. **Split the cross-package refactor from the folder fix.** Carve the `@mog/geometry` guide-math relocation and the Rust `ooxml-types` contract touch into a separate prerequisite change (or clearly mark them optional/deferrable) so catalog completion + `upArrow` can ship without waiting on the highest-blast-radius work.
3. **Mandate a pre-move characterization baseline** for both guide evaluators (shape-engine `evaluateGuides` and `connection-points.ts`) over a shared formula corpus, asserted before and after the move within stated tolerances.
4. **Resolve the manifest timestamp conflict:** default to no timestamp in the committed JSON/manifest; put run metadata in a non-committed log if needed.
5. **Verify the named Rust gate exists** (`shape_preset_spec_xml_roundtrip`) or replace it with the actual test, and reconcile the count contract to a single asserted number (187 OOXML registry targets + `textBox` extension), then update the `186`/`188` comments in `spec-presets.ts` and `index.ts` to match.
6. **Name a concrete production-path verification surface** — an app-eval (or drawing-lab) scenario inserting `upArrow` and one shape per family — instead of "run the app and exercise insertion."
