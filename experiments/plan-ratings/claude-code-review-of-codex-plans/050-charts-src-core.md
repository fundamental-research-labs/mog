# Review of 050-charts-src-core.md

## Summary judgment

This is a strong, evidence-grounded plan. Its description of `charts/src/core` matches the actual code closely: the long `if`-chain dispatcher in `config-to-spec/index.ts`, the `__mog*` IR field constants in `chart-ir/fields.ts`, the raw owner-key template strings scattered across `data-row-style.ts`, `resolver.ts`, and `pie-doughnut-geometry.ts`, the `isPreservedOnlyChartExFamily` (treemap/sunburst/regionMap returning empty rows/layers), the `HIDDEN_CHART_CELL`/`valueState` semantics, and the `literal`/`cacheFallback` source-kind policy in imported extraction. The plan is not a hallucinated wishlist; the author clearly read the folder.

The core proposal is a contract-driven refactor: an exhaustive chart-family registry, named `configToSpec` pipeline phases, an IR row schema, explicit extraction plans, centralized typed style owner keys, a unified style/color authority resolver, a normalized axis bundle, and registry-backed family builders — all "behavior-preserving first," with strong invariant documentation and concrete verification gates. The invariants section (lines 75–127) is the best part of the document: specific, testable, and tied to real failure modes.

The main reasons this is an 8 and not a 9–10: the scope is very large for a single plan (eight substantial implementation sections touching dozens of files), the user-visible payoff is thin (it is almost entirely internal architecture cleanup plus closing two real gaps — heatmap/violin), and the behavior-equivalence verification strategy is slightly underspecified relative to the risk a refactor this size carries.

## Major strengths

- **Accurate scope and ownership boundaries.** Purity (DOM-free, workbook-state-free), the kernel/`@mog/charts` split, and the canonical contracts location are all stated correctly and match the code. The non-goals (lines 33–38, 366–372) are crisp and prevent scope creep into the grammar compiler, renderer, or workbook state.
- **Excellent invariant capture.** Lines 84–127 enumerate the subtle contracts a refactor here would most easily break: hidden-vs-blank distinction, `sheetId` pass-through, imported live-vs-cache precedence, render order (`collectMarks`), source-series identity fields, and the color fallback chain. These read like they were extracted from the tests and code, not invented.
- **Type-level enforcement.** Using `satisfies Record<ChartType, ChartFamilyContract>` to force a compile error when a new `ChartType` is added is the right mechanism and mirrors the existing `MARK_TYPE_MAP: Record<ChartType, MarkType>` guard.
- **Identifies a genuine gap, not a cosmetic one.** Section 8 correctly notes heatmap and violin fall through to generic `rect`/`violin` marks via `MARK_TYPE_MAP` with no documented data/axis/style contract (confirmed: neither has a dispatch branch in `index.ts`). This is a real, defensible improvement.
- **Concrete, runnable verification gates** (`pnpm --filter @mog/charts test`/`typecheck`, cross-package `pnpm typecheck`, named new suites) plus sensible sequencing and parallelization with explicit dependency ordering.

## Major gaps or risks

- **Scope is too large to land as one unit.** Eight implementation sections, each a multi-file refactor (registry, pipeline phases, row schema, extraction plans, owner keys, authority resolver, axis bundle, family builders), plus a fixture matrix across ~50 chart types. There is no proposed minimal first slice that delivers value and de-risks the rest. The parallelization section helps, but a refactor of this breadth needs an explicit "land registry + heatmap/violin fix first, defer the rest" cut so the work can ship incrementally rather than as a giant in-flight branch.
- **Behavior-equivalence is the central risk but the harness for it is vague.** The plan repeatedly promises "behavior-preserving first" and "after output equivalence is locked down," yet section 9 says "use snapshots sparingly … prefer targeted structural assertions." For a pure function like `configToSpec`, the strongest guard against silent regressions is a comprehensive before/after golden snapshot over a corpus of configs — exactly what the plan soft-pedals. There is tension between "guarantee equivalence" and "avoid brittle full snapshots" that is never resolved with a concrete mechanism (e.g., a frozen pre-refactor spec corpus diffed in CI, then relaxed to tolerant assertions only for numeric-geometry fields).
- **Partial redundancy with existing guards.** Objective 2's exhaustiveness goal is already partly enforced: `MARK_TYPE_MAP` is `Record<ChartType, MarkType>`, so a new `ChartType` already breaks the build there. The plan should acknowledge this and frame the registry as adding *policy* exhaustiveness (axis/legend/blank/export) on top of the existing *mark* exhaustiveness, rather than implying nothing enforces it today.
- **No effort/size estimate or rollback story.** Given the owner-key and IR-field migrations touch grammar and OOXML deep importers, there is no statement of how large the change is, how long output-equivalence locking takes, or how to revert a slice if a consumer test regresses mid-migration.
- **"Internal diagnostics, no public output change" is asserted but unverified.** Sections 2/4 add diagnostics "without changing public output." Because the public return is `ChartSpec` and rows carry `__mog*` fields consumed by grammar/export, any added field is effectively semi-public. The plan should state explicitly that diagnostics must not be attached to returned rows/spec unless versioned (it gestures at this but does not make it a hard gate).

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. The field-ownership rule (one owning builder per row-field group, line 102), the owner-key fallback priority, the color authority fallback order (line 120), and the render-order contract are all specific enough to write tests against, and the plan does propose those tests (`chart-ir-row-schema.test.ts`, `style-owner-keys.test.ts`, `style-authority-contract.test.ts`, `all-chart-types-smoke.test.ts`).

Verification gates are concrete and correct for the package. The gap is not *which* commands to run but *what defines pass* for a behavior-preserving refactor: there is no defined equivalence oracle (frozen corpus diff) and no acceptance criterion for "output unchanged" beyond per-suite green. The proposed `all-chart-types-smoke.test.ts` ("compiles or explicit preserved-only empty result") is a good coverage net but only checks shape, not equivalence with pre-refactor output.

One factual nuance to keep honest in implementation: `MARK_TYPE_MAP` *does* assign `rect`/`arc` to treemap/sunburst/regionMap, but those entries are never reached because the dispatcher short-circuits them through `isPreservedOnlyChartExFamily`. The plan's claim that these "preserve data/export semantics without rendering placeholder geometry" is correct in effect — the registry work should make that explicit so the unreachable `MARK_TYPE_MAP` entries don't mislead a future reader.

## Concrete changes that would raise the rating

1. **Add an explicit phased delivery cut.** Define a minimal first slice (family registry + the heatmap/violin policy fix + the equivalence harness) that ships standalone, then sequence the remaining sections behind it. State that no later slice merges until the equivalence harness is green.
2. **Specify the equivalence oracle.** Commit to a concrete before/after mechanism: freeze a corpus of `configToSpec` outputs across all chart types/subtypes pre-refactor, diff structurally per slice, and relax to tolerant numeric assertions only for the named geometry-sensitive families (stock, radar, pie/doughnut, Excel cartesian). Resolve the "snapshots sparingly" tension explicitly.
3. **Acknowledge the existing `Record<ChartType, …>` guard** and reframe the registry as policy-level exhaustiveness, so the plan doesn't over-claim novelty for mark-type completeness.
4. **Make "no public output change" a hard gate.** State that no diagnostic field may appear on returned rows/`ChartSpec` without a versioned contract change, and add a test asserting the set of `__mog*` fields on output is unchanged by the refactor (the row-schema test can double as this guard).
5. **Add a rollback/size note** for the owner-key and IR-field migrations, given grammar/OOXML deep-import consumers, so a mid-migration consumer regression has a defined recovery path.
6. **Tie heatmap/violin closure to a defined target.** "Document the contract" is open-ended; specify the intended render policy (rendered, rendered-with-diagnostics, or preserved-only) so the gap-closing work has an acceptance criterion rather than just "don't fall through to generic."
