Rating: 8/10

# Review of Plan 058 — `runtime/embed/src`

## Summary judgment

This is a strong, evidence-grounded plan. I cross-checked its central claims against the actual source tree and they hold up: there genuinely are two competing event maps with divergent `sheetChange` shapes, the web component leaks `private _*` field names through TS `private` (not `#`), the API Extractor rollup only covers `index` and `react` (not `web-component`/`config`), `tsup.config.ts` emits both `client` and `internal/views-host` entries, and the `iframe`/`publish` barrels are stamped `@stability public-experimental` while `EXPOSURE.md` classifies them `reserved`. The plan also references real verification scripts (`check:declaration-rollups`, `check:api-snapshots`, `check:external-fixtures`, `check:private-leaks`, `check:platform-dependencies`, `validate:packages`, `build:public-artifacts`) that all exist in the root `package.json` — these are not invented gates. The architecture framing (facade over kernel/sheet-view, host-policy as the only source/effective-state authority, same-page ≠ trust boundary) matches the code.

The main thing holding it back from a 9–10 is scale and a few under-decided contract points. It bundles ~12 large workstreams (shared controller, both wrapper refactors, config schema overhaul, save/export submodule, declaration rollups, artifact stripping, iframe hardening, publish disposition, external fixtures, browser E2E, docs) into one plan. Each is justified, but the plan is closer to an epic than a single executable unit, and a few of its contract decisions are deferred to the implementer rather than resolved.

## Major strengths

- **Accurate, falsifiable observations.** Nearly every "current state" claim is verifiable in source. Example: line 112/156's complaint about the web-component `{ index }` sheet-change detail is confirmed at `mog-sheet-element.ts:420` (`new CustomEvent('mog-sheet-change', { detail: { index } })`), while `MogEmbedEventMap.sheetChange` in `config.ts` is `{ index; name; sheetId }`. This is a real, well-targeted inconsistency.
- **Identifies the dual event-map problem precisely.** `types.ts` exports `EmbedEventMap` (`sheetChange: number`, `select`) and `config.ts` exports `MogEmbedEventMap` (richer, `selectionChange`, `dirtyChange`, etc.). Both are re-exported from `index.ts`. The plan's call to make `MogEmbedEventMap` the real emitted contract and reconcile wrappers is the correct resolution.
- **Strong trust/authority modeling.** The "source and trust" invariants (opaque `MogEmbedSourceRef` only, `resolveSource` as sole materialization hook, callbacks never grant authority, save requires effective `save` + `host-callback`/`autosave`, export requires effective `export` + non-`none` save policy) exactly match `effective-state-gates.ts` and `same-page-host.ts`. The plan strengthens rather than reinvents.
- **Real verification gates with the right altitude.** It explicitly refuses to let "typecheck + source tests" count as done for a browser package and demands packed external fixtures + browser E2E through public entrypoints. The positive/negative external-import fixture matrix (must import `react`/`web-component`/`config`; must NOT import `client`/`iframe`/`publish`/`host-adapters/*`/`internal/views-host`) is the most valuable single test idea in the plan.
- **Reserved-surface discipline.** Catching that `iframe/index.ts` and `publish/index.ts` advertise `public-experimental` while EXPOSURE says reserved is a genuine, shippable-risk finding (these are externally confusing stability tags on code that isn't even a package export).
- **Sensible parallelization with a real dependency gate** ("once the shared session contract is agreed"), and a clean, accurate non-goals list.

## Major gaps or risks

- **Scope/sequencing.** Twelve workstreams under one plan number is a lot. The shared `same-page-embed-session.ts` controller (step 1) is the keystone everything else depends on, yet it is described at roughly the same granularity as a docs update (step 12). I'd want step 1 broken into its own sub-plan with the `SamePageEmbedSessionHandle` surface fully enumerated before the wrapper refactors begin, because the React and web-component refactors cannot start in parallel until that interface is frozen — the plan says this but doesn't make the controller contract concrete.
- **Range-parser contract left open.** Step 6 / edge cases say to support "single cells, rectangular ranges, absolute refs, lowercase refs, optional whitespace, and sheet-qualified refs *if the public config allows them*" — but the plan never decides whether sheet-qualified ranges are in scope. That's the one ambiguity that will produce divergent wrapper behavior again. The plan should pick a grammar and make it the contract, not hand the decision to the implementer.
- **Event-map deprecation not explicit.** The plan says reconcile, but never states "delete `EmbedEventMap` from `types.ts` and re-point `index.ts`/the renderer's `renderer.on('sheetChange', …)` to the unified type." Since the renderer emits `sheetChange(index: number)` (used at `mog-sheet-element.ts:418`), the unification has a concrete downstream edit the plan glosses over.
- **`requestExport` return-type coupling.** `same-page-host.ts` types `requestExport` as returning `Blob | null`, which ties the public host policy to a DOM `Blob`. The plan tightens export *gating* but doesn't examine whether `Blob` in the public `MogEmbedHostPolicy` is the intended cross-environment contract. Worth a line.
- **Version/asset drift not surfaced.** `index.ts`'s header advertises a CDN URL pinned to `v/0.1.0/embed.js` while the package is at `0.8.0` and `SDK_VERSION` is build-injected. The docs step (12) is the natural place to catch this but doesn't mention it.
- **Migration/compat is asserted, not analyzed.** The plan leans on "this project has no external compatibility burden" to justify aggressive config-schema tightening. That's plausible for `public-experimental`, but stricter validation that rejects currently-tolerated host metadata is a behavior change the plan should at least enumerate (which fields move from tolerated to rejected), even if it then accepts the break.

## Contract and verification assessment

Contracts are unusually well-specified for a plan of this breadth. The "production-path contracts and invariants" section reads like an acceptance spec: entrypoint exposure per subpath, source/trust rules, lifecycle semantics of `ready`, error-normalization rule, and package-boundary forbidden-dependency list. The forbidden declaration-string assertions (`@mog/types-`, `@mog-sdk/kernel`, `MogClient`, `EmbedRenderOrchestrator`, `Workbook`, private field names) are concrete and testable, and they map onto the real `check-private-leaks` / declaration-rollup tooling.

Verification is the plan's strongest dimension. It correctly distinguishes three tiers (unit/package, packed-artifact + external-consumer, browser E2E), and it ties each tier to commands that exist. The one weakness: it lists the gates but does not state pass/fail acceptance thresholds or which gate is the blocking one for promotion-to-stable vs. merely informational. "Renders nonblank content" and "selection emits events through real pointer input" are good behavioral assertions but would benefit from a named fixture workbook and expected values (the plan does point at `runtime/embed/public/showcase.xlsx` or a generated sample, which is reasonable).

One inaccuracy to flag: the plan asserts API Extractor rollup entries should be *added* for `web-component` and `config` — correct, and confirmed (`scripts/rollup-public-types.mjs` `ROLLUP_ENTRIES` only has `index` + `react`, and only `api-extractor.json` / `api-extractor-react.json` exist on disk). No config files for the other two subpaths exist yet, so this is net-new work the plan correctly scopes rather than under-counts.

## Concrete changes that would raise the rating

1. **Split out the session controller as a prerequisite sub-plan** and fully enumerate `SamePageEmbedSessionOptions` and `SamePageEmbedSessionHandle` (method signatures + the exact event payloads) before any wrapper refactor — make it explicit that steps 2 and 3 are blocked on this interface freezing.
2. **Decide the `navigateToRange()` / `config.range` grammar now.** State whether sheet-qualified refs are supported; if not, specify the exact rejection behavior. This is the residual source of wrapper divergence.
3. **Make the event-map unification a concrete edit list:** delete/deprecate `EmbedEventMap` in `types.ts`, re-point `index.ts` exports, and reconcile the renderer's `sheetChange(index: number)` emission with the `{ index, name, sheetId }` public detail.
4. **Enumerate the config fields that move from tolerated to rejected** under the new schema validator, so the behavior change is auditable even though compatibility is not a constraint.
5. **Add explicit promotion thresholds** to the verification section: which gates block stable promotion, and a named fixture workbook with expected rendered/selection values for the browser E2E assertions.
6. **Add a line on `requestExport` returning `Blob`** and the `SDK_VERSION` vs. hardcoded CDN-version drift in `index.ts` to the docs/contract cleanup so the public surface is internally consistent.
