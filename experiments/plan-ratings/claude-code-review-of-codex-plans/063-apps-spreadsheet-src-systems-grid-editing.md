- **Production-path discipline.** It explicitly scopes out test-harness-only optimizations, compatibility shims, and simulator-only paths, and repeatedly insists read-only/mutation policy applies only to human UI paths, not kernel/API calls â€” which matches the existing test's `does not affect direct kernel-style function calls` assertion.
- **Invariants-first.** The "contracts and invariants to preserve or strengthen" section is excellent and would serve as a regression-protection checklist even if only part of the plan is executed.
- **Layered verification.** Unit/coordinator â†’ integration (`GridInteractionSimulator`, `SheetSimulator`) â†’ browser/UI is the correct ladder for a subsystem whose correctness depends on DOM focus, pointer bubbling, IME, and clipboard. - **Awareness of the public boundary.** It flags that `exports.ts` makes actor/event/type changes breaking for package consumers and points to `check:publish-readiness:fast`.

## Major gaps or risks

- **Scope is a program, not a plan.** Objectives 1â€“10 each map to a substantial refactor of a 43k-line subsystem. Seven parallel workers with cross-cutting shared contracts (session identity, input intents) is high-coordination, high-merge-risk work. The plan never estimates effort, never defines a minimal first increment that delivers value, and offers no fallback if only 2 of 7 workstreams land. Realistically this risks a half-migrated subsystem with two coexisting models â€” the exact state it's trying to eliminate.
- **Contract files are named but not specified.** Objective 1 proposes `contracts/actor-topology.ts`, `lifecycle.ts`, `input-intents.ts`, `mutation-policy.ts`, `edit-session.ts` and says "they are not documentation-only," but no actual TypeScript interface signatures are given. "Contract clarity" is high at the invariant/prose level and low at the type level. Without pinned signatures, seven parallel workers will diverge on the shared types the plan itself says they must share.
- **Delta vs. existing intent is sometimes fuzzy.** Machines already don't import each other (verified); generation checking already exists in `edit-entry-service.ts`; blur-is-not-commit is already enforced. The plan says "strengthen," which is fair, but for objectives 2/3 the incremental work over what exists is underspecified â€” a reader can't tell how much is net-new vs. relabeling.

## Contract and verification assessment

Verification gates are the plan's second-strongest section and are appropriate. The command list (`pnpm --filter @mog/app-spreadsheet test -- src/systems/grid-editing`, typecheck, publish-readiness when exports change, browser/app-eval for DOM-dependent behavior) is correct and correctly deferred to implementation time. The unit/integration/browser split is well chosen for this subsystem.

The weakness is on the contract side, not the verification side. The plan asserts contracts will be enforceable code but never shows a single concrete interface. For a plan whose central thesis is "turn comments into enforceable contracts," the absence of even one fleshed-out `interface GridEditingInputIntent` / `EditSessionId` / `MutationDecision` signature is a real gap â€” especially because the parallelization model explicitly requires workers B and C to "share the same session identity and input intent types." Those shared types are the integration seam and should have been pinned in the plan itself.

The "no public method returns placeholder null/no-op for a declared required capability" test (objective 5 / public API tests) is a genuinely good, checkable acceptance gate and directly addresses the verified `subscribeToCellPropertyChanges`/`getSlicerCache` defects.

## Concrete changes that would raise the rating

1. **Pin the shared contract types in the plan.** Add concrete TS signatures for at least `GridEditingInputIntent`, `EditSessionId`/edit-session identity, the `GridEditingMutationPolicy` interface with its `MutationDecision`/denial-reason shape, and the lifecycle-node type. These are the cross-worker seams; specifying them is what makes the parallelization claim credible.
2. **Define a minimal viable first slice with its own acceptance criteria.** E.g., "Phase 0: fix the two verified public-API defects (`subscribeToCellPropertyChanges`, `getSlicerCache`) + replace `read-only-mode.test.ts` source-inspection block with behavior tests against a real `GridEditingSystem`." That delivers value, exercises the ESM-linker problem the existing test dodges, and de-risks the rest.
3. **Concretize "lock current behavior."** Name the mechanism (characterization tests via `GridInteractionSimulator` for the top ~15 invariants, listed) and make "baseline captured" an explicit gate before any code moves.
4. **Scope the upstream-events dependency (Worker F).** State which workbook/kernel events are missing today, who owns that surface, and the interim behavior if they don't land (keep current bootstrap/sheet-switch refresh, ship the coalescing index ready to consume events later).
5. **Add a coexistence/rollback strategy.** Define how the old and new lifecycle/clipboard/input paths coexist during migration and the criteria for deleting the old path, so a partially-landed program doesn't leave two models permanently.
6. **Correct the read-only-test characterization** to acknowledge the existing behavioral dispatcher block, and frame the work as "extend the behavioral block to cover all mutation paths and drop the source-inspection block" rather than implying the file has no behavior coverage.
7. **Right-size or explicitly stage the program.** Acknowledge this is a multi-increment program and sequence it so each of the seven workstreams has an independently shippable, independently verifiable deliverable rather than presenting it as one atomic refactor.

Overall: accurate, production-focused, invariant-rich, and well-verified â€” clearly above average. Held to 8 by program-sized scope without per-increment acceptance criteria and by contracts that are described in prose but not pinned as types.
