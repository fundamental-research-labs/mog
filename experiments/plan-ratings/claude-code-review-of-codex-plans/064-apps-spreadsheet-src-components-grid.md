Rating: 8/10

# Review of Plan 064 — apps/spreadsheet/src/components/grid


## Summary judgment

This is a strong, unusually well-grounded plan. I verified its specific claims against the live source and nearly all of them are accurate to the line. The author clearly read the code rather than describing it generically: the "migration pending" comments in `useRenderContextConfig.ts` (lines 267, 396), the regex A1 parsing and `String.fromCharCode()` column increment in `useSparklineCFIntegration.ts` (lines 177/203/217/219), the `rects[0] ?? null` silent single-rect picks in five editor overlays, the render-time `getViewport()` call in `ScrollContainer.tsx` (line 154), the "Split view coming soon" placeholder boxes (lines 199–253), the `coordinator: any // SheetCoordinator type` in `useSearchHighlights.ts`, the `let table: any` in `AccessibilityAnnouncer.tsx`, and the rich-text editor's "Convert segments back to plain text … TODO: Store segments directly" all exist exactly as described. `SpreadsheetGrid.tsx` is 1059 lines, matching the "1000-line" characterization.

The plan correctly identifies the central architectural problem for this folder: it is an app-integration layer that has accreted a large, weakly-typed surface of renderer callbacks, async pushes, overlay anchoring, and editor mounting, and the highest-value work is to convert that surface into explicit typed contracts with one disposition per data source and one race policy for async reads. The objectives, invariants, and implementation steps are coherent and trace cleanly to observed code.

The main reason this is an 8 and not a 9–10 is scope discipline. As written it is a multi-agent, multi-quarter program (14 implementation steps, 9 parallel agent tracks, several steps that are themselves large projects), with no MVP slice, no explicit ordering of must-do vs. nice-to-have, and no captured behavioral baselines for the render-isolation goals it calls "measurable." It reads more like an excellent backlog charter than a single executable plan.

## Major strengths

- **Evidence-based scope.** Every file named exists; the specific defects cited are real and located precisely. This dramatically lowers the risk that an implementer chases a phantom.
- **Clear separation of concerns.** It explicitly carves the app-boundary contracts (which callbacks exist, which overlays mount, which input path wins, how async is sheet-scoped) away from renderer/canvas internals, and repeatedly states those internals are out of scope — consistent with the real package boundaries (`canvas/grid-renderer`, `canvas/grid-canvas`, `views/sheet-view`, which all exist).
- **Strong invariants section.** The "production-path contracts to preserve" list is the best part: sheet-aware callbacks, binary viewport data as the source of truth, generation/cancellation-scoped async, coordinate-space-by-construction, multi-rect for frozen/split, real-input E2E, and render isolation as a contract. These are exactly the failure modes this folder is prone to.
- **Concrete, named artifacts.** `GridSurfaceRuntime`, `GridRenderDataSourceRegistry`, `useSheetScopedAsyncValue`, `grid-overlay-geometry`, `GridEditorSurface`, `grid-print-preview-adapter` give implementers tangible targets rather than aspirations.
- **No-shim discipline.** "Update production callers directly; no compatibility shims" is the right call given there are no external users, and it is stated explicitly (step 14, non-goals).
- **Good edge-case and risk coverage.** StrictMode double-mount, sheet-switch-during-edit/async, multi-region selections, IME + autocomplete, multi-letter columns, 1900/1904, high-DPR/fractional zoom, repeated SR announcements — these are the genuinely hard cases.

## Major gaps or risks

- **Over-scoped; no prioritization or sequencing within the program.** 14 steps with no "if you only do three things" ranking. Steps 3 (typed registry replacing the 30+ callback effect, with new coordinator/SheetView capabilities), 7 (real rich-text editor with segment storage + selection), and 11 (print preview adapter) are each substantial projects. The plan acknowledges some depend on capabilities that may not exist yet (registry → coordinator/SheetView, rich text → editor machine + Worksheet API accepting segments) but does not de-risk them with a spike or a "verify the capability exists first" gate. If those capabilities are absent, several agent tracks stall.
- **`GridSurfaceRuntime` is a large architectural bet with no intermediate checkpoint.** "Make `SpreadsheetGrid` a thin surface" over a 1059-line file touching dozens of subscriptions is high-risk for exactly the regression the plan warns about (re-widening subscriptions, reintroducing per-frame renders). The plan says "add render isolation tests before moving large blocks" — good — but doesn't define the baseline numbers those tests must hold, so "measurable" is asserted, not specified.
- **Verification gate factual error.** The plan lists `cd mog/print-export && pnpm test` and describes `mog/print-export` as an adjacent folder. That path does not exist; the package lives at `mog/file-io/print-export` (the import `@mog/print-export` in `ViewportTableDataProvider.ts` is correct, but the filesystem gate command as written would fail). Minor, but it's a verification gate that wouldn't run.
- **No acceptance baselines.** Render-count targets, scroll-frame budgets, and "context update frequency" are named as things to measure but no current numbers or target thresholds are captured, so there's no objective pass/fail for the performance/isolation objectives.
- **Browser/E2E gates are unanchored.** It mandates real-input E2E and lists what to exercise, but names no specific app-eval scenarios or harness, despite app-eval being the established mechanism in this repo. An implementer must invent the scenario set.
- **Registry's contract shape is under-specified.** The disposition fields (stable callback / push / async refresh / direct update; undefined=leave, null=clear; required dirty signal; scope key) are good, but the actual TypeScript shape and how exhaustiveness is enforced at compile time vs. test time is left entirely to implementation. Given exhaustiveness is the whole point, a sketch of the type would harden it.

## Contract and verification assessment

Contract clarity is above average for this experiment. The `undefined`-means-leave / `null`-means-clear convention, the per-source dirty/invalidation requirement, and the sheet/selection/generation scoping are precise and directly address real bugs (stale-sheet repaint is a recurring class here). The overlay-geometry helper API (`useCellPageAnchor`, `useRangePageAnchors`, `useCellContainerRect`, `useScrollSyncedEditorAnchor`) is a clean contract that maps to the observed coordinate-space confusion and the `rects[0]` problem, and extending the existing overlay coordinate lint to force a page/container choice is a smart enforcement hook.

Verification is mostly solid: focused package gates (`apps/spreadsheet` test + typecheck, `mog` typecheck) are correct and runnable, and the dependent-package gates are reasonable — except the `print-export` path error noted above. The "specific tests to add" list is genuinely useful and well-matched to the defects. The weaknesses are the lack of captured baselines for the isolation/perf claims and no named E2E scenarios. The plan is honest that it ran no commands (the queue forbids it).

## Concrete changes that would raise the rating

1. **Add a phased MVP slice with a hard ordering.** Designate a Phase 1 that delivers value with bounded risk and no cross-package dependency — e.g., (a) the generation-scoped async adapter + migrate the highest-risk stale-sheet readers, (b) overlay-geometry helpers + kill silent `rects[0]`, (c) the `as any`/regex-A1 cleanup. Defer `GridSurfaceRuntime`, the typed registry, rich-text completion, and the print adapter to later phases gated on Phase 1 landing.
2. **Fix the print-export gate path** to `mog/file-io/print-export` (or drop the path and reference the `@mog/print-export` package via its workspace filter).
3. **Capture baselines before refactoring.** Record current `SpreadsheetGrid` render counts under selection/scroll/cache-update and state explicit target thresholds the isolation tests must enforce — otherwise "make render isolation measurable" has no pass condition.
4. **De-risk capability dependencies up front.** Add a spike step that confirms the coordinator/SheetView data-source capability and the Worksheet-API rich-text-segment-without-flattening path actually exist (or scopes the work to add them) before the dependent agent tracks start.
5. **Name the E2E surface.** Specify the app-eval (or equivalent) scenarios that will cover editing, formula autocomplete, IME, paste options, pickers, split/scrollbar, and accessibility live regions, rather than only describing what to exercise.
6. **Sketch the registry type.** Include a minimal TypeScript shape for a `GridRenderDataSourceRegistry` entry and state whether exhaustiveness is enforced by a discriminated union at compile time or by the inventory test, so the central contract isn't left ambiguous.
