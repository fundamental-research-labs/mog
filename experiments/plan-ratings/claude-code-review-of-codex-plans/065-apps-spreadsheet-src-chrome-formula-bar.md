## Summary judgment

This is a strong, evidence-grounded plan. Its diagnosis of the `chrome/formula-bar` folder is not generic boilerplate â€” nearly every concrete gap it names is verifiable in the current source, and several are precise enough to act on immediately. The plan correctly identifies the folder's real role (it coordinates four user-visible contracts, not "a text field"), preserves the right invariants (editor state machine owns editing, keyboard coordinator owns Enter/Tab/Escape, Workbook/Worksheet APIs stay the durable source), and proposes a sensible extraction (typed view model + unified text control + typed name-box directory) rather than a vague "clean this up."

The main reason it is an 8 and not higher: the scope is very large for a single folder workstream (10 implementation steps, 5 parallel agents, ~8 unit-test families and ~8 app-eval scenario families), and a few objectives quietly depend on changes outside the folder (new Worksheet API metadata for projected-cell source) that are framed as "if needed" rather than committed and sequenced. Some of the proposed test surface is aspirational and would be hard to fully land. None of this is disqualifying; it is an ambition/sequencing risk, not a correctness problem.

## Major gaps or risks

- **Scope vs. one coherent landing.** This is effectively a folder rewrite (view model + text control + highlighter rework + autocomplete + context-menu command service + name-box directory/navigator + chrome-control typing + caller updates). The five-agent split is plausible but several streams share files (`FormulaBar.tsx`, `FormulaBarContainer.tsx`, `internal-api.ts`), so the "parallel" framing understates merge coordination. The plan would be stronger with an explicit ordering: land the view-model + invalidation contract first (it's the shared dependency), then text control, then the rest.
- **Cross-folder API dependency is soft.** Step 3 / risks both say projected-cell source readback "may require Worksheet API metadata that is not yet exposed" and that the fix is to add public API surface. That is the right instinct, but it is a dependency on the contracts/kernel layer that is left as a conditional. If that metadata is in fact missing, this becomes a multi-folder change with its own review surface, and the plan does not size or gate it. This should be investigated up front, not discovered mid-implementation.
- **Async guarding is slightly mischaracterized.** The plan implies refresh is purely "ad hoc counters." In reality the container already uses `cancelled`-flag cleanup guards around the async `getCell`/`getRawCellData`/`refreshActiveCellData` reads. The plan's stronger proposal (sheet/cell/generation guards so a late read can't repaint for a *newer* active cell, beyond same-effect re-runs) is a real improvement, but the writeup should acknowledge the existing guard so the implementer doesn't assume there's none and reason about the actual delta.
- **"No compatibility shims / update callers directly" is asserted without a caller inventory.** Step 10 forbids dual APIs and says to update consumers in the same workstream, but the plan never enumerates the consumers of `FormulaBarProps` / autocomplete / context-menu helpers. Without that inventory the blast radius of the typing changes is unknown, which is the kind of thing that derails "update all callers" steps.
- **Test surface is large and partly aspirational.** IME/composition, multiline scroll-sync alignment, and clipboard activation behavior are notoriously hard to assert reliably in unit tests; the plan lists them as required without flagging which are best-effort vs. firm gates. Some prioritization (which tests are blocking vs. nice-to-have) would make the gate honest.

## Contract and verification assessment

The verification section is above average. It separates focused package gates (`pnpm --filter @mog/app-spreadsheet test -- src/chrome/formula-bar`, the autocomplete/FunctionSuggestions test, `typecheck`) from a repo-level `pnpm typecheck` triggered only if public contract types change â€” that conditional is the correct discipline. The unit-test families map cleanly onto the view-model fields and the source-resolution inventory, and the app-eval families map onto real, existing scenarios plus named new ones, with a concrete DOM-readback instruction (`[data-formula-bar] input, [data-formula-bar] textarea`) that forces proof of the user-visible value rather than internal state. The explicit "tests must use real UI input paths; direct state mutation is unit-only, not proof the feature works" rule is exactly right for this folder's history.

Two weaknesses: (1) the gates do not state expected pass criteria or a baseline (e.g., are the named new scenarios expected to be authored and green, or just exercised?), and (2) there is no gate tied to the cross-folder Worksheet-API-metadata contingency, which is the riskiest part. The contracts themselves are clear and testable; the brace policy, hidden-formula, forced-text, and calculated-column distinctions are all stated with enough precision to write assertions against.

## Concrete changes that would raise the rating

1. **Add a dependency-ordered sequencing section.** Land the typed view model + named invalidation first (shared dependency), gate it, then fan out text-control/highlighter/autocomplete/name-box. Name which files each agent owns to surface the real merge conflicts on `FormulaBar.tsx`/`FormulaBarContainer.tsx`/`internal-api.ts`.
2. **Promote the projected-cell metadata question to a step-0 investigation with a go/no-go.** Determine now whether Worksheet exposes source metadata for CSE/dynamic-array/Data-Table/pivot cells. If not, scope and gate the contracts/kernel API addition as its own reviewed sub-plan instead of an inline "if needed."
3. **Add a caller inventory for the typed contracts** (`FormulaBarProps`, autocomplete props, context-menu helpers) so Step 10's "update all callers, no shims" has a known blast radius.
4. **Acknowledge the existing `cancelled` guard** and frame the generation-guard work as the specific delta (cross-cell/cross-sheet late-read suppression), with a test that fires a stale `getCell` resolution after an active-cell change.
5. **Tier the test list** into blocking gates vs. best-effort, and state expected outcomes (which app-eval scenarios must be authored-and-green). Call out IME/scroll-sync/clipboard-activation as the hard-to-assert items.
6. **Add one quick win callout** that the multiline-highlighter visibility bug (`isFormula && !isMultiLine` + `text-transparent`) is independently shippable ahead of the full rework, since it is a clear correctness regression with a small surface.
