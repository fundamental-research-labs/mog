# Review of `072-contracts-src-bridges.md`

## Summary judgment

This is an accurate, boundary-aware, and verification-rich plan for a small, well-understood folder. Its description of the current state matches the code exactly: I confirmed that seven of the eight per-bridge files are already minimal `export type * from '@mog/types-bridges/<bridge>'` shims, that `index.ts` already does `export type * from '@mog/types-bridges/bridges'` plus a value re-export of `DEFAULT_RECOGNITION_THRESHOLDS`, and that `contracts/src/bridges/ink-recognition-bridge.ts` is a **byte-for-byte duplicate** of `types/bridges/src/ink-recognition-bridge.ts` (`diff` reports identical). The runtime inventory entry for `@mog/types-bridges/bridges` exists and points at the contracts ink file with `runtimeExports: ["DEFAULT_RECOGNITION_THRESHOLDS"]`, and every `pnpm check:*` gate the plan names resolves to a real script in `package.json`. The plan's grasp of the public/private split and the declaration-rollup leak risk is genuine and well-targeted.

The main weakness is scope calibration. A meaningful fraction of the proposed "improvement" (the per-shim consistency conversion) is already done, while a large speculative apparatus — production implementation-conformance reconciliation across five subsystems and a dozen typecheck/test gates, plus a five-agent parallelization plan — is attached to what is fundamentally a one-file dedup plus two small tooling/fixture additions. Most of that apparatus degrades gracefully to no-ops, so it isn't wrong, but it overstates effort and would mislead an estimator about the true size of the task.

## Major strengths

- **Verified-accurate current-state model.** Every factual claim I checked holds: the ink duplicate, the already-minimal shims, the `index.ts` shape, and the runtime-inventory entry (`sourceOfTruth: contracts/src/bridges/ink-recognition-bridge.ts`). Plans that misdescribe their target waste implementer time; this one does not.
- **Correctly isolates the one real code smell.** The byte-identical ink duplicate with a second `DEFAULT_RECOGNITION_THRESHOLDS` literal is the genuine defect here, and step 2 attacks it directly, including the right follow-on question (does any internal consumer still import the private constant, and if not, collapse to one runtime owner).
- **Strong public/private boundary discipline.** Keeping `@mog-sdk/contracts/bridges` public and `@mog/types-bridges` private, owning the runtime constant in contracts rather than re-exporting a private package's runtime value, and guarding with declaration-identity/rollup/external-fixture gates is exactly the failure mode this surface is prone to. The negative `types-star-import` fixture the plan wants preserved already exists and already lists `@mog/types-bridges`.
- **Runnable, real verification gates.** All named scripts (`check:contracts-runtime-inventory`, `check:contract-runtime-imports`, `check:contracts-declaration-identity`, `check:declaration-rollups`, `check:api-snapshots`, `check:external-fixtures`) exist, and the API snapshot file it targets exists. Ordering API-snapshot regen *after* rollup proof is the correct sequencing.
- **Sober risk section.** The DOM-canvas-types-in-declarations risk, the "removing the only runtime value" risk, and the "don't add per-bridge subpaths piecemeal" risk are all real and correctly framed as packaging decisions rather than local hacks.

## Major gaps or risks

- **Overstated shim work (objective 5, step 3).** "Convert per-bridge shim files into a systematic, auditable projection layer" implies conversion that has already happened. Seven of eight shims are already the exact minimal form the plan prescribes. The plan should say so plainly and reduce step 3 to "ink is the only non-conforming file; everything else is already canonical."
- **Speculative conformance work (step 8) is unlikely to fire.** Because the ink files are byte-identical and the other shims already project the canonical `types/bridges` definitions, the plan tightens **no interface**. With no contract tightening, the promised "drift exposed in kernel/charts/diagram/drawing/equations/text-effects" essentially cannot materialize. Pulling those six implementation files plus `@mog/charts`, `@mog/canvas-grid-canvas`, and `@mog-sdk/runtime-sdk` typecheck gates into a pure type-shim cleanup is disproportionate. Step 8 should be demoted to a contingent "if the dedup somehow changes a declared type, reconcile" note, not a first-class step.
- **Five-agent parallelization mismatches the scope.** Assigning whole subsystems (kernel, canvas, charts, apps, runtime) to a plan whose folder is nine files of one-liners plus one duplicate reads as scope inflation. Agents D and E in particular have little real work given no interface changes.
- **The one genuinely new mechanism is under-specified.** The drift-detection check (step 7) is the most valuable net-new deliverable, but the plan does not say where the allowlist of value-owning shims lives, what manifest format encodes it, or which tool owns it. The repo already has `tools/contracts-runtime-inventory.json` (with per-module `runtimeExports`) and `check-contracts-runtime-inventory.mjs`; the plan misses the chance to extend that existing manifest instead of inventing a parallel one. As written, the implementer must design the mechanism from scratch.
- **Stale comment fix is mislocated.** Step 3 says to fix misleading comments "in downstream code," but the actual stale comment ("Interface defined here in contracts/", "Implementation in engine/src/bridges/", "@see contracts/src/ink/types.ts") lives inside the ink file itself — and it imports from `@mog/types-objects/ink/types`, so the `@see` path is already wrong. Step 2 deletes the contracts copy, but the surviving canonical copy in `types/bridges/src/ink-recognition-bridge.ts` will keep the stale comment unless the plan explicitly says to correct it there. The plan should name that file.

## Contract and verification assessment

The contract invariants enumerated (chart `renderCached()` stays synchronous; pivot `compute()` is a pure read path that does not materialize/notify; schema validation sourced from Rust recalc; locale normalization advisory-only; complete ink thresholds; lifecycle cleanup; equation `Result<MathNode[], EquationParseError>`) are specific and read as correct domain knowledge — they are the right invariants to *preserve*, but note none of them is actually *changed* by this plan, which reinforces that step 8 is precautionary rather than load-bearing.

Verification coverage is the plan's strongest dimension and is concrete enough to execute. Two genuine coverage gaps the plan correctly targets, which I confirmed are real: (1) the positive runtime-values fixture (`fixtures/external/positive/contracts-runtime-values/inventory-runtime-smoke.ts`) currently only smoke-tests `toCellId` and does **not** import `DEFAULT_RECOGNITION_THRESHOLDS`; (2) there is **no** positive external fixture importing bridge *types* (`IChartBridge`, `IPivotBridge`, etc.) — `rg` finds none. Filling both is worthwhile. One precision note: the plan says "verify all expected threshold keys exist" but does not pin the expected key set (line, rectangle, ellipse, triangle, arrow, star, text), so the assertion's strength is left to the implementer.

## Concrete changes that would raise the rating

1. **Recalibrate scope explicitly.** State up front that 7/8 shims already conform and the only file changes are the ink dedup, the stale comment in the surviving canonical file, and two new fixtures plus one new/extended check. Demote step 8 and the kernel/charts/canvas/runtime gates to a contingency that runs only if a declared type actually changes (it won't, given byte-identical dedup).
2. **Specify the drift check against existing tooling.** Define it as an extension of `tools/contracts-runtime-inventory.json` + a new `check-contracts-bridges-shims.mjs` (or a clause inside the existing inventory checker): each `contracts/src/bridges/*.ts` must be either a pure `export type *` re-export of the matching `@mog/types-bridges` file, or an inventory-listed value-owning shim whose runtime exports equal the inventory's `runtimeExports`. Name the authority file and the failure message.
3. **Pin the fixture assertions.** Have the runtime fixture assert the exact seven threshold keys and assert the imported object is `===`-shaped to a frozen expectation; have the type fixture include compile-time assignability checks for each of the ten named types without fabricating production objects.
4. **Name the comment-fix target file.** Explicitly require correcting the stale "engine/src/bridges" / "contracts/" / `@see contracts/src/ink/types.ts` comment in `types/bridges/src/ink-recognition-bridge.ts`, since that is where it survives after dedup.
5. **Resolve the runtime-owner decision in the plan, not at implementation time.** A quick `rg` for internal importers of `DEFAULT_RECOGNITION_THRESHOLDS` from `@mog/types-bridges` would let the plan state definitively whether to delete the private runtime export or keep it behind an equality test — turning step 2's open question into a decision.
6. **Trim the parallelization section** to match the real surface (one agent for contracts+inventory+fixtures+drift check is sufficient), reserving subsystem agents only for the contingent conformance branch.
