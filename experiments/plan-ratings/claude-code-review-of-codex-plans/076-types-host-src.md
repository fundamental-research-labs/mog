Rating: 8/10

# Review of 076-types-host-src.md

## Summary judgment

This is a strong, evidence-grounded plan for hardening the `@mog-sdk/types-host` contract shard. Nearly every factual claim it makes about the current source is verifiable in the tree, and the weaknesses it targets are real production-path defects, not stylistic nits. The plan correctly understands that this is a *type shard* whose authority is enforced elsewhere (trusted adapters, `@mog/kernel-host-internal`, kernel gates), and it consistently refuses to move enforcement into the type package. Its main liability is breadth: it bundles roughly six independently large workstreams (export surface, primitives, trusted construction, untrusted protocol, fingerprint migration, binding/lifecycle typing, diagnostics, adapter migration) into one plan, and the most security-sensitive piece — the canonicalization/fingerprint migration — carries coordination risk that the plan acknowledges but does not fully sequence.

## Major strengths

- **Claims are accurate.** I spot-checked the strongest assertions and they hold:
  - `index.ts` does re-export `TrustedDocumentHostContext`, `TrustedHostBase`, `TrustedHostKind` from root, contradicting the narrow-subpath policy (verified).
  - `untrusted.ts` is genuinely a placeholder (`clientKind` + `protocolVersion` only).
  - The diagnostics misuse is real and egregious: `host-import-source.ts` has an `emitSuccess()` that emits `kind: 'documentAuthorization.denied'`; `validate.ts:847` emits success as `kind: 'hostConstruction.invalid'` with `code: 'HOST_VALIDATION_SUCCESS'`; `host-storage-preflight.ts` emits success via `kind: 'storage.failure'` with `providerRefId: '*'`. This is a correctness/auditability bug, not cosmetic.
  - Broad casts exist exactly where claimed: `as unknown as TrustedDocumentHostContext` at standalone-browser-host:999, node-headless-host:715, iframe-child-host:172, and the literal `kernel: null as unknown` TODO at iframe-child-host:165.
  - `bindings.ts` does use `unknown` and free strings (`createTransportConfig(): unknown`, `attach(rustDocument: unknown)`, `kind: string`, `role: string`, `has(sourceKind: string)`).
  - `fingerprints.ts` `HostCanonicalFingerprintProof.canonicalization` is hardcoded `'jcs-rfc8785'` while `canonicalJsonStringify` silently coerces functions/symbols/`undefined` to `'null'` (trailing `return 'null'`) and JSON-stringifies `NaN`/`Infinity` to `null` — so the JCS claim is false and the non-JSON-safe handling is unsafe, as alleged.
  - The duplicate canonicalizer exists: node-headless-host.ts:201 reimplements `canonicalJsonStringify` locally.
  - The ESLint subpath layering the plan relies on is already present in `import-boundaries.cjs` (per-layer view/shell/app rules, `/trusted` restricted to named construction modules/runtime facades/test fixtures).
- **Correct architectural framing.** The trusted/untrusted boundary invariants (origin authority from `MessageEvent.origin`, no kernel/handoff/raw-byte leakage to untrusted payloads, fail-closed expiry with `expiresAt === now` expired, exact tenant/workspace markers, replay-before-materialize ordering) match the existing bindings/validate contracts and are stated as invariants to *preserve*, not invent.
- **Verification gates are concrete and runnable.** Real package filters, real test names (`host-integration`, `host-operation-gate`, `host-storage-preflight`, `host-import-source`), boundary lint + external negative fixtures, and declaration-leak checks for the workspace-private guarantee. The "do not stop at typecheck" instruction plus security-specific assertions (untrusted payloads cannot carry kernel/handoff/raw bytes; replay consumed before resolver; domain-separation tests) are exactly the right gates for this folder.
- **Risks and non-goals are mature.** It explicitly forbids root compat shims, forbids weakening kernel validation/replay/raw-byte policy to make adapters compile, forbids treating the deterministic test host as the production path, and warns that changing canonicalization changes hash values and must be migrated in one coordinated pass.

## Major gaps or risks

- **Scope is very large for one plan.** Ten implementation sections across six parallel tracks (A–F) is closer to an epic than a single executable plan. The parallelization notes help, but tracks are not as independent as presented — e.g. Track B (fingerprint helpers) feeds Track E (adapter migration) and Track D (validation comparison helpers), and Track A (removing root export) can break consumers that Track C/E also touch. A reader cannot execute this as one atomic change; it needs an explicit phase ordering, not just a track list.
- **Canonicalization migration sequencing is under-specified.** The plan correctly says "migrate all production helpers in one coordinated change," but does not specify the ordering between (a) tightening/renaming canonicalization in `fingerprints.ts`, (b) replacing the duplicated canonicalizers in `validate.ts`/`host-storage-preflight.ts`/`host-import-source.ts`/`node-headless-host.ts`, and (c) recomputing expected hashes in tests/adapters. Because these hashes participate in security joins (decision id + nonce + principal/resource fingerprints), a half-applied migration could silently break a replay/materialization join rather than fail typecheck. This is the highest-risk part of the plan and deserves a step-by-step cutover (introduce v2 proof shape → migrate all producers/consumers together → flip → delete v1), which it does not give.
- **`/untrusted` vs `runtime/embed` ownership is left as "converge."** Section 4 invents a full protocol family (`HostBoundaryMessageEnvelope`, request/response/error, origin validation, redacted snapshots) while `runtime/embed` already owns the live iframe protocol. The plan says to "migrate toward these contract types … where the semantics overlap" but never decides who is canonical or how to avoid a second, competing vocabulary. This risks producing two protocol definitions and a stalled migration.
- **Several decisions are deferred rather than made.** "decide whether `source.ts` should become a `/source` subpath," "if file splitting is not worth the churn, keep `kernel.ts` as the barrel." These are reasonable to leave open in a design doc, but they reduce the plan's prescriptiveness — an implementer must still make non-trivial export-surface calls that affect `package.json` and lint rules.
- **Blast radius of removing the root export is asserted, not measured.** The plan relies on `pnpm typecheck` + the export-surface test to catch hidden root importers and forbids a shim. That is the right policy, but a quick `rg` for `from '@mog-sdk/types-host'` (bare) consumers would have let the plan state the actual count and name the files to fix, rather than leaving it to discovery.

## Contract and verification assessment

The contract section is the plan's strongest part. It enumerates the storage/export/management handoff field sets, the raw-byte and source-handle policies, the replay-before-materialize ordering, and the diagnostics redaction rules in a way that maps cleanly onto `bindings.ts`, `kernel.ts`, and the kernel gates. The proposed `HostCanonicalFingerprintProof` enrichment (purpose/domain separation in the hashed payload, `coveredFields`, honest `canonicalization` label, explicit byte-vs-JSON-byte-list identity) directly fixes a genuine security weakness — today two structurally-identical-but-semantically-different payloads (principal vs resource context) can collide, and the `'jcs-rfc8785'` label is a lie. Verification gates are specific, executable, and security-aware. The one gap is that the gates verify *behavior and types* but the plan does not propose a gate that detects partial fingerprint-scheme migration (e.g., a test asserting no production module still imports a local canonicalizer), which would be the cheapest guard against the highest risk.

## Concrete changes that would raise the rating

1. **Add an explicit phase order with gates between phases**, not just parallel tracks — e.g. Phase 1 export surface + primitives (independent), Phase 2 fingerprint scheme cutover (atomic, its own gate), Phase 3 binding/lifecycle/diagnostics typing, Phase 4 adapter migration, Phase 5 untrusted protocol. State which phases can land independently and which must land together.
2. **Specify the fingerprint migration as a step-by-step cutover** and add a guard test asserting no production package retains a local `canonicalJsonStringify`/sha256 helper once `types-host/fingerprints` is the single source.
3. **Decide `/untrusted` ownership** relative to `runtime/embed`'s existing protocol: name the canonical definition, the deprecation path for the embed-private shapes, and a gate proving there is exactly one protocol vocabulary.
4. **Resolve the deferred export-surface decisions** (`/source` subpath; `kernel.ts` split vs barrel) with a recommendation, since both affect `package.json` and `import-boundaries.cjs` that other tracks depend on.
5. **Quantify the root-export blast radius** by listing the current bare-root importers to fix, so removing the root re-export is a known edit set rather than a typecheck-driven discovery.
6. **Split into a small lead plan + linked sub-plans** if execution is meant to be incremental; as written it is one very large unit of work whose risk concentrates in the middle (fingerprint + diagnostics) phases.
