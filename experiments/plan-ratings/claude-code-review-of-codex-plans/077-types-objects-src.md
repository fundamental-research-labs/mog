Rating: 8/10

# Review of Plan 077: Harden `@mog/types-objects` Public Contracts


## Summary judgment

This is a strong, evidence-grounded plan. Spot-checking against the live source confirms nearly every concrete claim the plan makes: the package root really is `export {}` (`src/index.ts`), `ObjectPosition` is the loose optional bag described (`anchorType` discriminant with everything else optional — `objects/floating-object-types.ts:72`), `FloatingObjectBase.sheetId` is an explicit backward-compat alias of `containerId` (`:140-148`), `FloatingObjectKind` includes `camera`, `slicer`, and `formControl` (`:115-127`), `ChartObject.chartConfig` is `Record<string, unknown>` (`floating-objects.ts:482`), the `ShapeType` / `DiagramShapeType` overlap across folders is real and even self-documented in `index.ts`, and the diagram contracts carry exactly the prose-count comments the plan wants replaced ("exactly 14", "all 41", "all 64", "all 25", `text-effects/types.ts` "41 from DrawingML"). The cited cross-folder dependencies also exist: `tools/contracts-runtime-inventory.json`, `kernel/src/bridges/compute/floating-object-mapper.ts`, `domain-types/src/domain/floating_object/*`, and the `contracts/src/objects/*` re-export shims.

So this is not a hallucinated plan — it is a real audit of a genuinely underspecified contract folder, with a coherent target architecture (raw OOXML vs domain vs API vs resolved-render separation, discriminated anchors, unit-named numeric fields, canonical value tables, serialized-vs-runtime Map/Record split). The main reasons it is not a 9–10 are scope and altitude: it reads as a multi-folder epic rather than a single landable change, and it specifies by objective rather than by concrete type signature, leaving a few mechanisms ("generate or add `satisfies` fixtures if generation is not yet available") deliberately undecided.

## Major strengths

- **Accurate problem inventory.** Every named defect is verifiable in the tree. The plan clearly read the code rather than guessing, which is the single biggest predictor of a usable plan.
- **Correct architectural framing.** Separating raw OOXML / persisted domain / public API / resolved-render models, and making `Map`-vs-`Record` boundaries explicit at serialization, attacks the actual root cause of the drift rather than patching symptoms.
- **Sequencing is sound and safety-first.** Step 1 ("inventory and freeze the exported surface before changing it" + add declaration/API-report fixtures) is exactly right for a package whose `.d.ts` flows into the public SDK. Locking the baseline before mutating it is the correct first move and de-risks everything downstream.
- **Verification spans production paths, not just the type package.** It explicitly chains TS typecheck → contracts build + runtime-inventory check → SDK API report → type-level fixtures → kernel mapper tests → canvas render tests → XLSX round-trip → `cargo test -p domain-types` → manual browser pass. The "every Rust wire variant has an exhaustive TS mapper case" and "every object kind maps to exactly one supported/preserved contract" fixtures are precisely the gates that would catch the camera/slicer fallback-to-shape coercion it warns about.
- **Strong invariant section.** Preserving `sheetId`↔`containerId` and `position`↔`anchor` aliases "by construction, not by convention", the no-dependency-cycle constraint, the `mog` must-not-depend-on `mog-internal` rule, and the "unsupported imports must not silently become generic shapes" policy are all real, load-bearing constraints.
- **Honest non-goals and risks.** Calling out declaration churn becoming an SDK break, the chart/slicer cycle hazard, lossy `Map`→`Record` conversion, and "do not revive slicer JSON blobs" shows it understands the failure modes of its own proposal.

## Major gaps or risks

- **Epic-sized, not atomic.** The scope deliberately spans `types/objects`, `contracts`, `runtime/sdk`, Rust `domain-types` + compute/file-io, kernel, canvas, spreadsheet UI, and `typeset/math-engine`. The parallelization section acknowledges this and proposes seven agents, but there is no explicit landing strategy (single PR vs stacked vs feature-flagged) and no statement of which steps are independently shippable. A contract change that must touch Rust hydration + canvas + SDK reports in one shot is high-risk; the plan should commit to incremental, independently-mergeable slices behind the frozen baseline.
- **Spec-by-objective, not spec-by-signature.** For a *contract* plan, the absence of even one concrete proposed type is a real gap. It says "replace `ObjectPosition` with `TwoCellObjectPosition`, `OneCellObjectPosition`, `AbsoluteObjectPosition`" but never writes the discriminated shapes, and "introduce `CssPx`, `Emu`, `Degrees`, `Radians` using the least invasive branded or documented type strategy already used in core contracts" without identifying what that existing strategy actually is. The reviewer/implementer is left to rediscover it.
- **Undecided generation mechanism.** The canonical value tables and "generate const arrays from one source" repeatedly fall back to "or add exhaustive `satisfies` fixtures if generation is not yet available." Whether codegen exists is knowable now and materially changes the work; leaving it open weakens the estimate and risks two implementers picking different answers.
- **Cross-folder claims asserted but not co-located.** The "duplicate `ImportObjectStatus` definitions" live in the file-io bridge, not in this folder (this folder defines it once in `floating-object-types.ts`). The plan is right to flag it, but several of its highest-value asserts (Rust variant exhaustiveness, file-io duplicate) depend on folders outside the nominal scope; the plan should mark which gates can be satisfied within `types/objects` alone vs which require the partner folders to move first.
- **Root-export decision left as a fork.** Step 1 / objective 6 offers "intentionally empty tested root OR namespace-only root" without choosing. Given the existing `export {}` plus the documented `ShapeType` collision, this is decidable now; leaving it open invites churn in a public surface.
- **Branded-unit migration is under-bounded.** Objective/risk both note that introducing `Emu`/`Degrees`/etc. "can expose latent bugs in existing renderers." That is a large blast radius across canvas. The plan should scope a phased rollout (new/revised contracts only first, as it hints) more concretely, with a fixture proving no resolved-pixel regression, rather than treating it as one objective.

## Contract and verification assessment

The contract direction is correct and the verification design is unusually thorough for a type package — it explicitly refuses to "optimize test harnesses or mock paths instead of production paths," and routes through real Rust hydration, kernel mapping, canvas rendering, and XLSX round-trips. The type-level fixture set (kind→contract bijection, exhaustive Rust mapper cases, anchor-discriminant field requirements, JSON-safe serialized payloads, value-table coverage) is the right shape and would genuinely catch the regressions the plan fears. The runtime-export angle is handled correctly: it ties any runtime value movement to `tools/contracts-runtime-inventory.json` (which exists) and to regenerated SDK reports.

What is missing on the verification side is *acceptance specificity*: the gates are listed as commands and fixture intentions, but there are no concrete pass/fail assertions (e.g., "the API report diff for `@mog-sdk/contracts/objects` is empty except for the intentionally migrated symbols," or "the exhaustive-mapper fixture enumerates all N `FloatingObjectKind` arms"). Counts are referenced ("computed counts instead of comments") but no target numbers are pinned, so a reviewer cannot tell when the work is done. The manual browser pass is described well but lacks the specific imported-fixture matrix it should exercise.

## Concrete changes that would raise the rating

1. **Add a landing strategy.** State explicitly which of the 10 steps are independently shippable, in what order they merge, and how the frozen-baseline fixtures (Step 1) gate each subsequent slice. This converts the epic into a sequence of low-risk PRs and would move it toward 9.
2. **Write the headline type signatures inline.** Show the discriminated `TwoCell/OneCell/AbsoluteObjectPosition` union, the branded unit aliases (and name the existing core-contracts branding mechanism they reuse), and the variant-safe patch type that replaces `Partial<FloatingObject>`. One code block removes most interpretation risk for a contract plan.
3. **Resolve the open forks now.** Decide root-export policy (recommend: keep a tested-empty root + namespace-only re-exports, given the `ShapeType` collision) and confirm whether contract codegen exists — if not, commit to `satisfies` fixtures as the standing answer, not a fallback.
4. **Pin acceptance assertions and counts.** For each fixture, state the concrete invariant (kind count, mapper-arm count, empty-except-migrated API diff). Replace "exactly 14 / all 41 / all 64" prose with the computed-count source and assert the fixture fails on drift.
5. **Bound the unit migration explicitly.** Scope it to new/revised contracts in phase 1, with a resolved-pixel round-trip fixture proving canvas output is unchanged, before touching any existing renderer field.
6. **Tag each gate by scope ownership.** Mark which verification gates are satisfiable within `types/objects` vs which require `domain-types` / file-io / kernel / canvas to land first, so the partner-folder dependency order is unambiguous.
