## Major strengths

- **Ground-truth accuracy.** The "Current role" and "duplicate/divergent paths" sections are factually correct. Misleading APIs (`setUsedRange`, unchecked `getFirstId`, clamped `setFrozenPanes`) are named with the right rationale.
- **Clean ownership model.** "Rust owns durable state; `domain/sheets` owns TS adaptation, validation preflight, and sync rendering fallbacks" is stated up front and held consistently. The explicit non-goals (no TS formula rewriting, no grid-index mutation, no state migration out of Rust) are the right guardrails and match how `structures.ts` already delegates.
- **Invariants are specific and testable**, not platitudes: at-least-one-visible-sheet, `veryHidden` counts as non-visible, stable IDs across rename/move/hide but new ID on copy, zero-height reads for hidden dimensions in sync getters, bulk hidden enumeration must include dimensions outside the used range. These map directly to the edge-case list.
- **Three-surface metadata separation** (public `SheetMeta` / runtime `SHEET_META_SCHEMA` / `MirrorSheetMeta`) is a genuinely useful distinction and the schema-parity test idea (`Object.keys(SHEET_META_SCHEMA)` vs `SheetMetaField` with an explicit exclusion list) is a concrete, high-signal gate.
- **Cache-lifecycle diagnosis is precise.** Keying the module-level cache only by `sheetId` (not document identity) is a real cross-document leak risk on reused sheet IDs; the recommended fix (own the cache in document context / mutation handler, or key by document+sheet) is correct.
- **Verification section is layered and named** ŌĆö kernel test/typecheck, targeted compute-core tests, bridge regeneration/parity gate, `check:publish-readiness:fast`, and UI flows ŌĆö with characterization-tests-first sequencing.

## Major gaps or risks

- **Unverified bridge-endpoint assumptions.** Several keystone steps hinge on endpoints the plan only conditionally posits: a single `getSheetMeta` to collapse the five roundtrips (┬¦4); bulk `computeBridge.getHiddenRows`/`getHiddenColumns` (┬¦6); a first-class Rust print-settings and a page-break *set-snapshot* endpoint (┬¦9). The plan hedges ("if available or can be added"), which is honest, but the work and risk differ by an order of magnitude between "route to an existing method" and "add a Rust bridge method, regenerate bindings, add parity tests." The plan should have confirmed which already exist ŌĆö note that `dimensions.ts:223-224` already references `getHiddenRowsBitmap`/`getHiddenColumnsBitmap`, suggesting bulk-style endpoints may already exist and the plan under-leverages that signal.
- **Scope vs. queue item.** The folder is five files, but the plan effectively proposes re-homing workbook lifecycle CRUD, consolidating worksheet view/print/structure facades, and touching Rust invariants. That is a multi-week, multi-worker effort. The parallelization notes acknowledge this, but a reviewer/executor could reasonably want a smaller, shippable first slice explicitly carved out (e.g., "await + typed errors in lifecycle and dimensions, plus cache keying" as Phase 1) rather than a single large migration.
- **Last-visible-sheet enforcement location is left ambiguous.** ┬¦3 says keep enforcement in Rust "where possible" and add a compute invariant "if Rust currently allows" hiding the last visible sheet ŌĆö but the plan never establishes which is true today. This is the highest-stakes correctness invariant in the folder; whether it needs a Rust change should have been determined, not deferred.
- **`_maps`/`_origin` removal is under-decided.** ┬¦7 says remove `_maps` "only if all callers can migrate cleanly" without enumerating callers. Since `structures.ts` signatures are public-ish within the kernel and table row paths call them, a caller census belongs in the plan, not as a runtime discovery.
- **No explicit migration-safety net for receipts.** The plan repeatedly insists public receipts be derived from post-mutation canonical state, but doesn't propose a snapshot/golden test of current receipt shapes *before* the lifecycle re-home ŌĆö the most likely place for a silent regression.

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. The lifecycle function signatures in ┬¦2 are typed and return mutation results instead of booleans, directly fixing the error-swallowing in `sheet-crud-operations.ts`. The invariant catalogue is precise enough to drive tests. The wire-compatibility triangle (`store-types.ts` `SheetMeta` Ōåö generated bridge `SheetMeta` Ōåö `domain-types/src/domain/sheet.rs`) and the visibility-string parity check are exactly the contracts that drift silently, and tying them to `mirror-coverage.test.ts` extension is the right anchor since that is correctly identified as the strongest existing guard.

Verification gates are concrete and correctly scoped to where behavior changes (TS-only vs. Rust/bridge vs. public-contract vs. UI). The "characterization tests before implementation" instruction is appropriate for a refactor of live behavior. Weaknesses: the gates assume bridge regeneration is needed in several branches without confirming a Rust change is actually required, and there is no named gate for the cross-document cache-leak fix (the one concrete test in ┬¦6 covers hidden-zero and headless-override, but not the document-identity keying that ┬¦6 itself flags as the cleanest fix).

## Concrete changes that would raise the rating

1. **Resolve the endpoint assumptions up front.** State which already exist (`getSheetMeta`, bulk hidden reads ŌĆö check the `*Bitmap` methods already referenced in `dimensions.ts`, page-break set-snapshot, first-class Rust print settings) and which require new bridge work. Split steps that need new Rust/bridge surface into their own dependency-gated phase.
2. **Carve an explicit Phase 1** that ships value without the full migration: await + typed-error propagation in `lifecycle`/`dimensions`, document-identity cache keying with invalidation hooks, and the schema-parity test. Defer the worksheet view/print/structure consolidation to later phases.
3. **Determine the current last-visible-sheet behavior in Rust** and state definitively whether a compute-side invariant must be added, instead of "if Rust currently allows."
4. **Add a caller census for `structures.ts` `_maps`/`_origin`** and for each public API said to bypass `domain/sheets`, so the migration slices in ┬¦12 are sized against real call sites.
5. **Add a pre-migration receipt golden-snapshot gate** for `sheetRemove`/`sheetMove`/`sheetRename`/`sheetHide`/`sheetShow` to lock public behavior before lifecycle is re-homed.
6. **Tighten the `setUsedRange` decision** to a single recommendation plus the exact import/recompute call sites to check, rather than offering two open options.
