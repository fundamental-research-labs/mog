Rating: 8/10

# Review of 088 — compute-chart-render Native Chart Raster Surface

## Summary judgment

This is a strong, technically grounded plan. Its description of the current state is accurate against the source: `compute-chart-render/src` is indeed a single 1730-line `lib.rs`, it dispatches marks via a stringly-typed `RawMark`/`RawStyle`/`RawClip` shape with `match` on `"rect" | "path" | "line" | "area" | "arc" | "symbol" | "text"`, it accepts only version `1`, and the N-API bridge in `compute/napi/src/chart_render.rs` returns exactly `{ bytes, format, width, height }` and wraps every failure in the generic `chart render failed: {e}` message the plan flags. The core thesis — that the SDK serializer (`runtime/sdk/src/chart-export/node-chart-image-exporter.ts`) lossily projects the rich `ChartMark` IR down to `fill`/`stroke` strings and drops `fillPaint`/`strokePaint`/`line`/gradients/`shadow`/`richText` before native render — is verifiably correct: `chart-bridge.ts` defines all those fields (`fillPaint`, `strokePaint`, `line`, `shadow`, `richText`, `maxWidth`, `lineHeight`, `underline`, `strikethrough`, `miterLimit`) and the serializer collapses paints to `style.fill`/`style.stroke` strings. The objectives, contracts, module decomposition, and verification gates are coherent and the cited test files (`node-chart-image-exporter.test.ts`, `chart-export.test.ts`) and primitives (`charts/src/primitives/marks/symbol.ts` with `'x'`/`'dash'`) actually exist.

The main reasons this is not a 9–10: the scope is enormous for a single plan with no phasing/MVP ordering, the central verification gate (browser↔native golden parity) leaves its single biggest infrastructure dependency unspecified, and the plan repeatedly points at `charts/src` as the behavior-of-record without acknowledging that the public `charts/src` is minified/bundled output (e.g. `paintToCanvasStyle()` does not exist as a named symbol there — the logic lives in a minified `n()` in `rect.ts`), which makes "match `paintToCanvasStyle()`" harder to execute than the prose implies.

## Major strengths

- **Accurate current-state diagnosis.** Every concrete claim I checked held up: single `lib.rs`, `RawMark` string dispatch, version-1-only, the `{ bytes, format, width, height }` result contract, the generic `chart render failed` wrapper, and the lossy serializer projection. A plan built on a correct read of the code is rare and valuable.
- **Explicit, testable invariants.** The "contracts and invariants" section is the best part: logical-pixel coordinates with `pixelRatio` as the only logical→physical transform, returned dims equal `width*pixelRatio`/`height*pixelRatio`, paint order = mark order with single compositing, strict version rejection, and deterministic rejection paths for empty marks / bad formats / non-finite numbers / impossible allocations. These are assertable, not aspirational.
- **Clean module decomposition** (`request/style/color/path/geometry/symbols/text/surface/encode`) that explicitly preserves the public entry points, so the refactor is non-breaking by construction.
- **Right architectural boundary.** The plan firmly keeps chart grammar/scale/data extraction in `@mog/charts` and limits the crate to rasterizing already-compiled marks — and reinforces this as an explicit non-goal. This is the correct fit for the existing system.
- **Concrete verification gates** naming real commands (`cargo test -p compute-chart-render`, `cargo clippy`, `pnpm typecheck`) and real existing test files, plus robustness/fuzz gates for path and color parsing and a contract-fixture requirement that every mark variant crosses the TS→Rust boundary with no dropped fields.
- **Sensible parallelization** with five agent lanes keyed off an agreed schema, plus honest tolerance-based parity expectations for text/gradient/shadow rather than pretending byte-equality is achievable across engines.

## Major gaps or risks

- **Scope is an epic, not a plan.** "Make native a production-path peer of browser Canvas2D" bundles a full CSS color parser, linear/radial/rectangular gradients, shadow/blur offscreen passes, full SVG arc curve approximation, rich-text layout with wrapping/decorations/run overrides, a memory budget, diagnostics, and a cross-engine golden harness. Any one of gradients, shadows, or rich text is a multi-day effort. There is no MVP ordering or "ship in slices" sequencing — if time-boxed, there is no guidance on what to land first. The parallelization lanes mitigate but do not replace prioritization.
- **The golden parity harness is underspecified — and it is the load-bearing gate.** The plan asks to "render the same `ChartMark[]` through browser Canvas2D and the native N-API backend, then compare decoded pixels." It never says *how* the browser side runs in CI: headless Chromium, `node-canvas`, jsdom, Playwright? Each has very different fidelity, cost, and determinism. `node-canvas` (Cairo) is not the same renderer as browser Canvas2D, so a "browser" comparison using it would be measuring the wrong baseline. This is the single biggest unstated risk; the whole parity claim rests on infrastructure the plan does not design.
- **`charts/src` as behavior-of-record is partly illusory.** The plan tells implementers to "match `paintToCanvasStyle()`" and keep symbols "in lockstep with `symbol.ts`," but the public `charts/src` is minified — `paintToCanvasStyle()` is not a real symbol there (it appears as a minified `n()` inside `rect.ts`). Matching browser behavior by reading minified output is far harder than the prose suggests; the plan should point at the canonical un-minified source location or at observable Canvas2D semantics, not function names that don't resolve.
- **Undefined magic numbers.** Tolerances ("documented tolerances for text antialiasing and gradients"), the memory budget ("max physical pixel count," "max encoded work estimate"), and JPEG quality mapping are all deferred to implementation. Deferring tolerances is defensible; leaving the pixel-count budget with no even-rough ceiling weakens the "bound costs before allocation" objective into a TODO.
- **N-API test gate is conditional and vague:** "if the package has tests for the bridge, otherwise add one." `compute/napi` ships a `smoke-test.mjs` and `chart_render.rs` has no `#[test]`; the plan should just commit to adding the bridge test rather than hedging.
- **"Clipping in logical coordinates after DPR normalization"** is internally ambiguous — clip regions arrive in logical pixels but the pixmap is scaled by DPR; the plan asserts the ordering without describing the transform, which is exactly the kind of coordinate-space detail that causes subtle off-by-`pixelRatio` bugs.

## Contract and verification assessment

The contract section is the plan's strongest dimension and is genuinely production-path relevant: it pins the public N-API result shape, the version-strictness rule, the single-transform pixel-ratio invariant, paint-order semantics, and a concrete list of deterministic rejection cases. The requirement to evolve the TS serializer and Rust schema together (no compat shims, single-branch model) is consistent with how this repo works and removes a class of versioning risk.

Verification is mostly excellent — real commands, real test files, fuzz/property tests for the two riskiest parsers (SVG path, CSS color), a zero-tolerance metadata assertion (format/dims/MIME) separated from fuzzy pixel tolerances, and an XLSX-export gate (`Workbook.toXlsx()` with charts) that exercises the native backend through the real embedded-fallback path. The one weak link is the browser↔native golden harness: it is the gate that justifies the entire "production peer" objective, yet it is the least specified, with no decision on the browser-side renderer or how nondeterministic antialiasing is bounded. Until that is pinned, "parity" is unfalsifiable.

## Concrete changes that would raise the rating

1. **Add explicit phasing/MVP ordering.** e.g. Phase 1: typed schema + serializer + contract fixtures (no rendering change); Phase 2: paints/gradients/lines; Phase 3: shadows + clipping + arcs; Phase 4: rich text; Phase 5: budgets + golden harness. State what is shippable after each phase so the work can land incrementally rather than as one mega-merge.
2. **Specify the golden-harness baseline.** Name the browser-side renderer (headless Chromium via Playwright is the only true Canvas2D match) and explain how nondeterministic AA is bounded (per-channel SSIM / max-delta threshold, region masks for text). Note explicitly that `node-canvas`/Cairo is *not* an acceptable browser proxy.
3. **Stop citing minified symbols as the spec.** Replace "match `paintToCanvasStyle()`" / "lockstep with `symbol.ts`" with pointers to the canonical (un-minified) source of truth or to the observable Canvas2D behavior, and acknowledge that public `charts/src` is bundled output.
4. **Commit to concrete budget and tolerance numbers** (or at least order-of-magnitude ceilings) for max physical pixels, max encoded-work estimate, and the initial golden tolerance, even if tuned later — a budget objective with no number is not enforceable.
5. **Make the N-API bridge test unconditional:** add a `compute-napi` bridge test for the new request/result contract rather than gating on whether tests already exist.
6. **Add one sentence on the clip/DPR transform order** with the actual coordinate math, since "logical coordinates after DPR normalization" is the kind of phrasing that hides a `pixelRatio` scaling bug.
