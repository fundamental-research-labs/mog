# Review of Plan 089: Compute PyO3 Source Boundary Improvements

Reviewer: Claude Code review worker 89 of 100
Source folder: `mog/compute/pyo3/src`
Plan: `mog-internal/plans/active/experiments/plan-ratings/codex-plans/089-compute-pyo3-src.md`

## Summary judgment

This is a strong, evidence-grounded plan. Its core observations are factually correct against the live source, not hallucinated: `src/lib.rs` does only register `ComputeEngine` and `pivot_detect_fields` (lib.rs:108-116) while leaving every other `bridge_pyo3::generate!`-emitted free function unregistered; the PyO3 `generate_class!` list does omit the eight descriptor groups the plan names (`core_cells`, `core_sync`, `core_theme`, `objects_floating`, `objects_groups`, `objects_hyperlinks`, `objects_z_order`, `screenshot`), all of which are present in both `wasm/src/lib.rs` and `napi/src/lib.rs`; the manual bare-import block (lib.rs:22-56) is exactly as described; the `hasattr(self._engine, "compute_set_cell_value_parsed")` fallback branch exists in `python/mog/_bridge.py`; and the disposition counts are quoted *exactly* right (`api_surface.json` reports 1054 dispositions split 86 implemented / 174 renamed / 285 unsupported / 379 omitted / 129 python_only / 1 out_of_scope). A plan whose factual base reproduces this precisely earns trust in its prescriptions.

The central thesis — that the PyO3 native ABI should be a generated, snapshotted contract checked against the *built* `mog._native` and installed wheel rather than against source-tree imports — is the right architectural framing for a binding crate, and the plan repeatedly resists the tempting wrong fix (hand-registering individual functions, writing Python compatibility shims, faking success). That discipline is the plan's biggest asset.

It loses points for breadth-vs-boundary tension: although nominally scoped to one ~117-line `lib.rs`, the real work lives almost entirely in adjacent crates (`bridge-pyo3`, `compute-api`, the Python package), and several steps (remove all fallbacks, release the GIL, structured error propagation) are individually large initiatives folded into one plan without per-step done-criteria or effort sizing.

## Major strengths

- **Accurate, falsifiable problem statement.** Every "observed gap" maps to a verifiable line of source. The plan distinguishes between accidental drift and intentional omission and demands the difference be *encoded*, not assumed.
- **Correct root-cause targeting.** Missing free-function registration is treated as a missing generator capability in `bridge-pyo3` (step 5, a paired `register!` macro driven by the same descriptor expansion), not papered over with a hand-written `m.add_function(...)` list. This makes "generated-but-unregistered" structurally impossible, which is the only durable fix.
- **Production-path realism.** The plan refuses source-tree import as proof of behavior, insists wheels/native-extension loading are the product path (step 16, objective 8), and forbids placeholder XLSX bytes and silent no-ops. The "no fake-success" invariants (objective 6, invariants on returning `None`/`{}`/`[]`) are precisely the failure mode that makes a binding look healthy while being hollow.
- **Contract clarity around the FFI boundary.** Parameter tags (`[serde]`, `[str]`, `[prim]`, `[bytes]`, `[parse]`, tagged-enum) and return encodings (JSON strings, `(bytes, metadata_json)` tuples) are spelled out as the contract, with the explicit rule that Python wrappers must not guess serialization from value or method name. The one-shot `take_lifecycle_result` invariant is stated for both snapshot and Yrs-state construction.
- **Verification gates are concrete and runnable**, naming real targets (`cargo test -p bridge-pyo3`, `maturin develop/build`, `pyright --verifytypes`, `pnpm check:python-sdk`) and a clean-temp-env wheel install that guards against `PYTHONPATH` source-tree bleed-through. The contract-test list (ABI snapshot, parity/disposition, registration, param/return metadata, lifecycle, structured error, unsupported-path, no-fallback audit, XLSX round-trip, wheel content) is comprehensive.
- **Sensible sequencing and ownership map.** Integration order (manifest + ABI inventory first → bridge-pyo3 support → lib.rs consumption → Python refactor → stubs/docs/hygiene) is dependency-correct, and the dependency section correctly assigns ownership to `compute-api`, compute-core, `xlsx-api`, and `bridge-pyo3`.

## Major gaps or risks

- **Scope inflation under a folder-shaped title.** The literal source surface is one short file, yet the bulk of the plan rewires `bridge-pyo3`, `compute-api` error types, and the entire Python package. The scope section is honest about this, but the plan never sizes the work or flags that several steps (9 "remove fallbacks category by category," 11 structured errors, 17 GIL release) are multi-week efforts in their own right. A reader could mistake this for a single landable change.
- **No per-step acceptance criteria / definition of done.** Steps say what to build and which gates to run, but rarely state the pass condition (e.g. step 2's manifest "should fail on current drift" — fail *how*, with what exit contract? step 6's "every expected free function present" — expected list derived from what authority?). Done-ness is implied by "gates green," but the gates are partly new artifacts the plan is itself creating.
- **Snapshot-governance gap.** The plan rests heavily on generated/snapshotted ABI and surface metadata, but says nothing about how baselines are reviewed and updated, or how to prevent the snapshot from silently absorbing a regression on `--accept`/regen. For a "snapshotted contract" plan, baseline-update discipline is a load-bearing omission.
- **"Register all free functions then classify low-level separately" risks surface bloat.** The plan acknowledges (risk #2) that registering everything may expose low-level exports before high-level wrappers exist, and proposes the manifest classify them separately — but does not commit to whether unwrapped low-level exports are publicly importable, deprecated, or hidden. The public/experimental SDK could leak a large untyped surface in the interim.
- **GIL-release step is under-specified for its danger.** Step 17 correctly warns against unsound borrowing, but offers no criteria for *which* generated wrappers are eligible (owned args only?) nor how the generator decides. As written it's an aspiration; without a concrete eligibility predicate it is the most likely step to either be skipped or to introduce unsoundness.
- **Clock semantics deferred rather than decided.** Step 13 correctly identifies that the stateless thread-local `ClockBridge` path may not reach the PyO3 dispatch thread, but leaves the resolution conditional ("if not valid, move to instance/session path... mark stateless setter unsupported until true"). This is the one place the plan punts a genuine design decision into implementation instead of resolving it.
- **Python version / ABI3 matrix asserted but not gated.** Risk list mentions "Python 3.9 ABI3 compatibility must be preserved while testing newer Python versions locally," but no gate pins the abi3 minimum or runs a 3.9-target check; the local `.venv` gates use whatever interpreter is present (the repo's compiled `.pyc` is `cpython-314`).

## Contract and verification assessment

Contract definition is the plan's strongest dimension. The FFI boundary is described as a typed contract with explicit param tags, return encodings, lifecycle one-shot semantics, structured error propagation, and per-instance session/principal state — all stated as invariants to preserve, with the matching "must not" list (no value-based serialization guessing, no global session state, no placeholder bytes, no swallowed failures). The `UnsupportedApiError` shape is specified down to its fields (`api_path`, `python_path`, `reason_code`, `owner_package`).

Verification is well-covered at the gate level and unusually good at guarding the *real* artifact (built extension + clean-env wheel install, not source import). The weakness is reflexive: many of the strongest gates (ABI snapshot, disposition manifest, FFI metadata) are *outputs of this plan*, so "run the gate" presupposes the gate has been correctly authored. The plan would be more trustworthy if it named the authority each generated expectation is derived from (descriptor expansion) versus the thing it checks (built module), and committed to snapshot-update review discipline. The cross-target parity test (PyO3 vs WASM vs N-API vs generated surface) is the right keystone check and is correctly placed first in the integration order.

## Concrete changes that would raise the rating

1. **Add per-step acceptance criteria.** For each numbered step, one line stating the pass condition and the authority for any "expected" set (e.g. "expected free functions = the set emitted by `bridge_pyo3::generate!` minus the per-target skip list; test fails if `dir(_native)` differs").
2. **Resolve the clock decision now.** State whether Python time injection moves to the `ComputeService` instance path or the stateless setter is marked unsupported, rather than leaving it conditional. This is a design call, not an implementation detail.
3. **Define low-level export visibility policy.** Commit to whether unwrapped `_native` exports are private (underscore/hidden), publicly importable-but-undocumented, or gated — so step 6 doesn't silently widen the public surface.
4. **Add snapshot-baseline governance.** A short paragraph on how ABI/surface baselines are regenerated, who reviews diffs, and how a regression is prevented from being absorbed by a blind regen.
5. **Specify a GIL-eligibility predicate.** Give the concrete rule the generator uses to decide a wrapper is `allow_threads`-safe (e.g. all args owned/`Send`, no borrowed `Py` refs held across the boundary), or explicitly mark step 17 as a follow-up out of this plan's first landing.
6. **Pin the abi3 floor in a gate.** Add an explicit minimum-version build/check (and document the supported Python matrix) so 3.9 ABI3 compatibility is enforced, not merely asserted.
7. **Size and stage the mega-steps.** Split or explicitly label steps 9, 11, and 17 as independent follow-on workstreams with their own landing points, so the plan reads as a program rather than a single change.

Overall: a high-quality, honest, source-accurate plan with excellent contract and production-path instincts, held back from a 9 by scope/sizing ambiguity, missing per-step done-criteria, one deferred design decision, and absent snapshot-governance — all addressable with edits rather than rework.
