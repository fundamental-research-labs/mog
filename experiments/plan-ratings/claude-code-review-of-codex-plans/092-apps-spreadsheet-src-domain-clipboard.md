# Review of 092 — Spreadsheet Clipboard Domain Production Improvement Plan

## Summary judgment

This is a strong, unusually well-grounded plan. Its central diagnosis — that the clipboard "contract" is not singular and that overlapping codecs, capture paths, and routing logic let one path silently diverge from another (formulas, formats, hidden rows, images, cut semantics) — is correct and is the right problem to attack for this folder. I spot-checked nearly every concrete code claim and they hold:

- The cross-folder map is accurate. `actions/handlers/clipboard.ts` (1218 lines), `hooks/editing/use-clipboard.ts` (1380), `systems/grid-editing/machines/clipboard-machine.ts` (1265), `coordination/paste-integration.ts` (660), and `infra/utils/clipboard-utils.ts` (545) all exist with the stated roles.
- `copyToSystem`/`readFromSystem` appear only inside `clipboard-service.ts` itself — confirming the plan's claim that the exported `ClipboardService` singleton has no production grid consumer. `rg clipboardService` outside `domain/clipboard` returns nothing.
- `parseHTML`, `rangeToTSV`, `rangeToHTML` exist in `infra/utils/clipboard-utils.ts`; `parseExternalData`, `parseExternalText`, `buildClipboardDataFromParsed`, `truncateExternalCellText`, and `MAX_EXTERNAL_CELL_TEXT_CHARS = 32767` exist in the clipboard machine at the cited locations. The duplicate-parser claim is real.
- `executePaste()` (paste-executor.ts:804), the core `copyRange` fast path with its eligibility gate (paste-executor.ts:888, :1123), and the preview calculator re-importing `transposeData`/`filterByPasteType`/`filterBlanks`/`getClipboardDimensions` and re-orchestrating them independently (paste-preview-calculator.ts:117–127) all match the plan's "preview can drift from execution" claim.

The plan reads like it was written by someone who actually traced the production paths rather than skimming filenames. The invariants section (synchronous write reservation under user activation, cut single-use-after-commit, system-clipboard-wins-over-stale-internal, canceled-text suppression, sparse geometry must describe the intended rectangle not the populated cells) captures exactly the subtle, regression-prone contracts that a refactor here would otherwise break.

The main reason this is an 8 and not a 9–10 is scope and sequencing discipline: it is an epic dressed as a plan, with several genuinely deferred design decisions and no minimal first slice that ships value behind a gate before the big consolidation lands.

## Major strengths

- **Evidence-grounded.** Specific symbols, files, and line-level behaviors are named and verifiable. It correctly identifies dead-ish surface area (`ClipboardService` singleton, `copyToSystem`/`readFromSystem`) rather than treating all exports as load-bearing.
- **Right architectural target.** "One capture/export pipeline, one importer, one router, one plan/apply split, preview derived from the plan" is the correct shape. Collapsing the two HTML parsers and merging their complementary strengths (colspan/value-inference vs. style/MSO/color extraction) is a concrete, defensible merge rather than a vague "unify."
- **Production-path fidelity is a first-class constraint.** The plan repeatedly forbids the failure modes that actually matter: no awaits before write reservation, no direct state mutation, no test-only paste shortcuts, no mock-only production path, keep core `copyRange` for formula rebasing, no reduction of paste-special/secondary-payload coverage. These are exactly the corners a careless refactor cuts.
- **Contract boundaries are explicit.** It keeps `ClipboardData` compatible with the public actor contracts, refuses to invent a third unowned runtime representation, and explicitly forbids public `mog` depending on `mog-internal`. The kernel-clipboard coordination section ("pick one owner per phase, assert ownership in tests, avoid a half-migration") is mature and shows awareness of an adjacent in-flight effort.
- **Verification gates are concrete and layered** — focused folder tests, coordination/handler tests, app + repo typecheck, public-API/publish-readiness checks gated on the specific files that would trigger them, and a real-browser E2E pass with the explicit rule that E2E must seed through real clipboard/event paths.

## Major gaps or risks

- **Scope is very large with weak internal sequencing.** Ten implementation steps and six parallel agent streams (A–F) touch types, codecs, capture, routing, host boundary, executor, preview, machine, handlers, hooks, and E2E. The "parallelization notes" list agents but do not encode the real dependency DAG: E (planner) depends on A (types); F (integration/E2E) depends on B/C/D/E; the router (D) depends on the host boundary and capture (C). Without an ordered, gated phasing, this reads as "do everything at once," which is the highest-risk way to refactor a timing-sensitive, multi-entry-point subsystem.
- **No minimal shippable first slice.** Nothing in the plan can land and prove value while the rest is in flight. A safer framing would isolate the lowest-risk, highest-leverage win first — e.g., consolidate the two HTML/TSV importers behind one tested codec and route the machine through it — then build the router/host/plan-apply on top once the codec is proven. As written, partial completion leaves the codebase with both old and new paths simultaneously, which is exactly the divergence risk the plan exists to eliminate.
- **Genuinely deferred design decisions are left to implementation.** Step 3 explicitly says to "decide and document external formula policy" (do `=`-prefixed strings enter as formulas, stay literal, or depend on paste mode) — this is a user-visible contract question, not an implementation detail, and leaving it open weakens the "one contract" thesis. Step 9 is an "audit whether `ClipboardService` should remain" — an investigation, not a decision. A plan at this level of detail should pre-resolve these or call them out as explicit blocking decisions with owners.
- **Plan/apply purity is asserted but under-specified for failure ordering.** Secondary payloads are applied after the core `copyRange` write; the plan says the split should "make ordering and failure behavior visible" but does not specify what happens when a secondary payload (e.g., validation, merge) fails after the core value write has already committed. Partial-failure semantics and undo grouping across the core write + TS secondary writes are the single most likely source of subtle regressions and deserve a stated contract, not just "expected undo grouping" as a plan field.
- **No explicit definition-of-done per step beyond "add tests."** The test lists are excellent in breadth but there is no acceptance criterion tying behavior parity to the old paths (e.g., "captured TSV/HTML for fixture X is byte-identical before/after migration" or a golden-fixture differential). For a consolidation whose whole point is "stop divergence," a differential/parity gate against the current production output would be the strongest possible verification and is conspicuously absent.

## Contract and verification assessment

Contract clarity is the plan's strongest dimension. The new type vocabulary (`SpreadsheetClipboardTransfer`, `ClipboardSourceKind`, `ClipboardGeometry`, `ClipboardSignatureSet`, `ClipboardImportResult`, `ClipboardCaptureResult`, `ClipboardPasteIntent`) is mapped explicitly onto the existing public `ClipboardData`/`ClipboardPayload` rather than replacing them, and the invariant-helper set (`normalizeClipboardGeometry`, `assertClipboardDataInvariants`, `deriveSourceRangeGeometry`, `computeClipboardSignatures`, `isSameSystemPayload`) is concrete enough to implement directly. The sparse/full-shape geometry invariant in particular is the right thing to pin down — it is exactly where blank-clearing, size-mismatch checks, and selection-after-paste break today.

Verification gates are concrete, correctly scoped, and appropriately conditional (publish-readiness only when the named public files change). The insistence on real-input E2E and the explicit non-goal of direct actor seeding is the correct posture. The one missing gate is a parity/differential check against current behavior; the plan tests the *new* code well but does not prove *equivalence* with the *old* code it deletes, which is the riskiest part of a consolidation.

## Concrete changes that would raise the rating

1. **Add an explicit phase DAG with gates.** Order the work as: (P1) codec consolidation behind one importer + machine routed through it, gated by codec parity tests; (P2) capture/export unification, gated by TSV/HTML golden-fixture parity against today's output; (P3) host boundary + router, with `unifiedPaste()` and native paste both delegating; (P4) plan/apply split + preview-from-plan; (P5) `ClipboardService` resolution and kernel coordination. State that each phase must land green before the next starts, and that no phase may leave two live paths for the same flow.
2. **Define a differential/golden-fixture parity gate.** For a representative fixture corpus (Excel HTML, Google Sheets HTML, browser tables, sparse full-row/col copies), assert that captured TSV/HTML and imported `ClipboardData` are equivalent before and after each migration step. This directly verifies the "stop divergence" thesis.
3. **Pre-resolve the deferred contract decisions.** Make the external `=`-formula policy a decided contract in the plan (recommend: literal text on external paste unless an explicit "paste as formula" mode, matching mainstream spreadsheet behavior), and convert Step 9 from "audit whether" into a stated decision (recommend: narrow `ClipboardService` to pure cross-view payload helpers, remove the unused singleton export after a public-export impact check).
4. **Specify partial-failure and undo-grouping semantics for plan/apply.** State what happens when a secondary payload write fails after the core `copyRange` commit (roll back via undo group? best-effort with diagnostic? abort before core?), and require a test for "core write succeeds, merge/validation write throws."
5. **Map agent streams to the phase DAG and name the cross-stream interfaces** (the `ClipboardHost` interface, the importer signature, the `PasteWritePlan` shape) as the contracts that must be frozen first so A–F can proceed without churn.
6. **Add a one-paragraph rollback/coexistence note** describing how old and new paths coexist mid-migration (e.g., a single feature switch at the router boundary) so a partially-landed migration is never in a divergent state.
