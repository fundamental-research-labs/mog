Rating: 8/10

# Review of Plan 093 — Spreadsheet Domain Editor State Behavior


## Summary judgment

This is a strong, evidence-grounded plan. Its central diagnosis is verifiably correct: `domain/editor/formula-context.ts` and `domain/editor/formula-range-parser.ts` are duplicated **byte-for-byte** under `systems/shared/utils/` (confirmed via `diff` — IDENTICAL for both files), and the live editor machine (`systems/grid-editing/machines/editor/autocomplete.ts`, `types.ts`) imports the helpers from `shared/utils`, not from `domain/editor`. So the "two import paths for the same behavior" problem the plan leads with is real and is the right thing to attack first. The plan also correctly identifies a genuine lifecycle bug in `formula-metadata-cache.ts` (stale-snapshot publication after invalidation/disposal), a dead `_cursorOffset` parameter in `cursor-position.ts`, and the regex-only nature of `formula-range-parser.ts` (`CELL_REFERENCE_PATTERN`) that cannot distinguish references inside strings or structured-reference column names.

The plan is specification-rich: it separates preserved invariants from improvement objectives, ties each consumer to a concrete production path, and provides real, runnable-looking verification gates with correct filter targets. The main weakness is scope: it bundles a low-risk, high-value consolidation with a speculative, large "grammar-aware token model" rebuild, and proposes six parallel agents whose dependencies are tighter than the parallelization section admits.

## Major strengths

- **Accurate, falsifiable root-cause claims.** The duplication, the import-path split, the metadata-cache stale-publish race, and the unused cursor-offset param all check out against the actual source. This is not a hand-wavy plan.
- **Real lifecycle bug correctly characterized.** In `FormulaMetadataCache.request()`, the resolved `.then()` publishes `status: 'ready'` using `this.snapshot.version + 1` without capturing a generation at request start. An `invalidate()` (or `dispose()`) that fires mid-flight is therefore overwritten by a stale `ready` snapshot when the load resolves. Section 6's "cache generation counter captured when a request starts" + disposal guard is exactly the right fix, and the test list (in-flight invalidation, disposal during load, unsubscribe during notification) targets the actual failure modes.
- **Invariant-first framing.** The "contracts and invariants to preserve or strengthen" section is the best part: function-frame vs grouping-paren distinction (`SUM((A1+B1),C1)` must not pop `SUM`), separator counting only at current depth and outside literals, cross-sheet identity carried explicitly instead of `ref.text.includes('!')`, and non-editable typed results for unrepresentable references. These are concrete, testable behaviors, not aspirations.
- **Consumer migration is enumerated, not assumed.** It names `use-formula-autocomplete`, `InlineCellEditor`, `FormulaBarContainer`, `render-context-coordination`, `use-formula-range-drag`, `actions/handlers/editor.ts`, and `formula-auditing.ts` — all of which do consume these helpers — so the blast radius is acknowledged up front.
- **Verification gates are concrete and correctly scoped.** The `pnpm --filter @mog/app-spreadsheet test -- src/domain/editor` family plus the repo-level typecheck trigger condition ("if public actor contracts / app exports change") shows awareness of the public/internal boundary. The app-eval list drives real keyboard/mouse/selection paths rather than direct helper calls, which matches the stated invariant that UI-sensitive behavior must be exercised through production input.

## Major gaps or risks

- **Scope is two plans wearing one coat.** Sections 1 and 9 (consolidate to one import path, migrate consumers, delete the duplicate) are low-risk, high-value, and could ship independently with immediate benefit. Sections 2–4 (a token-aware formula editor model that "aligns with Mog's compute parser") are a large, open-ended undertaking whose payoff for *editor UI helpers* is much less certain. Bundling them means the cheap, safe win is gated behind the expensive, risky one. The plan would be stronger if it explicitly sequenced consolidation as a shippable Phase 0 before committing to the tokenizer rebuild.
- **The token-model bet is under-justified and self-acknowledged as uncertain.** Section 2 hinges on whether the compute parser/tokenizer can expose source spans and tolerate partial/incomplete formulas. The plan flags this as a risk but does not require a spike to answer it *before* committing the dependent context/reference rewrites (Sections 3–4) to it. If the compute parser can't do partial parse, the fallback ("a small editor-tolerant lexer that shares grammar with compute") is itself a substantial new component with no sizing. This is the single largest source of estimate risk and should be de-risked first.
- **One contract claim is speculative and slightly inaccurate.** Section 1 says to align `FormulaContext`/`FunctionStackEntry` with the public actor/editor contract type, "If `@mog-sdk/contracts/actors` remains the public source." A search of `packages/contracts` finds **no** `FormulaContext` or `FunctionStackEntry` there — the type's real source is `shared/utils` (i.e. the duplicate itself), re-imported by the editor machine. The plan hedges with "If," but readers may take the premise as established fact. The actual canonicalization decision is simpler than implied: make `domain/editor` the single declaration and have the machine import from it.
- **Parallelization optimism.** Agent B (token model) is a hard upstream dependency for Agent C (reference extraction on tokens) and most of Agent F's fixtures. Running them "in parallel" really means C and F stall until B's token API stabilizes. The dependency note mentions the coupling but the workstream split presents more independence than exists. A serialized B→(C,D)→E,F ordering would be more honest.
- **Behavior-change risk to highlighting is large and only partially bounded.** Replacing `CELL_REFERENCE_PATTERN` changes which spans light up in the formula bar, inline editor, and canvas range boxes simultaneously. The plan calls this out and proposes broad fixtures, but there is no explicit "golden snapshot of current highlighting on a fixed corpus before/after" gate to catch silent regressions in colors/spans that unit tests on new shapes won't notice.

## Contract and verification assessment

The contract surface is well chosen. Strengthening `FormulaRangeReference` with `kind`, `sheetId`/`isCurrentSheet`, absolute-marker metadata, `editable`, and a rewrite policy directly removes the current downstream string-sniffing (`ref.text.lastIndexOf('!')`, the brittle `buildReferenceText` absolute-marker regexes in lines 280–290 that re-derive `endColAbsolute` twice). The "non-editable typed result rather than mutating text" rule is the correct safety contract for structured references and named ranges, which `parseRefToRange` currently cannot represent.

Verification is above average for this plan set: gates are real commands with correct filter paths, the JSDOM-vs-browser split for cursor/selection is sensible, and the app-eval scenarios target the genuine input paths. Two gaps: (1) no regression/golden gate for existing highlight spans and colors across a formula corpus, which is the most likely place for silent breakage; (2) the metadata-cache concurrency tests are listed but the plan does not specify the deterministic mechanism (fake timers / controllable promise resolution order) needed to make "invalidate during in-flight load" reproducible rather than flaky. Naming that harness would raise confidence that the generation-counter fix is actually provable.

## Concrete changes that would raise the rating

1. **Split into a shippable Phase 0.** Make Sections 1 + 9 (single `domain/editor` declaration, migrate the editor machine and all named consumers off `shared/utils`, delete the duplicate, add the no-duplicate static check) a self-contained first deliverable with its own gates. This banks the verified win independent of the parser rebuild.
2. **Require a token-model spike before Sections 3–4.** Add an explicit investigation task: can the compute parser/tokenizer emit UTF-16 source spans and tolerate incomplete input? Gate the token-based context/reference rewrites on the answer, and size the editor-tolerant-lexer fallback if the answer is no.
3. **Correct the contract-source claim.** State as fact that `FormulaContext`/`FunctionStackEntry` currently originate in `shared/utils` (not `contracts/actors`), and define the single canonical home as `domain/editor`; drop the conditional framing.
4. **Add a highlight-parity golden gate.** Snapshot current reference spans/colors for a fixed formula corpus and assert before/after equivalence (minus intended fixes) when `CELL_REFERENCE_PATTERN` is replaced, so silent UI regressions are caught.
5. **Specify the deterministic concurrency harness** for the metadata-cache tests (controllable promise ordering / fake timers) so the generation-guard and disposal-during-load cases are reproducible, not flaky.
6. **Re-serialize the workstream graph.** Present B (tokens) as upstream of C and the fixtures, rather than as a peer parallel lane, to set realistic expectations.
