# Review of 098 — Apps Spreadsheet Renderer System

## Summary judgment

This is a strong, evidence-grounded plan. I verified its central factual claims directly against `mog/apps/spreadsheet/src/systems/renderer`, and they hold up: the `async`-via-`void`-inside-`try/catch` transition bug, the unwired `PageBreakCoordinator.setDependencies(...)`, the unwired `setupLayoutCoordination`, the untyped `RenderContextConfig { [key: string]: unknown }` placeholder, the surviving `getRenderer()` / `__mogInternalGridRenderer` raw access, and the no-op simulator viewport methods are all real. The plan correctly diagnoses that the folder is "close to intended architecture but contracts are incomplete," and it proposes a coherent migration toward a single activation boundary, typed context projection, a unified event manifest, real page-break wiring, a single view-sync transaction, and production-path tests. The objectives, invariants, sequencing, verification gates, risks, edge cases, and parallelization are unusually thorough.

The main reason this is not a 9–10 is scope and dependency risk: the plan spans ~9 systems and explicitly defers the most load-bearing dependency (the SheetView capability contract that gates items 4 and 8) to "the adjacent SheetView plan" without specifying that contract's shape here. Several work items therefore cannot be completed within this folder's workstream alone, and the plan does not fully de-risk that coupling.

## Major strengths

- **Claims are real and checkable.** Spot-verified:
  - `execution/renderer-execution.ts:452-454` — `try { void executeStateTransition(...) } catch` over an `async` function (line 205). The plan's critique that async rejections escape the synchronous catch is correct.
  - `pageBreakCoordinator.setDependencies(...)` is never called from `render-system.ts`; `hitTest` returns `{ hit: false }` when `deps` is null (`page-break-coordination.ts:231-233`). The plan's claim that page-break drag has "no committed dependency path" is accurate.
  - `setupLayoutCoordination` has no call site (only exported in `coordination/index.ts`). Confirmed.
  - `types.ts:122-125` `RenderContextConfig { [key: string]: unknown }` placeholder and `updateContext(config: Partial<RenderContextConfig>)` confirmed.
  - `renderer-simulator.ts:134` documents that `setScrollPosition/setFrozenPanes/setZoom` are no-ops — the "no-op simulator" critique is correct.
- **Catches a documentation/architecture contradiction the plan implicitly relies on.** `render-system.ts:19` advertises "Constructor-complete … no setDependencies()", yet the live setters the plan lists (`setRendererDependencies`, `setContextConfig`, `setSelectionActorForViewportFollow`, `setSparklineManager`, `setSparklineConfig`, `setTableAutoExpansionConfig`) all exist. The plan's item 2 (consolidate to one audited activation boundary) and item 10 (fix the stale comment) directly target this drift.
- **Invariants section is the best part.** Strict attach ordering, generation guards on sheet switch, InputCoordinator physics resync on programmatic scroll, sheet-aware callbacks not closing over stale sheet ids, and "cache invalidation cannot depend on whether a renderer is attached" are precise, testable production constraints rather than platitudes.
- **Verification is concrete and largely real.** Referenced app-eval scenarios mostly exist on disk (`viewport-follow/*`, `sheet-switch-roundtrip/{freeze-after-switch,hit-test-after-switch,selection-scroll-restoration,switch-and-back-rendering}.spec.ts`, `zoom/{zoom-freeze-pane-alignment,zoom-persists-per-sheet,view-zoom-to-selection}.spec.ts`, `view/view-page-break-preview-*.spec.ts`). The package gates are scoped to the right filters.
- **Tests-first framing is right.** Item 1 deliberately adds failing contract tests that expose unwired modules before refactoring — good discipline that makes the gaps non-deniable.

## Major gaps or risks

- **The critical dependency is deferred, not specified.** Items 4 (typed projection) and 8 (remove raw `GridRenderer`) both hinge on "once SheetView exposes the missing object hit-region and interaction capabilities," delegated to a separate SheetView plan. No interface sketch is given here for that capability surface (object hit-region, interaction state, typed data-source update). Without it, the two highest-value items are blocked on out-of-folder work whose contract is undefined. This is the largest specification gap.
- **Scope is very large for one plan.** Ten work items touching `coordinator`, `components/grid/effects`, `systems/input`, `systems/objects`, `systems/grid-editing`, SheetView, and app-eval. The parallelization split (Agents A–F) is sensible, but `getRenderer()` removal (item 8) cuts across renderer, coordinator, object system, hooks, and action handlers simultaneously; a half-migrated state would leave both the raw and typed paths live. The plan says "do not leave a compatibility shim," which is correct in principle but raises the risk of a large, hard-to-land single change. A staged migration (introduce capability, dual-wire behind it, then delete) is not described.
- **Minor app-eval reference drift.** `view/view-normal-clears-preview-rendering-not-breaks.spec.ts` does not exist; the closest real file is `view-normal-view-restores-from-page-break-preview.spec.ts`. A handful of others are globs (`arrow-*`, `*freeze*`) that match, but the page-break drag-through-canvas scenario is explicitly new and unwritten. Low impact, but the spec list reads as if all exist.
- **Sequencing between item 1 and item 2 is slightly under-specified.** Several item-1 "failing tests" (e.g., activation-precondition test, projection-field-owner test) presuppose the `RendererRuntimeBindings` contract and typed projection from items 2 and 4. The plan should state whether item 1 tests assert against *current* APIs (and are later rewritten) or against the *target* contracts (and thus can't be authored until 2/4 land).
- **"Constructor-complete vs. activation boundary" is left as an either/or.** Item 2 and item 7 both allow "either truly constructor-complete or one audited activation boundary." React must supply some callbacks post-construction, so the realistic answer is the activation boundary — the plan would be stronger committing to it rather than presenting it as an open choice, since the decision shapes every downstream test.

## Contract and verification assessment

- **Contract clarity: good within the folder, deferred at the boundary.** The typed projection groups in item 4 are enumerated field-by-field (visual actor state, data sources, structure/overlay, page/print, preview, workbook settings) and the plan demands exhaustive compile-time mapping (`satisfies`) plus removal of the `[key: string]: unknown` escape hatch — this is genuinely strong contract work. But the SheetView-side capability interfaces that those groups must call into are named, not specified. The renderer plan cannot fully own a contract whose other half lives in another plan.
- **Verification gates: appropriate and falsifiable.** Package `test`/`typecheck` filters are correct; the "add failing tests that prove the gap" approach gives real gates rather than aspirational ones. The insistence on real UI input + canvas/readback assertions (not direct state mutation) matches the folder's actual failure mode (the no-op simulator). The edge-case list (dispose between MOUNT and LAYOUT_READY, double sheet switch before first completes, page-break drag over hidden rows, CF on inactive sheet, table auto-expansion under stale sheet id) is excellent and directly traces to the invariants.
- **Gap in gates:** no explicit gate for "InputCoordinator physics resync on programmatic scroll" despite it being called out as an invariant; the `keyboard/ctrl-backspace-scroll-to-active.spec.ts` reference is the nearest proxy but doesn't target the physics-state desync (`unfreeze after scrolling a frozen pane with non-zero input physics` edge case has no matching named scenario). Worth a dedicated assertion.

## Concrete changes that would raise the rating

1. **Specify the SheetView capability contract inline** (even as a TypeScript interface sketch): object hit-region, interaction/selection state, and the typed data-source `update` surface that items 4 and 8 depend on. State which plan owns it and make the dependency a hard prerequisite with a checkable artifact, not a forward reference.
2. **Add a staged-migration path for `getRenderer()` removal**: introduce the capability, dual-route callers behind it, verify, then delete in a final pass — rather than a single cross-system cut. Name the call-site inventory (`renderer-execution`, `event-subscriptions`, `sheet-coordinator`, `use-renderer-actions`, `page-navigation`) as a checklist with a "no remaining callers" gate (the plan already lists these — promote them to a tracked inventory).
3. **Commit to the single activation boundary** (drop the "truly constructor-complete" alternative), since React supplies callbacks post-construction. Fix the contradictory `render-system.ts:19` comment as part of item 2, not just item 10.
4. **Clarify item-1 test targets**: state explicitly whether the first failing tests assert current APIs or target contracts, and which item-1 tests must wait on items 2/4.
5. **Correct the app-eval spec references** (`view-normal-clears-preview-rendering-not-breaks` → the real `view-normal-view-restores-from-page-break-preview`), and mark which referenced specs are new vs. existing so the verification list is trustworthy.
6. **Add an explicit gate for InputCoordinator physics resync** on programmatic scroll / unfreeze-with-physics, matching the stated invariant and the listed edge case.
