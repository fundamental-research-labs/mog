# Improve `mog/kernel/src/domain/pivots`

## Source folder and scope

Source folder reviewed: `/Users/guangyuyang/Code/mog-all/mog/kernel/src/domain/pivots`

Queue item 17 covers pivot table domain behavior and state transitions in the public Mog source tree. The folder currently contains only:

- `style-normalization.ts`
- `__tests__/style-normalization.test.ts`

The actual pivot behavior is spread across nearby production paths:

- `kernel/src/api/workbook/pivot-styles.ts`
- `kernel/src/api/worksheet/pivots.ts`
- `kernel/src/bridges/pivot-bridge.ts`
- `kernel/src/bridges/pivot-event-bridge.ts`
- `types/data/src/data/pivot.ts`
- `types/events/src/pivot-events.ts`
- `docs/internals/spreadsheet/pivot-tables.md`

The improvement scope is to make `kernel/src/domain/pivots` the canonical TypeScript domain layer for pivot style catalogs, placement identity, pure config transitions, source-change reconciliation, update policy, and event refresh decisions. It should not take ownership of Rust persistence, Rust compute, or UI session state.

## Current role of this folder in Mog

`domain/pivots` currently normalizes pivot style names only. It accepts aliases like `light16` or `pivotstylemedium04`, canonicalizes valid built-in IDs to `PivotStyleLight16` / `PivotStyleMedium4`, preserves custom style names, and maps nullish or blank values to `null` / `undefined`.

Production callers use this helper in two places:

- `WorkbookPivotTableStylesImpl` normalizes default pivot styles before delegating to `ComputeBridge`.
- `WorksheetPivotsImpl.setStyle()` normalizes `style.styleName` before `PivotBridge.updatePivot()`.

Everything else that behaves like pivot domain logic lives outside this folder:

- Placement IDs are generated by both `WorksheetPivotsImpl.makePlacementId()` and `PivotBridge.createStablePlacementId()`, with different identity formats.
- Placement add/update/remove/move/reset transitions are duplicated between the worksheet API, the bridge, and synchronous handle facades.
- `setDataSource()` embeds source field redetection, duplicate header handling, stale reference migration, calculated-field invalidation, filter migration, and atomic error reporting inside `kernel/src/api/worksheet/pivots.ts`.
- Update reasons and refresh policies are repeated as inline objects across pivot mutators.
- Pivot event materialization decisions are split between `PivotBridge.setupObservers()` and `PivotEventBridge`, creating room for duplicate refreshes or divergent dirty/read behavior.
- Mutation receipts are assembled ad hoc with pseudo revisions (`0` or `Date.now()`) even though public contracts model kernel/config/result/materialization revisions.

## Improvement objectives

1. Promote `domain/pivots` from a style helper into the kernel's canonical pivot domain module.
2. Centralize all pure pivot config transitions so API and bridge code delegate behavior instead of open-coding array mutations.
3. Make placement identity stable, placement-id-first, and consistent across worksheet API, bridge API, handles, UI command handlers, and receipts.
4. Move source-change reconciliation into a pure, testable module that returns either complete config updates or structured invalid-reference errors before any mutation.
5. Centralize pivot update reasons, refresh policies, and event refresh decisions in a domain policy table.
6. Strengthen style catalog behavior with a single built-in style list, default style constant, alias parser, and custom-style preservation contract.
7. Keep Rust/Yrs storage and Rust compute as the source of persisted truth; the TypeScript domain layer should prepare valid transitions and metadata, not shadow persisted state.
8. Turn known limitations into explicit contracts, especially expansion state ownership and dirty-only read paths after source changes.

## Production-path contracts and invariants to preserve or strengthen

- Persisted pivot config state lives in Rust/Yrs storage and is reached through `ComputeBridge`; no TypeScript `PivotStore` resurrection.
- `PivotBridge.compute()` remains a pure read path through `pivotComputeFromSource()` and must not materialize cells, force viewport refresh, clear dirty state, or notify subscribers.
- `PivotBridge.refresh()` remains the explicit production materialization path through `pivotMaterialize()` and `forceRefreshAllViewports()`.
- Every `PivotBridge.updatePivot()` producer must pass explicit `PivotUpdateOptions` with a `reason` and `refreshPolicy`; preserve the existing invariant test.
- `sourceRangeChanged` remains `dirtyOnly`; placement, aggregate, Show Values As, sort, filter, reset, layout, style, formatting, calculated-field, and UI config changes remain `refreshAndMaterialize` unless a new contract explicitly says otherwise.
- Placement IDs must be stable identities, not display labels. The same field may appear multiple times in Values, calculated fields may be placed beside source fields, and sorting/filtering may target a specific measure placement.
- Existing persisted `placementId` values must be preserved. Fallback IDs may be synthesized for legacy configs, but no transition should rewrite stable IDs unnecessarily.
- Within each placement area, positions are dense zero-based integers after add, remove, move, or insert.
- A failed source change must be atomic: no `updatePivot()` call when any placement, filter, Show Values As base field, top/bottom value field, or calculated field reference cannot be reconciled.
- Duplicate source headers must report all candidate resolutions instead of picking an arbitrary field.
- `sourceSheetId` is authoritative when present; `sourceSheetName` is display/legacy fallback.
- Pivot style normalization must preserve custom styles, canonicalize valid built-ins, reject blank inputs as unset, and keep the public default as `PivotStyleLight16`.
- Pivot errors with `PIVOT_*` codes continue to come from typed factories in `kernel/src/errors/pivot.ts`.
- UI E2E coverage must drive real UI input paths for pivot workflows; no direct state mutation to satisfy tests.

## Concrete implementation plan

1. Add `kernel/src/domain/pivots/styles.ts` and make `style-normalization.ts` a focused compatibility export or replace it with re-exports after updating imports.

   - Define `DEFAULT_PIVOT_STYLE_ID = 'PivotStyleLight16'`.
   - Define `PIVOT_STYLE_RANGES` and a generated `BUILT_IN_PIVOT_STYLE_IDS`.
   - Export `normalizePivotStyleIdForCompute()`, `publicPivotStyleId()`, `isBuiltInPivotStyleId()`, `listBuiltInPivotStyleIds()`, and `pivotStyleInfo()`.
   - Update `WorkbookPivotTableStylesImpl` to use the catalog for `getCount()`, `getItem()`, and `list()` instead of rebuilding the same list locally.
   - Update worksheet style mutation to use the same style normalizer.

2. Add `kernel/src/domain/pivots/ids.ts`.

   - Export branded helper functions for `PlacementId`, `CalculatedFieldId`, `PivotMemberKey`, and `PivotTupleKey` where the kernel currently casts locally.
   - Export `getPivotPlacementId(placement)` that preserves existing IDs and synthesizes only for legacy configs.
   - Export `createStablePivotPlacementId({ pivotId, area, sourceId, position, existingPlacements })` with collision handling.
   - Replace local `makePlacementId()`, `getBridgePlacementId()`, and equivalent casts where they are domain identity concerns.

3. Add `kernel/src/domain/pivots/placements.ts`.

   - Implement pure transition helpers:
     - `addPivotPlacement(config, spec)`
     - `updatePivotPlacement(config, placementId, patch)`
     - `removePivotPlacement(config, placementId)`
     - `movePivotPlacement(config, placementId, toArea, toPosition)`
     - `resetPivotPlacement(config, placementId)`
     - `resolvePivotPlacement(config, selector, options)`
     - `readPivotPlacement(config, placement)`
   - Each helper returns a new placement array plus effect metadata; it must not call `ComputeBridge`.
   - Ensure area positions are dense after every transition and preserve stable IDs.
   - Use this module from `WorksheetPivotsImpl`, `PivotBridge`, and `PivotTableHandle` construction so all placement paths share one behavior.

4. Add `kernel/src/domain/pivots/update-policy.ts`.

   - Define a single map from domain transition names to `PivotUpdateReason`, `PivotRefreshPolicy`, and expected mutation effects.
   - Export helpers such as `pivotUpdateOptionsFor(reason)` and `shouldMaterializePivotEvent(event)`.
   - Replace repeated inline `{ reason, refreshPolicy }` objects in pivot API and bridge code with domain policy helpers.
   - Keep the invariant test that forbids missing update options, and extend it to detect hard-coded pivot reason/policy objects outside the domain policy module.

5. Add `kernel/src/domain/pivots/source-reconciliation.ts`.

   - Move the pure parts of `WorksheetPivotsImpl.setDataSource()` into this module.
   - Inputs should include the current config, old effective fields, newly detected fields, calculated fields, and the user-facing data source string.
   - Output should be either:
     - `{ sourceSheetName, sourceRange, fields, placements, filters }`, or
     - a complete `PivotInvalidReference[]` list for the typed error factory.
   - Preserve current handling for stale placements, filters, top/bottom value fields, `sortByValue.valueFieldId`, `showValuesAs.baseField`, duplicate headers, and calculated-field placements.
   - Replace the current regex-only calculated-field formula scan with a deterministic reference extractor aligned with the Rust calculated-field grammar. If reusing Rust parser output is not available through compute, introduce a small tokenizer with exhaustive tests for identifiers, quoted field names, operators, string literals, and function calls.

6. Add `kernel/src/domain/pivots/config-transitions.ts`.

   - Centralize non-placement config transitions:
     - rename
     - layout merge
     - style merge
     - formatting options
     - filters
     - calculated-field add/update/remove/add-and-place
     - source reconciliation application
   - Return immutable `updates` objects and receipt/effect metadata.
   - Do not perform async reads in this module; callers remain responsible for resolving sheets, detecting fields, and fetching current config.

7. Add `kernel/src/domain/pivots/receipts.ts`.

   - Centralize construction of `PivotKernelMutationReceipt` and `PivotPlacementMutationReceipt` for TypeScript-originated transitions.
   - Preserve effect metadata (`placementAdded`, `placementUpdated`, `placementRemoved`, `calculatedFieldAdded`, etc.).
   - Prefer real revision values from compute mutation results when available. Where compute does not yet expose them, make the missing data explicit in one place rather than scattering `0` and `Date.now()` revisions.

8. Add `kernel/src/domain/pivots/event-policy.ts` or include event helpers in `update-policy.ts`.

   - Make the single materialization decision for `pivot:created`, `pivot:updated`, `pivot:deleted`, and `pivot:expansion-changed`.
   - Audit whether both `PivotBridge.setupObservers()` and `PivotEventBridge` can refresh the same event. Keep one owner for production materialization and make the other responsible only for UI notification if needed.
   - Preserve dirty-only invalidation without materialization.

9. Add `kernel/src/domain/pivots/expansion.ts`.

   - Define canonical helpers for `PivotExpansionKey` construction, comparison, and legacy rendered-header fallback.
   - Specify the current contract: `PivotBridge.setExpansion()` returns a receipt but app-level `PivotExpansionStateProvider` owns actual UI expansion state.
   - If persistent expansion is desired, route it through a real config/storage transition instead of leaving a receipt-only method.

10. Add `kernel/src/domain/pivots/index.ts`.

   - Export the public domain helpers intentionally.
   - Update call sites to import from `../../domain/pivots` instead of deep-importing individual local files where practical.

## Risks, edge cases, and non-goals

Risks:

- Rewriting placement ID generation can break persisted workbooks, UI selections, expansion keys, and command receipts if existing IDs are not preserved exactly.
- Centralizing event materialization can change timing; avoid regressions where UI needs a refresh callback after materialization completes.
- Moving source reconciliation may expose hidden behavior differences between name-based legacy facades and placement-id-first APIs.
- Calculated-field reference parsing is easy to under-specify. Do not rely on broad regex matching once the domain module owns this contract.
- Fire-and-forget handle methods currently hide errors by API shape. The refactor should preserve compatibility while moving actual transition logic to tested pure helpers.
- Style catalog changes must not reject custom workbook styles or imported OOXML style IDs.

Edge cases to include in tests:

- Same source field placed multiple times in Values with different aggregate functions and display names.
- Same display name on multiple placements, requiring an ambiguity error for name/field facades.
- Placement move within the same area and across areas with out-of-range target positions.
- Calculated field IDs that differ from display names.
- Source sheets renamed after pivot creation.
- Source ranges with duplicate, blank, numeric, boolean, or missing headers.
- Filters with include/exclude values and top/bottom value field references.
- Show Values As with base field/base item references.
- Legacy configs without `sourceSheetId`, `placementId`, `calculatedFieldId`, `rowItems`, or `colItems`.

Non-goals:

- Do not move Rust compute, storage, or materialization logic into TypeScript.
- Do not add a new TypeScript persisted pivot store.
- Do not optimize benchmark-only or mock-only paths.
- Do not change public API names or remove deprecated facades in this workstream.
- Do not implement group/ungroup UI unless it is explicitly included in a later pivot UI task.
- Do not treat `compute/api/src/sheet/pivots.rs` as the shipped pivot CRUD path; current production storage runs through compute-core/Yrs bridge methods.
