## Current role of this folder in Mog

`compute-wire/src` is the Rust source of truth for the binary viewport and mutation protocols between the Rust compute engine and TypeScript renderer. The crate writes compact little-endian byte buffers so TS can read viewport cells through `DataView` and apply mutation patches directly to existing viewport buffers without JSON decoding.

The mutation path currently has three nested formats:

- A single mutation blob: `[16-byte header][sheet id][N x 40-byte cell patches][string pool][optional spill section][optional palette section]`.
- A cell patch: `row u32`, `col u32`, then the same 32-byte cell record used by viewport serialization.
- A multi-viewport envelope: `[u16 viewport_count]`, then repeated `[u8 id_len][viewport id][u32 patch_len][patch bytes]`.

The production Rust storage engine generates one mutation blob per registered viewport, optionally including spill patches and palette deltas, then packs them for TS. TypeScript routes packed entries to viewport coordinators, heuristically distinguishes full viewport binaries from mutation binaries, constructs `BinaryMutationReader`, applies regular and spill patches to `BinaryViewportBuffer`, appends patch strings to the overflow pool, and updates overlay state for fetch reapplication.

Observed strengths:

- Constants and flags are generated into `kernel/src/bridges/wire/constants.gen.ts`.
- Viewport headers embed `WIRE_VERSION` in the header flags byte.
- The shared cell record offsets are centralized in Rust constants and TS generated constants.
- Positive-path coverage is substantial: Rust mutation unit tests, Rust proptests, multi-viewport tests, viewport deserializer negative tests, generated fixtures, and TS cross-language roundtrip tests.
- Palette binary deserialization already returns structured errors for malformed buffers.
- `security_filter.rs` uses bounds checks before in-place redaction of viewport buffers.

Observed correctness gaps:

- Mutation blobs do not carry a wire version or payload-kind marker. `viewport-coordinator-registry.ts` detects full viewport binaries by inspecting byte 30 for viewport version bits; mutation byte 30 can be ordinary patch data, so routing is heuristic rather than contractual.
- Many writer paths use `debug_assert!` followed by release-mode casts into `u16`, `u32`, or `u8` fields. Oversized sheet ids, viewport ids, viewport counts, patch counts, string pools, palette record counts, and patch lengths can silently truncate or wrap in optimized builds.
- `concat_multi_viewport_patches` caps the header count at `u16::MAX` while still appending all entry bytes, which can make TS ignore valid trailing entries.
- Mutation and viewport string lengths are `u16`; existing tests codify UTF-8-boundary truncation above `u16::MAX`. That is a hidden product contract and not tied to a user-visible truncation/error policy.
- `MUT_HAS_ERRORS` is set from `RecalcResult.errors`, but the inspected mutation writer does not encode a corresponding errors section. The flag is therefore a phantom signal unless the intended contract is documented and tested.
- The TS mutation reader trusts header counts, section lengths, string offsets, and optional-section flags. Malformed data can be silently decoded as empty palette/spill data or throw late `DataView` range errors instead of a typed protocol error.
- Optional mutation sections are positional and flag-driven, not tagged. A single inconsistent flag or length shifts all later interpretation.
- Mutation serialization uses raw `Vec<u8>` string pools while viewport serialization uses a deduplicating pool; the two paths share the same cell record but not one typed writer/validator.
- Docs and comments have drift: TS docs still refer to `mutation_binary.rs`, kernel README describes older viewport sizes, and README value-type docs omit current image/type flag details.

## Improvement objectives

1. Make the wire protocol self-identifying and versioned for both viewport and mutation payloads, including packed multi-viewport entries.
2. Replace silent release-mode truncation with checked, typed serialization contracts and structured `WireEncodeError` failures.
3. Define one authoritative mutation layout model that Rust writers, Rust test decoders, TS readers, TS test builders, generated constants, and docs all derive from or verify against.
4. Resolve the mutation errors contract: either encode `RecalcResult.errors` in a real section or remove/rename the flag so consumers cannot infer nonexistent data.
5. Turn mutation decoding into a validated contract on the TS side, with typed protocol errors for malformed headers, section bounds, string references, palette deltas, and unknown versions.
6. Make string-pool behavior a deliberate product/protocol decision rather than implicit truncation. If full display/error text must be preserved, bump the protocol and widen length representation; if truncation remains allowed, expose and test the exact render contract.
7. Make optional sections extensible without positional ambiguity by using tagged section headers or an explicit section directory in the next protocol version.
8. Strengthen cross-language and malformed-buffer tests so every byte-level invariant is exercised by the production writer and production reader.

## Production-path contracts and invariants to preserve or strengthen

- `compute-wire` remains the single Rust source of truth for viewport, mutation, palette, and generated TS constants. TS must not carry hand-maintained offsets that can drift silently.
- All binary fields remain little-endian and are read through `DataView` on the TS hot path.
- Mutation patch cell records remain semantically identical to viewport cell records, so `BinaryViewportBuffer.applyBinaryMutation()` can splice fields without semantic conversion.
- A packed multi-viewport entry must identify whether its payload is a mutation blob or a full viewport blob through an explicit contract, not by inspecting arbitrary payload bytes.
- `WIRE_VERSION` or a successor version field must gate every incompatible layout change before any reader consumes counts or offsets.
- Header counts and byte lengths must exactly describe the buffer. Readers must reject trailing bytes, section underflow/overflow, invalid UTF-8 references, unknown required sections, and unsupported versions.
- Writers must not silently wrap, truncate, or clamp protocol fields in release builds. Any intentional limit must be enforced before bytes are written and surfaced to the production caller.
- Patch order remains deterministic and row/col values remain absolute zero-based sheet coordinates. Viewport filtering remains inclusive via `ViewportBounds`.
- Regular patches and spill patches must not encode contradictory same-cell writes within one mutation result unless the scheduler contract explicitly allows last-writer-wins and tests it end to end.
- Palette indices remain stable `u16` sheet-session indices unless a future version intentionally changes the palette index width and updates all consumers.
- Palette deltas in mutation blobs must apply before patched cells with new `format_idx` values are rendered.
- String offsets remain relative to the mutation blob string pool for mutation readers and relative to the viewport main/overflow pool after application.
- `NO_STRING` plus zero length remains the only no-string sentinel. Offset/length pairs must never point outside their declared pool.
- `security_filter.rs` must continue to operate only on valid viewport buffers and must update header string-pool bytes when it appends placeholders.
- Public dependency direction is unchanged: `mog` must not depend on `mog-internal`.

## Concrete implementation plan

1. Write a formal wire schema in `compute-wire`.
   Add Rust types for `WirePayloadKind`, `WireVersion`, `MutationHeader`, `MutationSectionKind`, `MutationSectionHeader`, `MultiViewportHeader`, `MultiViewportEntryHeader`, and shared `CellRecordLayout`. Keep these as the canonical internal model for layout size, field offsets, maximum representable values, required sections, optional sections, and version compatibility.

2. Introduce checked writers.
   Replace direct casts in mutation, multi-viewport, viewport, and palette writers with checked helpers such as `write_u16_len`, `write_u32_len`, `write_u8_len`, and `checked_section_size`. Return `Result<Vec<u8>, WireEncodeError>` from production serialization entrypoints or introduce checked entrypoints and migrate production callers to them in the same change set. The old infallible path should not remain the production path.

3. Version and tag mutation payloads.
   Design the next mutation format so the reader can validate payload kind and version before parsing section counts. Prefer adding an explicit payload header or section directory over stealing bits from existing flags. If backward compatibility is required for persisted/local test buffers, implement decoding for the old format only as a clearly bounded legacy reader; newly produced production bytes should use the tagged version.

4. Make the multi-viewport envelope typed.
   Extend packed entries with a payload kind and entry version, or define a new packed-envelope version that includes them. Remove the TS registry heuristic that treats byte 30 as a full-viewport marker. Reject oversized viewport ids, too many entries, and oversized patch payloads at the Rust writer boundary instead of truncating `u8`, `u16`, or `u32` fields.

5. Fix `concat_multi_viewport_patches`.
   Parse and merge typed packed envelopes rather than concatenating raw bodies. If the merged count exceeds the protocol limit, return an encode error. Never write a capped count with more entry bytes than the count declares.

6. Resolve the mutation error section.
   Audit production TS/kernel consumers for `BinaryMutationReader.hasErrors`. If `RecalcResult.errors` should reach TS through binary patches, add a tagged error section with `sheet_id`, cell id or row/col where available, error code/text, and count/length checks. If the binary channel should only carry patchable cell errors, remove `MUT_HAS_ERRORS` from new-format mutation headers and update generated constants/docs/tests accordingly.

7. Define the string length contract.
   Decide whether display/error strings must preserve full UTF-8 content. If yes, bump the cell record format and widen string lengths or move lengths into a string table that supports `u32` lengths. If no, keep `u16` lengths but make truncation explicit: use one shared `WireStringPool` that reports truncation, adds tests for multibyte boundaries, and surfaces metrics or flags if render text was truncated.

8. Unify string-pool and patch-record construction.
   Replace mutation's raw `Vec<u8>` string interning with a shared `WireStringPool`/`CellRecordWriter` used by viewport and mutation paths. Keep optional deduplication configurable if mutation ordering or memory behavior requires it, but make offset/length validation identical.

9. Add a Rust mutation decoder/validator.
   Add a test-utils mutation decoder parallel to the viewport deserializer, with structured errors for short headers, unsupported versions, bad flags, section overflow, invalid section order, invalid string refs, invalid UTF-8, palette decode errors, trailing bytes, and count overflows. Use it in existing Rust mutation tests and proptests instead of ad hoc offset reads.

10. Harden the TS reader.
   Update `BinaryMutationReader` to validate the full buffer in its constructor before exposing accessors. Add a typed `BinaryMutationProtocolError` with code/context fields. Validate header version/kind, declared lengths, patch and spill section bounds, string references for every patch, palette section length and decode result, trailing bytes, and unknown required sections.

11. Generate more TS protocol metadata.
   Extend `generate_constants_ts()` to emit payload kind constants, version constants for viewport/mutation/envelope, section-kind constants, header offsets, maximum field values, and any error-code names. Add a freshness test that fails when `constants.gen.ts` is stale.

12. Update production producers.
   Update `compute/core/src/storage/engine/viewport/patches.rs` and structural/range/format paths to handle checked encode errors deliberately. The correct behavior should be explicit: return a compute error, force a full viewport refresh when a mutation patch is too large but a full viewport can be encoded, or fail the mutation if neither path can represent the result. Do not silently drop patches.

13. Update production TS consumers.
   Update `viewport-coordinator-registry.ts` to route by typed packed-entry payload kind. Update `binary-viewport-buffer.ts` and `viewport-coordinator.ts` to handle reader construction failures deterministically, likely by marking hydration deficit and requesting a full viewport refresh while reporting the protocol error.

14. Regenerate and expand cross-language fixtures.
   Generate fixtures for old-format rejection if retained, current mutation vNext, full viewport payloads inside the packed envelope, spill patches, palette deltas, image cells, CF color overrides, long UTF-8 strings, empty patches, and malformed fixtures. Make TS tests consume real Rust-produced bytes for all positive cases.

15. Refresh docs from the schema.
   Update `compute-wire/README.md`, crate-level docs, `kernel/src/bridges/wire/README.md`, and TS comments from the same schema facts. Remove stale file names and old header/stride descriptions.

## Risks, edge cases, and non-goals

Risks:

- Versioning mutation blobs changes the hot path shared by Rust and TS. The implementation must land Rust writer, generated constants, TS reader, packed routing, fixtures, and storage-engine callers together.
- Widening string lengths or adding section directories can increase patch size and reduce cache locality. Correctness comes first, but the new layout should still be measured on production mutation paths.
- Checked serialization errors need product-level handling. Some oversized cases may be best represented by full viewport refresh, while others should fail the mutation; this must be explicit per case.
- If legacy mutation decoding is retained, it can become a permanent compatibility surface. Keep it bounded to tests or migration windows unless persisted production bytes require it.
- Tagged optional sections must preserve palette-before-render semantics. A reader that applies cell patches before palette deltas can briefly render wrong formats.

Edge cases to cover:

- Empty mutation with non-empty sheet id and no sections.
- Empty string display text versus no display string.
- `NO_STRING` with nonzero length, non-sentinel offset with zero length, and offset exactly at pool end.
- UTF-8 truncation or widening behavior for multibyte strings.
- Image cells using metadata in the error/string slot and `HAS_CELL_IMAGE`.
- Spill projection cells with generated display text, CSE formula flags, sentinel `u32::MAX` row/col guards, and cross-sheet viewport filtering.
- Palette delta with `start_index > current TS palette length`, empty delta, full palette limit, and malformed palette bytes.
- Multi-viewport packet containing a full viewport payload, a mutation payload, an empty payload, and an entry for an unregistered viewport.
- Full viewport and mutation payloads whose bytes would collide with the old byte-30 routing heuristic.
- Security-filtered viewport buffers whose string-pool size changes before later sections.

Non-goals:

- Do not introduce a JSON mutation fallback as the main fix for binary protocol correctness.
- Do not optimize benchmark-only or test-only serialization paths instead of production writers and TS readers.
- Do not keep silent truncation, clamping, wrapping, or patch dropping for production bytes.
- Do not add dependencies from public `mog` code to `mog-internal`.
- Do not broaden this plan into formula evaluation or scheduler semantics except where their outputs must satisfy the wire contract.
