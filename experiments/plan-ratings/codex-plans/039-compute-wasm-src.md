## Current role of this folder in Mog

`compute/wasm/src/lib.rs` is the browser compute binding crate's entire Rust source surface. It does not implement spreadsheet algorithms directly; it assembles the production browser boundary around `compute-api`, `compute-core`, and `xlsx-api`.

Observed responsibilities:

- Re-export `compute_api` as the clean Rust facade for downstream Rust consumers.
- Initialize the WASM module with `console_error_panic_hook` and browser console tracing through `tracing-web`.
- Keep tracing formatter timestamps disabled because `SystemTime::now()` is unsupported on `wasm32-unknown-unknown`.
- Import every bridge type that appears as a bare identifier in generated descriptor expansions.
- Generate browser exports for the stateful `ComputeService` lifecycle and engine methods.
- Generate stateless pure bridge exports for pivot, table, chart, formatting, schema, conditional formatting, clock/time, and XLSX parsing.
- Merge XLSX parser commands into the same `@mog-sdk/wasm` module so browser transport uses one WASM package.

Important observed integration points:

- `compute/wasm/src/lib.rs` manually lists the same `ComputeService` descriptor groups as `compute/napi/src/lib.rs`.
- A source comment documents a prior production failure where a missing WASM descriptor caused `Unknown WASM function` for `compute_wb_security_drain_events`.
- `infra/transport/src/wasm-transport.ts` dispatches to WASM by string command name and `Object.values(args)`, so the WASM export set and parameter order are production contracts.
- `infra/transport/src/command-metadata.gen.ts` already carries generated command metadata for bytes tuples, recalc/time injection, N-API serde params, and security levels, but it does not prove the built WASM package exports every command.
- `compute/wasm/npm/compute_core_wasm.d.ts` is generated by wasm-bindgen and exposes broad `any` shapes for most structured parameters and returns.
- `compute/wasm/build.sh` appends a `__wbindgen_reset()` export to the generated JS glue so trap recovery can instantiate a fresh module after a wasm32 trap.

## Improvement objectives

1. Make the browser WASM export surface mechanically complete.
   The `@mog-sdk/wasm` module must export every command the generated TypeScript bridge, command metadata, table bridge, chart bridge, and XLSX browser path can call.

2. Eliminate descriptor-list drift between WASM and native binding targets.
   WASM and N-API should consume one shared, generated or macro-backed binding-surface contract for the intended common compute browser/native command set.

3. Replace positional argument order assumptions with generated metadata.
   Browser dispatch should not depend on object property insertion order when the bridge generator already knows the Rust parameter order.

4. Make WASM startup and reset behavior explicit and idempotent.
   Panic hook, tracing setup, module reset, service registry lifetime, and recreate-after-trap behavior should be tested on the production module path.

5. Tighten the typed contract around generated WASM exports.
   Keep wasm-bindgen output as the low-level package artifact, but generate or derive a command signature map that TypeScript transport and smoke tests can use to catch argument, return, and bytes-tuple drift.

6. Reduce manual bare-type import fragility.
   Shared binding prelude or generated import coverage should make new descriptor payload types fail clearly and early instead of requiring repeated manual import edits in WASM and N-API binding crates.

7. Verify real browser compute behavior, not only compile-time codegen.
   Add a production-path WASM smoke that initializes the actual `@mog-sdk/wasm` package, creates an engine, mutates cells, recalculates, exercises bytes-tuple and XLSX/pure commands, destroys the engine, and reloads after reset.

## Production-path contracts and invariants to preserve or strengthen

- `compute/wasm` remains a public browser binding layer for `compute-api`; it must not depend on `mog-internal`.
- `compute-core` and `compute-api` stay built with native-only features disabled for the WASM target.
- Browser compute remains single-threaded unless a separate explicit threading architecture is introduced.
- `#[wasm_bindgen(start)]` must never panic during repeated module initialization, HMR-style reloads, trap recovery rebuilds, or test harness reloads.
- WASM tracing must not use formatter paths that call `SystemTime::now()` on `wasm32-unknown-unknown`.
- `@mog-sdk/wasm` must keep exporting the wasm-pack web target module and `./wasm` binary asset.
- Every generated stateful service command takes `doc_id` first at the JS boundary.
- Every generated stateless pure command keeps the command name and parameter order emitted from bridge descriptors.
- Commands returning `(Vec<u8>, MutationResult)` must surface to browser JavaScript as `[Uint8Array, MutationResult]`.
- Structured object returns can still be normalized by the case-normalizing transport, but the raw WASM return shape must remain serializable and deterministic.
- Recalc-triggering commands must still receive `compute_set_current_time` injection before the recalc command crosses the WASM boundary.
- Trap recovery must clear both the transport's module reference and wasm-bindgen's private cached module/typed-array views before the next load.
- XLSX parser commands stay merged into the same browser module unless an explicit architecture replaces the single-module browser runtime.
- Missing exports must fail in tests before they can become runtime `Unknown WASM function` errors.

## Concrete implementation plan

1. Build a canonical binding-surface manifest.
   Add a generated or macro-backed manifest that lists all intended WASM/N-API compute binding descriptors: `ComputeService` lifecycle/core/domain groups, stateless pure bridges, `ClockBridge`, and `XlsxParser`. The manifest should encode target disposition, not only a raw list. WASM and N-API should consume this manifest so adding a descriptor group in one target cannot silently skip the other.

2. Replace the manual WASM descriptor list with the shared manifest.
   Refactor `compute/wasm/src/lib.rs` so `bridge_wasm::generate!` is fed from the shared compute binding surface. Keep the file as the browser entrypoint, but remove the hand-maintained long descriptor list as the source of truth. Apply the same shared surface to `compute/napi/src/lib.rs` where the same descriptor set is intended.

3. Add descriptor parity tests.
   Add a Rust test in the bridge/codegen area that compares the descriptor groups used by WASM and N-API. The test should explicitly allow only documented target differences. It must fail for the exact class of issue already observed: a command available in generated TS metadata or N-API but missing from WASM.

4. Generate WASM command parameter metadata.
   Extend bridge-ts command metadata generation to include `WASM_COMMAND_PARAM_ORDER`, keyed by snake-case command name with ordered JS argument names. Include lifecycle `docId`/`doc_id`, pure bridge params, bytes params, parse params, and serde params. This should be emitted from the same parsed bridge API that writes `compute-bridge.gen.ts` and `command-metadata.gen.ts`.

5. Update browser dispatch to use generated param order.
   Change `createWasmTransport` to call `fn(...paramOrder.map(name => args[name]))` for generated commands. Keep a strict failure path for missing metadata on bridge commands, and only use a narrow fallback for intentionally external/manual commands if such commands exist. This removes reliance on `Object.values(args)`.

6. Generate a WASM export expectation set.
   Emit a generated list of expected low-level WASM export names from the descriptor manifest plus wasm-bindgen lifecycle/reset exports that are production-required. The list should include compute service commands, pure table/chart/pivot/format/schema/cf/clock commands, and XLSX parser commands.

7. Add an actual module export smoke.
   Add a test that imports the built `@mog-sdk/wasm` package, initializes it through the same loader path used by production, and asserts every expected export exists and is a function. This test should run after `bash compute/wasm/build.sh --profile dev` so it validates the real wasm-pack output, not only macro tokens.

8. Add production behavior smoke coverage.
   Add a focused WASM smoke that uses the actual transport stack: load module, wrap with time injection, bytes-tuple normalization where relevant, and case normalization; create a compute engine from a minimal workbook snapshot; set cells; run formula recalc; exercise a bytes-tuple mutation; call one stateless table command; call one chart/stat command; call one XLSX parser command with a small valid fixture; destroy the engine.

9. Test lifecycle registry semantics.
   Cover create, duplicate create for the same `doc_id`, destroy, destroy missing id, command after destroy, and recreate after destroy. The test should assert errors are structured through the transport path and that no stale instance receives calls after lifecycle teardown.

10. Test trap reset semantics on the real loader.
    Keep the existing `__wbindgen_reset()` production contract, but add a build-output gate that fails if the generated JS or `.d.ts` lacks the export after build. Pair it with a loader test that calls `resetWasmModule()`, reloads, and verifies a fresh engine can be created. If wasm-bindgen later exposes a first-class cache reset API, replace the post-bindgen patch with that supported mechanism and keep the same test contract.

11. Make `wasm_start` idempotent.
    Replace unconditional tracing subscriber installation with a non-panicking setup path such as `try_init()` plus a documented ignored already-initialized case. Preserve `console_error_panic_hook::set_once()` and the no-time formatter. Add a wasm-bindgen-test or loader-level test that initializes the module more than once through the supported reset/reload path.

12. Centralize the binding prelude.
    Move the repeated bare-type imports shared by WASM and N-API into a public binding prelude module in the appropriate public Rust crate, or generate the prelude from descriptors. `compute/wasm/src/lib.rs` and `compute/napi/src/lib.rs` should import the prelude instead of carrying parallel allow-unused blocks. New descriptor payload types should be added once and checked by compile tests.

13. Add return-shape contract tests.
    For representative commands in every return category, assert raw WASM return shapes before TypeScript normalization: primitive, string, `Uint8Array`, serde object, nullable object, array of objects, and `[Uint8Array, metadata]`. Include at least one mutation result and one stateless pure bridge result.

14. Add error-shape contract tests.
    Exercise a parse/serde failure, a missing service id, and an engine compute error through the real WASM transport. Assert the error message retains enough bridge-formatted information for `TransportError.fromCommand` and kernel trap/error handling to classify it without conflating ordinary errors with wasm32 traps.

15. Add generated artifact freshness checks.
    Expand bridge generation gates so `compute-bridge.gen.ts`, `manifest.gen.ts`, `command-metadata.gen.ts`, WASM parameter metadata, and WASM export expectation metadata are regenerated together and compared against checked-in files. Descriptor edits should have one obvious regeneration command and one failing freshness test when stale.

16. Document the browser boundary contract.
    Update public architecture docs after implementation to state that `@mog-sdk/wasm` is the low-level binary wrapper, that command names/param order are generated from bridge descriptors, that XLSX is merged into the same module, and that trap recovery depends on the reset export generated by the WASM build.

## Risks, edge cases, and non-goals

Risks and edge cases:

- Descriptor centralization can expose converter gaps that manual lists previously hid. The correct outcome is to add converter/type support or an explicit target disposition.
- Param-order metadata must handle JS camelCase argument names from generated TypeScript methods and snake_case wasm-bindgen function names without mixing the two naming layers.
- Some generated `.d.ts` names use snake_case Rust parameter names while TypeScript bridge methods pass camelCase object keys. Metadata generation must define the exact mapping.
- WASM module initialization can behave differently in browser bundlers, Node tests, and Vite dev. Smoke tests should cover both browser-oriented bundler loading and the Node bytes path already implemented in `wasm-loader`.
- The reset export is appended after wasm-bindgen output today. A gate should make it a verified production artifact, but the long-term target should remain a first-class reset mechanism if wasm-bindgen supports one.
- Return-shape tests must inspect raw WASM results before case normalization and production transport results after normalization; both contracts matter.
- XLSX smoke fixtures should be tiny and valid, but still go through the real parser command in the merged WASM module.
- Browser performance can regress if export/param metadata adds per-call overhead. The transport should cache each command's ordered param array and avoid allocations beyond the existing argument list where practical.

Non-goals:

- Do not hand-write separate browser-only commands that bypass bridge descriptors.
- Do not optimize test-only code paths instead of the production `@mog-sdk/wasm` package and transport.
- Do not split XLSX into another browser module unless a separate architecture plan changes the single-module runtime contract.
- Do not use compatibility fallbacks to hide missing WASM exports.
- Do not make `compute/wasm` depend on private/internal packages.
- Do not weaken trap classification by treating ordinary bridge errors as wasm32 traps.
