Rating: 8/10

Summary judgment

This is a strong plan. It identifies the central architectural problem in `kernel/src/security`: the folder-local subject-based services are effectively orphaned, while the public `@mog-sdk/kernel/security` barrel exposes the older AppId-keyed `services/capabilities` system that production callers actually use. The plan is unusually well grounded in source evidence and correctly treats this as an authorization-boundary problem, not ordinary cleanup.

It falls short of a 9 or 10 because the migration contract is not yet executable enough for a high-risk security subsystem. The plan says "make the subject-based model canonical", but the current production path includes stores, denials, audit events, requesters, consent UI, gated API checks, scope parsing, dependency expansion, package exports, and shell consumers. Those are acknowledged, but not specified deeply enough to prevent another partial migration or a fourth capability model.

Major strengths

- The plan is production-path aware. It correctly distinguishes the live `@mog-sdk/kernel/security` barrel from the dead folder-local implementations, and it names the shell/app-launch/settings consumers that depend on the current AppId-based surface.
- The core defects are real and important: order-dependent matching in `grant-service.ts`, exact-key-only revoke, unused registry/trust-policy constructor dependencies, implied grants losing scope/expiry, first-come namespace ownership, short-ID namespace bypass, first-party auto-grant of critical capabilities, and contract drift against the app-platform type declarations.
- The proposed invariants are the right kind for an authorization layer: contract conformance, deny precedence, revocation completeness, scope soundness, critical-capability auto-grant restrictions, and reserved namespace authority.
- The plan preserves important public-surface constraints, especially that `@mog-sdk/kernel/security` must keep resolving for existing consumers and that the barrel should not expose store implementations, requesters, sensitive handlers, re-auth providers, or audit implementations.
- The verification section goes beyond smoke tests. It calls for focused unit tests, type-level contract conformance, and an app-launch integration path after removing the legacy adapter.
- The sequencing recognizes a real hard dependency on the app-platform contract before implementation starts, which is the right instinct for a security contract migration.

Major gaps or risks

- The target package boundary is ambiguous. The current contract package appears to be `@mog-sdk/types-app-platform`, while the plan repeatedly says `@mog-sdk/app-platform` and says factories should live there. If that is historical naming, the plan should say so. If factories really move there, the dependency direction and package ownership need to be spelled out because app-platform currently looks like a type package, not a kernel implementation package.
- "Contract conformance" conflicts with "evolve the contract to list-form `TrustPolicyDecision`". Both may be valid options, but the plan does not define the final source-of-truth shape. It also leaves `expiresAt` as a proposed contract addition. Until those decisions are recorded as exact interfaces, the contract gate cannot be implemented.
- The registry contract drift is under-inventoried. The plan covers `CapabilitySubject`, `CapabilityGrant`, and trust-policy shape, but less directly addresses `CapabilityRegistration` versus `CapabilityMetadata`: `ownerPackage` versus `owner`, `id: string` versus branded `CapabilityId`, `implies` versus `impliedCapabilities`, missing `dependentCapabilities`, missing `compatibilityProfile`, and missing consent requirement fields.
- The live `services/capabilities` system is much richer than the new `grant-service.ts`. It has `IGrantsStore`, memory/SQLite/cloud stores, denials, subscriptions, audit logging, dependency expansion, cascading revoke, cleanup, requester/consent flows, and gated API use sites. Phase 6 says to make the AppId store a thin detail or replace it, but does not define the new store interface, event payloads, audit subject shape, denial semantics, or compatibility layer for current `ICapabilityRegistry` callers.
- Scope semantics are not specified tightly enough. Reusing `scopeMatches` only works for the current string `CapabilityScope` format, while the app-platform contract uses `Record<string, unknown>` and the registry has `scopeSchema`. The plan needs to choose the canonical scope representation, validation library/strategy, and requested-scope matching contract instead of saying both "JSON Schema" and "align with `scopeMatches`".
- The deny/revoke precedence lattice needs exact examples. The summary says deny wins at same-or-broader specificity, but Phase 2 says denied/revoked grants at `>= query specificity`, which reads like the opposite relation. For a security-sensitive matcher, the plan should define broader/narrower ordering, tie-breaks, tombstone coverage, and re-grant supersession in precise truth tables.
- Persisted grant migration is acknowledged but not planned. Existing stores key by `AppId` and old capability strings such as non-`mog:` IDs, with timestamps in Unix milliseconds and separate denial records. A production plan needs an explicit versioned migration, rollback behavior, roundtrip tests, and compatibility for existing local SQLite/cloud data.
- Trust hardening is directionally correct but not yet tied to an authoritative provenance source. "Authenticate bundled/signature/enterprisePolicyId" needs a named install-record producer, verification API, and failure behavior; otherwise implementers may simply move the self-asserted flags one layer over.
- The no-shim stance is too absolute for the public surface. It is fine to delete `LegacyGrantServiceAdapter`, which is dead today, but existing production APIs still accept `AppId`. The plan should distinguish "delete the dead folder-local adapter" from "preserve public AppId convenience methods as compatibility wrappers over subject storage until consumers are migrated".

Contract and verification assessment

The plan has the right contract instincts but needs sharper preconditions, postconditions, and type names. The final contracts should state the exact subject type source, capability ID format, grant timestamp/expiry fields, grant decision lifecycle, requested-scope representation, implied-capability scope propagation, denial/tombstone behavior, and public barrel exports. They should also define how the old `ICapabilityRegistry` interface maps to the new subject service during migration.

The proposed tests are strong for the folder-local services, but incomplete for the actual production path. Any implementation would need concrete commands and gates for the kernel package, the app-platform/types package, and shell consumers. It should include public export validation for `@mog-sdk/kernel/security`, unit coverage for store migration and denial/subscription/audit event behavior, and an app-launch or shell consent flow that drives the same UI/service path production uses. Because this crosses package contracts and shell imports, a narrow kernel-only typecheck would not be enough.

Concrete changes that would raise the rating

- Add a Phase 0 contract table with the final exact interfaces for `CapabilitySubject`, `CapabilityId`, `CapabilityGrant`, `TrustPolicyDecision`, `ICapabilityGrantService`, registry metadata, and requested/granted scope. Mark each field as preserve, rename, add, remove, or migrate.
- Correct the app-platform package naming and dependency direction. If implementation factories belong somewhere other than `kernel`, name that package and explain how it can legally depend on kernel services.
- Inventory every live production capability surface: `ICapabilityRegistry`, `IGrantsStore`, denials, audit entries, registry events, requester/consent flows, gated API checks, shell settings UI, and persistent stores. For each, state the subject-based replacement or compatibility wrapper.
- Define the authorization resolution algorithm as a table: direct allow, broader allow, narrower allow, broader deny, narrower deny, expired grant, revoked tombstone, implied grant, scoped grant, re-grant after revoke, and deterministic tie-break behavior.
- Specify the canonical scope model. If scope remains the existing string format, say how it maps to app-platform `Record<string, unknown>`. If JSON-schema scope becomes canonical, define validation and requested-scope matching and avoid depending directly on the old `scopeMatches` contract.
- Add a versioned persisted-data migration plan for AppId-keyed grants, denials, Unix millisecond timestamps, old capability IDs, SQLite/cloud stores, and audit compatibility.
- Replace the generic trust-policy hardening language with named provenance inputs and verifier APIs for bundled, signature, and enterprise-policy trust.
- Turn verification into concrete gates, including kernel unit tests, kernel typecheck/build or public export validation, app-platform/types typecheck, shell typecheck for consumers, store migration tests, and a production-path app-launch/consent scenario.
